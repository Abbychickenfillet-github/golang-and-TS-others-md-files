DATA STRUCTURE NOTE　/　INDEXING

B+樹與索引結構：叢集索引 vs 非叢集索引

核心結論先講：不論是叢集索引還是非叢集索引，底層骨架都是同一棵 B+樹。 兩者唯一的本質差異，發生在樹的最底層——葉子節點（Leaf Node）裡到底放的是「資料本身」還是「指向資料的指標」。

01

B+樹的整體骨架

不管哪種索引，畫出來都是這個樣子：Root（根節點）負責往下分流， Internal Node（內部節點）負責路由，真正的內容只會出現在最底層的 Leaf Node（葉子節點）。葉子節點之間還有一條雙向鏈結串列互相串起來， 這就是範圍查詢可以飛快的關鍵。

02

葉子節點裡到底放了什麼：三種情境對照

把上面那棵樹的葉子節點放大來看，三種情境的內容完全不一樣：

叢集索引
葉子節點 ＝
完整資料列
（PK／Name／Email／…全部欄位）
↓
✓ 直接拿到完整資料
不需要再查第二次
非叢集索引
（表有叢集索引）
葉子節點 ＝
叢集索引鍵值
（例如 PK：Id = 87）
↓
再走一次叢集索引 B+樹
＝回表 / Bookmark Lookup
↓
✓ 取得完整資料列
非叢集索引
（Heap Table，無叢集索引）
葉子節點 ＝
RID
（檔案編號：分頁編號：列位移）
↓
直接定位磁碟上的實體位置
↓
✓ 取得完整資料列

也因為叢集索引的葉子節點就是實體資料本身，一張表只能有一個叢集索引——資料在磁碟上物理排列方式只能有一種，不可能同時依兩種順序排好。

03

為什麼是 B+樹，不是二元搜尋樹？

① 樹高極低、極扁平

B+樹 每個節點可以塞進成百上千個子節點指標，百萬、千萬筆資料的樹高通常只要 3～4 層。樹越扁，從 Root 走到 Leaf 需要的磁碟 I/O 次數就越少——這是效能的關鍵，而不是單純的「比較次數」。

② 葉子節點之間有鏈結串列

所有葉子節點用雙向鏈結串列串起來。範圍查詢只要定位到起點，就能沿著串列往後讀，完全不必每次都回到 Root 重新搜尋一次——上面 01 的示範按鈕就是在演這個過程。

04

⚠️ 妳在用的 PostgreSQL，其實是特例

⚠️ 重要警告：上面整套說法，主要對應 MySQL（InnoDB）或 SQL Server

PostgreSQL 的儲存機制不一樣，這點對正在用 Go + GORM + PostgreSQL 的妳特別重要：

・PostgreSQL 的 Heap（資料表的實體儲存區）預設是無序的，沒有「持續維護」的叢集索引這種東西。

・所有索引（包含 Primary Key 用的 B-tree 索引）的葉子節點，存的都是 TID（Tuple Identifier，概念上等同於 RID：page + offset）， 指向 heap 裡的實體資料列。也就是說，PostgreSQL 裡每個索引在結構上其實都更接近「非叢集索引」。

・PostgreSQL 有一個 CLUSTER 指令，可以依某個索引手動、一次性把實體資料重新physically排序一次，效果類似叢集索引。 但這只是「當下整理一次」，之後新插入或更新的資料不會自動維持那個排序，要的話得自己重複手動執行。

05

進階延伸（選讀）

涵蓋索引（Covering Index）怎麼避免回表？

回頭看 02 的第二欄：如果那個非叢集索引的葉子節點裡，本來就已經包含查詢需要的全部欄位（不只 PK）， 資料庫就完全不需要再回去走一次叢集索引——因為要的東西在葉子節點上就拿得到了。這就是涵蓋索引的核心原理：用「索引多放一點欄位」去換「省掉一次回表」。

B-Tree 跟 B+-Tree 差在哪？

傳統 B-Tree 的內部節點本身也能放資料，葉子節點之間也沒有鏈結串列。B+-Tree 把資料全部趕到葉子節點，內部節點只專心做路由—— 節點更精簡，能塞進更多鍵值，樹更扁；加上葉子節點之間的鏈結串列，範圍查詢的效率明顯更好。這就是資料庫普遍選 B+-Tree 而不是 B-Tree 的原因。

是非題

T1.一個資料表可以同時擁有兩個叢集索引。
錯。叢集索引的葉子節點就是實體資料本身，資料在磁碟上的物理排列只能有一種順序，所以一張表只能有一個叢集索引。
T2.非叢集索引一定比叢集索引查詢速度慢。
錯。如果非叢集索引已經是「涵蓋索引」——葉子節點本身就包含查詢需要的所有欄位——就不需要回表，反而可能比叢集索引掃描更快。
T3.B+樹 比二元樹更適合做資料庫索引的原因之一，是樹的高度較低、能減少磁碟 I/O 次數。
對。B+樹 每個節點能容納大量子節點，樹高通常只有 3～4 層，每多一層就多一次磁碟存取，扁平的樹能大幅減少 I/O。
T4.在 PostgreSQL 中，Primary Key 索引預設就是叢集索引，資料列會自動依照 Primary Key 物理排序儲存。
錯，這是常見誤解。PostgreSQL 的 Primary Key 索引在結構上其實是非叢集索引（葉子節點存的是 TID），heap 預設無序。要讓資料暫時依某索引重新排列一次，需要手動執行 CLUSTER 指令，而且之後新資料不會自動維持排序。

申論題（先自己寫，再點開參考答案對照）

E1.請用自己的話解釋：「回表」為什麼會造成效能成本？在什麼情境下可以靠「涵蓋索引」避免回表？
回表＝多走一次 B+樹（從非叢集索引的葉子節點拿到鍵值或 RID 後，還要再用這個鍵值去查一次叢集索引，或直接定位磁碟分頁）。 每一次「走一棵 B+樹」在最差情況下都可能是一次額外的磁碟 I/O，資料量越大、要回表的列數越多，這個成本就越明顯。 如果非叢集索引在建立時，把查詢實際需要的欄位都包含進去（不限於 PK），葉子節點本身就拿得到答案，就完全不用回表——這正是涵蓋索引的設計目的。
E2.在妳目前 Go + GORM + PostgreSQL 的後端架構裡，PostgreSQL「所有索引結構上都更接近非叢集索引」這個特性，對妳設計查詢或索引時會有什麼實際影響？
因為 PostgreSQL 沒有「自動持續維護」的叢集索引，靠 Primary Key 查詢時，理論上每一次都隱含一次「鍵值 → TID」的查找（雖然 PK 本身常駐記憶體快取，實務上感受不明顯）。 實際影響：① 對高頻、效能敏感的查詢，可以考慮建立涵蓋索引（GORM 可透過自訂 migration 或 raw SQL 加 INCLUDE 欄位）來省掉回表； ② 如果某張表有「幾乎只依某欄位範圍查詢」的明顯熱點模式（例如時間序資料表常依時間範圍掃描），可以評估手動執行 CLUSTER 搭配排程維護，但要清楚這不是自動持續的，新資料寫入後順序會慢慢「鬆散」，需要重新評估維護頻率。
