> [!info] 為什麼是 1 Byte 而不是 1 Bit？
> **核心原因：Byte 是電腦硬體（CPU 和記憶體）進行數據讀寫、定址的基本最小單位。電腦無法直接單獨對一個單一的 Bit 進行記憶體定址。**

## 0. 從最底層說起：bit 是什麼？為什麼 8 個 bit = 256 種？

### bit = binary digit（二進位數字）

**1 bit = 1 個二進位數字，只有兩種狀態：`0` 或 `1`。** 就這樣，沒別的了。

### 為什麼 8 個 bit 能表示 **256** 種？

因為**每多一個 bit，可能性就乘以 2**。想成「N 個燈泡，每個可以開或關，總共幾種排列？」

```
1 個 bit：  0, 1                            → 2 種    = 2¹
2 個 bit：  00, 01, 10, 11                  → 4 種    = 2²
3 個 bit：  000,001,010,011,100,101,110,111 → 8 種    = 2³
4 個 bit：  0000 ~ 1111                     → 16 種   = 2⁴
...
8 個 bit：  00000000 ~ 11111111             → 256 種  = 2⁸
```

> **每個 bit 都是一次獨立的「二選一」。8 次二選一 = 2×2×2×2×2×2×2×2 = 2⁸ = 256。**
>
> 這就是 $2^8 = 256$ 的來源——**8 是 bit 的數量（指數），2 是「每個 bit 有幾種狀態」（底數）**。

### 為什麼是 **0～255**，不是 1～256？

因為**從 0 開始數**。256 種可能性，第一種是 0，第 256 種就是 255：

```
二進位        十進位
00000000  =    0     ← 第 1 種
00000001  =    1     ← 第 2 種
00000010  =    2
   ⋮
11111110  =  254
11111111  =  255     ← 第 256 種（最後一種）
```

`11111111` 怎麼算出 255？**每個位置代表 2 的不同次方，全部相加**：

```
 位置：   7    6    5    4    3    2    1    0
 權重： 128   64   32   16    8    4    2    1
 值：     1    1    1    1    1    1    1    1
        ───────────────────────────────────────
        128 + 64 + 32 + 16 + 8 + 4 + 2 + 1  =  255
```

### 4 個 bit = 1 個 16 進位數字（這就是 hex 好用的原因）

4 個 bit 有 2⁴ = **16** 種組合 → 剛好對應 16 進位的一個位數（0-9, A-F）：

```
0000 = 0    0100 = 4    1000 = 8     1100 = C (12)
0001 = 1    0101 = 5    1001 = 9     1101 = D (13)
0010 = 2    0110 = 6    1010 = A(10) 1110 = E (14)
0011 = 3    0111 = 7    1011 = B(11) 1111 = F (15)
```

所以：

> **2 個 hex 數字 = 8 個 bit = 1 byte。完美對齊。**
> 這就是為什麼工程界表示記憶體、色碼、檔案內容都愛用 16 進位——**十進位跟 byte 沒有這種漂亮的對應關係**。
>
> 例：`0x4D` → `0100 1101` → 十進位 77 → 字元 `'M'`
> （`0x` 前綴就是在說「後面是 16 進位」。）

## 1. 記憶體最小定址單位 (Byte Addressability)

### 📐 記憶體實際長什麼樣子

**記憶體 ＝ 一整排連號的格子。每一格恰好 1 byte（8 bits），每一格有一個專屬的「門牌號碼」（地址）。**

```
        ┌─ 門牌號碼（地址，用 16 進位寫）
        │              ┌─ 格子裡的內容（固定 8 個 bit）
        ↓              ↓
   ┌──────────┬──────────────────┬─────────────┐
   │  地址     │  內容（8 bits）   │  當成數字看  │
   ├──────────┼──────────────────┼─────────────┤
   │  0x1000  │  0 1 0 0 1 1 0 1 │     77      │
   ├──────────┼──────────────────┼─────────────┤
   │  0x1001  │  0 1 0 1 1 0 1 0 │     90      │
   ├──────────┼──────────────────┼─────────────┤
   │  0x1002  │  0 0 0 0 0 0 0 1 │      1      │  ← 就算你只想要最右邊那 1 個 bit…
   ├──────────┼──────────────────┼─────────────┤
   │  0x1003  │  1 1 1 1 1 1 1 1 │    255      │  ← 8 個 bit 全開 = 最大值
   ├──────────┼──────────────────┼─────────────┤
   │  0x1004  │  0 0 0 0 0 0 0 0 │      0      │  ← 全關 = 最小值
   └──────────┴──────────────────┴─────────────┘
                ↑           ↑
                │           └── 沒有「0x1002 的第 3 個 bit」這種地址！
                └────────────── bit 是住在 byte 裡的「房客」，沒有自己的門牌
```

### 🔑 關鍵：CPU 只能指名「門牌」，不能指名「房客」

CPU 對記憶體說話只有一種方式：**「把 `0x1002` 這一格給我。」**

它**沒有辦法**說「把 `0x1002` 的第 3 個 bit 給我」——**因為那個地址根本不存在。**

所以當你想讀「某一個 bit」時，硬體實際上做的是：

```
Step 1  把整個 0x1002（8 個 bit 一起）搬進 CPU 暫存器
        暫存器： 0 0 0 0 0 0 0 1

Step 2  用「位元遮罩 bitmask」把不要的 bit 蓋掉
        暫存器： 0 0 0 0 0 0 0 1
        遮罩：   0 0 0 0 0 0 0 1   ← 只留最右邊那位
        AND 運算 ─────────────────
        結果：   0 0 0 0 0 0 0 1

Step 3  （必要時）用「位移 shift」把它挪到定位
```

> **結論：想「只存 1 bit」不但沒省到空間，還多花了 CPU 的步驟。**

### 為什麼硬體要這樣設計？

因為**地址線是有限的資源**。如果讓每個 bit 都有門牌，同樣數量的地址線能定址的記憶體就少了 8 倍，而且絕大多數程式根本用不到「單獨存取一個 bit」。**用 byte 當最小單位是效率與實用性的權衡結果**，1970 年代定下來，沿用至今。

## 2. `TINYINT` 的設計初衷並非只存 0 和 1

`TINYINT` 的名字是「微小整數」，設計目的是儲存一個**微小範圍的整數**，不是二進位開關：

* 1 Byte = 8 bits
* $2^8 = 256$ 種
* 可存 `-128 ~ 127`（有符號 signed）或 `0 ~ 255`（無符號 unsigned）

> **為什麼有符號是 -128~127？** 因為要拿 1 個 bit 出來當正負號，剩下 7 個 bit 表示大小 → 2⁷ = 128 種，分給正負兩邊。總數還是 256 種，只是**切法不同**。

適合用 `TINYINT` 的欄位：月份 (1-12)、一週天數 (1-7)、狀態碼 (0-5)、年齡 (0-120)、評分 (1-5)。

## 3. 💡 「0～255」你其實天天看到——這才是它的用處

**這是「1 byte 到底用在哪」的答案。看到上限 255，就知道背後是 1 byte。**

| 你看過的東西 | 為什麼上限是 255 |
|---|---|
| **CSS `rgb(255, 0, 0)`** | 紅綠藍**各佔 1 byte** → 每個顏色 0～255。這就是為什麼純紅是 255 而不是 100 |
| **色碼 `#FF0000`** | `FF` = 16 進位的 255。**每 2 個 hex 數字 = 1 byte** → 6 個 hex = 3 bytes = RGB 三色 |
| **IP 位址 `192.168.1.1`** | 每一段都是 1 byte → **你永遠不會看到 `192.168.1.300`**（超過 255 就不合法） |
| **`TINYINT` 0～255** | 資料庫的 1 byte 整數 |
| **一個 [[ASCII-美國資訊交換標準碼\|ASCII]] 字元** | 1 byte → 英文字母、數字、符號全塞得下（實際只用了 0-127，因為 ASCII 只用 7 bits） |
| **一張 24 位元全彩圖的一個像素** | 3 bytes（R、G、B 各 1 byte）→ 「24 位元」= 3×8 bits |

> 🎨 **RGB 那條線索最好記**：`rgb(255,255,255)` 是白色，因為三個 byte 全部拉滿 = `#FFFFFF` = `11111111 11111111 11111111`。
> 詳見 [[進位制-二進制-十六進制-Bytes與RGB]]

## 4. 那資料庫有 1 bit 的類型嗎？

**有，MySQL 中叫 `BIT(1)`。** 定義後確實只允許存 `0` 或 `1`。

### 但 `BIT(1)` 真的比較省空間嗎？

**不一定。** 在 MySQL (InnoDB) 中，為了配合前面的「Byte 定址原則」，即使只定義一個 `BIT(1)` 欄位，它在磁碟與記憶體中通常還是會**佔用 1 Byte 的對齊空間**——因為**沒辦法只配一個 bit 的門牌給它**。

只有當你在同一張表**連續定義多個 BIT 欄位**時：

```sql
bit_a BIT(1),
bit_b BIT(1),
bit_c BIT(1)   -- 這三個會被「打包」進同一個 byte
```

資料庫才會聰明地把這幾個 1-bit 欄位塞進同一個 Byte，這時才真正省到空間。

> **原則：byte 是硬體的最小單位，所以「省空間」只能以 byte 為單位去省。** 零星一個 bit 省不到。

## 5. 延伸：byte 在「執行檔」裡長什麼樣

同樣的 byte 觀念，換個場景就是**檔案格式**。

任何檔案本質上都是**一長串 byte**，而「格式」就是**「這串 byte 該怎麼解讀」的約定**。所以每種格式都會在開頭放一個固定標記，叫 **magic number（魔術數字）**，讓程式一眼認出「這是什麼檔」：

| 檔案類型 | 開頭的 magic number（前幾個 byte） |
|---|---|
| Windows 執行檔 `.exe` / `.dll` | `0x4D 0x5A` → 字元 **`MZ`** |
| PNG 圖片 | `0x89 0x50 0x4E 0x47` → **`‰PNG`** |
| PDF | `0x25 0x50 0x44 0x46` → **`%PDF`** |
| ZIP | `0x50 0x4B` → **`PK`** |

### PE = Portable Executable（可攜式可執行檔）

**PE 是 Windows 執行檔的「檔案格式規範」**——規定 `.exe` / `.dll` 內部的 byte 該怎麼排版，作業系統才知道怎麼載入它。

- **Portable（可攜式）**：這個名字的原意是「同一套格式可以跨不同 CPU 架構使用」（x86、x64、ARM 都用 PE），不是指「可以隨身帶著跑」。
- `MZ` 這個 magic number 取自 MS-DOS 開發者 **M**ark **Z**bikowski 的名字縮寫，1980 年代的彩蛋，沿用至今。

各作業系統的執行檔格式：

| OS | 執行檔格式 | 動態函式庫 |
|---|---|---|
| **Windows** | **PE** (Portable Executable) | `.dll` |
| Linux | **ELF** (Executable and Linkable Format) | `.so` |
| macOS | **Mach-O** | `.dylib` |

> 這就是為什麼 Windows 的 `.exe` 不能直接在 Linux 上跑——**連 byte 的排版方式都不一樣，OS 根本看不懂**。

親手驗證（讀出檔案的前 2 個 byte）：

```powershell
[System.IO.File]::ReadAllBytes("C:\Program Files\Git\bin\bash.exe")[0..1]
# → 77, 90   （十進位）
# → 0x4D, 0x5A（十六進位）
# → 'M', 'Z'  （當成字元看）
```

**同樣的 2 個 byte，用不同方式「解讀」就得到不同意義**——這正是「byte 本身沒有意義，格式賦予它意義」的最好示範。

PE 的內部結構詳見 [[VSCode-Cursor-終端機找不到Git-Bash]] 的 §8。

## TL;DR

1. **1 bit = 1 個二進位數字**（0 或 1）。**1 byte = 8 bits**。
2. **2⁸ = 256**：8 個 bit 各自二選一，2 的 8 次方 = 256 種組合。從 0 數起 → **0～255**。
3. **記憶體的門牌（地址）是配給 byte 的，不是配給 bit 的**。CPU 無法定址單一 bit，只能整個 byte 搬進來再用 bitmask 挑。
4. 所以 `TINYINT` 佔 1 byte、`BIT(1)` 也常佔 1 byte——**省空間只能以 byte 為單位省**。
5. **看到「上限 255」就想到「1 byte」**：`rgb(255,0,0)`、`#FF0000`、IP 位址的每一段、[[ASCII-美國資訊交換標準碼|ASCII]] 字元。
6. **2 個 hex 數字 = 1 byte**，這就是工程界愛用 16 進位的原因。
7. 檔案 = 一長串 byte；**格式**（PE / PNG / PDF）＝「這串 byte 怎麼解讀」的約定，開頭的 **magic number** 是身分證。
