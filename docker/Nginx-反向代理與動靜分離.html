重點整理

Nginx 的「多重身份」

Nginx 在同一份設定檔（nginx.conf）裡可以同時扮演兩個角色：

  • 靜態網頁伺服器（Web Server）：請求索取 index.html、圖片等檔案時，直接從硬碟把檔案送出去。
  • 反向代理（Reverse Proxy）：請求是「查資料庫 / 登入」等動態邏輯時，Nginx 自己處理不了，於是把請求轉發給後端容器（Node.js、Python…）。

把前端打包好的檔案放進 Nginx，不代表 Nginx 變成後端。Nginx 像是派送貨物的卡車，只負責把檔案送出去，本身不生產內容（不執行後端邏輯）。

反向代理是「每次」都會運行嗎？

不一定，取決於路由規則（routing rules）。在 nginx.conf 裡依路徑分流：

情境 路徑 Nginx 行為 反向代理是否觸發
請求靜態資源 example.com/ 直接回傳 HTML 檔案 否
請求 API 資料 example.com/api/login 轉發給後端 API 容器 是

這種「動靜分離」設計：靜態檔案像不動的裝潢、由 Nginx 直接送；動態請求才轉給後端現場製作。

.conf 是什麼

.conf = Configuration（設定檔／配置檔案）。讓軟體在不改原始碼的情況下改變運作行為；程式「啟動時」會先讀取它，決定 Port、資料路徑、允許的使用者等。

  • 常見：nginx.conf（流量轉發/靜態檔位置）、resolv.conf（DNS）、sysctl.conf（核心參數）。
  • 現代等價替代：.yaml/.yml（如 docker-compose.yml，可讀性高）、.json、.ini、.env（放環境變數/密碼）。

Proxy（代理伺服器）是什麼

Proxy 是一個技術概念／功能角色（中間人），不是某一間公司的產品。就像「汽車」是概念、Toyota 才是產品；「保全」是職位、Nginx 是某家公司派來的明星保全。

實作 Proxy 的常見軟體：Nginx（最受歡迎的反向代理，由 Nginx 公司開發、後被 F5 收購）、Apache、Squid（專做正向代理＋快取）、HAProxy（高效能負載平衡）。

核心價值：隱私（隱藏真實 IP）、安全（第一線防火牆）、效能（快取 Cache 減少重複請求）。

正向代理 vs 反向代理

類型 方向 誰知道它存在 主要用途
正向代理 Forward Proxy 使用者 → Proxy → 網路 使用者主動設定 翻牆、隱藏身份、企業內部管控
反向代理 Reverse Proxy 網路 → Proxy → 後端伺服器 使用者不知道（以為它就是目標網站） Nginx 主打：負載平衡、保護後端、SSL 加密

為什麼 Proxy 能省資源？

盲點是以為每個人各自在自己電腦架 Proxy——那樣確實無法省資源。能省資源的 Proxy 是很多人共用同一台、且架在伺服器端：100 個同事共用機房裡一台高效能 Proxy，第一個人抓過的熱門頁面會被快取，後面的人直接從快取拿，不必再跑一次網際網路。

靜態 vs 動態：關鍵不在「有沒有 JS」

  • 「靜態」指的是伺服器端的行為：伺服器只是「檔案夾」，你要 index.html 它就原封不動丟給你，不跑程式、不查資料庫組裝內容。
  • 靜態網頁絕對可以有 JS——那個 JS 是傳到瀏覽器（Client）後才執行的（動畫、AJAX/Fetch）。只要伺服器端是「單純傳檔案」，架構上仍算靜態。
  • 純靜態：每個使用者拿到的檔案一模一樣。動態：伺服器跑 PHP/Node.js → 查資料庫 → 把資料塞進 HTML 模板 → 產出「專屬於你」的 HTML（例如登入後的 Facebook）。
  • 因此用「有無 JS」分辨純不純靜態並不準確——判準是「誰在組裝內容」（伺服器 vs 瀏覽器）。
