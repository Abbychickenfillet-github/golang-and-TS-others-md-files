# 訂單報名序號功能

## 需求背景
客戶需求：「民眾購票系統依照報名順序給予序號，方便後續發送贈品」。

## 設計決定
- **不新增資料庫欄位**：直接用 `created_at` + `payment_status = "PAID"` 計算序號
- **序號範圍**：每個活動獨立編號（活動 A 有 1,2,3...，活動 B 也從 1 開始）
- **涵蓋訂單類型**：所有訂單類型
- **計算方式**：前端根據已付款訂單的 `created_at` 排序，依序編號

## 實作計畫（純前端，不改後端）

### 1. Dashboard 後台（主辦方）
**檔案**: `futuresign.dashboard/src/routes/_layout/orders.tsx`
- 從 API 取得的訂單列表中，篩選 `payment_status === "PAID"` 的訂單
- 依 `created_at` 升序排序
- 計算每筆訂單在同活動中的排名作為序號
- 在訂單列表表格中新增「序號」欄位顯示

### 2. Official Website 前台（消費者）
- 消費者在「我的訂單」頁面也能看到自己的報名序號
- 需要從同活動的所有已付款訂單中計算自己的排名
- ⚠️ 注意：前台可能只拿到自己的訂單，無法知道全域排名
  → 可能需要後端提供一個簡單的 API 回傳排名，或在現有 API 回應中加入

## 注意事項
- 序號是動態計算的，不存在資料庫中
- 如果訂單被取消/退款，序號會重新計算（後面的人會往前遞補）
- 前台消費者要看到自己的序號，可能需要後端配合（因為前台只能查到自己的訂單）

## 關鍵檔案
| 檔案 | 修改內容 |
|------|----------|
| `futuresign.dashboard/src/routes/_layout/orders.tsx` | 後台列表顯示序號 |
| `futuresign.official_website` 相關訂單頁面 | 前台顯示序號 |

## 驗證方式
1. Dashboard 訂單列表確認已付款訂單有正確的序號
2. 序號按 `created_at` 排序
3. 不同活動的序號獨立
4. 前台消費者能看到自己的序號
