💻 C. 程式碼填空：彈窗 → 重定向點灰塊看答案

把彈窗改成重定向，要換 import、換方法，還要在載入時接結果。

import { getAuth, ,
         , GoogleAuthProvider }
  from "https://www.gstatic.com/firebasejs/10.14.0/firebase-auth.js";

const auth = getAuth();
const provider = new GoogleAuthProvider();

// 點按鈕 → 整頁跳到 Google
btn.onclick = () => (auth, provider);

// 跳回來後（頁面重載時）接結果
(auth)
  .then((result) => { if(result) console.log(result.user); })
  .catch(console.error);

D1. 用「分工」的角度說明 OAuth 和 RESTful API 的關係。

OAuth 2.0 負責授權：核發 access token，讓第三方 App 在不取得帳密的情況下被允許存取使用者資源。RESTful API 負責操作資源：用 HTTP 動詞對資源做 CRUD。流程上先過 OAuth 拿 token，再帶著 token 去打 REST API。兩者相關但職責不同。

D2. signInWithRedirect 登入「沒效」，你會怎麼排查？至少講兩點。

①確認頁面重載後有用 getRedirectResult() 接結果（它只在重定向回來那一次回傳）。②Firebase Console → Authentication → Sign-in method 是否已啟用 Google 登入。③你的網域是否在授權網域白名單。④打開 F12 Console / Network 看錯誤訊息。

D3. 「電腦關機 LINE 就會登出嗎？」請說明登入狀態維持取決於什麼、何時會被中斷。

不一定關機就登出，取決於憑證與裝置設定。會被中斷的情況：手動登出、清除 cookies／App 資料、安全性驗證（偵測到異地或異常登入）、憑證過期（長期沒活動）。勾選「自動登入」可讓關機後仍保持登入。

D4.〔對照本專案〕同樣是 Google 登入，「前端 Firebase 流程」和「後端 Authorization Code 流程」差在哪？

前端 Firebase：瀏覽器端 SDK（signInWithPopup/Redirect）直接跟 Google 互動，拿到 Firebase 使用者憑證。後端 Authorization Code（本專案 fastapi-sso）：/auth/google/login → Google → 後端 /auth/google/callback，後端核發自家 JWT 再導回前端。後者 client secret 不外洩、較安全。詳見 [[GOOGLE_OAUTH_SETUP]]。

⭕❌ E. 是非題判斷對錯

每題先判斷⭕/❌，再點看答案。

E1. OAuth 是用來做「身分認證（authentication）」的，跟授權無關。
❌ 錯。OAuth 2.0 本質是授權（authorization）框架——「准你拿 token 存取資源」。認證另有 OpenID Connect 在 OAuth 之上處理。別把授權跟認證混為一談。
E2. 呼叫 Google 的 REST API 前，通常要先透過 OAuth 取得授權。
⭕ 對。先過 OAuth 拿 access token，再帶 token 打基於 HTTP 的 REST API（Gmail/Drive…）。
E3. getRedirectResult() 每次頁面載入都會持續回傳上次的登入結果。
❌ 錯。它只在重定向回來後的那一次回傳結果；沒在當下處理就接不到，這正是「重定向沒效」的常見原因。
E4. 彈窗登入失敗（閃一下就消失）可以改用 signInWithRedirect 規避彈窗被擋的問題。
⭕ 對。重定向是整頁跳轉，不依賴彈窗，較不會被瀏覽器攔截；代價是要處理跳轉回來的結果與狀態。
E5. 只要關機，登入狀態一定會被清掉、下次一定要重新登入。
❌ 錯。是否要重登取決於憑證效期與是否開自動登入，不是「關機 = 登出」。清 cookie、安全驗證、憑證過期才會強制重登。
E6.〔對照〕Session 是「有狀態」（狀態存伺服器），JWT 是「無狀態」（狀態由 token 自己帶），所以 JWT 較難主動撤銷。
⭕ 對。Session 登出刪伺服器那筆即可；JWT 無狀態，要撤銷得另作黑名單（jti）或縮短效期＋refresh token。對照 JWT_TOKEN_EXPLANATION、LOGOUT_AND_LOGIN_EXPLANATION。
E7.〔對照〕本專案的 Google 登入跟本筆記的 Firebase 彈窗/重定向，是同一種 OAuth 流程。
❌ 錯。本專案是後端 Authorization Code 流程（fastapi-sso，後端 callback 後發 JWT）；本筆記是前端 Firebase SDK。都叫 OAuth 2.0，但 grant 流程與 token 落點不同。見 GOOGLE_OAUTH_SETUP。
