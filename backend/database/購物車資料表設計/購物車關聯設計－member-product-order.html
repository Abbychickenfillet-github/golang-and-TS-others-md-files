01 核心觀念：order 和 product 是「多對多」

先想清楚每對關係是 1 對多還是多對多，關聯怎麼建就出來了。

關係型態白話怎麼實作
member → order1 對多 (1:N)一個會員有很多張訂單；一張訂單只屬於一個會員在 order 放 member_id 外鍵
order ↔ product多對多 (M:N)一張訂單有多個商品；一個商品出現在多張訂單開一張中間表 order_item
order_item → order多對 1每筆明細屬於一張訂單order_item.order_id 外鍵
order_item → product多對 1每筆明細指向一個商品order_item.product_id 外鍵
關鍵原則

關聯式資料庫沒辦法直接存「多對多」，一律要拆成一張中間表（join／junction table），把 M:N 變成「兩個 1:N」。這裡的中間表就是 order_item（訂單明細），它一邊連 order、一邊連 product。

02 ER 關聯圖（點表格高亮關係）

點上面切換，或點圖中的表格看它連到誰。
member cart / cart_item（結帳前） order（結帳後） order_item（中間表） product

讀法：member 有很多 order；每張 order 有很多 order_item；每個 order_item 指向一個 product。購物車這側用 cart / cart_item 平行對應，只是它是「結帳前」的暫存。

03 購物車 vs 訂單：為什麼要分兩套？（價格快照）

這是最多人忽略、面試也最愛問的點：cart_item 不存價格，order_item 一定要存價格。

cart_item（結帳前）

用「即時價格」

購物車只是「想買的清單」，價格應該即時去 product 表撈現價顯示。商品改價、打折，購物車就跟著變——這是對的。所以 cart_item 只存 product_id + quantity，不存價格。

order_item（結帳後）

用「凍結快照」

一旦下單，成交價就是當下那個價，必須把 unit_price 寫死進 order_item。日後商品漲價，舊訂單金額也不能變——這是帳務正確性與法律憑證。

反例（錯誤設計）

order_item 只存 product_id

如果訂單金額靠「即時 JOIN product.price 算」，那麼商品一改價，歷史訂單金額全變，對帳、退款、發票全爆。這是新手最常踩的雷。

一句話記

購物車跟著現價走（不存價）；訂單凍結成交價（存 unit_price）。 同樣的「明細」概念，因為「時間點」不同，設計就不同。

04 GORM 模型怎麼寫（Go）

用 GORM 的關聯標籤把上面的關係翻成 struct。重點：has many、belongs to 靠「外鍵欄位 + 對應結構」自動接起來。

type Member struct {
    ID     uint   `gorm:"primaryKey"`
    Name   string
    Email  string `gorm:"uniqueIndex"`
    Orders []Order        // has many：一個會員多張訂單
    Cart   Cart           // has one：一個會員一台購物車
}

type Product struct {
    ID    uint   `gorm:"primaryKey"`
    Name  string
    Price int    // 現價（以「分」為單位避免浮點誤差）
    Stock int
}

type Order struct {
    ID        uint   `gorm:"primaryKey"`
    MemberID  uint   `gorm:"index"`            // 外鍵 + 索引（查某會員的訂單）
    Status    string `gorm:"index"`
    Total     int                              // 下單當下總額快照
    CreatedAt time.Time
    Items     []OrderItem                      // has many
}

type OrderItem struct {
    ID        uint `gorm:"primaryKey"`
    OrderID   uint `gorm:"index"`              // 外鍵 → order
    ProductID uint `gorm:"index"`              // 外鍵 → product
    Quantity  int
    UnitPrice int                              // ★ 價格快照：下單當下的單價
}

type Cart struct {
    ID       uint `gorm:"primaryKey"`
    MemberID uint `gorm:"uniqueIndex"`         // 一個會員一台車
    Items    []CartItem
}

type CartItem struct {
    ID        uint `gorm:"primaryKey"`
    CartID    uint `gorm:"index"`
    ProductID uint `gorm:"index"`
    Quantity  int                             // 注意：沒有價格欄位，現價即時撈
}

小細節：金額用整數（分/最小貨幣單位）存，別用 float，避免 0.1+0.2 那種浮點誤差。GORM 的 index 標籤就會幫你在那欄建 B+樹索引——正好對應你最近學的索引筆記。

下單時把購物車「轉成」訂單（價格在這一刻凍結）

// 結帳：讀 cart_item 的現價，寫進 order_item 當快照
func Checkout(db *gorm.DB, memberID uint) error {
  return db.Transaction(func(tx *gorm.DB) error {        // 包在交易裡，全成功或全失敗
    var cart Cart
    tx.Preload("Items").Where("member_id = ?", memberID).First(&cart)

    order := Order{MemberID: memberID, Status: "pending"}
    var total int
    for _, ci := range cart.Items {
      var p Product
      tx.First(&p, ci.ProductID)                          // 撈現價
      order.Items = append(order.Items, OrderItem{
        ProductID: p.ID, Quantity: ci.Quantity,
        UnitPrice: p.Price,                                // ★ 凍結這一刻的價格
      })
      total += p.Price * ci.Quantity
    }
    order.Total = total
    if err := tx.Create(&order).Error; err != nil { return err }
    return tx.Where("cart_id = ?", cart.ID).Delete(&CartItem{}).Error // 清空車
  })
}

05 對應的 SQL 結構（含外鍵與索引）

CREATE TABLE member (
  id     BIGINT PRIMARY KEY AUTO_INCREMENT,   -- 窄＋遞增主鍵（叢集索引友善）
  name   VARCHAR(100),
  email  VARCHAR(255) UNIQUE
);

CREATE TABLE product (
  id    BIGINT PRIMARY KEY AUTO_INCREMENT,
  name  VARCHAR(200),
  price INT,            -- 現價（以分為單位）
  stock INT
);

CREATE TABLE `order` (
  id         BIGINT PRIMARY KEY AUTO_INCREMENT,
  member_id  BIGINT NOT NULL,
  status     VARCHAR(20),
  total      INT,
  created_at DATETIME,
  CONSTRAINT fk_order_member FOREIGN KEY (member_id) REFERENCES member(id),
  INDEX idx_order_member (member_id)          -- 查「某會員的所有訂單」
);

CREATE TABLE order_item (
  id         BIGINT PRIMARY KEY AUTO_INCREMENT,
  order_id   BIGINT NOT NULL,
  product_id BIGINT NOT NULL,
  quantity   INT,
  unit_price INT,                              -- ★ 價格快照
  CONSTRAINT fk_oi_order   FOREIGN KEY (order_id)   REFERENCES `order`(id),
  CONSTRAINT fk_oi_product FOREIGN KEY (product_id) REFERENCES product(id),
  INDEX idx_oi_order (order_id),               -- 查「一張訂單的所有明細」
  INDEX idx_oi_product (product_id)            -- 查「某商品被誰買過」
);

-- 購物車（結帳前）
CREATE TABLE cart (
  id        BIGINT PRIMARY KEY AUTO_INCREMENT,
  member_id BIGINT NOT NULL UNIQUE,            -- 一個會員一台車
  CONSTRAINT fk_cart_member FOREIGN KEY (member_id) REFERENCES member(id)
);
CREATE TABLE cart_item (
  id         BIGINT PRIMARY KEY AUTO_INCREMENT,
  cart_id    BIGINT NOT NULL,
  product_id BIGINT NOT NULL,
  quantity   INT,                              -- 沒有價格欄
  CONSTRAINT fk_ci_cart    FOREIGN KEY (cart_id)    REFERENCES cart(id),
  CONSTRAINT fk_ci_product FOREIGN KEY (product_id) REFERENCES product(id),
  UNIQUE KEY uq_cart_product (cart_id, product_id) -- 同商品在同一台車只一列，加量改 quantity
);

每個外鍵（member_id、order_id、product_id）都建索引——這跟你 event 表那堆 fk_*、idx_* 是同個道理：JOIN 和「依外鍵查子表」都靠它加速。cart_item 的 UNIQUE(cart_id, product_id) 確保同一台車同商品不會出現重複列，再加一件就是把 quantity +1。

06 小測驗（點黃框看答案）

填 1

order 和 product 是多對多，必須開一張 點我 來拆解。

填 2

member 和 order 是 1 對多，外鍵 member_id 要放在 點我。

填 3

下單後即使商品改價、舊訂單金額也不能變，是因為 order_item 存了 點我。

填 4

購物車的 cart_item 之所以不存價格，是因為它要顯示商品的 點我。

填 5

結帳這個動作（讀車→建訂單→清空車）應該包在一個 點我 裡，確保全成功或全失敗。
