00 先分清楚：三個同名、毫不相關的 heap

① 資料結構

🌳 二元堆積 Binary Heap

完全二元樹，父 ≥/≤ 子，最大/最小值永遠在頂端。做優先佇列、heapsort。跟資料庫無關。

② 記憶體

🧠 記憶體堆區 Heap Memory

程式動態配置記憶體那塊（相對 stack）。malloc/new 拿到的就在這。跟資料庫無關。

③ 資料庫 ✅

🗄️ 堆積表 Heap Table

「沒依任何鍵排序、一頁頁隨意堆放」的資料表儲存方式。PostgreSQL 預設、SQL Server 沒建 clustered index 時。

最容易誤會

你學索引看到的 heap（heap-organized、回 heap 撈資料）是第 ③ 種，跟刷題的二元堆積①、記憶體②只是同名、完全無關。資料庫的 heap ＝「一堆沒排序的資料頁」，跟叢集索引（照鍵排好）正好相反。

01 你的問題：heap-organized ＝ 非叢集索引嗎？是很多顆小樹嗎？

先給結論

① heap-organized ≠ 非叢集索引——它們是兩個不同維度：一個講「表怎麼存」，一個講「某個索引的類型」。
② 「樹」是索引，不是 heap：每個索引各自是一棵 B+樹（可以很多顆），但 heap 本身不是樹，是平的資料頁堆。

兩個維度，別黏在一起

名詞它在描述什麼答案是
heap-organized【表】怎麼存（儲存組織方式）一堆無序的資料頁
非叢集索引【某個索引】的類型葉子放指標、要回表

一個講表、一個講索引，不是同一件事。但兩者有因果關係：當一張表是 heap-organized（沒有叢集索引當家），那它上面所有索引「只能」都是非叢集索引——因為沒有任何索引的葉子是資料本身，全都得靠指標回 heap。

🎨 圖解：左邊 heap 表 vs 右邊叢集表

B+樹（索引） HEAP（無序資料堆，不是樹） 叢集樹（葉子＝資料）

所以針對你的兩個問題

Q：heap-organized ＝ 非叢集索引嗎？

不是。 heap-organized 是「表」的儲存方式（無序資料堆）；非叢集索引是「索引」的一種（葉子放指標）。但在 heap 表上，所有索引都只能是非叢集索引——這是因果，不是相等。

Q：那是很多顆小樹嗎？

對一半。 ✅ 每個索引各自是一棵 B+樹，一張表有幾個索引就有幾棵樹（可以很多顆）。❌ 但 heap 本身不是樹，它是平的資料頁堆。樹是「索引」，heap 是「被索引指向的資料」——別把這兩個角色混在一起。

1½ heap 無序＝沒有 PK 嗎？ctid 是 PK 的替代品嗎？

先破除一個誤會

「heap 無序」跟「有沒有 PK」是兩件不同的事，別黏在一起。 heap 表照樣可以有 PK！PK 只是「另一棵索引樹」，負責保證唯一、給你和外鍵當識別——但它不會把資料排序，資料仍躺在無序的 heap 裡。PostgreSQL 正是這樣：每張表都是 heap，PK 只是一個 unique 的 B+樹索引，葉子存 ctid 指回 heap。

PK 和 ctid/RID 是「兩個不同角色」，不是替代關係

　PK（主鍵）ctid / RID
是什麼邏輯識別碼（你選的、有意義，如 id=5）實體位址（檔案／分頁／列號）
誰在用你、外鍵、JOIN、API儲存引擎內部定位用
穩不穩定穩定，不會變會變！UPDATE／VACUUM FULL／CLUSTER 後就改變
怎麼來的你定義 + 唯一索引強制不重複引擎自動給，跟著實體位置走
能當外鍵嗎能不能（會變，不可靠）

所以 ctid 不是「PK 的替代品」，它是「索引用來找到 heap 那一列的低階指標」。所有索引（包含 PK 索引）的葉子都存 ctid。查 WHERE id=5 時：走 PK 索引那棵樹 → 拿到 ctid → 跳到 heap 對應的那一列讀整列（這就是 PG 幾乎每次查詢都要回表的原因）。

關鍵分離

「實體順序」（heap vs 叢集）和「邏輯識別」（PK）是兩件獨立的事。
· heap 表：可以有 PK，但 PK 不決定實體排序（資料仍無序，靠 ctid 定位）。
· 叢集表（InnoDB）：PK 同時決定實體排序（資料照 PK 排好，葉子就是資料）。
差別只在「PK 有沒有順便排序資料」，不在「有沒有 PK」。

實用警告

ctid 會變（PG 的 UPDATE 會寫新版本→新 ctid，VACUUM FULL / CLUSTER 也會重排）。所以千萬別把 ctid 當永久 ID 或寫進程式邏輯／外鍵。要穩定識別一律用 PK。
