01 一棵會動的 B+樹：範圍查詢演給妳看

點下「示範範圍查詢 WHERE id BETWEEN 30 AND 70」，看資料庫怎麼定位一次、然後沿著葉子節點的鏈結串列一路掃，不用每次回到 Root。

點按鈕開始。觀察底部那條黃色葉子鏈。
Root（根，只負責導航） Branch（中間層，只存鍵與指標） Leaf（葉子層，真正存資料／指標）

為什麼範圍查詢不用每次回 Root？

找 BETWEEN 30 AND 70 時，資料庫只從 Root 往下走一次，定位到「30 落在哪片葉子」。接著因為葉子之間用鏈結串列串好了，直接沿著葉子往右掃到 70 就結束——中間完全不必再爬回樹根。這就是 B+樹做範圍查詢特別快的關鍵。

B+樹分三種角色：Root（根）、Branch / Internal（中間節點）、Leaf（葉子節點）。葉子節點是樹的最底層，也是唯一「真的放有用內容」的一層：

· 在 叢集索引裡，葉子節點放的就是整列實體資料。
· 在 非叢集索引裡，葉子節點放的是索引鍵 + 一個指回真資料的指標。

中間的 Root 和 Branch 都只是「路標」，只存鍵值和往下的指標，幫你快速縮小範圍，本身不存你要查的那一列。

鏈結串列是一種資料結構：每個節點除了存資料，還存一個「指向下一個節點的指標」，像火車一節一節串起來。B+樹的葉子層用的是雙向鏈結串列（Doubly Linked List）——每片葉子同時記得「左邊鄰居」和「右邊鄰居」。

好處：一旦定位到起點葉子，就能像走鐵軌一樣左右順著掃，不需要回頭重走整棵樹。ORDER BY 順序輸出、BETWEEN 範圍掃描，靠的都是這條鏈。

02 三欄對照：葉子節點到底放了什麼？

同樣是 B+樹，差別全在葉子節點存的東西。這三張卡把妳問的「葉子放什麼」一次說清楚。

叢集索引 Clustered

葉子 ＝ 實體資料本身

  • 索引結構和資料列綁在一起
  • 走到葉子＝直接拿到整列完整資料
  • 資料在磁碟上的實體排序就照這個鍵
  • 所以一張表只能有一個叢集索引
葉子內容：[ PK=60 → 整列: name, age, email… ]
非叢集索引（表有叢集索引）

葉子 ＝ 鍵 ＋ 叢集鍵值

  • 是一棵完全獨立的 B+樹
  • 葉子不放整列，放的是叢集鍵（如 PK）
  • 要拿完整資料 → 再走一次叢集索引
  • 這個二次查找＝書籤查找／回表
葉子內容：[ age=25 → PK=60 ]，再拿 PK=60 回叢集索引撈整列
非叢集索引（Heap Table）

葉子 ＝ 鍵 ＋ RID

  • 表沒有叢集索引 → 資料是一堆無序 Heap
  • 葉子放的是實體位址 RID
  • RID（Row Identifier，列識別碼）＝檔案編號＋分頁編號＋列號
  • 拿 RID 可直接跳到磁碟那一頁
葉子內容：[ age=25 → RID(file 3, page 12, slot 4) ]

💡 一句話記：叢集索引的葉子是「資料」，非叢集索引的葉子是「往資料的地址」——地址有兩種：叢集鍵（要回表）或 RID（Row Identifier，列識別碼）（直接跳）。

澄清：回表只發生在「從非叢集索引出發」時（起點是哪一棵樹很關鍵）

原本的三欄圖沒畫出「你是從哪一棵樹開始查的」，這才是會卡住的根本原因。把兩種情況分清楚：

情況 A — 直接用叢集鍵(PK)查，例如 WHERE id = 5
→ 只走一次 B+樹（叢集索引那棵），葉子節點本身就是完整資料，沒有指標、沒有回表。你的直覺在這個情境是對的。

情況 B — 用別的欄位查，例如 WHERE company_id = 7（company_id 不是 PK）
→ 你走的第一棵樹是 company_id 的非叢集索引那棵，不是叢集索引！它的葉子放的不是資料，是「PK 的值」。拿到 PK 後，才需要再走第二棵樹（叢集索引）換到真資料——這個「走第二次」就是回表。

所以重點不是「叢集索引也有指標、要走兩次」，而是：叢集索引本身永遠只走一次（直接到資料）；回表只發生在你從「非叢集索引」出發查詢的時候。兩棵是不同的樹，做的事也不一樣。

拿 company_id 去問（相當於跟櫃台說名字）→ 非叢集索引給你一張號碼牌（PK 值） → 你拿著號碼牌去另一個窗口（叢集索引那棵樹）兌換 → 拿到真正的東西（完整資料列）。這整段「拿號碼牌去兌換」就是回表。

如果你一開始就知道號碼牌號碼（直接用 PK 查 = 情況 A），就直接去兌換窗口拿東西，根本不用先問名字拿牌——全程只走一次。

03 為什麼選 B+樹？（扁平 + 鏈結串列）

資料庫不用一般二元搜尋樹或紅黑樹，因為 B+樹有兩個對「磁碟」特別友善的特性：

① 樹高極低（扁平）。 B+樹每個節點可以有成百上千個子節點，所以即使資料量上百萬、上千萬筆，樹高通常也只有 3～4 層。每往下一層約等於一次磁碟 I/O——層數少＝讀寫次數少。

② 範圍查詢極快。 所有葉子之間用雙向鏈結串列相連，定位一次後就能順著掃，BETWEEN / ORDER BY 都受惠。

3. B+樹 vs Binary Tree（二元樹）差在哪？

面向Binary（二元搜尋）樹 / 紅黑樹B+樹
每個節點子節點數最多 2 個成百上千個（高扇出 fan-out）
百萬筆資料的樹高約 20+ 層約 3～4 層
磁碟 I/O 次數多（層數＝I/O 次數）少，非常適合磁碟
資料放哪每個節點都放資料只有葉子放資料／指標
範圍查詢要中序走訪、反覆上下爬葉子鏈結串列直接掃，超快
適合場景記憶體內的資料結構磁碟上的資料庫索引

直覺：二元樹「又高又瘦」，每下一層只砍一半，磁碟要跳很多次；B+樹「又矮又胖」，一層就砍掉幾百分之一，跳幾次就到底。對動輒一次 I/O 就是毫秒等級的磁碟來說，矮胖永遠贏。

3½ B+樹的樹高是固定三層嗎？（矮胖神樹的真面目）

很多範例都畫三層，容易讓人以為「Root / Internal / Leaf」就固定三層。其實樹高會隨資料量變動——這節用一個互動計算機演給你看。

不是固定的！ 樹高完全取決於「資料量多寡」與「每個節點能塞多少 Key（分支度）」。資料量少時可能只有 1~2 層；大到幾億筆時，可能長到 4~5 層。

那為什麼範例老是畫「三層」？

① 教學好畫：三層剛好能完整表現「根節點 → 中間導航層 → 底部葉子層」的完整結構。
② 現實威力：真實資料庫裡，光是 3 層的 B+樹就已經能容納上百萬到上千萬筆資料了——所以三層範例其實一點都不小。

答案是「分支度 Fan-out」！普通二元樹一個點只能分 2 岔，但資料庫一個頁面（常見 8KB）可以塞上百個 Key。這種「一個節點能分上百個岔」的特性，讓樹長得極寬、極扁。

🧮 互動：分支度 × 樹高 = 容量

拉動滑桿改變「每個節點的分支度（Fan-out）」，看每多一層，容量怎麼用次方暴衝。

樹高結構最多容納（= Fan-out ^ 層數）大概是
1 層Root 兼 Leaf100百筆
2 層Root → Leaf10,000萬筆
3 層Root → 1 層 Internal → Leaf1,000,000百萬筆
4 層Root → 2 層 Internal → Leaf100,000,000億筆
5 層Root → 3 層 Internal → Leaf10,000,000,000百億筆

分支度 100 時：3 層就裝得下一百萬筆、4 層一億筆。難怪查幾億筆只要 3~4 次磁碟 I/O。

📈 它是怎麼「長高」的？（由下往上 Node Splitting）

它是「由下往上」擠上去的！當某頁葉子節點滿了 → 分裂成兩頁，並把中間值往上推給上一層。如果連上一層也滿了，就一路往上推，直到逼 Root 分裂，整棵樹才往上長高一層。所以 B+樹永遠是「從底部長、往上頂」，不是從上面加。

🛡️ 平衡樹的鐵律

無論變幾層，所有 Leaf 節點永遠在同一層。絕不會出現「左邊葉子深度 3、右邊葉子深度 5」。這就是它叫 Balanced Tree（平衡樹） 的原因——任何一筆資料的查找路徑長度都一樣，效能穩定可預測。

💡 大總結：樹高不固定，但因為 B+樹「太能裝」，通常 3~4 層就足以應付世上 95% 以上系統的資料量。查資料時通常只要 3~4 次磁碟 I/O 就能從幾億筆裡撈出目標——這就是「矮胖型神樹」最恐怖的威力。

04 ⚠ PostgreSQL 特例：沒有「持續維護」的叢集索引

給 Abby 的紅字警告

妳目前的 stack 是 Go + GORM + PostgreSQL，而 PostgreSQL 跟 SQL Server / MySQL InnoDB 不一樣：

PostgreSQL 沒有持續維護的叢集索引。它的資料表預設是無序的 Heap，所有索引（包含 Primary Key）結構上都更接近「非叢集索引」——索引和資料是分開的兩棵 / 兩塊東西。也就是說，在 PG 裡查資料幾乎都會發生「回表」（除非用到涵蓋索引 / Index-Only Scan）。

那 CLUSTER 是什麼？要在哪裡下？

PostgreSQL 提供 CLUSTER 指令，能照某個索引把整張表的實體資料重新排序一次。注意關鍵字是「一次性」：它只在你執行的當下排好，之後新增 / 更新的資料不會自動維持順序，所以這比較像「模擬一次叢集」，不是真正持續維護的叢集索引。

在哪裡下？ 直接在連到該資料庫的 SQL 介面執行即可——psql、TablePlus / DBeaver / pgAdmin 的 query 視窗都行。它是 DDL/維護指令，不會放進 GORM 的一般 query，通常寫在 migration 或維護腳本裡。

-- 1) 該表要先有一個索引可以拿來當排序依據
CREATE INDEX idx_orders_created_at ON orders (created_at);

-- 2) 照這個索引，把 orders 表的實體資料重排一次（一次性）
CLUSTER orders USING idx_orders_created_at;

-- 3) 之後可以只打 CLUSTER（PG 會記住上次用的索引）
CLUSTER orders;

-- 4) 一次重排所有「曾經 CLUSTER 過」的表
CLUSTER;

-- 注意：CLUSTER 會鎖表（ACCESS EXCLUSIVE），且需要額外磁碟空間重寫整張表，
-- 線上尖峰時段別亂跑。新進資料不會自動維持順序，要定期重跑。

CLUSTER 到底照什麼順序排？（數字大小 還是 a→z？）

照你指定的那個索引「本身已經排好的順序」——CLUSTER 只是把實體資料照著它重抄一遍。而索引的順序，取決於 欄位型別 ＋ 排序方向(ASC/DESC) ＋ collation：

· 整數 / 數字欄 → 照數字大小（預設由小到大）
· 文字欄 → 照該欄的 collation（多半 a→z；中文依排序規則）
· 日期欄 → 由舊到新
· 索引建成 DESC → 反向（大到小 / z→a）

-- ① 照時間索引：由舊到新
CREATE INDEX idx_orders_created_at ON orders (created_at);
CLUSTER orders USING idx_orders_created_at;

-- ② 照主鍵（PK 也是一個索引）：id 數字由小到大
CLUSTER orders USING orders_pkey;

-- ③ 照文字欄索引：order_code 字母 a→z（依 collation）
CREATE INDEX idx_orders_code ON orders (order_code);
CLUSTER orders USING idx_orders_code;

-- ④ 複合索引：先 customer_id，再 created_at
CREATE INDEX idx_orders_cust_date ON orders (customer_id, created_at);
CLUSTER orders USING idx_orders_cust_date;

-- ⑤ DESC 索引：amount 由大到小
CREATE INDEX idx_orders_amount_desc ON orders (amount DESC);
CLUSTER orders USING idx_orders_amount_desc;

-- ⑥ 之後重跑（PG 記得上次用哪個索引）
CLUSTER orders;   -- 用上次的索引重排這張表
CLUSTER;          -- 重排所有「曾經 CLUSTER 過」的表

一句話：CLUSTER 跟著「索引」走、不自己決定順序。想要什麼排法，就先建一個那樣排的索引，再 CLUSTER ... USING 它。

5. 「回表」到底是什麼？

「回表」＝Bookmark Lookup（書籤查找）。當你用非叢集索引找到目標葉子時，葉子裡只有索引鍵和一個指標，沒有你 SELECT * 要的其它欄位。資料庫只好拿著那個指標（叢集鍵或 RID）再跑一趟去把整列撈回來——這「第二趟」就叫回表。

回表會多一次（甚至多次隨機）I/O，所以是效能關注點。避免回表的招數：涵蓋索引（Covering Index）——把查詢要用到的欄位都塞進索引，讓資料庫光看索引葉子就湊齊答案，根本不用回表（PostgreSQL 叫 Index-Only Scan，可用 INCLUDE 加欄位）。

-- PostgreSQL：把 status 也帶進索引，讓 "查 user_id 取 status" 不用回表
CREATE INDEX idx_orders_user ON orders (user_id) INCLUDE (status);
-- 之後 EXPLAIN 會看到 "Index Only Scan"，代表沒有回表 🎉

想看真的回表長怎樣？可以在 PG 跑 EXPLAIN (ANALYZE, BUFFERS) SELECT ...，觀察是 Index Scan（會回表）還是 Index Only Scan（不回表）。需要的話我可以幫妳實際截一張 query plan 給妳看。

4½ 釐清：PostgreSQL 是叢集還是非叢集？＋ heap 是什麼

這題會搞混，是因為「底層資料結構」和「儲存組織方式」是兩個不同維度，被混在一起問。拆開就清楚了。

兩個維度，別混在一起

維度在問什麼PostgreSQL 的答案
底層資料結構索引用哪種樹？B+樹（跟幾乎所有 RDBMS 一樣）
儲存組織方式葉子放整列資料，還是放指標？非叢集 / Heap（葉子放指標，取整列要回表）

所以一句話：PostgreSQL 用 B+樹（資料結構相同），但組織方式是非叢集 Heap。 「所有關聯式資料庫底層都用 B+樹」這句是對的——但那是在講資料結構，不是在講叢集 / 非叢集。

heap（堆積）是什麼

這裡的 heap 是資料庫專有名詞：指「沒有依任何鍵排序、一頁一頁隨意堆放的資料表」。新資料插進來就找個有空位的頁塞，所以要靠指標（PG 叫 ctid、SQL Server 叫 RID）才能定位某一列。PostgreSQL 的表預設就是 heap → 這就是它「非叢集」的原因。

⚠ 別跟另外兩個同名詞搞混：Heap 資料結構（優先佇列/最大最小堆）與 記憶體 Heap（動態配置）都跟這裡無關。資料庫的 heap = 「一堆沒排序的資料頁」，跟叢集索引（照鍵排好）正好相反。

三家資料庫切換：葉子到底放什麼？

非叢集 / Heap

PostgreSQL — 永遠 Heap（非叢集）

小結：用 B+樹這件事三家一致；差別在「表本身怎麼擺」——InnoDB 照 PK 叢集、SQL Server 可選、PostgreSQL 永遠 heap。

05 選讀：涵蓋索引 ＆ B-Tree vs B+-Tree

涵蓋索引（Covering Index）一句話

如果一個索引「涵蓋」了某查詢需要的所有欄位，資料庫就能只讀索引、不碰主資料——這個索引對那個查詢就是 covering index，效果是消滅回表。代價是索引變大、寫入稍慢，是典型的空間換時間。

4. B-Tree 跟 B+-Tree 差在哪？

面向B-Tree（B 樹）B+-Tree（B+ 樹）
資料放哪每一層的節點都可能放資料資料只放在葉子層，內部節點只放鍵與指標
葉子之間沒有互相串連有雙向鏈結串列串起來
範圍查詢要在樹裡上上下下中序走訪，較慢定位一次→沿葉子鏈掃，快
單點查詢運氣好在上層就命中，可能更快一律走到葉子才命中
節點扇出 / 樹高內部節點要存資料，能放的鍵較少 → 較高內部節點只存鍵 → 扇出大、更扁平
資料庫採用較少絕大多數 RDBMS 索引都用它

重點記憶：B+樹＝「資料全趕到葉子 + 葉子串成鏈」。這兩個改動讓它內部更扁平、範圍掃更快，正好是資料庫最在意的兩件事。

② 每一欄在說什麼？

欄位意思
Non_unique1 = 允許重複值（非唯一索引）；0 = 唯一索引。你這些 idx_* 全是 1。
Key_name索引名稱。PRIMARY=叢集主鍵；idx_*=你自建的二級（非叢集）索引；fk_*=外鍵附帶的索引。
Seq_in_index在「複合索引」裡的欄位順序。例如 idx_event_member_created 是 member_id(1)→created_at(2)→deleted_at(3) 的三欄複合索引。
Collation = AA = Ascending，這欄在 B+樹裡由小到大排序。這就是「照鍵排好」的證據。
Cardinality估計的不重複值數量，越高代表越「有鑑別度」、越值得當索引前導欄。你 dev 資料量小所以數字都不大。
Sub_part / Packed前綴索引長度 / 壓縮設定，NULL = 整欄索引、未壓縮。
Null = YES該欄允許 NULL。注意 deleted_at 都是 YES。
Index_type = BTREE底層資料結構＝B+樹（MySQL 顯示成 BTREE）。整張表清一色 BTREE。

③ 看出你的設計意圖（蠻不錯的）

① 幾乎每個複合索引最後都掛一個 deleted_at——這是 GORM 軟刪除（gorm.DeletedAt）的經典優化：因為 GORM 預設查詢都會加 WHERE deleted_at IS NULL，把 deleted_at 放進索引尾巴，能讓「軟刪除過濾」也吃得到索引。

② idx_event_member_created (member_id, created_at, deleted_at) 很明顯是為了「查某會員的活動、依建立時間排序、且只看未刪除」這種查詢量身打造，符合複合索引「等值欄在前、範圍/排序欄在後」的原則。company_created 同理。

③ start_at / end_at 各自有索引 → 是為了「依活動起訖時間做範圍查詢」。這正是 01 節範圍查詢動畫在演的場景。

5⅞ 叢集索引效能會比較好嗎？（取捨 + 回表圖解）

沒有絕對答案——叢集索引是取捨。主鍵相關的查詢它很快，但把成本轉嫁到「二級索引變大 + 回表 + 非遞增主鍵的寫入」上。

先把「回表」一次說死

回表（Bookmark Lookup）＝第二趟撈資料。 用二級索引查到目標時，葉子裡只有「索引鍵 + 指標（InnoDB 是主鍵值）」，沒有你要的其他欄位；資料庫只好拿指標再走一次叢集索引把整列撈回來。這多出來的第二趟就是回表。

贏 / 輸場景對照

場景叢集索引表現為什麼
用主鍵單點查（WHERE id=?）✅ 很快走一次樹到葉子，葉子＝整列，不用回表
用主鍵範圍／排序（BETWEEN、ORDER BY id）✅ 很快資料實體相鄰、順序 I/O，葉子鏈結串列直接掃
用「非主鍵欄位」查並取整列⚠ 多一次回表二級索引葉子只有 PK，要再走叢集索引撈整列
主鍵很寬（如 UUID／字串）⚠ 二級索引全胖一圈每個二級索引葉子都要存主鍵 → 佔空間、變慢
非遞增主鍵的大量寫入❌ 頁分裂、碎片化新列要插進「中間」，搬資料、分頁，插入變慢
遞增主鍵（BIGINT AUTO_INCREMENT）寫入✅ 順手永遠往尾端 append，不會頁分裂

給你兩個 stack 各一句

🟢 MySQL InnoDB（你截圖那張）：PK 本來就是叢集 → 維持「窄＋遞增的整數主鍵」最關鍵，熱查詢再用涵蓋索引消回表，就能拿好處又避開頁分裂。

🔵 PostgreSQL（你另一邊）：根本沒有持續的叢集索引（永遠 heap），這題基本不適用。優化主力是設計好二級索引 + 涵蓋索引（INCLUDE → Index-Only Scan），而不是糾結叢集；要一次性物理排序才偶爾用 CLUSTER。

一句話總結：主鍵相關的點查與範圍查，叢集索引明顯快；代價是二級索引變大、回表、以及非遞增主鍵的寫入。選對主鍵（窄＋遞增）才能拿到好處不踩雷。

C. 申論（2 題）— 先自己寫，再看參考方向

申論 1

用你自己的話說明：為什麼 B+樹做「範圍查詢」比二元搜尋樹快？請至少提到「葉子層」與「鏈結串列」兩個關鍵字。

參考方向：二元樹做範圍查詢要靠中序走訪，在樹中反覆上下爬、跨子樹跳，且樹高動輒 20+ 層、I/O 多。B+樹則是：①所有資料 / 指標都在葉子層；②葉子之間用雙向鏈結串列相連。於是只要從 Root 往下定位一次找到範圍起點葉子，之後就沿著鏈結串列「水平」一路掃到終點，不必每筆都回 Root，也不必跨樹枝跳，所以範圍查詢明顯更快。
申論 2（連結你的架構）

情境：你的後端是 Go + GORM + PostgreSQL，有一張 orders 表，最常見的查詢是 SELECT id, status FROM orders WHERE user_id = ? ORDER BY created_at。 請說明：(a) 在 PG 這個查詢預設會不會回表、為什麼；(b) 你會怎麼設計索引來避免回表；(c) 為什麼 CLUSTER 不是長久解。

參考方向：

(a) 預設會回表。 PG 的表是無序 Heap、索引與資料分離，user_id 的一般索引葉子只有 user_id → ctid，撈 status 必須拿 ctid 回 Heap，那就是回表（EXPLAIN 顯示 Index Scan + Heap Fetches）。

(b) 建涵蓋索引，把 created_at 當排序鍵、status 用 INCLUDE 帶進去：
CREATE INDEX idx_orders_user ON orders (user_id, created_at) INCLUDE (status);
這樣 WHERE user_id=? ORDER BY created_at 能用索引排序，id（PK）與 status 都在索引內 → 走 Index-Only Scan，不回表。

(c) CLUSTER orders USING ... 只是一次性把 Heap 物理重排，新寫入不會維持順序，高寫入的 orders 很快又亂掉，且 CLUSTER 會鎖表重寫，無法線上常態維護。所以正解是涵蓋索引 / 適當複合索引，而不是靠 CLUSTER。
