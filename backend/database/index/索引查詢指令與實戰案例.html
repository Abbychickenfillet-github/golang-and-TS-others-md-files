01 怎麼查索引？用什麼指令（PG vs MySQL）

A. 看「這張表有哪些索引」

MySQL

SHOW INDEX

SHOW INDEX FROM event;
-- 重點欄位：
-- Key_name(索引名) Seq_in_index(複合索引第幾欄)
-- Non_unique(1=非唯一) Cardinality(不重複值估計)
-- Index_type=BTREE Collation=A(升冪)
PostgreSQL

\d 或 pg_indexes

\d event                 -- psql 內看表結構含索引
-- 或查系統視圖：
SELECT indexname, indexdef
FROM pg_indexes
WHERE tablename = 'event';

B. 看「這條查詢有沒有用到索引 / 有沒有回表」

MySQL 
EXPLAIN
SELECT * FROM event WHERE member_id = 5;
-- 看 key 欄位=用了哪個索引；
-- type=ref/range 好，type=ALL 是全表掃描(壞)；
-- Extra: "Using index" = 涵蓋索引不回表 🎉

EXPLAIN ANALYZE SELECT ...;  -- 8.0+ 給實際耗時
PostgreSQL 
EXPLAIN (ANALYZE, BUFFERS)
SELECT * FROM event WHERE member_id = 5;
-- Index Scan      = 用索引(會回表)
-- Index Only Scan = 涵蓋索引(不回表) 🎉
-- Seq Scan        = 全表掃描(壞)
-- Heap Fetches    = 回表次數

C. 看「哪些索引重複多餘 / 從沒被用過」（清索引）

想知道MySQL 指令需要 performance_schema？
重複/多餘索引（結構面）SELECT * FROM sys.schema_redundant_indexes;不需要 ✅
從沒被用過的索引（執行面）SELECT * FROM sys.schema_unused_indexes;需要開著 ❌

PostgreSQL 對應：用過幾次看 pg_stat_user_indexes 的 idx_scan 欄（=0 代表沒被用過）。細節見 [[鎖與交易與並發控制]] 第 01 章。

02 cluster / non-cluster / heap-organized：PG vs MySQL

　MySQL InnoDBPostgreSQL
表怎麼存照 PK 叢集（clustered）永遠 heap-organized（無序堆積）
PK 索引葉子＝整列資料（找到即拿到）＝指標 ctid（仍要回 heap）
二級索引葉子放 PK 值 → 回表走叢集索引放 ctid → 回表跳 heap
有沒有真叢集索引有（就是 PK）沒有，CLUSTER 只能一次性重排
避免回表涵蓋索引（多欄索引把欄位都帶上）INCLUDE → Index-Only Scan
一句話

底層都是 B+樹（同資料結構）；差別在「表本身怎麼擺」：InnoDB 照 PK 叢集、PostgreSQL 永遠 heap。所以同一句 SQL，在兩邊的 query plan 不一定一樣。heap 這個詞容易誤會，另見 [[三種Heap的差異釐清]]。

03 實戰案例：查詢結果會遇到什麼

案例 1：非唯一索引（Non-unique index）

允許欄位有重複值的索引，SHOW INDEX 裡 Non_unique = 1。大多數二級索引都是非唯一，例如 status、member_id。對比：唯一索引 Non_unique=0（如 email），會額外保證「不重複」。

CREATE INDEX idx_event_status ON event (status);   -- 非唯一：很多列 status 都=published
CREATE UNIQUE INDEX uq_member_email ON member (email); -- 唯一：每個 email 只能一列

查詢時：非唯一索引 WHERE status='published' 會回傳很多列（type=ref，可能很多筆）；唯一索引 WHERE email=? 最多一列（type=const/eq_ref，最快）。

案例 2：複合索引（Composite index）＋ 最左前綴

多欄合成一個排序鍵 (member_id, created_at)。規則：等值欄在前、排序/範圍欄在後，且查詢必須用到最左前綴才吃得到。

CREATE INDEX idx_member_created ON event (member_id, created_at);

-- ✅ 吃得到（用到最左欄 member_id）
WHERE member_id = 5 ORDER BY created_at;
WHERE member_id = 5 AND created_at > '2026-01-01';
-- ❌ 吃不到（跳過最左欄，直接查 created_at）
WHERE created_at > '2026-01-01';

驗證：對上面三句各跑一次 EXPLAIN，看 key 是否顯示 idx_member_created；吃不到的那句會變 Seq Scan / type=ALL。

案例 3：Hot Row（熱點列）— 並發瓶頸

Hot row ＝ 被大量並發交易同時讀寫的「同一列」，變成鎖競爭的瓶頸。索引能讓你「找得快」，但找得快不代表「改得快」——大家搶改同一列時，會卡在那列的 X 鎖排隊上。

怎麼發現 hot row

MySQL：查 performance_schema.data_locks / data_lock_waits 看誰在等誰；或看 SHOW ENGINE INNODB STATUS 的 LATEST DETECTED DEADLOCK / 等鎖資訊。症狀：某列更新延遲飆高、鎖等待暴增。

注意：hot row 是並發/鎖問題，不是索引能解的——索引解「找」，hot row 要靠「分散寫入」解（見下個案例）。

案例 4：全站按讚計數器（把上面全部串起來）

需求：一篇貼文的「總讚數」，全站使用者同時按。天真設計就是經典 hot row 雷：

-- ❌ 天真：所有人搶改同一列 → hot row，按讚一多就卡在這列的 X 鎖
UPDATE post SET like_count = like_count + 1 WHERE id = 123;

改法 A：分片計數器（sharded counter）

-- 把一個計數拆成 N 列，隨機挑一列加
UPDATE post_like_shard
SET cnt = cnt + 1
WHERE post_id=123 AND shard = FLOOR(RAND()*10);
-- 讀總數時把 N 列加起來
SELECT SUM(cnt) FROM post_like_shard WHERE post_id=123;

把「一個 hot row」拆成 10 個 warm row，分散鎖競爭。

改法 B：明細表 + 索引（可查誰按過）

CREATE TABLE post_like (
  post_id BIGINT, user_id BIGINT,
  PRIMARY KEY (post_id, user_id)   -- 複合主鍵：防重複按
);
-- 複合索引(post_id,user_id)：查「某貼文被誰按」走最左前綴
SELECT COUNT(*) FROM post_like WHERE post_id=123;

用複合主鍵天然防止同一人重複按；COUNT(*) 走索引。量大時再加維護式總數或快取(Redis)。

這個例子同時用到：複合索引（post_id,user_id）、非唯一/唯一（PK 防重複）、hot row（天真寫法的病）、FOR UPDATE/原子更新（lock 那篇）。索引讓你查得快，但寫入熱點要靠「分散」而非索引解決。
