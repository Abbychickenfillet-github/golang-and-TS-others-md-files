## 速查指令
| 想知道 | MySQL | PostgreSQL |
|---|---|---|
| 表有哪些索引 | `SHOW INDEX FROM 表` | `\d 表` 或 `pg_indexes` |
| 查詢有沒有用索引/回表 | `EXPLAIN` / `EXPLAIN ANALYZE` | `EXPLAIN (ANALYZE, BUFFERS)` |
| 重複多餘索引（免 perf_schema） | `sys.schema_redundant_indexes` | 比對 `pg_indexes` 定義 |
| 從沒被用過的索引 | `sys.schema_unused_indexes`（要 perf_schema） | `pg_stat_user_indexes.idx_scan = 0` |

## 四個案例
1. **非唯一索引**：`Non_unique=1`，允許重複（status、member_id）；唯一索引保證不重複（email）。
2. **複合索引 + 最左前綴**：`(member_id, created_at)` 要用到最左欄才吃得到；等值欄在前、排序欄在後。
3. **Hot row**：被大量並發搶讀寫的同一列 → 鎖競爭瓶頸。索引解「找」，不解「寫熱點」。
4. **全站按讚計數器**：天真 `UPDATE post SET like_count=like_count+1` = hot row 雷。解法：分片計數器（拆多列）/ 明細表 `post_like(post_id,user_id)` 複合主鍵防重複 + 快取。
