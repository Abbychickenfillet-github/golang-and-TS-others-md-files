EXPLAIN 查詢計畫解讀

資料庫到底怎麼執行你的 SQL？看懂它有沒有走索引、會不會回表、有沒有 filesort
放在 backend/database/index/ · 搭配 [[索引查詢指令與實戰案例]]

00 EXPLAIN 是什麼？跟 EXPLAIN ANALYZE 差在哪

EXPLAIN 讓資料庫把「它打算怎麼執行這條 SQL」（查詢計畫 query plan）攤開給你看——走哪個索引、掃多少列、要不要排序。它是你判斷「查詢慢在哪」最直接的工具。

指令會不會真的執行給你什麼
EXPLAIN ...不執行，只估計預估的計畫、成本(cost)、預估列數
EXPLAIN ANALYZE ...會真的跑一次實際耗時、實際列數（估計 vs 實際差很多 = 統計過期）
EXPLAIN (ANALYZE, BUFFERS)（PG）會跑再加上讀了幾個 page（看 I/O）
⚠ 注意

EXPLAIN ANALYZE 會真的執行你的 SQL！對 UPDATE/DELETE 用它會真的改資料——要測寫入請包在交易裡最後 ROLLBACK。

01 PostgreSQL：三種 Scan 一定要會認

計畫節點意思好壞
Seq Scan（循序掃描）從頭到尾掃整張表，沒用索引大表上是警訊（小表反而正常）
Index Scan走索引定位，再回 heap 撈整列（回表）好，但有回表成本
Index Only Scan欄位都在索引裡，不回表（涵蓋索引）最好 🎉
Bitmap Index/Heap Scan命中列數中等時，先收集再批次回表中等選擇性時的折衷

看計畫先抓三個數字：cost=啟動..總成本、rows=預估列數、actual time（ANALYZE 才有）。估計 rows 跟 actual rows 差很多 → 該 ANALYZE 表; 更新統計。

點開看實際 plan（PG）

EXPLAIN ANALYZE SELECT * FROM orders WHERE total > 100;

Seq Scan on orders  (cost=0.00..18.50 rows=300 width=64)
                       (actual time=0.012..0.480 rows=280 loops=1)
  Filter: (total > 100)
  Rows Removed by Filter: 720      ← 掃了 1000 列、丟掉 720 列

整表掃描、靠 Filter 一列列篩。total 沒索引時就會這樣。資料量大就慢 → 考慮 CREATE INDEX ON orders(total)。

EXPLAIN ANALYZE SELECT * FROM orders WHERE id = 5;

Index Scan using orders_pkey on orders
       (cost=0.28..8.30 rows=1 width=64) (actual time=0.02..0.02 rows=1)
  Index Cond: (id = 5)         ← 用索引條件定位

走 PK 索引找到位置，再回 heap 撈 SELECT * 的其它欄位（回表）。單筆查詢很快。

EXPLAIN ANALYZE SELECT id FROM orders WHERE id = 5;

Index Only Scan using orders_pkey on orders ...
  Index Cond: (id = 5)
  Heap Fetches: 0          ← 0 = 完全沒回表 🎉

要的欄位都在索引裡 → 只讀索引就湊齊答案。Heap Fetches: 0 是「沒回表」的鐵證。用 INCLUDE 加欄位可製造這效果。

02 MySQL：看懂 type / key / Extra

MySQL 的 EXPLAIN 是一排欄位，最關鍵看三個：type（存取方式好壞）、key（實際用了哪個索引）、Extra（額外動作）。

type 等級（左邊最好 → 右邊最壞）

const / eq_ref最多一列（PK/唯一）
ref用非唯一索引
命中數列
range索引範圍掃
(BETWEEN/>)
index掃整個索引
(全索引)
ALL全表掃描
(沒用索引)

看到 type=ALL 且表很大 → 通常要加索引。key=NULL 也代表沒用到任何索引。

常見 Extra 值

Extra意思好壞
Using index涵蓋索引，只讀索引、不回表很好 🎉
Using where取回後還要再用 WHERE 過濾中性
Using index condition索引條件下推(ICP)，在索引層先濾好
Using filesort排序沒辦法靠索引，要額外排序注意（大資料量會慢）
Using temporary要建暫存表（常見於 GROUP BY/DISTINCT）注意

點開看實際 plan（MySQL）

EXPLAIN SELECT * FROM orders WHERE member_id = 5;

id type  key                possible_keys      rows  Extra
1  ref   idx_order_member   idx_order_member   3     Using where

type=ref + key=idx_order_member ＝ 有走索引、命中少數列。健康。

EXPLAIN SELECT * FROM orders WHERE total > 100;

id type  key   possible_keys  rows   Extra
1  ALL   NULL  NULL           10000  Using where

type=ALL、key=NULL ＝ 沒任何索引可用，掃一萬列。total 該建索引。

-- 索引 (member_id, status)，只查這兩欄
EXPLAIN SELECT member_id, status FROM orders WHERE member_id = 5;

id type  key                 rows  Extra
1  ref   idx_member_status   3     Using index   ← 不回表 🎉

Using index ＝ 答案全在索引葉子裡，省掉回表。等同 PG 的 Index Only Scan。

EXPLAIN SELECT * FROM orders WHERE member_id=5 ORDER BY created_at;
-- 若索引只有 (member_id)，沒含 created_at：

id type key               rows Extra
1  ref  idx_order_member  3    Using where; Using filesort

Using filesort ＝ 排序得另外做。把索引改成複合 (member_id, created_at)，排序就能直接吃索引、filesort 消失。

03 PG ↔ MySQL 名詞對照

PostgreSQL

Seq Scan ＝ 全表掃描
Index Scan ＝ 走索引 + 回表
Index Only Scan ＝ 不回表
Heap Fetches: 0 ＝ 沒回表
看 I/O：加 BUFFERS

MySQL

type=ALL ＝ 全表掃描
type=ref/range ＝ 走索引
Extra: Using index ＝ 不回表
（無「Using index」就是會回表）
Using filesort/temporary ＝ 額外排序/暫存

三個自我檢查問題

看任何 plan，問自己：① 有沒有走索引？（PG 不是 Seq Scan / MySQL 不是 ALL）② 有沒有回表？（PG 看 Index Only / MySQL 看 Using index）③ 有沒有額外排序？（filesort / Sort 節點）。三題都漂亮，這條查詢就穩了。

04 小測驗（點黃框看答案）

填 1

會「真的執行」SQL、給你實際耗時的是 點我。

填 2

PG 計畫出現 點我 代表整表掃描、沒走索引（大表是警訊）。

填 3

PG 的 點我 代表用涵蓋索引、完全不回表。

填 4

MySQL 的 type 出現 點我、key=NULL，就是沒用到任何索引。

填 5

MySQL Extra 出現 點我 代表涵蓋索引、不回表（＝PG 的 Index Only Scan）。

填 6

看到 Using filesort，可把索引改成 點我 來消除它。
