01 為什麼 UPDATE 要排他，卻還要有「共享鎖」？

因為不是所有動作都要改值。讀跟讀之間從不衝突——兩個人同時看同一個值，誰都不會把對方看壞。若只有排他鎖，連「只想讀、但要鎖住防止被改」的交易也會互卡，純粹浪費效能。所以鎖分兩種基本型：

S 鎖 Shared（共享／讀鎖）

📖 借閱影印本

很多人可同時各拿影印本看同一本書，互不妨礙。多個交易能同時持有同一列的 S 鎖。語法：SELECT ... FOR SHARE（MySQL 舊：LOCK IN SHARE MODE）。

X 鎖 Exclusive（排他／寫鎖）

✍️ 把正本借回家改寫

原書正在被改，誰都不能同時碰。一次只有一個交易能持有，排斥其他所有鎖。語法：SELECT ... FOR UPDATE，以及所有 UPDATE/DELETE 自動上的鎖。

唯一綠格＝S + S。記法：讀讀不擋，只要牽涉到寫就互斥。
實際場景：外鍵檢查只要 S 鎖

InnoDB 檢查外鍵約束時（確認被參照那列還在），只上 S 鎖——只是要「確認資料還在、暫時別被刪」，不需要、也不該擋住另一個同時做同樣檢查的交易。這就是為什麼 S 鎖要存在：讓「單純讀、但要防止被改」不必付排他的代價。

除了列上的 S/X，InnoDB 在表層級還有意向鎖 IS / IX。用途：當某交易想鎖整張表時，不必逐列檢查「有沒有人鎖了某列」，只要看表上有沒有 IX/IS 旗標就知道。等於在大門口掛個牌子「裡面有人在改某些列」，加速衝突判斷。它們彼此大多相容，只跟「表層 X」衝突，平常不太需要手動管。

03 鎖競爭（Lock Contention）：效能怎麼被吃掉的

鎖競爭＝很多交易想要同一個資源的衝突鎖，只好排隊等。 隊伍越長、持鎖時間越久，吞吐量越низ。三個放大鎖競爭的因素：

因素為什麼變慢怎麼緩解
持鎖時間太長交易裡夾了慢查詢、外部 API call、人為等待，鎖一直不放交易盡量短；先算好再進交易；別在交易裡呼叫外部服務
熱點列 Hot row大家搶改同一列（如全站計數器），全卡在那列 X 鎖分片計數器、批次累加、用 Redis 原子計數
鎖範圍太大沒走索引 → 掃描鎖了一堆不相關的列（甚至整表）讓 WHERE 走索引，把鎖縮到目標列（見下方警告）
索引也會影響鎖範圍！

InnoDB 是對「掃到的索引記錄」上鎖。如果 UPDATE ... WHERE x=? 的 x 沒有索引，它可能掃描並鎖住很多列甚至整張表，鎖競爭瞬間爆炸。所以「WHERE 條件走索引」不只查得快，也讓鎖得準、鎖得少。這就是索引與鎖的交會點。

04 工程師如何「主動」加鎖？正確 vs 不好的寫法

平常 UPDATE/DELETE 會自動上 X 鎖。但「先讀、判斷、再改」這種情境，讀的那一步預設不上鎖，中間就有空窗 → 要主動加鎖。

兩個主動加鎖語法

語法上什麼鎖用在
SELECT ... FOR UPDATEX 鎖讀完馬上要改（扣庫存、轉帳）
SELECT ... FOR SHARE
(舊:LOCK IN SHARE MODE)		S 鎖讀完不改、但要確保這段時間別人不能改（一致性讀）

❌ 不好的寫法（會 lost update）

-- 讀和改分兩句，中間沒鎖，資料庫不知道它們是「同一件事」
SELECT stock FROM products WHERE id=1;     -- A、B 都讀到 stock=10
-- 程式各自算 10-1=9
UPDATE products SET stock=9 WHERE id=1;     -- A、B 都寫回 9（少扣一次！）

問題：兩句之間的空窗完全沒保護，兩個交易的修改互相覆蓋＝lost update。

✅ 正確寫法之一：悲觀鎖 FOR UPDATE

START TRANSACTION;
SELECT stock FROM products WHERE id=1 FOR UPDATE;  -- 讀就上 X 鎖，B 被卡住排隊
-- 確認 stock>0 後
UPDATE products SET stock = stock - 1 WHERE id=1;
COMMIT;                                            -- 放鎖，B 才進來，讀到的已是 9

假設「衝突很常發生」就先鎖起來。代價：並發被序列化、要小心死鎖。

✅ 正確寫法之二：原子更新（最簡單時優先）

-- 不先讀，直接讓資料庫在一句內完成「讀+算+寫」，天然沒有空窗
UPDATE products SET stock = stock - 1 WHERE id=1 AND stock > 0;
-- 看「影響列數」：=1 成功；=0 代表沒庫存或被搶完

能用一句 x = x - 1 解決就別先 SELECT。單純扣減這是最省事、最快的做法。

✅ 正確寫法之三：樂觀鎖（衝突少時）

-- 用版本號，不先鎖；改的時候比對版本
SELECT stock, version FROM products WHERE id=1;     -- 讀到 version=7
UPDATE products SET stock=stock-1, version=version+1
WHERE id=1 AND version=7;                            -- 影響列數=0 → 被別人搶先 → 重試

假設「衝突很少」，不上鎖、衝突了才重試。並發高、衝突低時效能最好。

怎麼選

單純加減 → 原子更新；要先讀複雜邏輯再改、且衝突常見 → FOR UPDATE（悲觀）；衝突很少、想要高並發 → 版本號（樂觀）。三者都能擋 lost update，差在「假設衝突多還是少」。

4½ 三個常卡住的澄清：version / 主動 S 鎖 / 新舊語法

① 樂觀鎖跟 version 到底什麼關係？

悲觀鎖＝「假設一定撞，先鎖」；樂觀鎖＝「假設很少撞，不鎖，等要寫時才檢查有沒有被插隊」。那個 version 欄就是檢查依據——它是一支「我讀到之後有沒有人動過？」的探針。

步驟A 交易B 交易
讀讀到 stock=10, version=7讀到 stock=10, version=7
寫UPDATE ... SET stock=9, version=8 WHERE id=1 AND version=7 → 影響 1 列（version 變 8）... WHERE id=1 AND version=7 → 現在 version 是 8，撞不到 → 影響 0 列
結果成功影響列數=0 → 知道被搶先 → 重讀(version=8)、重算、再試

關鍵信號就是「影響列數 = 0」＝有人在你讀完後改過。全程沒有 DB 鎖，純靠版本對不上來偵測衝突，所以一樣能擋 lost update。對照：FOR UPDATE 是「事前就鎖、別人等」；樂觀鎖是「事後才查、撞了重試」。

② 主動上 S 鎖（FOR SHARE）為什麼還要「主動」？

因為普通的 SELECT 在 InnoDB 預設下根本不上鎖——它讀的是 MVCC 快照，讀完別人照樣能改那列。所以當你需要「我讀到的這列，在交易結束前別人不准改/刪」，就得主動用 FOR SHARE 把「無鎖快照讀」升級成「上鎖讀」。

為什麼用 S 不用 X？

S 鎖讓別人也能讀（S+S 相容），只擋「改」。典型場景：你讀一筆 parent 列、待會要插 child 參照它，需確保 parent 在 commit 前不被刪——FOR SHARE 剛好：大家能一起讀、但誰都不能刪它，又不像 FOR UPDATE 把其他讀的人也擋掉。

一句話：普通 SELECT 不鎖，要鎖就得自己講；FOR SHARE =「大家可以一起讀，但這段時間誰都別改它」。

③ LOCK IN SHARE MODE 是舊語法？

　舊寫法新寫法（建議）
上 S 鎖SELECT ... LOCK IN SHARE MODESELECT ... FOR SHARE
引入版本一直都有MySQL 8.0.1 起
支援 NOWAIT / SKIP LOCKED不支援支援（撞鎖不等、或跳過被鎖的列）

兩者效果一樣，新版用 FOR SHARE。對應的寫鎖一直是 FOR UPDATE。PostgreSQL 一直都是 FOR SHARE / FOR UPDATE。NOWAIT=撞鎖立刻報錯不等；SKIP LOCKED=跳過已被鎖的列（常用於搶工作佇列）。

4¾ 鎖耗不耗效能？「樂觀」在哪？問號是什麼？

① 鎖會耗效能嗎？是哪個零件的效能？

先講清楚：鎖的成本主要不是磁碟、也不是 RAM，而是「時間／吞吐量」（交易被序列化、使用者等更久）。只有「鎖表記帳」用到一點點 RAM/CPU。

成本來源說明吃哪個零件
取鎖/放鎖的記帳鎖管理器在記憶體維護鎖表「誰鎖了什麼」RAM（少量）+ CPU（少量）
阻塞等待（最大宗）交易被卡住空等、work 被序列化 → 吞吐↓、延遲↑時間/吞吐量（等待中不燒 CPU、不用磁碟，純粹「卡著等」）
死鎖偵測 + 回滾 + 重試CPU + 白做的工
跟磁碟的關係（重要，別混淆）

鎖不直接造成磁碟 I/O。但若交易持鎖期間還去做慢磁碟讀寫，鎖就被佔更久 → 放大競爭。對比記住：索引省的是磁碟 I/O（少讀 page）；鎖省的是等待時間/吞吐量——兩者是不同零件、不同維度。

降成本：交易短、WHERE 走索引把鎖縮小、持鎖期間別做慢 I/O、衝突少時改用樂觀。InnoDB 普通 SELECT 是 MVCC 快照、不上鎖，所以「讀」不會卡在鎖等待，別亂用 FOR UPDATE 把讀也鎖起來。

② version 只是探針，為何叫「樂觀鎖」？樂觀在哪？

「樂觀／悲觀」講的是對「會不會撞」的態度，不是機制本身：悲觀＝賭一定撞 → 先鎖保平安；樂觀＝賭應該不撞 → 先不鎖、直接做，只在寫的那刻用 version 驗證賭對沒。

樂觀的部分

＝你樂觀地選擇不上鎖、直接賭沒人動；version 只是事後驗證這個賭注的工具。若你悲觀，你會選「先鎖」而不是「先賭」。其實樂觀鎖根本沒拿任何 DB 鎖，「鎖」是借用的名字，本質是一種並發控制策略。

③ WHERE version=? 的問號是什麼？實際是問號還是數字？

? 是參數佔位符（placeholder），不是真的打一個問號。它代表「這裡之後綁一個值進來」，用在參數化查詢以避免 SQL injection。

· SQL 工具手打（psql/HeidiSQL）：直接寫數字 → WHERE version = 7
· 程式碼裡：用佔位符、值另外傳。

// GORM（你的 stack）
result := db.Model(&Product{}).
    Where("id = ? AND version = ?", id, ver).
    Updates(map[string]any{"stock": gorm.Expr("stock - 1"), "version": ver + 1})
if result.RowsAffected == 0 {
    // 被別人搶先 → 重讀重試
}

// database/sql：MySQL 用 ? ；PostgreSQL 用 $1,$2
db.Exec("UPDATE products SET stock=stock-1, version=? WHERE id=? AND version=?", ver+1, id, ver)

重點：用佔位符讓驅動綁值，別字串拼接（拼接有 SQL injection 風險）。MySQL 佔位符是 ?，PostgreSQL 是 $1、$2。

05 死鎖（Deadlock）：兩邊互等對方放鎖

A 鎖了列1、想要列2；B 鎖了列2、想要列1 → 互相等，永遠不會結束。InnoDB 會自動偵測並回滾其中一個交易（犧牲者收到 deadlock error），另一個得以完成。

-- A: UPDATE ... WHERE id=1;  然後想 UPDATE ... WHERE id=2;
-- B: UPDATE ... WHERE id=2;  然後想 UPDATE ... WHERE id=1;
-- → 交叉等待，InnoDB 砍掉其中一個

怎麼避免：① 多個交易用固定順序存取資源（都先小 id 再大 id）；② 交易盡量短；③ 接到 deadlock error 就重試整個交易。查死鎖：SHOW ENGINE INNODB STATUS 的 LATEST DETECTED DEADLOCK 區塊。

06 隔離等級（順帶建立座標）

鎖的鬆緊由「交易隔離等級」決定。由鬆到嚴：

等級會發生的異常備註
Read Uncommitted髒讀、不可重複讀、幻讀幾乎不用
Read Committed不可重複讀、幻讀PostgreSQL 預設
Repeatable Read（InnoDB 用 gap lock 連幻讀也大致擋掉）MySQL InnoDB 預設
Serializable無（最嚴）等於把交易序列化，最慢

名詞：髒讀=讀到別人還沒 commit 的資料；不可重複讀=同一交易兩次讀同一列值不同；幻讀=兩次範圍查詢列數不同。InnoDB 在 RR 下用 gap lock / next-key lock 鎖住「區間」來防幻讀——這也是為什麼有時你會鎖到「還不存在的列」。
