# 鎖與交易與並發控制

> [!info] 互動筆記（需 HTML Reader 外掛）
> 含 S/X **2×2 四格相容矩陣**、鎖競爭、主動加鎖正確/錯誤寫法、死鎖、隔離等級、點黃框測驗。
![[鎖與交易與並發控制.html]]

## S/X 相容性（四格矩陣）
|  | 對方 S | 對方 X |
|---|---|---|
| **我要 S** | ✔ 相容（讀+讀） | ✘ 等 |
| **我要 X** | ✘ 等 | ✘ 等 |

→ 唯一綠格是 **S+S**。記法：**讀讀不擋，只要牽涉寫就互斥。** InnoDB 檢查外鍵只要 S 鎖。

## 主動加鎖（正確 vs 不好）
- ❌ 不好：`SELECT` 再 `UPDATE` 分兩句 → 中間空窗 → **lost update**。
- ✅ 悲觀鎖：`SELECT ... FOR UPDATE`（上 X 鎖，衝突常見時用）。
- ✅ 原子更新：`UPDATE ... SET stock=stock-1 WHERE id=? AND stock>0`（單純加減最優先，看影響列數）。
- ✅ 樂觀鎖：`WHERE version=?`，影響列數 0 就重試（衝突少、要高並發時）。
- 主動上 S 鎖：`SELECT ... FOR SHARE`（舊：`LOCK IN SHARE MODE`）。

## 三個澄清（version / 主動 S 鎖 / 新舊語法）
- **樂觀鎖 × version**：version 是「我讀完後有沒有人動過」的探針。`UPDATE ... WHERE id=? AND version=讀到的值`，**影響列數=0 就是被搶先** → 重讀重試。全程不上鎖。
- **FOR SHARE 為何要主動**：普通 `SELECT` 在 InnoDB 預設不上鎖（MVCC 快照），要「別人不准改我讀的這列」就得主動 `FOR SHARE`。用 S 不用 X 是因為要讓別人也能讀、只擋改（如 parent 列防被刪）。
- **LOCK IN SHARE MODE = 舊語法**：MySQL 8.0.1 起改用 `FOR SHARE`（還支援 `NOWAIT` / `SKIP LOCKED`）。寫鎖一直是 `FOR UPDATE`。PostgreSQL 一直用 `FOR SHARE`/`FOR UPDATE`。

## 鎖效能 / 樂觀在哪 / 問號
- **鎖耗效能嗎？是哪個零件？**：主要不是磁碟、也不是 RAM，而是**時間/吞吐量**（交易被序列化、等更久）。只有「鎖表記帳」用到一點 RAM/CPU。阻塞等待中的交易**不燒 CPU、不用磁碟**，只是卡著等。
- **鎖 vs 索引省的是不同零件**：索引省**磁碟 I/O**（少讀 page）；鎖省**等待時間/吞吐量**。鎖不直接造成磁碟 I/O，但持鎖期間做慢 I/O 會拉長持鎖時間。普通 SELECT 是 MVCC 快照不上鎖。
- **樂觀在哪**：「樂觀/悲觀」是對「會不會撞」的態度。樂觀=賭不撞、**選擇不上鎖直接做**，version 只是事後驗證工具。樂觀鎖其實沒拿任何 DB 鎖。
- **`?` 是佔位符**（placeholder）不是真打問號：工具手打寫數字 `version=7`；程式碼用佔位符值另傳（GORM `Where("version = ?", ver)`）防 SQL injection。MySQL 用 `?`，PostgreSQL 用 `$1`。

## 鎖競爭 Lock Contention
持鎖太久 / hot row / 鎖範圍太大（沒走索引）→ 排隊變慢。
**關鍵：`UPDATE` 的 `WHERE` 沒走索引，可能鎖住很多列甚至整表** → 索引影響鎖範圍。

## 死鎖 Deadlock
A 鎖列1要列2、B 鎖列2要列1 → 互等。InnoDB 自動偵測回滾其一。
避免：固定順序存取資源、交易短、接到 error 就重試。查：`SHOW ENGINE INNODB STATUS`。

## 隔離等級
RU < RC（PG 預設）< RR（**InnoDB 預設**，gap lock 防幻讀）< Serializable。
異常：髒讀 / 不可重複讀 / 幻讀。
