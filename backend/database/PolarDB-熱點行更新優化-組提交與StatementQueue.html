重點整理

先搞懂問題：什麼是「熱點行」排隊等鎖

當很多事務在短時間內更新同一行資料（熱點行 hot row）時，只有一個事務能更新，其他必須等待。

關鍵釐清：這些等待的事務並不是「各自有一個鎖」，而是排隊等待取得「同一個」行鎖。持有鎖的事務 commit 或 rollback 釋放鎖後，下一個排隊者才拿得到。

[!note] 名詞釐清
PolarDB 是阿里雲自研的「雲原生資料庫（Cloud-Native Database）」，名稱跟 Hulu 影集、MySQL 都無關（對話中 Gemini 一度語音辨識錯亂，特此標註）。

優化技術一：熱點行 UPDATE 優化（Hot row optimization）

針對「多事務搶同一行鎖」的核心優化，靠流水線分組（Group Update）＋組提交（Group Commit）。

Leader / Follower 機制（消除申請行鎖的等待）：

  • 組內第一個更新操作 = Leader：負責讀取目標資料行並加鎖。
  • 後續更新操作 = Follower：要對同一行加鎖時，發現 Leader 已持有行鎖，就無需等待，直接獲得行鎖。
  • 效果：減少行鎖的加鎖次數與時間開銷。

組提交流程： 多筆更新先收集進記憶體（Collect and Update Memory Record / Row Cache 暫存），由 Leader 統一收集更新記憶體紀錄，再以群組方式一起提交（Group Committing）→ 大幅減少磁碟 I/O 次數與鎖衝突。

優化技術二：Statement Queue（語句隊列）

如果沒有開啟上面的熱點行優化，PolarDB 還提供另一種排隊機制。

  • 痛點：原生 MySQL 中，多條語句對同一行並發操作時，執行緒會在 InnoDB 行鎖上「硬碰硬」競爭，造成嚴重鎖等待與衝突，系統吞吐量暴跌。
  • 做法：透過 Hint 語法，把這些衝突語句在進入資料庫引擎前，先用 Hash 演算法路由到一個共享的「排隊桶（Bucket）」，讓語句在隊列中依序排隊，而不是直接去頻繁申請/釋放鎖 → 降低內核的鎖衝突開銷。

它如何把高並發熱點行更新做到「並行處理」

把針對同一熱點行的更新操作分組，再用類似 Hash 演算法把不同的組並行處理，最大化並發吞吐。

數字累加 vs 字串更新的差別

  • 數字（1 + 1 + 1）本質是累加，組內可以一次合併算完。
  • 字串一直被改寫時，會按先後順序分批處理與提交，最終值取決於「最後一個成功執行的更新」。

Gemini： 在 PolarDB 架構中主要有兩種技術處理高並發排隊與行鎖等待：
1.《熱點數據行 UPDATE 操作的性能優化》（Hot row optimization）— 核心是組提交與流水線：大量並發更新同一行時，內核自動按到達順序分組（Group Update）。「組內第一個更新操作為 Leader，其讀取目標數據行並且加鎖。後續更新操作為 Follower，其對目標數據行加鎖時，如果發現 Leader 已經持有行鎖，無需等待，直接獲得行鎖。」藉此減少行鎖的加鎖次數和時間開銷。
2.《使用 Statement Queue 優化高並發性能》— 原生 MySQL 多語句針對相同行並發時，執行線程會在 InnoDB 行鎖上硬碰硬競爭，吞吐量暴跌。PolarDB 允許透過 Hint 把衝突語句在進入引擎前以 Hash 路由到共享排隊桶（Bucket）中依序排隊，降低鎖衝突開銷。

使用者： 這些是怎麼對高並發熱點行更新做到並行處理？

Gemini： PolarDB 透過將針對同一熱點行的更新操作分組，並使用類似哈希演算法的技術將不同的組並行處理，從而最大程度實現並發更新、提高整體效能。

使用者： 圖解 sequence：update the same hot row、row 快取，快取的內容是什麼？

使用者： 透過 group commit 的機制是什麼？

Gemini： 先把同一熱點行的更新請求收集在記憶體（圖中的 Collect and Update Memory Record），再在適當時機一起提交（Group Committing），減少磁碟 I/O、提高效能。
