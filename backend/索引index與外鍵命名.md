# 索引 (Index) 是什麼？外鍵與索引的命名

> 起因：在 event 表看到 `fk_event_updated_by`，以為它是欄位，搞不清楚跟 `updated_by` 的關係。

## 一句話釐清

`fk_event_updated_by` **不是欄位**，它是一個「名字」，這個名字同時被兩樣東西共用：

| 名稱 | 是欄位嗎 | 是什麼 |
|------|---------|--------|
| `updated_by` | ✅ 是欄位 | event 表裡真正存資料的欄位，存「是哪個 user 改的」的 id |
| `fk_event_updated_by` | ❌ 不是欄位 | **外鍵約束**的名字，同時也是 MySQL 幫這個外鍵**自動建立的索引**的名字 |

它們不是同一個欄位 —— `fk_event_updated_by` 是「**掛在** `updated_by` 這個欄位上」的東西。

命名慣例：`fk_<表名>_<欄位名>`，一看就知道「這是 event 表、updated_by 欄位的外鍵」。
（索引另一種常見前綴是 `idx_`，外鍵約束用 `fk_`。）

---

## 索引 (Index) = 資料庫的目錄

就像一本書後面的「索引頁」。

假設 event 表有 10 萬筆，要找「updated_by = 某個 user」的全部活動：

- **沒有索引**：從第 1 筆翻到第 10 萬筆，一筆一筆比對 → 慢（full table scan 全表掃描）
- **有索引**：先查「目錄」，直接跳到對的位置 → 快

```
書本類比：想查「光合作用」在哪一頁
  ❌ 沒索引 → 從第 1 頁翻到最後一頁找
  ✅ 有索引 → 翻到書末索引頁「光合作用 ... p.245」，直接翻過去
```

### 核心是「排序」不是「分組」（常見誤解）

直覺常以為索引是「把姓陳的分一堆、姓林的分一堆」。
更精準的說法是：**索引把欄位的值先「排好順序」**，並記下每個值在原表的第幾筆。

像通訊錄姓名照筆劃排好：

```
找「陳大文」：
  ❌ 沒排序的名單 → 從第一個開始念到他
  ✅ 排好序的通訊錄 → 翻到「陳」附近 → 找「大文」，幾步就到
```

因為「排好序」，連在同一堆裡都能用二分法快速逼近：

```
排好序時找一個值：
  翻到正中間 → 目標在前？往前半找；在後？往後半找
  每翻一次排除一半 → 10 萬筆約 17 步就找到
```

> 比喻：原本的資料表像「照報名順序堆的報名表」（姓名是亂的）；
> 索引像「另外抄一份照姓名排序的清單，每個名字旁註明在原表第幾張」。
> 查的時候先看排好序的索引 → 找到第幾張 → 直接抽原始資料。
> 所以索引是**額外多出來的一份排序目錄**（也因此佔空間、寫入要多維護）。

### 代價
索引不是免費的：
- 會佔額外儲存空間
- 每次 INSERT / UPDATE / DELETE 都要順便維護索引 → 寫入稍微變慢
- 所以「常用來查詢 / JOIN 的欄位」才值得加索引，不是每個欄位都加

## 為什麼 `updated_by` 上面會有索引？

因為對它建了**外鍵 (FOREIGN KEY)**：

```sql
ALTER TABLE event
  ADD CONSTRAINT fk_event_updated_by
    FOREIGN KEY (updated_by) REFERENCES `user`(id);
```

**MySQL 規則**：建外鍵時，如果該欄位還沒有索引，MySQL 會**自動幫你建一個同名的索引**。
所以 `fk_event_updated_by` 就同時身兼「外鍵約束名」+「索引名」。

外鍵本身的作用：保證 `updated_by` 存進去的 id 一定對應到 `user` 表裡真的存在的一筆，不會是亂填的孤兒值。

---

## 怎麼查一個表有哪些索引？

```sql
-- 看 event 表所有索引（含外鍵自動建的）
SHOW INDEX FROM event;

-- 看建表語句，連外鍵 / 索引定義一起看
SHOW CREATE TABLE event;
```

`SHOW INDEX FROM event;` 結果裡 `Key_name` 那欄就會看到 `fk_event_updated_by`，
`Column_name` 那欄會顯示 `updated_by` —— 證明「索引名 fk_event_updated_by」是建在「欄位 updated_by」上。

## 名詞對照小抄

| 你看到的 | 中文 | 白話 |
|----------|------|------|
| Column（欄位） | `updated_by` | 真正存資料的格子 |
| Index（索引） | `fk_event_updated_by` | 查資料用的目錄，讓搜尋變快 |
| Foreign Key（外鍵約束） | `fk_event_updated_by` | 規定這格的值一定要對應到別張表存在的 id |
