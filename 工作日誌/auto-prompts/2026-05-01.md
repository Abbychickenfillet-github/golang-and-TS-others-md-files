## 12:45 — 2. **過早下結論** - 看到 Redis 錯誤就認定是根本原因 像我後來有學redis我要怎麼用redis的語法去檢查

**Claude:**

針對「看到 Redis 錯誤就跳結論」這件事，重點是**先用指令驗證假設**，而不是看訊息字面就判定。下面是排查順序，從外層到內層：

## 1. 先確認「Redis 本身」還活著

```bash
redis-cli -h host -p 6379 PING        # PONG = 連得上
redis-cli INFO server | head -20      # 版本、uptime、是否剛重啟
redis-cli INFO replication            # role: master/slave？replica 寫入會錯
redis-cli CLUSTER INFO                # cluster 模式才用
```

**排除什麼**：`Connection refused` 不一定是 Redis 掛，可能是網路 / port / 容器；先 PING 確認。

## 2. Key 真的存在嗎？型別對嗎？

```bash
EXISTS user:1                         # 0/1
TYPE user:1                           # string/hash/list/set/zset/stream
TTL user:1                            # -2=不存在, -1=無 TTL, >0=剩餘秒數
OBJECT ENCODING user:1                # 內部編碼（ziplist? hashtable?）
```

**排除什麼**：`WRONGTYPE` 錯誤幾乎都是程式拿型別 A 的 key 用型別 B 的指令；先 `TYPE` 看一下實際是什麼。

## 3. Value 真的是預期內容嗎？

依型別用對應指令（**不要全部用 GET**，會吃 `WRONGTYPE`）：

```bash
GET k                                 # string
HGETALL k                             # hash
LRANGE k 0 -1                         # list
SMEMBERS k                            # set
ZRANGE k 0 -1 WITHSCORES              # zset
XRANGE k - +                          # stream
MEMORY USAGE k                        # 這個 key 佔多少 bytes
```

## 4. 是不是 Server 資源問題？

```bash
INFO memory                           # used_memory_human / maxmemory / evicted_keys
INFO clients                          # connected_clients / blocked_clients
INFO stats                            # total_connections_received / rejected_connections
INFO persistence                      # 是否在 AOF rewrite / RDB save（會 fork 卡住）
DBSIZE                                # 這個 DB 多少 key
CONFIG GET maxmemory-policy           # noeviction? allkeys-lru?
```

**排除什麼**：`OOM command not allowed` 不一定是「記憶體不夠」，常常是 `maxmemory-policy=noeviction` + key 沒 TTL，要先看 policy 與 evicted_keys。

## 5. 找慢查詢 / 阻塞元兇

```bash
SLOWLOG GET 10                        # 最近 10 筆慢查詢（含原始指令）
SLOWLOG RESET
CONFIG GET slowlog-log-slower-than    # 門檻（微秒）

CLIENT LIST                           # 看誰連著、idle 多久、執行什麼指令
CLIENT LIST | grep cmd=               # 過濾出正在跑的指令
LATENCY LATEST                        # 各事件最近一次延遲
LATENCY DOCTOR                        # Redis 自己給的診斷建議（很神）
```

## 6. 找肥大的 key（常見效能殺手）

```bash
redis-cli --bigkeys                   # 各型別最大的 key
redis-cli --hotkeys                   # 最熱的 key（要開 LFU 才有用）
redis-cli --memkeys                   # 記憶體用最多的 key
DEBUG OBJECT k                        # 單 key 詳細資訊
```

## 7. 看「真實發生什麼」（核彈級，最後手段）

```bash
redis-cli MONITOR                     # 即時印出每個指令（耗效能！只看幾秒就 Ctrl+C）
redis-cli MONITOR | grep "user:1"     # 只看相關 key
```

⚠️ Prod 上 `MONITOR` 會降效能 50%+，只在開發或 staging 用。

## 常見「跳結論」的對照表

| 錯誤訊息 | 第一直覺（常常錯） | 真正要做的驗證 |
|---|---|---|
| `Connection refused` | Redis 掛了 | `PING` host、看容器網路、`CONFIG GET bind` |
| `OOM` | 記憶體不夠要擴容 | `INFO memory` 看 evicted_keys 與 policy，`--bigkeys` 找元兇 |
| `WRONGTYPE` | Redis bug | `TYPE k` 看實際型別，是程式邏輯問題 |
| `MOVED 5798 host` | cluster 壞了 | 用 `redis-cli -c` 連，或客戶端沒開 cluster mode |
| `READONLY` | 主從掛了 | `INFO replication` 確認連到的是 master 還 replica |
| `LOADING` | Redis 掛了 | 正在從 RDB 載入，等就好 |
| Timeout | Redis 慢 | `SLOWLOG GET`、`LATENCY DOCTOR`、看是不是在 AOF rewrite |

---

**排查心法**：每一層都用指令**取證**再往下走，不要跳。看到 Redis 錯誤先問三件事：
1. 連得到嗎？(`PING`)
2. Key 存在且型別對嗎？(`TYPE` / `EXISTS`)
3. Server 資源有問題嗎？(`INFO memory` / `SLOWLOG`)
