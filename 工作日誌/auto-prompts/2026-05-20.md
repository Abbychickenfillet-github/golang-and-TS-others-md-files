## 這個工具是什麼

**claude-log-cli** 是一個用來瀏覽和搜尋你本機 **Claude Code 對話記錄**的 CLI 工具。

Claude Code（你現在正在用的這個 CLI）會把每一次 session 的對話 log 存在本機（通常在 `~/.claude/projects/` 底下，就像你 MEMORY.md 所在的那個目錄）。這些 log 是 JSONL 格式，平常很難翻閱。這個工具就是專門用來翻這些 log 的。

## 用途

作者寫這個工具的核心動機（連結到他的部落格「self-improving CLAUDE.md files」）是：

> **回頭翻自己過去跟 Claude 的對話，從中找出該寫進 CLAUDE.md 的規則、回饋、踩過的坑**，讓你的 CLAUDE.md 越用越聰明。

對你來說特別實用，因為你的 MEMORY.md 已經有非常多 `feedback-*.md` 檔案（像 `feedback-frontend-file-size.md`、`feedback-planning-workflow.md`），這個工具能幫你：

1. **`sessions list`** — 列出這個專案的所有 session
2. **`sessions prompts <id>`** — 只看你問過的問題（適合回顧你常重複交代的事 → 寫成 memory）
3. **`sessions search <query>`** — 跨所有 session 全文搜尋（例如搜「不要」「stop」「don't」可以挖出你給過的所有 correction）
4. **`sessions tools`** — 看 tool 使用統計
5. **`projects list`** — 列出所有 Claude Code 專案

## Symbiont 是什麼

**Symbiont** 是一個在背景跑的 Python daemon，目的是「**讓 Claude Code 變成有持續記憶的長期夥伴**」，而不只是一個一次性的對話 session。

作者把它命名為「共生體（Symbiont）」是有寓意的：不是主僕關係，而是 **Claude 給 agent 指導 → agent 的回饋反過來精煉 Claude 自己的規則**，雙向互相進化。

---

## 它解決的三個痛點

Claude Code 的 session 是「短暫的」——一次結束，context 就消失。這帶來三個問題：

| 痛點 | 你的真實處境 |
|------|------|
| **每次都要重新教 Claude 你的偏好** | 你今天教了「Commit 不加 Co-Authored-By」，下次 session 又要重講 |
| **Memory 檔案會悄悄過時** | 你的 `feedback-*.md` 寫了幾個月可能已經不準確，但沒人去 review |
| **遠端 agent 在你離線時就停擺** | 你睡覺 / 沒開電腦時，VM 上的 agent 就沒人能對話 |

---

## 五大模組（你可以挑著用）

| 模組 | 做什麼 | 何時跑 |
|------|------|------|
| **`evolve.py`** | 讀完一個 session log → 萃取「行為規則」→ 寫進 `~/.claude/CLAUDE.md` | 每次 Claude Code session 結束（Stop hook 觸發） |

## 為什麼是 `NameError: name 'Vehicle' is not defined`

`v = Vehicle()` 和 `v.show()` 被縮排到 **class body 裡面**了。Python 在執行 class body 時，`Vehicle` 這個名字**還沒建立**（要等整段 class body 跑完，class object 才會被綁定到 `Vehicle` 這個名字）。所以裡面寫 `Vehicle()` 自然找不到。

## 修正成你應該要的樣子

把那兩行往左移到 class **外面**：

```python
class Vehicle:
    def show(jessie):
        print("Show Vehicle")

v = Vehicle()
v.show()
```

這樣會印出 `Show Vehicle`，**不會有任何錯誤**。

## 為什麼 `show(jessie)` 不會出錯？

你可能以為「第一個參數一定要叫 `self`，叫 `jessie` 會壞掉」── 不會。Python 完全不在乎名字，它只看**位置**：呼叫 `v.show()` 時，Python 會自動把 `v` 當作第一個 positional argument 傳進去，於是 `jessie = v`。`self` 只是慣例。

## 如果你真的想看 `TypeError`，試這個

```python
class Vehicle:

Vehicle.show()   # ← 沒有 instance 自動帶入，jessie 沒人填
```

這會丟：

## 13:37 — 好崩潰喔 沒問題了 但NameError這也算編譯錯誤嗎？

**Claude:**

不算，`NameError` 是 **執行期錯誤（runtime error）**，不是編譯錯誤。

## Python 的兩個階段

Python 雖然常被叫「直譯語言」，但它其實也會先**編譯成 bytecode**，再由 VM 執行：

```
原始碼 .py
   │
   ▼
[編譯階段] ──► 只檢查「語法」對不對
   │
   ▼
bytecode (.pyc)
   │
   ▼
[執行階段] ──► 真的跑起來，去查名字、查型別、查屬性…
```

## 哪些算編譯錯誤？

只有「語法層面」的問題會在**還沒開始跑**就被擋下來：

| 錯誤 | 階段 | 範例 |
|---|---|---|
| `SyntaxError` | 編譯期 | `def show(:` |
| `IndentationError` | 編譯期 | 縮排亂掉 |
| `TabError` | 編譯期 | tab 和 space 混用 |

判斷方法：如果一個檔案裡**只要有一行語法錯**，整個檔案連第一行都不會執行 → 那就是編譯錯誤。

## 為什麼 `NameError` 是 runtime？

因為 Python 編譯時**根本不知道 `Vehicle` 這個名字存不存在**。它只看「`Vehicle()` 這個寫法合不合文法」，合 → 通過編譯。等到真的執行那行，去當前 scope 找 `Vehicle` 找不到，才丟 `NameError`。

證據：

```python
def never_called():
    print(完全不存在的變數)   # 這行絕對會壞

print("Hello")
```

跑起來會印 `Hello`，不會報錯 ── 因為 `never_called` 沒被呼叫，那行 `NameError` 永遠沒機會發生。如果是編譯錯誤，整個檔案連 `Hello` 都印不出來。

## 同類的 runtime 錯誤

- `NameError`（名字不存在）
- `TypeError`（型別不對 / 參數數量不對）
- `AttributeError`（物件沒這個屬性）
- `ZeroDivisionError`、`KeyError`、`IndexError`…

這些全部都要「跑到那一行」才會發生。
