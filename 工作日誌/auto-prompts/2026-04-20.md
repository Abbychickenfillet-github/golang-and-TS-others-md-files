### ① O(n) 是 10 萬嗎？跟 log 有關？

**n 不是特定數字，是變數**，代表「資料有幾筆」。10 萬只是我隨便舉的例子。

大 O 在講的是**「資料量變大時，時間怎麼長」**：
- **O(n)** = 資料翻倍、時間翻倍（10 萬筆做 10 萬步）
- **O(log n)** = 對數成長，10 億筆也只要 ~30 步 ← 近乎免疫

`log₂(n)` 就是「n 能被 2 除幾次變 1」。1,000,000 ≈ 20 次、1,000,000,000 ≈ 30 次。

所以 O(n) 跟 O(log n) 是**對比關係**——一個是暴力掃、一個是聰明跳著找。

### ② 絕對 vs 相對只能一個

就是「**XOR 互斥**」：
- **absolute**（絕對）= 「**改成** 0.9」(像冷氣設為 26 度)
- **delta**（相對）= 「**加** 0.1」(像按 +1 度)

兩個都填 → 系統不知道聽誰的（矛盾）
兩個都不填 → 沒說要改啥（無效）
所以規則強制**恰好一個有值**。

### ③ 12 個操作每次都用嗎？

**不會，每次只用 1 個。** 12 個是工具箱的總量，不是出動清單。

就像廚房 12 把刀，做菜時拿 1 把；複雜任務可能串 3–5 次呼叫（例如 Retrieve → Summarize → Encode → Demote），**但每次 JSON 只帶一個 op**。原子操作的價值就在「一次只做一件事」。

## Big O 是什麼的縮寫？

**O = Order (階 / 數量級)**

這個記號來自 19 世紀末的數學家：
- **Paul Bachmann** 1894 年在《Analytische Zahlentheorie》首次使用
- **Edmund Landau**（德國數學家）1909 年把它推廣，所以也叫 **Landau notation**

「O」原本是德文 **"Ordnung"**（英文 Order，中文「階」或「數量級」）的縮寫——意思是「**成長的等級**」。它不是 "Of" 或 "Omega" 之類的縮寫。

所以 `O(n)` 唸作 "**order of n**"，意思是「**這個東西的成長階數是 n**」。

## 開平方 vs log₂ 的差別

### 開平方 `√n` = 找一個數，自己乘自己等於 n
- `√100 = 10` 因為 `10 × 10 = 100`
- `√1,000,000 = 1,000` 因為 `1,000 × 1,000 = 1,000,000`

**關鍵：乘 2 次**

### `log₂(n)` = 找一個數 x，讓 2 自己乘 x 次等於 n
- `log₂(1,024) = 10` 因為 `2 × 2 × 2 × 2 × 2 × 2 × 2 × 2 × 2 × 2 = 1,024`（2 乘 10 次）
- `log₂(1,000,000) ≈ 20` 因為 `2 乘 20 次 ≈ 1,000,000`

**關鍵：乘很多次 2**

所以：
- `√1,000,000 = 1,000`（答案是一千）
- `log₂(1,000,000) ≈ 20`（答案是二十）

**兩個答案差 50 倍！根本不一樣。**

## 為什麼不是開平方？

如果是開平方，100 萬的開平方是 **1,000**，不是 **20**。差距是：

## ① 「向量引擎」≠「讀圖片」(這是個誤解)

**向量引擎跟「能不能讀圖片」沒關係**。兩個不同的事：

### 向量 (vector) 是什麼？
一筆記憶（不管是文字、圖片、聲音）會先被 LLM / embedding 模型**轉成一串數字**，例如：
```
"我喜歡暗色主題"  →  [0.12, -0.45, 0.88, ..., 0.03]  ← 1536 個浮點數
```
這串數字叫 **embedding / 向量**。**文字記憶也是向量，不是只有圖片才有。**

### 「向量引擎」是幹嘛的？
是**專門做向量相似度搜尋**的資料庫引擎（例如 Pinecone、Qdrant、pgvector、Milvus）。它做一件事：**「給我一個向量，從幾百萬個向量裡找最像的前 10 個」**，而且要**快**。

### 為什麼 text2mem 沒有「真正的向量引擎」就不能上生產？
跟圖片無關，是**效能**問題：
- text2mem 現況：向量塞進 SQLite 的 TEXT 欄位，**每次搜尋都要全表掃 + JSON.parse**（O(n) 暴力掃）
- 生產級：用 HNSW / IVF 索引，資料 100 萬筆也能 20 步內找到（O(log n)）

**「不適合上生產」的真正原因（回到筆記的三個短版）**：
1. 只有 SQLite，**併發弱**（很多人同時用會搶鎖）
2. 向量**暴力掃**，資料變多會越來越慢
3. **沒 HTTP API**，只能給 Python 程式內部用

圖片讀取是另一個獨立議題（叫 multimodal embedding），跟這三個短版沒關係。

## ② 「讓外部服務呼叫」是什麼意思？(筆記 13 展開)

用**餐廳**比喻最好懂：

### 現況：Library (函式庫) — 像「自家廚房」
text2mem 現在是一個 **Python 套件**。使用方式：
```python
from text2mem import Memory
mem = Memory()
mem.encode("我喜歡暗色主題")
```
**限制**：這段 code 只能在**你的 Python 程式裡面**跑。你寫的 Node.js 網站、iOS App、同事的 Go 微服務……**全部都用不到**，因為他們不是 Python。等於你家廚房只有你家人能進去煮。

### 缺的：HTTP API (對外介面) — 像「外送餐廳」
如果 text2mem 有 HTTP API，它會變成一個**一直在背景跑的 server**，別人可以用網路呼叫：
```bash
POST https://my-memory-server.com/memory
{"op": "Encode", "args": {"text": "我喜歡暗色主題"}}
```
這時候：
- 你的 Node.js 網站可以呼叫 ✅
- 你的 iOS App 可以呼叫 ✅
- 同事的 Go 服務可以呼叫 ✅
- 甚至另一台電腦、另一個國家的服務都能呼叫 ✅

**「讓外部服務呼叫」= 讓其他程式（不管什麼語言、哪台機器）可以透過網路用到它**。沒有 HTTP API 就沒辦法變成「memory-as-a-service」，只能當一個內嵌的小工具。

和「能不能連網」無關 — **text2mem 自己內部還是會上網呼叫 LLM 做 Summarize**。缺的是「**別人怎麼呼叫它**」，不是「**它怎麼上網**」。

## 11:02 — embedding 不是嵌入的意思嗎 怎麼會是向量?

**Claude:**

超棒的語感問題！你其實**沒理解錯**，是**英文這個詞一詞多義**，中文翻譯造成的混淆。

## Embedding 既是「動作」也是「結果」

### 英文原意：embedding = 嵌入（動作）
`embed` 這個動詞就是「把某個東西**塞進 / 嵌入**到另一個東西裡」。例如：
- 把鑽石**嵌入**戒指 → embed a diamond in a ring
- 把影片**嵌入**網頁 → embed a video in a webpage

**你的直覺沒錯，embedding 的確是「嵌入」。**

### 但在 ML / NLP 的術語中，它同時指「嵌入後的那個東西」

這是關鍵：**英文名詞 `embedding` 不只是「嵌入」這個動作，還可以指「被嵌入的那個結果」**（英文名詞可以是動作也可以是結果，像 `recording` 既是「錄音這件事」也是「錄音檔」）。

---

## 用「地圖」比喻最好懂

想像有一張超高維的地圖（叫**向量空間 / embedding space**），每個詞彙、句子、圖片都要被放到這張地圖的某個位置上：

```
                    🗺️ 嵌入空間 (embedding space)
                   ┌─────────────────────────────┐
"我喜歡暗色主題" ──►│  放在座標 [0.12, -0.45, ...]│
"我愛深色模式"   ──►│  放在附近 [0.11, -0.44, ...]│  ← 意思相近 → 位置相近
"今天天氣好"     ──►│  放在遠處 [-0.8, 0.2, ...]  │  ← 意思不同 → 位置遠
                   └─────────────────────────────┘
```

在這個比喻裡：
| 中文 | 英文 | 在比喻裡 |
|------|------|---------|
| **嵌入（動作）** | embedding (verb) | 「把這個詞**放到**地圖上」的這個動作 |
| **嵌入空間** | embedding space | 這張**地圖本身** |
| **那個向量** | embedding (noun) / vector | **座標值** `[0.12, -0.45, ...]` |

**關鍵洞察**：座標 `[0.12, -0.45, ...]` 就是「**被嵌入的結果**」—— 英文就懶得另外命名，直接拿 `embedding` 這個名詞指這個結果。

所以你會看到兩種說法都對：
- 「這個詞的 **embedding** 是 `[0.12, -0.45, ...]`」（embedding 指向量）
- 「**embedding** 這個技術把詞變成向量」（embedding 指這個技術 / 動作）

---

## 為什麼中文社群會直接叫它「向量」？

因為在**實際使用時**，我們只在乎那串數字（座標），不太在意「嵌入」這個動作。所以中文工程師為了方便，直接叫它**向量**，因為那個結果**就是一串數字，數學上就叫向量 (vector)**。

中文翻譯有三種，都指同一個東西：
1. **嵌入** — 直譯，強調「放進空間」這個動作
2. **詞嵌入 / 詞向量** — 加個「詞」字限定，指文字版本
3. **向量 (vector)** — 強調結果本身是一串數字

**所以 embedding = 嵌入 = 向量，這三個詞在 ML 語境下指同一件事，只是不同角度**：
- 從**動作**角度看 → 嵌入
- 從**空間**角度看 → 嵌入空間裡的一個點
- 從**數學**角度看 → 向量

---

## 11:07 — "我喜歡暗色主題"  →  [0.12, -0.45, 0.88, ..., 0.03]  ← 1536 個浮點數 這是2進制嗎 1536是固定的嗎

## ① 這是二進制嗎？→ **不是，是十進制浮點數**

你看到的 `0.12`, `-0.45`, `0.88` 是**正常的十進制小數**（浮點數 / float）。

### 二進制長什麼樣？
```
0101101110100101...  ← 只有 0 和 1 才是二進制
```

### 浮點數長什麼樣？
```
0.12, -0.45, 0.88, 3.14159, -0.00031...  ← 這才是浮點數
```

**差別**：
| 種類 | 範圍 | 例子 |
|------|------|------|
| 二進制 | 只有 0 和 1 | `1010110` |
| 浮點數 | 任何小數（可正可負） | `0.12`, `-0.45`, `3.14` |

### 那「電腦內部是不是用二進制存？」
**是，但那是底層儲存方式**，使用者看到和操作的永遠是十進制。就像你打 `3.14` 進 Python，電腦底層用 **IEEE 754** 格式把它存成 64 個 bit 的 0/1，但你讀寫時看到的永遠是 `3.14`。

**結論**：embedding 向量就是一串普通的小數，不是 0/1。

---

## ② 1536 是固定的嗎？→ **不是，取決於用哪個模型**

1536 是 **OpenAI `text-embedding-ada-002`** 模型的輸出維度，不是通用標準。**每個 embedding 模型有自己的維度**：

### 常見模型 vs 維度

| 模型 | 提供者 | 維度 |
|------|--------|------|
| text-embedding-ada-002 | OpenAI | **1,536** |
| text-embedding-3-small | OpenAI | 1,536（可縮短）|
| text-embedding-3-large | OpenAI | **3,072** |
| textembedding-gecko | Google | 768 |
| embed-english-v3.0 | Cohere | 1,024 |
| BGE-small | 開源 | 384 |
| BGE-base | 開源 | 768 |
| BGE-large | 開源 | 1,024 |
| Voyage-large-2 | Voyage AI | 1,536 |

### 但有個重要規則：**同一個模型內，維度永遠固定**

- 如果你用 `text-embedding-ada-002` → **每一筆**記憶都是 1,536 維
- 如果你換成 `text-embedding-3-large` → **每一筆**都變 3,072 維
- **同一個資料庫裡不能混用不同維度**（維度不同就沒辦法比較相似度）

### 維度大小代表什麼？
- **維度越高** → 表達越細膩，但佔空間、算越慢
- **維度越低** → 省空間、快，但細節少

例如 384 維足夠做粗分類；3,072 維適合需要精細語意區分的場景。

## IR 是什麼的縮寫？

**IR = Intermediate Representation**（中間表示 / 中介表徵）

這個詞**原本來自編譯器設計**，後來被 LLM 系統借用。

### 先看編譯器的老例子（幫助理解）

```
C++ / Rust / Go  ─►  LLVM IR  ─►  x86 / ARM 機器碼
  (原始碼)        (中間表示層)     (真正執行的)
```

為什麼要中間多一層 IR？
- **解耦**：新出一個程式語言，不用重寫所有機器碼後端；新出一個 CPU 架構，不用重寫所有前端
- **驗證**：在 IR 層做優化、做檢查最容易

---

## 套到 LLM + 記憶系統

```
LLM (Claude/GPT/Gemini)  ─►  IR (JSON)  ─►  存儲層 (SQLite/pgvector/Qdrant)
  (產生意圖)              (12 原子操作 JSON)    (真正存資料的)
```

**IR 就是你之前問的那個 JSON 五元結構**：
```json
{
  "stage": "T",
  "op": "Update",
  "target": {"ids": ["mem_01"]},
  "args": {"text": "..."},
  "meta": {...}
}
```

text2mem 的 IR = **12 個原子操作 + 五元結構 JSON**。

---

## 你的第二個問題：「存儲」指什麼？

**不是 LLM 的 DB，是你自己應用的記憶資料庫**。

先釐清一個常見誤解：

### ❌ LLM 沒有「使用者可存取的 DB」
Claude、GPT、Gemini 這些 LLM 本身是一個**會說話的神經網路**，沒有讓你存東西的資料庫。你跟它對話結束，它就忘了（除非有 memory 功能）。

### ✅ 「存儲」= 你應用端的記憶儲存
在 text2mem 架構裡：
- **LLM**：負責「理解使用者、決定要記什麼、要取什麼」→ 它吐出 IR (JSON)
- **IR 層 (text2mem)**：驗證 JSON 合不合法、解析成具體操作
- **存儲 (Storage)**：真正存資料的地方，可能是：
  - SQLite（參考實現用的）
  - PostgreSQL + pgvector（生產級）
  - Qdrant / Pinecone / Milvus（專業向量 DB）

**「LLM 跟存儲中間差一層 IR」的白話版**：LLM 不直接寫資料庫，它只吐 JSON 指令；text2mem 在中間當翻譯 + 守門員，確認指令合法後再去動存儲層。

---

## 為什麼需要這層 IR？

| 如果沒 IR（LLM 直接操作 DB） | 有 IR (text2mem) |
|---------------------------|------------------|
| LLM 要學 SQL、vector DB 的各種 API | LLM 只要會吐統一 JSON |
| 換存儲（SQLite→Postgres）要重訓 LLM | 換存儲只要換 adapter，LLM 不變 |
| 換 LLM（Claude→GPT）要重寫介接 | 換 LLM 不影響存儲層 |
| LLM 亂講直接爆炸 | IR 層當守門員擋下 |

**IR 是「解耦 LLM 和存儲」的中介層，讓兩邊可以獨立演進**。

## 真正的架構：**ENC / STO / RET 是三個獨立階段**

這是經典**認知心理學的記憶三階段模型**（Atkinson-Shiffrin Model），text2mem 借用這個框架：

| 縮寫 | 全名 | 中文 | 做的事 |
|------|------|------|--------|
| **ENC** | **Enc**oding | 編碼 | 把資訊**寫進**記憶 |
| **STO** | **Sto**rage | 儲存 | 記憶**留存期間**的維護 |
| **RET** | **Ret**rieval | 提取 | 把記憶**取出來**用 |

（我之前錯把 STO 當成「Storage/Transform/Output」的三字縮寫，其實 STO 本身就是 Storage 一個階段的三字縮寫。等下更正筆記。）

---

## 你的疑惑：為什麼順序是 ENC → RET → STO，不是 ENC → STO → RET？

你的直覺**完全正確**，時間上本來就該是：
```
ENC (寫入) → STO (存放期間) → RET (之後讀出)
```
這是認知心理學標準順序。

**但影片刻意把順序換成 ENC → RET → STO，是教學策略，不是時間順序。**

### 為什麼這樣排？— 因為 STO 才是最有料的部分

把 12 個操作分配到三階段看：

| 階段 | 屬於它的操作 | 使用者熟悉度 |
|------|-------------|-------------|
| **ENC (寫)** | Encode, Update, Delete, Label | 直覺，人人都懂（像 INSERT）|
| **RET (讀)** | Retrieve, Search, Clarify | 直覺，人人都懂（像 SELECT）|
| **STO (存放期間的維護)** | **Summarize, Merge, Split, Promote, Demote, Expire, Lock** ← | **這才是整個系統的精華** |

影片用 **ENC → RET → STO** 的順序是**教學鋪陳**：
1. 先講 ENC（寫入）— 大家都懂，暖身
2. 再講 RET（取出）— 大家也都懂，繼續鋪
3. **最後講 STO**（存放期間） — **鏘鏘！這才是這個系統的獨特設計**

因為傳統資料庫**只有 ENC + RET**（INSERT/SELECT），根本沒有 STO 這個概念。text2mem 的**核心創新就在 STO**——「**記憶存在期間會自己演化**」：
- **Summarize**：久了自動壓縮成摘要
- **Merge**：相似記憶合併
- **Promote/Demote**：重要性隨用途調整
- **Expire**：過期淘汰

這些全是傳統 DB 沒有的概念。**所以影片壓軸講 STO，是為了強調「這個才是創新點」**。

### 兩種順序的差別

| 順序 | 視角 | 用途 |
|------|------|------|
| **ENC → STO → RET** | 時間順序 / 資料流向 | 解釋「一筆記憶的生命」|
| **ENC → RET → STO** | 教學順序 / 重要性鋪陳 | 解釋「系統設計的精華」|

**兩種都對，只是切入角度不同。**
