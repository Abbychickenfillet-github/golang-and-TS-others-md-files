## 15:10 — 更新一下set, get 跟我解釋一下意思

**Claude:**

# Zustand 的 `set` 與 `get`

在 `useLapTimerStore.js:9` 這行：

```js
(set, get) => ({ ... })
```

這是 Zustand 的 **store creator function**，它會傳入兩個工具：`set` 和 `get`。兩者是操作 store 的唯一正式途徑。

## `set` — 更新 state

**作用**：修改 store 裡的狀態，並**觸發所有訂閱該狀態的 React 元件重新渲染**。

### 兩種用法

**1. 物件形式（部分更新 / merge）**

```js
set({ title: '讀書', isRunning: true })
```

Zustand 會把新物件**淺層合併**到現有 state，沒列出的欄位保持不變。`useLapTimerStore.js:63-73` 就是典型例子：

```js
set({
  startTime: now,
  currentLapStartTime: now,
  isRunning: true,
  // ...其他欄位不動
})
```

**2. Function 形式（需要讀舊值時）**

```js
set((state) => ({ laps: [...state.laps, newLap] }))
```

不過你的 store 沒這樣用——它都是先 `get()` 再 `set()`（見下一節）。

### 重點
- **只能透過 `set` 改 state**，不能直接 `state.laps.push(...)`，否則元件不會重新渲染。
- **合併是淺層的**——如果寫 `set({ user: { name: 'A' } })`，整個 `user` 物件會被替換，不會跟舊 `user` merge。

---

## `get` — 讀取當前 state（在 actions 內部）

**作用**：在 action 函式內取得**當下最新的 state snapshot**。

### 為什麼需要 `get`？

因為 action 定義時，`state` 還不存在。你不能寫：

```js
startActivity: () => {
  if (!this.title) ...     // ❌ 沒有 this
  if (!state.title) ...    // ❌ state 從哪來？
}
```

所以 Zustand 提供 `get()`，呼叫時回傳最新的 store 物件。`useLapTimerStore.js:51-60` 示範：

```js
startActivity: () => {
  const state = get()              // 取當下完整 state
  if (!state.title.trim()) { ... }
  if (state.isRunning) { ... }
  set({ ...})
}
```

### 為什麼不用解構參數的 state？
Zustand 其實也允許 `set((state) => ...)` 形式，但那隻適合「純粹根據舊值算新值」的情境。當你需要：
- 先做**驗證**（如 `if (!state.title.trim()) return`）
- 根據多個欄位做**複雜運算**（如 `endActivity` 要算 `pausePeriods.reduce(...)`）

這時用 `const state = get()` 讀一次、在整個函式裡複用，邏輯更清楚。這也是你這份 store 的慣用風格。

---

## `set` vs `get` 速查

| | `set` | `get` |
|---|---|---|
| 做什麼 | 寫入 state | 讀取 state |
| 會觸發重新渲染？| ✅ 會 | ❌ 不會（只是讀） |
| 回傳值 | 無 | 當前完整 state 物件 |
| 典型位置 | action 結尾 | action 開頭 |

---

## 在你的 store 裡的實際模式

幾乎每個 action 都長這樣：

```js
actionName: () => {
  const state = get()          // 1. 讀當前狀態
  if (!state.xxx) return        // 2. 驗證
  const now = new Date()        // 3. 算新值
  set({ ...新狀態 })            // 4. 寫回
}
```

例如 `recordLap` (`useLapTimerStore.js:116-141`)：
1. `get()` 拿到 `currentLapStartTime` 和 `laps.length`
2. 檢查 `isRunning` 和 `isPaused`
3. 計算 `lapDuration`、組 `newLap`
4. `set({ laps: [...state.laps, newLap], currentLapStartTime: now })`

這個 **「get → 驗證 → 計算 → set」** 就是 Zustand 最常見的 action 結構。

## 15:14 — isRunning是要幹嘛用的

**Claude:**

# `isRunning` 的用途

`isRunning` 是分圈計時器的**「活動進行中」狀態旗標**——用來區分「計時器有沒有在跑」，與 `isPaused`（暫停中）合起來描述完整的生命週期。

## 三種活動狀態

由 `isRunning` 和 `isPaused` 兩個布林值組合而成（`useLapTimerStore.js:250-255` 的 `getActivityStatus`）：

| `isRunning` | `isPaused` | 狀態 | 說明 |
|---|---|---|---|
| `false` | `false` | **準備中** | 還沒開始，或已結束 |
| `true` | `false` | **進行中** | 正常計時 |
| `true` | `true` | **已暫停** | 活動還活著，但時間不累計 |

> 注意：`isRunning: true` 時即使在暫停，**活動仍然「存在」**，只是時間停擺。按「結束」才會把 `isRunning` 變 `false`。

## 它被用來做什麼

### 1. 防止重複操作（守衛條件）

在每個 action 開頭當作前置檢查：

- **`startActivity`** (`useLapTimerStore.js:57-60`)：如果 `isRunning` 已為 `true`，擋下重複開始
  ```js
  if (state.isRunning) { alert('活動已在進行中'); return }
  ```
- **`pauseActivity`** (`:79`)：必須 `isRunning` 才能暫停
- **`resumeActivity`** (`:94`)：必須 `isRunning` 才能恢復
- **`recordLap`** (`:118`)：必須 `isRunning && !isPaused` 才能記錄分圈
- **`endActivity`** (`:146`)：必須 `isRunning` 才能結束

沒有這個旗標，使用者可以亂按按鈕，邏輯會錯亂（例如在沒開始時就按暫停）。

### 2. 控制 UI 顯示

在 `components/timelog/LapTimer.js:72-77`，計時顯示器只在 `isRunning` 為 `true` 時每秒更新：

```js
useEffect(() => {
  if (isRunning) {
    const elapsedTime = getCurrentElapsedTime()
    setDisplayTime(formatTime(elapsedTime))
  }
}, [currentTime, isRunning, ...])
```

活動結束後（`isRunning: false`），顯示時間就凍結，不再跳動。

### 3. 決定按鈕顯示哪個

UI 層會依 `isRunning` 決定顯示「開始」還是「暫停/結束」按鈕（以你 `LapTimer.js` 的結構來看就是這個作用）。

### 4. 給 API 當「是否結束」的線索

`app/api/lap-timer/route.js:77, 199, 241`：

```js
isRunning: updatedTimeLog.endTime === null
```

後端沒有直接存 `isRunning` 欄位——它用 `endTime === null` 反推。這對應了前端「結束活動」的行為：`endActivity` 會同時 `set({ endTime: now, isRunning: false })`。

## 為什麼需要一個獨立旗標？不能用 `startTime != null` 推嗎？

理論上可以，但：

- **語意更清楚**：`if (isRunning)` 比 `if (startTime && !endTime)` 直觀
- **持久化行為更好**：`onRehydrateStorage` (`useLapTimerStore.js:328-331`) 從 localStorage 還原時，**刻意把 `isRunning` 重設為 `false`**——即使 `startTime` 還在，也強制回到準備狀態。這避免了「上次沒正常關閉 → 這次刷新 → 計時器以為還在跑、但時間早就亂了」的狀況。

這是一個**防呆設計**：用獨立旗標讓你能在還原時乾淨地「停下來」，而不用去動 `startTime` 之類的真實資料。

## 一句話總結

`isRunning` = **「這個分圈計時 session 是否活著」**——控制能按哪些按鈕、UI 要不要即時更新、API 要不要視為未結束的活動。

## Bug 1（關鍵）：`hideLoader` 根本沒作用

`hooks/use-loader/index.js:84`

```js

```

這行的語意是：**只有 `close=0` 時 `hideLoader` 才會關閉 loader**。

`app/dashboard/page.js:930-936`

```js

```

這個 effect 會在 `auth.hasChecked` 從 `false → true` 時再跑一次，**又呼叫一次 `showLoader()`**，把剛要關的 loader 計時器重置為新的 2 秒。所以即使 API 已經回來，loader 還是得再等一輪 2 秒。

## Bug 3：主畫面 render gate 也是白的

`app/dashboard/page.js:1159`

```js
if (!auth.hasChecked || isLoading) {
  return null
}
```

`isLoading` 初始 `true`，要等 `/api/timelogs` + `/api/featured-shares` 都回來才會變 `false`。在這之前頁面 **本身**就是空白的，loader 只是蓋在一個本來就空白的頁面上。

## 「寫了但沒 import」是什麼意思？

每個 Factory 內部有一張「**對照表**」：

```python
# VectorStoreFactory 裡面長這樣（簡化版）
provider_to_class = {
    "pinecone":  "mem0.vector_stores.pinecone.PineconeDB",
    "qdrant":    "mem0.vector_stores.qdrant.QdrantDB",
```

**真正的 import 只在你實際選那家時才跑**：

```python
def create(provider, config):
    class_path = provider_to_class[provider]   # 先查表拿路徑
    module = importlib.import_module(class_path)  # ← 這一行才真的 import
    return module(config)
```

- **廚房只備你點的那道的食材**（= 只 import 你配置的那家）
- **沒人點龍蝦，廚房不用進龍蝦**（= 沒用 pinecone，就不用 pip install pinecone）

## 「只要用的不是 pinecone 就不會掛」的意思

假設你：
- `pip install mem0` ✅
- 沒裝 `pinecone-client`
- config 寫 `provider="qdrant"`，你裝了 `qdrant-client`

**Mem0 運作順序：**

1. 讀你的 config，看到 `qdrant`
2. `importlib.import_module("qdrant_client")` ✅ 成功
3. **完全不碰** pinecone 那行字串
4. 正常運作

但如果你改成 `provider="pinecone"`：

1. 讀 config 看到 `pinecone`
2. `importlib.import_module("pinecone")` ❌ `ModuleNotFoundError`
3. **這時才會掛，而且是明確告訴你「你要用 pinecone 但沒裝」**

**結論**：掛不掛跟「Mem0 支援幾家」無關，只跟「你點的那家有沒有裝」有關。

## 2. importlib 是針對供應商的嗎？

**不是。importlib 是 Python 標準函式庫，本身跟「供應商」毫無關係。**

`importlib` 做的事只有一件：**「用字串的方式動態 import 一個 Python 模組」**。

### 普通 import vs importlib

```python
# 方式 A：寫死在原始碼裡（最常見）
import qdrant_client        # ← 檔案載入時就執行

# 方式 B：用字串，執行期才決定載什麼
import importlib
name = "qdrant_client"      # ← 這個字串可以來自 config、使用者輸入、資料庫
module = importlib.import_module(name)
```

兩者功能完全一樣，差別只在**「什麼時候決定要載哪個」**。

### importlib 的其他用途（跟供應商無關的例子）

- **外掛系統**：VSCode / Obsidian / Django 的 plugin 機制，都是用 importlib 在執行期載入外掛
- **熱重載**：開發時改 code 自動 reload，用 `importlib.reload()`
- **動態測試**：測試框架按檔名掃描並載入測試檔
- **配置驅動程式**：「config 寫哪個 class，我就載哪個」，不寫死

### 那為什麼我上一段講 Mem0 時一直在講供應商？

因為 **Mem0 拿 importlib 這個通用工具，應用在「供應商切換」這個場景**。就像「螺絲起子可以拆很多東西，我們剛剛只在講它拆水龍頭」——工具是通用的，場景是 Mem0 選的。

- **Pinecone** = 一家商業向量資料庫（我用它當 Mem0 支援的眾多家之一的例子）

importlib 是工具，供應商切換是 Mem0 怎麼用這個工具。兩者沒有綁定關係。

## 20:39 — 系統再把整數映射回真實UUID, User prompt只看user消息 agent prompt只看assistant消息為啥有ai assistant

**Claude:**

好問題，這是 Mem0 一個容易被忽略但**貢獻準確率很多**的設計。

## 為什麼要分「user prompt」和「agent prompt」？

**核心答案：因為對話裡有兩種「事實來源」，不能混在一起看。**

### 兩種事實來源

| 來源 | 產生什麼事實 | 舉例 |
|------|-----------|------|
| **使用者訊息 (user)** | 關於使用者的事實 | 「我喜歡暗色主題」「我住台北」「我不吃牛肉」|
| **AI 回覆 (assistant)** | 關於承諾、推薦、確認、執行的事實 | 「我已經幫你訂了週一的機票」「建議你選方案 B」「我會記得提醒你」|

**兩邊都是「值得記住的事實」，但性質完全不同。**

### 如果只抽 user 訊息會漏什麼？

想像一段對話：

```
使用者：「幫我訂一張週一到東京的機票」
AI：「已經幫你訂了！起飛時間早上 8:30，航班 JL123。」
```

如果 Mem0 **只看 user 訊息**，它會記住：
- ✅ 「user 想訂週一到東京的機票」

但會**完全漏掉**：
- ❌ 機票已訂、航班 JL123、起飛 08:30 ← **這些才是未來該被記住的重要事實**

下次使用者問「我的東京機票幾點起飛？」——系統完全答不出來，因為那個資訊**從來沒進記憶庫**。

### Mem0 的解法：兩條抽取管線平行跑

```
原始對話
    │
    ├── User Prompt：「從 user 訊息抽取關於 user 的事實」
    │      │
    │      ▼
    │   [user 偏好暗色主題]  [user 住台北]
    │
    └── Agent Prompt：「從 assistant 訊息抽取承諾 / 推薦 / 確認」
           │
           ▼
        [已訂 JL123 週一 08:30 東京機票]  [已推薦方案 B]
                │
                ▼
        兩邊的 facts 全部進同一個記憶庫
```

### 這就是之前提過的 "agent-generated facts as first-class data"

前面 §3 提到 Mem0 的抽取設計有一句話：

> "treats agent-generated facts as first-class data"

**這句話的精髓就是這件事**——AI 的回覆也產生事實，要跟 user 產生的事實**同等權重**存進記憶庫，不是隻當成「對話片段」就過了。

### 為什麼「兩個 prompt 分開」而不是「一個 prompt 抓全部」？

可以混在一起抽嗎？理論可以，但實測**準確率差很多**：

| 作法 | 問題 |
|------|------|
| ❌ 混在一起（一個 prompt 吃整段對話）| LLM 會搞混「誰說的」、「這事實歸誰」；容易把 AI 承諾錯記成 user 偏好 |
| ✅ 分開抽（兩個 prompt）| 每個 prompt 角色明確、只做一件事，抽取精準度高 |

這是一個 **「提示工程上的分工」** 設計——跟前面 text2mem 筆記的「**原子操作：每個 op 只做一件事**」是同一種哲學：**讓每個 LLM 呼叫負責一件乾淨的任務，不要混合職責**。

你描述的情境——「**講話 → 快速找到當初寫的東西**」——剛好是 Mem0 設計的正中紅心。

### 為什麼 Mem0 在這個場景贏？

回到前面 §2 的設計六「**多信號檢索融合**」，Mem0 對**每次查詢**同時跑三路評分：

| 信號 | 抓什麼 | 對你的幫助 |
|------|-------|----------|
| **語意相似度**（向量）| 意思相近但字不同 | 你問「暗色模式」，能找到當初寫的「深色主題」|
| **關鍵字匹配** | 精確字詞命中 | 你說「幫我找 Claude API」，會精準命中那個字 |
| **實體匹配**（NER）| 人名、地名、時間 | 你說「上週跟 John 的討論」，會抓 John 這個實體 |

三路融合取 top-k——**單一信號會漏的查詢，另外兩路會補上**。

## 🎨 Typography（typography）

- **`globals.scss:25` 字體堆疊** 只有 Noto Sans TC + 系統 fallback，沒有顯示字 + 內文字的對比配對。「登入」大標、副標、正文、數字全用同一字體，僅靠大小拉層級 → 符合檢查指引中「one well-chosen family 也 OK」，但缺乏個性。

## 📐 Spatial（spatial-design）

## 🎬 Motion（motion-design）

- **`.cloud-effect` + `mintFloat` + `glowPulse`**（`globals.scss:597,405,641`）三支無限循環動畫同時跑：登入頁面本應是任務導向短停留，persistent glow + 雲霧 + 背景漂移會分散對焦點（input）的注意力。

- **缺少 `prefers-reduced-motion`**：整個 globals.scss 沒有任何 reduced-motion 處理，違反參考 motion-design.md 中「這不是可選的」規定。
- **使用 `transition: all 0.3s ease`**（`.topnav-bootstrap`, `.themeToggle`）— reference 明確建議不要用 `ease`，且不要 `all`（難以 profile、performance 差）。

## 🖱 Interaction（interaction-design）

- **Input 缺少 focus 樣式覆蓋**：`.form-control:focus`（`globals.scss:815`）給了 `box-shadow: 0 0 0 0.2rem rgba(13, 202, 240, 0.25)` — 這是 `green` 主題色硬編碼，在 `warm`/`lavender`/`sand-barbie` 主題下 focus ring 不符合該主題。且沒用 `:focus-visible`，滑鼠點擊時也會出現 focus ring（令人分心）。
- **密碼欄「eye」按鈕無 `aria-label`**（`page.js:347-361`）。螢幕閱讀器使用者不知這是「顯示密碼／隱藏密碼」。
- **「忘記密碼」連結同時出現兩個裝飾 SVG**（勾勾 + 箭頭）— 違反 *"Never use Click here"*/冗餘裝飾原則；單一箭頭已足夠。
- **登入成功後用 `Swal.fire` 超級長的 HTML**（`page.js:43-64`）顯示 JWT/Cookie/Session 的技術細節：對使用者毫無意義，只會造成認知負擔。應改為單行「歡迎回來，Abby」。

## ✍ UX Writing（ux-writing）

## 📱 Responsive（responsive-design）

- **`page.js:112` `mt-5 row align-items-center`** — 沒有 mobile-first 的間距重設，`mt-5`（3rem）在 375px 小螢幕佔太多頂部空間（配合 main 本身 padding，會讓使用者看不到表單）。

> ## Snapshot Flags
> 
> The snapshot is your primary tool for understanding and interacting with pages.
> `$B` is the browse binary (resolved from `$_ROOT/.claude/skills/gstack/browse/dist/browse` or `~/.claude/skills/gstack/browse/dist/browse`).
> 
> **Syntax:** `$B snapshot [flags]`
> 
> ```
> -i        --interactive           Interactive elements only (buttons, links, inputs) with @e refs. Also auto-enables cursor-interactive scan (-C) to capture dropdowns and popovers.
> -c        --compact               Compact (no empty structural nodes)
> -d <N>    --depth                 Limit tree depth (0 = root only, default: unlimited)
> -s <sel>  --selector              Scope to CSS selector
> -D        --diff                  Unified diff against previous snapshot (first call stores baseline)
> -a        --annotate              Annotated screenshot with red overlay boxes and ref labels
> -o <path> --output                Output path for annotated screenshot (default: <temp>/browse-annotated.png)
> -C        --cursor-interactive    Cursor-interactive elements (@c refs — divs with pointer, onclick). Auto-enabled when -i is used.
> ```
> 
> All flags can be combined freely. `-o` only applies when `-a` is also used.
> Example: `$B snapshot -i -a -C -o /tmp/annotated.png`
> 
> **Flag details:**
> - `-d <N>`: depth 0 = root element only, 1 = root + direct children, etc. Default: unlimited. Works with all other flags including `-i`.
> - `-s <sel>`: any valid CSS selector (`#main`, `.content`, `nav > ul`, `[data-testid="hero"]`). Scopes the tree to that subtree.
> - `-D`: outputs a unified diff (lines prefixed with `+`/`-`/` `) comparing the current snapshot against the previous one. First call stores the baseline and returns the full tree. Baseline persists across navigations until the next `-D` call resets it.
> - `-a`: saves an annotated screenshot (PNG) with red overlay boxes and @ref labels drawn on each interactive element. The screenshot is a separate output from the text tree — both are produced when `-a` is used.
> 
> **Ref numbering:** @e refs are assigned sequentially (@e1, @e2, ...) in tree order.
> @c refs from `-C` are numbered separately (@c1, @c2, ...).
> 
> After snapshot, use @refs as selectors in any command:
> ```bash
> $B click @e3       $B fill @e4 "value"     $B hover @e1
> $B html @e2        $B css @e5 "color"      $B attrs @e6
> $B click @c1       # cursor-interactive ref (from -C)
> ```
> 
> **Output format:** indented accessibility tree with @ref IDs, one element per line.
> ```
>   @e1 [heading] "Welcome" [level=1]
>   @e2 [textbox] "Email"
>   @e3 [button] "Submit"
> ```
> 
> Refs are invalidated on navigation — run `snapshot` again after `goto`.

## viewBox 到底是什麼？「邏輯空間」變大 vs「實際 px」

**它們是兩回事，分開看：**

### 1. `width` / `height` = 實際 px 大小（外框）
```jsx
<svg width="20" height="20" ...>
```
這定義 **SVG 在 DOM 裡佔多少 CSS 像素**。20×20 就是真的 20×20 px 的框。這個沒變 → **實際 px 佔用沒變**。

### 2. `viewBox` = 內部座標系（邏輯空間）
```jsx
viewBox="0 0 24 24"   // 舊
viewBox="0 0 150 200" // 新
```
這告訴瀏覽器：「**當作** 裡面是一個 X×Y 大小的世界來畫」。不是實際像素，是給 `<circle>`、`<path>` 等用的「繪圖單位」。

### 類比：比例尺地圖

想像你有一個 **20cm × 20cm 的畫框**（= `width/height`），裡面要畫**一個真實世界的建築物**。

| viewBox | 等於在說 | 結果 |
|---|---|---|
| `0 0 24 24` | 「把 24m × 24m 的空間壓進 20cm 的畫框」 | 縮放比 ≈ 0.83，圖很大很清楚 |
| `0 0 150 200` | 「把 150m × 200m 的空間壓進 20cm 的畫框」 | 縮放比 ≈ 0.1，原本 r=10 的圓現在直徑 1px → **變成一個小點** |

### 你那個 icon 的實際後果

原本的 circle：`cx=12 cy=12 r=10`（畫在 24×24 的世界中央）
- 舊 viewBox 24×24：圓直徑 20 單位，佔滿 20px 畫布 ✅
- 新 viewBox 150×200：圓直徑 20 單位，但世界變大到 150×200 了，20/200 = 10% → 圓只佔畫布的 **10% = 2px 大小** ❌

**副作用 2：比例失真**
viewBox 150:200 是 3:4（直的），但畫布是 1:1（方的）。預設 `preserveAspectRatio="xMidYMid meet"` 會等比縮放，以「裝得下」為優先 → 圖會被擠在畫布中央，上下留白。

### 結論表

| | 實際像素 | 內部座標 | 視覺效果 |
|---|---|---|---|
| **width/height** | ✅ 變了 | — | 改畫布大小 |
| **viewBox** | ❌ 沒變 | ✅ 變了 | 改「縮放比例」|

**你這次改 viewBox 沒改實際 px**（還是 20×20 px），但**邏輯空間從 24 單位放大到 200 單位**，所以同樣大小的 `<circle r="10">` 被縮小到快看不到。

### 修復方式

改回 `viewBox="0 0 24 24"` 就好，或者如果你想用 150×200 的座標系，那裡面的 circle/path 座標都要同步等比放大：
```jsx
<circle cx="75" cy="100" r="60" />  // 等比放大 6.25x
```

## 16:41 — xmlns="http://www.w3.org/2000/svg" 寫這是啥 xmlns是啥

**Claude:**

## `xmlns` 是什麼

**`xmlns` = XML Namespace（XML 命名空間）**

它是 XML 世界的「身分證」，用來告訴解析器「接下來這些標籤屬於哪個規範」。

---

## 為什麼需要它

因為 XML 可以自訂標籤，不同規範可能用同一個名字但意思不同。例如：

- `<a>` 在 HTML 裡是超連結
- `<a>` 在某個自訂 XML 裡可能是「作者」

這時 namespace 就負責區分「我這個 `<a>` 是哪一家的」。

---

## `http://www.w3.org/2000/svg` 是什麼

這串看起來像網址，**但它不是真的要去連線的 URL**，只是一個**唯一識別字串**（URI）而已——就像身分證字號，長得像地址但不是真的地址。

意思是：「這份文件裡的 `<svg>`、`<path>`、`<circle>` 都是 W3C 在 2000 年訂的 SVG 規範裡定義的那個」。

你打開這個網址也會看到 W3C 的 SVG 說明頁，但瀏覽器渲染 SVG 時**不會去抓那個網址**，純粹拿字串做比對。

---

## 什麼時候要寫、什麼時候可以省略

| 情境 | 需要 `xmlns` 嗎？ |
|------|------------------|
| 獨立 `.svg` 檔案 | ✅ **必要**，不寫不會渲染 |
| 寫在 HTML5 文件裡 | ❌ 可省略（HTML5 parser 會自動認出 `<svg>`） |
| 寫在 XHTML / XML 裡 | ✅ **必要** |
| JSX / React 元件 | ❌ 通常可省略（但加上也沒差） |

### 舉例

**獨立 .svg 檔**（必要）：
```xml
<?xml version="1.0" encoding="UTF-8"?>
<svg xmlns="http://www.w3.org/2000/svg" viewBox="0 0 24 24">
  <path d="..." />
</svg>
```

**嵌在 HTML 裡**（可省略）：
```html
<!DOCTYPE html>
<html>
  <body>
    <svg viewBox="0 0 24 24">  <!-- 不寫 xmlns 也能正常顯示 -->
      <path d="..." />
    </svg>
  </body>
</html>
```

---

## 其他會遇到的 xmlns

有時會看到 SVG 多寫一個：
```html
<svg xmlns="http://www.w3.org/2000/svg"
     xmlns:xlink="http://www.w3.org/1999/xlink">
```

- `xmlns="..."` = **預設** namespace（沒加前綴的標籤都算這個）
- `xmlns:xlink="..."` = **取別名為 `xlink`** 的 namespace，之後可以用 `xlink:href="..."`

## 16:48 — viewBox的數字滿分值是100嗎

**不是，viewBox 沒有「滿分值」，數字可以任意設定。**

## viewBox 的本質

`viewBox="min-x min-y width height"` — 就是定義一個**虛擬座標系統**，你說它是幾就是幾。

```html
<!-- 這些全部合法 -->

<svg viewBox="0 0 100 100">   <!-- 內部座標 0~100（最常見） -->
<svg viewBox="0 0 24 24">     <!-- 內部座標 0~24（icon 常用） -->

<svg viewBox="0 0 1 1">       <!-- 內部座標 0~1 -->
```

實際渲染尺寸由外部的 `width`/`height` 或 CSS 決定，**跟 viewBox 數字無關**。

## 為什麼你會覺得是 100？

因為**教學範例很常用 `0 0 100 100`**，原因是：
- 好算（10% 位置 = 10，50% = 50）
- 適合示範用
- 對應百分比的直覺

但這只是習慣，不是規定。

## 實務上不同情境用什麼

| 情境 | 常用 viewBox | 原因 |
|------|-------------|------|

| Bootstrap Icons | `0 0 16 16` | 16px icon |
| Font Awesome | `0 0 512 512` | 保留精細度 |
| Logo / 示範圖 | `0 0 100 100` | 好算 |
| 照片、複雜插畫 | `0 0 1920 1080` 等 | 對應原始畫布尺寸 |

```html
</svg>

  <circle cx="500" cy="500" r="400" />
</svg>
```

## 16:50 — viewBox外面已經是 0 0 24 24 了 裡面rect x="25" 所以是看一個比值對嗎

**Claude:**

**不是比值，是絕對座標。** 這是常見誤解，要說清楚。

## viewBox 是「內部座標系統」，不是比例

`viewBox="0 0 24 24"` 的意思是：**內部可視範圍從 (0,0) 到 (24,24)**。

你給的座標數字就是**實際座標位置**，不是百分比。

```html
<svg viewBox="0 0 24 24" width="240" height="240">
  <rect x="25" y="0" width="10" height="10" fill="red" />
  <!-- 這個 rect 完全看不到！因為 x=25 已經在 viewBox 外面了 -->
</svg>
```

---

## 超出 viewBox 會怎樣？

**預設會被裁掉**（clip）。SVG 根元素的 `overflow` 預設是 `hidden`。

```html
<svg viewBox="0 0 24 24" width="240" height="240"
     style="border: 1px solid black;">
  <rect x="0"  y="0" width="10" height="10" fill="red" />      <!-- 看得到 -->
  <rect x="14" y="0" width="10" height="10" fill="blue" />     <!-- 看得到（剛好到邊） -->
  <rect x="20" y="0" width="10" height="10" fill="green" />    <!-- 只看得到左半部，右半被切 -->
  <rect x="25" y="0" width="10" height="10" fill="purple" />   <!-- 完全看不到 -->
</svg>
```

---

## 正確的心智模型

把 viewBox 想成「一張方格紙」：

```
viewBox="0 0 24 24"
=
一張 24×24 格的方格紙

rect x="5" y="5"  → 畫在第 5 格、第 5 格的位置
rect x="25"       → 超出紙張，畫不到（被裁掉）
```

然後這張方格紙**會被縮放**到 `width`/`height` 指定的外部尺寸。

---

## 比例是發生在「viewBox → 外部尺寸」的那一步

```html
<svg viewBox="0 0 24 24" width="240" height="240">
```

這時候的比例是 **24 → 240，放大 10 倍**。

所以內部畫一個 `<rect width="1">`，在螢幕上實際看到的是 10px 寬。

```html
<svg viewBox="0 0 24 24" width="240">
  <rect x="0" y="0" width="12" height="12" fill="red" />
  <!-- 內部 12×12，螢幕上實際 120×120px（放大 10 倍） -->
</svg>
```

---

## 如果你真的想用「百分比」

SVG 屬性其實支援 `%`：

```html
<svg viewBox="0 0 24 24">
  <rect x="50%" y="0" width="50%" height="100%" fill="red" />
  <!-- 這個才是「從中間開始，佔一半寬、整個高」 -->
</svg>
```

但這種寫法較少見，一般還是用絕對座標配合 viewBox 設計。

---

## 快速判斷表

| viewBox | 你給 `x="25"` | 結果 |
|---------|--------------|------|
| `0 0 24 24` | 超出右邊界 1 單位 | **看不到** |
| `0 0 100 100` | 在 25/100 的位置 | 看得到 |
| `0 0 50 50` | 超出右邊界 | **看不到** |
| `0 0 24 24` + `overflow: visible` | 超出但不裁切 | 看得到（要額外設定） |

**viewBox 裡的數字是絕對座標，不是比例。** 座標超過 viewBox 範圍就會被裁掉。比例只發生在「viewBox → 外部 width/height」這一步的縮放。

### `fill` 在 `<svg>` 上不是「背景色」

```html
  <path d="..." fill="white" />               <!-- 前景 -->
</svg>
```

**做法 2：用 CSS**
```html
<svg style="background: red;" viewBox="0 0 24 24">
  <path d="..." fill="white" />
</svg>
```

## 17:24 — 例如letter-spacing那裡placeholder

**Claude:**

Placeholder 可以單獨設字距，也會**繼承** input 的字距。實際看一下登入頁的情況：

## 目前 placeholder 的字距狀態

兩個 placeholder：
```jsx

placeholder="請輸入您包含英文大小寫、數字、特殊符號的密碼"  // 21 個字
```

**目前 `letter-spacing: normal`**（沒特別設），因為：
- input 沒設 letter-spacing
- 父層也沒設
- `globals.scss` 的 `.form-control::placeholder` 只設了 color，沒設字距

所以 placeholder 字距是瀏覽器預設（中文字之間幾乎貼著，看起來緊湊）。

---

## 要不要加字距？

**兩種情境**：

### 情境 A：placeholder 看起來太擠，想加寬一點

```scss
/* globals.scss 找到這段加上 */
.form-control::placeholder {
  color: var(--text-secondary, rgba(255, 255, 255, 0.6));
  letter-spacing: 0.5px;   /* 新增，輕微拉開 */
}
```

**建議值**：
- `0.3px ~ 0.5px` → 細緻、幾乎看不出但氣質變優雅
- `1px` → 明顯拉開，氣質感強
- `2px 以上` → 中文會太散，不建議

### 情境 B：只想讓密碼那個超長 placeholder 舒服一點

因為 `"請輸入您包含英文大小寫、數字、特殊符號的密碼"` 太長，可能還是應該先縮短文字而不是調字距。

---

## placeholder vs input 本體的字距會不一樣嗎？

會。`::placeholder` 會**繼承** input 的 `letter-spacing`，但你可以單獨覆蓋：

```scss
/* input 本體打字時字距緊湊 */
.form-control {
  letter-spacing: normal;
}

/* 但提示文字拉開一點看起來優雅 */
.form-control::placeholder {
  letter-spacing: 0.5px;
}
```

實際輸入時看到的字距（letter-spacing: normal）跟提示字距（0.5px）不一樣，這在 UI 設計上是常見做法，提示文字通常可以稍微拉開。

---

## 陷阱：input 的 `font-family` 也會影響感覺

就算字距都是 0，中文字和英數在預設字型下密度差很大：
```
請輸入您的信箱                 ← 中文字本身較寬
Please enter your email        ← 英文字本身較窄
```

所以有時候不是字距問題，是**字型問題**。如果你覺得中文看起來不夠通透，可以考慮設：
```scss
.form-control {
  font-family: 'Noto Sans TC', system-ui, sans-serif;
}
```

> ## Snapshot Flags
> 
> The snapshot is your primary tool for understanding and interacting with pages.
> `$B` is the browse binary (resolved from `$_ROOT/.claude/skills/gstack/browse/dist/browse` or `~/.claude/skills/gstack/browse/dist/browse`).
> 
> **Syntax:** `$B snapshot [flags]`
> 
> ```
> -i        --interactive           Interactive elements only (buttons, links, inputs) with @e refs. Also auto-enables cursor-interactive scan (-C) to capture dropdowns and popovers.
> -c        --compact               Compact (no empty structural nodes)
> -d <N>    --depth                 Limit tree depth (0 = root only, default: unlimited)
> -s <sel>  --selector              Scope to CSS selector
> -D        --diff                  Unified diff against previous snapshot (first call stores baseline)
> -a        --annotate              Annotated screenshot with red overlay boxes and ref labels
> -o <path> --output                Output path for annotated screenshot (default: <temp>/browse-annotated.png)
> -C        --cursor-interactive    Cursor-interactive elements (@c refs — divs with pointer, onclick). Auto-enabled when -i is used.
> ```
> 
> All flags can be combined freely. `-o` only applies when `-a` is also used.
> Example: `$B snapshot -i -a -C -o /tmp/annotated.png`
> 
> **Flag details:**
> - `-d <N>`: depth 0 = root element only, 1 = root + direct children, etc. Default: unlimited. Works with all other flags including `-i`.
> - `-s <sel>`: any valid CSS selector (`#main`, `.content`, `nav > ul`, `[data-testid="hero"]`). Scopes the tree to that subtree.
> - `-D`: outputs a unified diff (lines prefixed with `+`/`-`/` `) comparing the current snapshot against the previous one. First call stores the baseline and returns the full tree. Baseline persists across navigations until the next `-D` call resets it.
> - `-a`: saves an annotated screenshot (PNG) with red overlay boxes and @ref labels drawn on each interactive element. The screenshot is a separate output from the text tree — both are produced when `-a` is used.
> 
> **Ref numbering:** @e refs are assigned sequentially (@e1, @e2, ...) in tree order.
> @c refs from `-C` are numbered separately (@c1, @c2, ...).
> 
> After snapshot, use @refs as selectors in any command:
> ```bash
> $B click @e3       $B fill @e4 "value"     $B hover @e1
> $B html @e2        $B css @e5 "color"      $B attrs @e6
> $B click @c1       # cursor-interactive ref (from -C)
> ```
> 
> **Output format:** indented accessibility tree with @ref IDs, one element per line.
> ```
>   @e1 [heading] "Welcome" [level=1]
>   @e2 [textbox] "Email"
>   @e3 [button] "Submit"
> ```
> 
> Refs are invalidated on navigation — run `snapshot` again after `goto`.

## 關於 `var(--accent-color, #007bff)` 的 `#007bff` fallback

**那是 Bootstrap 5 的預設 primary 色**（2017 年定的經典 Bootstrap 藍）。

### 為什麼留這個 fallback

```scss
var(--accent-color, #007bff)
                     ^^^^^^^
                     這只是「萬一」的保險，正常情況不會出現
```

CSS 變數 fallback 語法：**當 `--accent-color` 沒被定義時才會用**。

在這個專案，`--accent-color` 在每個主題的 `:root[data-theme='...']` 都有設：
- `green`: `#49c1d9`（青）
- `pink`: `#e589ea5c`（粉紫）
- `lavender`: `#7b5fb1`（薰衣草）
- `warm`: `#8b4513`（咖啡）
- `mint`: `#20c997`（薄荷綠）
- `sand-barbie`: `#d3517c9e`（粉）

**只有當 HTML 根元素 `<html>` 沒有任何 `data-theme` 屬性時**，`--accent-color` 才會是 undefined，這時候才會吃到 `#007bff` 藍色。

### 所以藍色 fallback 會不會真的跑出來？

幾乎不會。會發生的唯一情境：
1. `<html>` 沒有 `data-theme="..."`
2. 且 JavaScript 還沒跑到 `theme-toggle` 去設預設主題
3. 使用者第一次進站的 **FOUC（flash of unstyled content）** 的一瞬間

這幾乎察覺不到。

### 為什麼作者會寫 `#007bff`

大部分情況是**複製貼上 Bootstrap 範例**時順手留下的，或是工程師習慣這樣寫保險。Bootstrap 官方範例碼的 focus ring 都用這個藍色。

### 要改的話

如果你想讓 fallback 更符合品牌（萬一真的出現藍色會很突兀），可以改成這個專案的自訂 primary：

```scss
// globals.scss L8 定義的
$primary: #583163;   // 深紫色

// 改 focus 規則為：
.form-control:focus {
  border-color: var(--accent-color, #583163) !important;
  box-shadow: 0 0 0 0.2rem
    color-mix(in oklab, var(--accent-color, #583163) 40%, transparent) !important;
}
```

或是用 sand-barbie 的粉色 `#d3517c` 當保險（最符合 default 主題）。

### 一句話

`#007bff` 是「萬一 CSS 變數壞掉」的保險藍色，實際上使用者看到的 focus ring 是**粉色**（跟著 `--accent-color` 主題色變）。
