## 為什麼是 `NameError: name 'Vehicle' is not defined`

`v = Vehicle()` 和 `v.show()` 被縮排到 **class body 裡面**了。Python 在執行 class body 時，`Vehicle` 這個名字**還沒建立**（要等整段 class body 跑完，class object 才會被綁定到 `Vehicle` 這個名字）。所以裡面寫 `Vehicle()` 自然找不到。

## 修正成你應該要的樣子

把那兩行往左移到 class **外面**：

```python
class Vehicle:
    def show(jessie):
        print("Show Vehicle")

v = Vehicle()
v.show()
```

這樣會印出 `Show Vehicle`，**不會有任何錯誤**。

## 為什麼 `show(jessie)` 不會出錯？

你可能以為「第一個參數一定要叫 `self`，叫 `jessie` 會壞掉」── 不會。Python 完全不在乎名字，它只看**位置**：呼叫 `v.show()` 時，Python 會自動把 `v` 當作第一個 positional argument 傳進去，於是 `jessie = v`。`self` 只是慣例。

## 如果你真的想看 `TypeError`，試這個

不要透過 instance 呼叫，直接從 class 呼叫但不傳參數：

```python

Vehicle.show()   # ← 沒有 instance 自動帶入，jessie 沒人填
```

## 13:37 — 好崩潰喔 沒問題了 但NameError這也算編譯錯誤嗎？

**Claude:**

不算，`NameError` 是 **執行期錯誤（runtime error）**，不是編譯錯誤。

## Python 的兩個階段

Python 雖然常被叫「直譯語言」，但它其實也會先**編譯成 bytecode**，再由 VM 執行：

```
原始碼 .py
   │
   ▼
[編譯階段] ──► 只檢查「語法」對不對
   │
   ▼
bytecode (.pyc)
   │
   ▼
[執行階段] ──► 真的跑起來，去查名字、查型別、查屬性…
```

## 哪些算編譯錯誤？

只有「語法層面」的問題會在**還沒開始跑**就被擋下來：

| 錯誤 | 階段 | 範例 |
|---|---|---|
| `SyntaxError` | 編譯期 | `def show(:` |
| `IndentationError` | 編譯期 | 縮排亂掉 |
| `TabError` | 編譯期 | tab 和 space 混用 |

判斷方法：如果一個檔案裡**只要有一行語法錯**，整個檔案連第一行都不會執行 → 那就是編譯錯誤。

## 為什麼 `NameError` 是 runtime？

因為 Python 編譯時**根本不知道 `Vehicle` 這個名字存不存在**。它只看「`Vehicle()` 這個寫法合不合文法」，合 → 通過編譯。等到真的執行那行，去當前 scope 找 `Vehicle` 找不到，才丟 `NameError`。

證據：

```python
def never_called():
    print(完全不存在的變數)   # 這行絕對會壞

print("Hello")
```

跑起來會印 `Hello`，不會報錯 ── 因為 `never_called` 沒被呼叫，那行 `NameError` 永遠沒機會發生。如果是編譯錯誤，整個檔案連 `Hello` 都印不出來。

## 同類的 runtime 錯誤

- `NameError`（名字不存在）
- `TypeError`（型別不對 / 參數數量不對）
- `AttributeError`（物件沒這個屬性）
- `ZeroDivisionError`、`KeyError`、`IndexError`…

這些全部都要「跑到那一行」才會發生。

## 四種中斷情境

| 中斷時機 | DB 狀態 | 重跑 incremental 會怎樣 |
|---------|--------|----------------------|
| **A. embedder 階段**（最常見，最慢的步驟） | 該檔舊 chunks 已被 `delete_file_chunks` 砍掉，新的還沒進去 → 該檔在 DB 完全消失 | `get_file_hashes()` 找不到該檔 → MD5 不符 → 重新處理 ✓ |
| **B. `insert_chunks_batch` 中間**（罕見） | 例如 10 個 chunks 只插入了 4 個。**那 4 個都帶新 hash** | `get_file_hashes()` 回傳新 hash → 跟檔案 MD5 相符 → **被跳過** ✗ 靜默損壞，該檔只有 4/10 個 chunks |
| **C. `--full` 模式跑到一半**（我們現在的情境） | truncate 已執行，已處理的檔案完整，未處理的不存在 | 已處理的 skip，未處理的補做 ✓（**注意：重跑時千萬別再加 `--full`，會把已完成的清掉重來**）|
| **D. chunker 階段** | 還沒碰 DB | 正常處理 ✓ |

## 為什麼有情境 B 的風險

`db.py:14` 寫了 `self.conn.autocommit = True`，意思是 **每一條 SQL 立刻 commit**，沒有 transaction 包住「delete + insert」這對動作。`insert_chunks_batch` 用 `executemany`，在 autocommit 下是逐筆提交。

理論上若剛好 kill 在那一瞬間，可能有部分 chunks 已寫入並帶著新 hash，騙過下次的增量檢查。

**實務上機率很低**：embedding 要花幾秒（CPU 跑 bge-m3），DB 寫入只要毫秒，中斷 99% 落在 embedding 階段而不是 insert 階段。

## 為什麼 `ModuleNotFoundError: No module named 'src.xxx'`？

核心觀念：**Python 找模組是看 `sys.path` 這個列表，不是看你 cd 到哪**。但 `sys.path[0]` 會根據你怎麼啟動 Python 自動帶不同值。

### `sys.path[0]` 規則表

| 啟動方式 | `sys.path[0]` 變成什麼 | `from src.xxx` 能不能用？ |
|---------|---------------------|------------------------|
| `python script.py` | **腳本所在的資料夾**（不是 cwd！） | 看腳本資料夾下有沒有 `src/` |
| `python -m foo` | **cwd**（你現在的工作目錄） | 看 cwd 下有沒有 `src/` |
| `python -c "..."` | cwd | 看 cwd 下有沒有 `src/` |
| `pytest` | cwd + conftest.py 所在資料夾 | 看 cwd 下有沒有 `src/` |

### 實際例子（你的專案結構）

```
abby-notes-rag/        ← 「專案根目錄」
├── src/
│   └── config.py
└── scripts/
    └── ingest.py
```

```powershell
cd C:\coding\futuresign\abby-notes-rag    ← cwd 是專案根

```

- `sys.path[0]` = cwd = `C:\coding\futuresign\abby-notes-rag`
- Python 找 `C:\coding\futuresign\abby-notes-rag\src\config.py` → **找到 ✓**

#### 情境 B：cd 錯地方


### 為什麼 `ingest.py` 卻能跑？

去看 `scripts/ingest.py:14-15`：

```python
sys.path.insert(0, str(Path(__file__).parent.parent))
```

這行就是手動把專案根塞進 `sys.path` 第一個位置。`__file__` = 腳本路徑、`.parent.parent` = 上兩層（`scripts/` → `abby-notes-rag/`）。

## Q2：sys.path[0] 是什麼？

`sys.path` 是一個 **Python 列表**（list），裡面是字串，每個字串是一個資料夾路徑。Python 找模組時 **從前往後一個一個試**：

```python
>>> import sys
>>> sys.path
['',                                                           # ← [0]
 'C:\\Users\\User\\AppData\\Local\\Programs\\Python\\Python313\\python313.zip',  # ← [1]
 'C:\\Users\\User\\AppData\\Local\\Programs\\Python\\Python313\\Lib',            # ← [2]
 'C:\\...\\abby-notes-rag\\venv\\Lib\\site-packages',         # ← [N] (venv 裝的套件)
 ...]
```

`sys.path[0]` = 列表第 0 個元素，**最優先搜尋的位置**。

當你 `from src.config import Config`：

```
1. 看 sys.path[0]/src/config.py 在不在 → 在的話用，停
2. 不在 → 看 sys.path[1]/src/config.py 在不在
3. 不在 → ... 一路往後找
N. 全找完都沒有 → 拋 ModuleNotFoundError
```

所以「為什麼 sys.path[0] 重要」= **它是第一個被檢查的位置，常常就是答案在不在的關鍵**。

「`系統路徑`」這個翻譯有點誤導。它不是「系統環境變數的 PATH」（那是給 OS 找 `.exe` 用的），是 **Python 自己內部的模組搜尋路徑**。兩個是完全獨立的東西。

## Q3：為什麼 Python 用 sys.path？太全域了吧？

**你的直覺是對的，這個設計確實很糟，是 Python 的歷史包袱**。Python 1991 年的設計，那時候沒人想到模組系統會變這麼複雜。

| 問題 | 原因 |
|-----|------|
| 兩個套件都叫 `utils` → 後裝的蓋掉先裝的 | sys.path 從前往後找，先找到的贏 |
| 不小心 import 到舊版本 | path 順序亂掉 |
| 在不同資料夾跑同一個腳本，行為不一樣 | sys.path[0] 隨啟動方式變 |
| 改 `sys.path.insert(...)` 的 hack 滿天飛 | 為了繞過上面的問題 |

**現代解法**（Python 社群慢慢演化出來的）：

| 工具 | 做什麼 |
|-----|-------|
| **venv / virtualenv** | 一個專案一個獨立的 Python 環境，套件不會打架 |
| **pyproject.toml + `pip install -e .`** | 用「正式安裝」而不是 sys.path hack，套件名是固定的 |
| **`python -m`** | 用模組路徑啟動，避開 `python script.py` 的 sys.path[0] 陷阱 |
| **uv / poetry** | 把 venv + 套件管理 + lock 都包好，少踩 path 雷 |

你的專案目前用的是 **venv + sys.path hack**（每個 script 開頭手動 insert）。這是「能跑就好」的等級。如果之後想升級成「比較不會踩雷」的結構，可以考慮加 `pyproject.toml` 把 src 包成正式 package。現在不是必要。

→ 答案總結：**sys.path 確實太全域、確實是設計缺陷**，業界已經發展出 venv、pyproject.toml 等工具來緩解。但 Python 為了相容性沒辦法砍掉舊機制。你只要記得：「踩到 ModuleNotFoundError 八成是 sys.path 問題」就夠了。

## Q4：`python -m foo` 是什麼意思？

`-m` = **以模組（module）方式執行**，相對於 `python foo.py` 的「以檔案方式執行」。

差別在 sys.path[0]：

| 啟動方式 | sys.path[0] | 適合什麼場景 |
|---------|------------|------------|
| `python foo.py` | foo.py 所在資料夾 | 獨立腳本（沒有跨資料夾 import）|
| `python -m foo` | **cwd**（你現在站的地方） | foo 是某個 package 的一員、需要從 cwd 看到它 |

#### 案例 B：跑單元測試

```powershell
cd C:\coding\futuresign\abby-notes-rag
venv\Scripts\python -m pytest tests\
```

這就是 `-m pytest`：把 pytest 當模組執行，cwd 當 sys.path[0]，所以 pytest 看得到你的 `src/`、`tests/`。

#### 案例 C：跑你自己的 script

## 5 個「為什麼這樣寫」的設計選擇

### ① 為什麼 `os.environ["X"]` 用方括號，不用 `.get("X")`？

```python
POSTGRES_HOST = os.environ["POSTGRES_HOST"]   # ← 找不到 → KeyError 立刻炸
# vs
POSTGRES_HOST = os.environ.get("POSTGRES_HOST")  # ← 找不到 → 回 None，默默壞
```

**故意要它早炸**（叫 fail-fast）。如果 `.env` 少寫一行，方括號版本會在 import 那瞬間就 KeyError，你立刻知道。`.get` 版本會默默回 `None`，等到後面 `psycopg2.connect(host=None)` 才壞，錯誤訊息看起來像連線問題，難 debug。

### ② 為什麼 `int(os.environ["POSTGRES_PORT"])` 要強制轉型？

```python
POSTGRES_PORT = int(os.environ["POSTGRES_PORT"])
```

因為 **`os.environ` 永遠回字串**，連 `"5433"` 都是字串不是數字。psycopg2 多半也接受字串，但：

- 顯式 `int()` 做兩件事：① 確認真的是數字（寫 `"abc"` 會立刻 ValueError）② 後面用的時候型別正確
- `EMBEDDING_DIM` / `CHUNK_*` 也都加 `int()` 同理

### ③ 為什麼 `NOTES_ROOT = Path(...)` 要包 Path？

```python
NOTES_ROOT = Path(os.environ["NOTES_ROOT"])
```

如果不包 Path，`NOTES_ROOT` 就是字串 `"C:\\coding\\futuresign\\Abby-notes"`，要拿來找檔案還得：

```python
import os
files = []
for root, dirs, files_in in os.walk(NOTES_ROOT):  # 麻煩
    ...
```

包了 Path 之後，`ingest.py:34` 可以直接 `.rglob("*.md")` — Path 物件的方法庫超齊全，跨 Windows/Linux 也不用煩惱反斜線。

### ④ 為什麼 `EMBEDDING_DIM = 1024` 要寫在 .env？模型自己不知道嗎？

技術上對 — 可以在 `embedder.py` 裡寫 `self.dim = self.model.get_sentence_embedding_dimension()` 自己問。

**但放在 .env 有「對帳」作用**：
- DB schema 寫死 `embedding vector(1024)`（看 `init_db.sql`）
- Config.EMBEDDING_DIM 也寫 `1024`
- 哪天你不小心把 `EMBEDDING_MODEL` 改成 `BAAI/bge-small`（512 維），但 `EMBEDDING_DIM` 還是 1024 → 程式啟動時可以對帳發現不一致，比 insert chunk 時 DB 噴 type error 友善多了

這是「**冗餘檢核**」設計，不是冗餘程式。

### ⑤ 為什麼 `dsn()` 是 `@classmethod`，不寫成屬性？

```python
@classmethod
def dsn(cls) -> str:
    return f"host={cls.POSTGRES_HOST} ..."
```

**vs** 如果寫成屬性：

```python
DSN = f"host={POSTGRES_HOST} ..."   # 在 class body 裡這樣寫，class 載入瞬間就求值
```

差別：
- 屬性版本 → class 載入時拼一次字串，之後固定
- method 版本 → 每次叫 `Config.dsn()` 才拼

實務上對 dsn 沒差（值不會變），但 **method 版本未來想改成讀檔、查 secrets manager、做動態替換**都方便。屬性版本一旦寫定就鎖死。這是「給未來留彈性」的習慣寫法。

## 整體心法

這個 Config 類別的設計哲學是：

1. **單一資料源**：所有設定從 `.env` 讀，**.env 是 single source of truth**，要改值改一個地方就好
2. **早早炸給你看**：少欄位 → KeyError，型別錯 → ValueError，不要默默壞
3. **集中存放**：散在各檔的 `os.environ[...]` 很難追蹤，集中到 `Config` 後 IDE 也會自動補全
4. **靜態屬性 vs classmethod 各司其職**：值用屬性，邏輯用 method

## Symbiont 是什麼

**Symbiont** 是一個在背景跑的 Python daemon，目的是「**讓 Claude Code 變成有持續記憶的長期夥伴**」，而不只是一個一次性的對話 session。
