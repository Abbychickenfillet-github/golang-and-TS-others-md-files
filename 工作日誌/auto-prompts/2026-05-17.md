# TDD 三色循環解釋

## 🔴 🟢 🔵 三個顏色對應三個階段

| 顏色 | 階段 | 動作 | 預期結果 |
|------|------|------|---------|
| 🔴 RED | 寫測試 | 先寫一個會失敗的測試 | **測試必須失敗** |
| 🟢 GREEN | 寫程式 | 寫最少的程式讓測試通過 | **測試必須通過** |
| 🔵 REFACTOR | 重構 | 整理程式碼結構 | **測試依然通過** |

---

## 「完全沒失敗」是什麼狀態？

**不是「沒有顏色」，而是「有問題」**。如果你寫完測試直接跑就通過，代表三種可能：

1. **測試根本沒在測東西** ← 最常見也最危險
   ```go
   // 假測試：永遠會過
   func TestAdd(t *testing.T) {
       result := Add(1, 2)
       _ = result  // 沒 assert，當然過
   }
   ```

2. **功能已經存在了**（那你不需要寫這個功能）

3. **測試條件太寬鬆**（例如 assert `result != nil`，但你想驗證的是 `result == 3`）

所以 RED 階段的失敗是**證明測試會抓到問題**的證據。沒看到紅燈 = 你的守門員可能在睡覺。

---

## 🔵 REFACTOR 是什麼意思？

**重構 = 改善程式碼結構，但「行為」不變**。

例如 GREEN 階段你可能寫了又醜又重複的程式碼：

```go
// GREEN：醜但能過
func CalculatePrice(qty int, price float64) float64 {
    if qty > 10 {
        return float64(qty) * price * 0.9
    }
    if qty > 5 {
        return float64(qty) * price * 0.95
    }
    return float64(qty) * price
}
```

REFACTOR 階段就把它整理乾淨：

```go
// REFACTOR：抽出折扣計算
func CalculatePrice(qty int, price float64) float64 {
    return float64(qty) * price * discountRate(qty)
}

func discountRate(qty int) float64 {
    switch {
    case qty > 10: return 0.9
    case qty > 5:  return 0.95
    default:       return 1.0
    }
}
```

**關鍵：重構過程中測試必須一直保持綠燈**。測試就是你的「安全網」——你知道改完結果沒變。沒有測試敢重構 = 在懸崖邊走鋼索。

---

## 完整循環範例

```
🔴 寫 TestEventCouponCanClaim → 跑 → FAIL（函式還不存在）
🟢 寫 CanClaim() 最簡版本 → 跑 → PASS
🔴 寫 TestEventCouponCannotClaimWhenExpired → 跑 → FAIL（沒處理過期）
🟢 加上過期判斷 → 跑 → PASS
🔵 發現過期判斷邏輯散在三個地方 → 抽成 isExpired() → 跑 → 全部 PASS
🔴 寫下一個 case...
```

**循環的核心精神**：小步前進，永遠有測試守門，重構時心裡有底。

## 13:57 — 這三層為什麼是循環

**Claude:**

# 為什麼是「循環」而不是「一次走完」？

因為**一個功能不是一次寫完的，是一個一個小行為堆出來的**。

---

## 一個功能 = 很多個小循環

假設要做「優惠券領取」功能。你不會一次寫完，而是這樣：

```
循環 1：基本領取
🔴 測試「會員可以領券」          → FAIL
🟢 寫最簡單的 Claim()            → PASS
🔵 （程式還很簡單，沒東西可重構）

         ↓ 再來一個行為

循環 2：過期不能領
🔴 測試「過期的券不能領」         → FAIL
🟢 加上過期判斷                  → PASS
🔵 （還好，繼續）

         ↓ 再來一個行為

循環 3：每人限領一張
🔴 測試「同會員不能領兩次」       → FAIL
🟢 加上重複檢查                  → PASS
🔵 發現三個判斷都散在 Claim()，抽成 validate() → 重構 → PASS

         ↓ 再來一個行為

循環 4：庫存用完不能領
🔴 ...
```

**每個循環只處理一個小行為**。功能完成 = 跑了幾十個循環。

---

## 為什麼不一次走完？

### ❌ 不循環的寫法（傳統）

```
1. 一次想清楚所有規則
2. 一次寫完所有程式碼
3. 最後寫測試補一補
```

問題：
- 想不清楚（人腦有極限）
- 寫到一半發現設計錯了，全部要改
- 測試補上去通常都會過（因為照著程式寫測試），抓不到 bug

### ✅ 循環的寫法（TDD）

```
每次只處理「一個行為」
寫測試 → 寫程式 → 重構 → 下一個行為
```

好處：
- **每次只想一件事**，腦袋負擔小
- **每一步都有測試守著**，改錯馬上知道
- **重構隨時做**，不會累積技術債到不能動

---

## 三層的角色：紅綠藍像三個分工

| 階段 | 在問什麼問題 |
|------|-------------|
| 🔴 RED | 「我要的行為是什麼？」（規格） |
| 🟢 GREEN | 「怎麼讓它動？」（實作） |
| 🔵 REFACTOR | 「怎麼讓它好？」（品質） |

**一次只回答一個問題**。三個問題混在一起想 = 容易亂。

- 寫測試時專心想「我要什麼」，不想實作
- 寫實作時專心想「怎麼過測試」，不想優雅
- 重構時專心想「怎麼變漂亮」，不想新功能

---

## 視覺化

```
        ┌──────────────┐
        │   🔴 RED     │ ← 想下一個行為
        │  寫失敗測試   │
        └──────┬───────┘
               ↓
        ┌──────────────┐
        │   🟢 GREEN   │ ← 讓它先動起來
        │  寫程式通過   │
        └──────┬───────┘
               ↓
        ┌──────────────┐
        │  🔵 REFACTOR │ ← 順手整理
        │   清理結構    │
        └──────┬───────┘
               │
               └─────→ 回到 🔴，下一個行為
```

**循環的意義**：軟體不是蓋好就不動的房子，是一直在長大的生物。每次長一點點，每次都健康。

## 🔵 循環 3：REFACTOR — 這次要重構

**這時候我看程式碼，鼻子聞到味道了**：

```go
func CalculateTotal(qty int, price float64) float64 {
    total := float64(qty) * price
    if total >= 3000 {              // ← 折扣邏輯
        return total * 0.8           // ← 跟主計算
    }                                // ← 混在一起
    if total >= 1000 {
        return total * 0.9
    }
    return total
}
```

問題：
- 折扣判斷會越來越多（之後可能還有滿萬 7 折）
- `CalculateTotal` 同時負責「算總價」和「決定折扣」，責任不單一

**抽出去**：

```go
func CalculateTotal(qty int, price float64) float64 {
    subtotal := float64(qty) * price
    return subtotal * discountRate(subtotal)
}

func discountRate(subtotal float64) float64 {
    switch {
    case subtotal >= 3000: return 0.8
    case subtotal >= 1000: return 0.9
    default:               return 1.0
    }
}
```

**跑測試**：三個都還是 PASS ✅
測試是你的保險絲——重構沒改壞行為。

# 誰決定要不要 REFACTOR？

**你決定，依據三個訊號**：

## 訊號 1：嗅到「程式碼壞味道」(code smell)

| 味道 | 意思 |
|------|------|
| 重複 | 同樣的邏輯出現兩三次 |
| 太長 | 一個函式做太多事 |
| 命名爛 | `tmp`、`data`、`x` 滿天飛 |
| 嵌套深 | if 裡面 if 裡面 if |
| 責任不單一 | 一個函式又算數又印 log 又寫 DB |

聞到味道 = 該重構。

## 訊號 2：三振原則（Rule of Three）

> 第一次寫：直接寫。
> 第二次寫類似的：忍住，再撐一下。
> 第三次寫類似的：**重構，抽成共用**。

太早抽象比晚抽象更危險（抽錯方向更難改回來）。

## 訊號 3：「下一個 RED 變難寫」

如果你準備寫下一個測試時，發現：
- 「我要先改一堆東西才能加這個測試」
- 「這個新功能根本塞不進現在的結構」

代表**現在的設計撐不住新需求**，先重構再繼續。

---

# REFACTOR 不是每次都要做

實際開發節奏大概是：

```
🔴🟢       (還很單純，跳過)
🔴🟢       (還行)
🔴🟢🔵    ← 第三次發現重複，抽出來
🔴🟢
🔴🟢🔵    ← 函式變太長，拆兩個
🔴🟢
```

**不重構也不會死，但會慢慢腐爛**。重構是給自己後路，不是給編譯器看的。

---

# 重點整理

| 問題 | 答案 |
|------|------|
| 誰決定要重構？ | **你**（寫程式的人） |
| 依據什麼？ | 程式碼壞味道、三振原則、下一步變難寫 |
| 什麼時候做？ | GREEN 之後、下一個 RED 之前 |
| 為什麼這時候做？ | 測試剛通過 = 行為正確 = 重構有安全網 |
| 不重構會怎樣？ | 短期沒事，長期變成技術債爛攤子 |

**核心心法**：GREEN 是「能動」，REFACTOR 是「能維護」。能動但不能維護的程式碼，三個月後就是別人（或你自己）的噩夢。

**Claude:**

在 VS Code 的搜尋（files to include）欄位，可以這樣寫：

```
Abby-notes/python/**
```

或更簡單：
```
Abby-notes/python
```

**幾種常見寫法：**

| 寫法 | 意思 |
|------|------|
| `Abby-notes/python` | 該資料夾下所有檔案（VS Code 會自動展開） |
| `Abby-notes/python/**` | 該資料夾下所有層級的檔案（明確 glob） |
| `Abby-notes/python/**/*.py` | 只搜尋 `.py` 檔 |
| `**/Abby-notes/python/**` | 不論 `Abby-notes` 在哪一層都搜（適合工作區有多 root 時） |

**注意事項：**
- 路徑是**相對於 workspace root**，不要寫絕對路徑 `C:\coding\...`
- 用正斜線 `/`，不要用反斜線 `\`（即使在 Windows）
- 多個路徑用逗號分隔：`Abby-notes/python, Abby-notes/待辦事項`
- 排除某資料夾用 files to exclude 欄位，或前面加 `!`

## 3. `.\venv\Scripts\Activate.ps1` 是什麼？

**啟用虛擬環境的 PowerShell 腳本**。

| 部分 | 意義 |
|---|---|
| `.\` | 當前目錄（PowerShell 規定執行本地腳本要明寫，避免誤跑同名系統指令） |
| `venv\Scripts\` | 剛才建的 venv 資料夾裡的 Scripts 子資料夾 |
| `Activate.ps1` | PowerShell 腳本檔（`.ps1` 是副檔名） |

執行後它做的事：
1. 把 `venv\Scripts\` 加到 `PATH` 最前面 → 之後輸入 `python` 或 `pip` 會優先用 venv 裡的，不再用全域
2. 改 PowerShell 提示字元，前面加 `(venv)` 提醒你「現在在虛擬環境裡」

```powershell
# 啟用前
PS C:\coding\my-rag>

# 啟用後
(venv) PS C:\coding\my-rag>     ← 這個 (venv) 就是視覺提示
```

**不同 OS / shell 用不同啟用腳本**：

| 環境 | 啟用指令 |
|---|---|
| Windows PowerShell | `.\venv\Scripts\Activate.ps1` |
| Windows CMD | `venv\Scripts\activate.bat` |
| Linux / Mac (bash) | `source venv/bin/activate` |

退出虛擬環境：直接打 `deactivate` 就回到全域。

## Part 2: 你說的那個 OS-level 的東西叫「容器（Container）」

你記憶裡的那個術語是 **Container（容器）**，技術名詞叫 **OS-level virtualization（作業系統層級虛擬化）**。

### 三層虛擬化的差別

```
┌─────────────────────────────────────────────────────────┐
│  VM（虛擬機）— 硬體層級虛擬化                              │
│  ┌──────────────┐  ┌──────────────┐                    │
│  │ Guest OS     │  │ Guest OS     │   每台 VM 自帶一個   │
│  │ (Windows)    │  │ (Ubuntu)     │   完整 OS、Kernel   │
│  │ + Kernel     │  │ + Kernel     │   重、慢、隔離強     │
│  └──────────────┘  └──────────────┘                    │
│         Hypervisor（VMware / VirtualBox）              │
│         Host OS                                         │
│         硬體                                            │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│  Container（Docker）— OS 層級虛擬化  ← 你忘記的那個      │
│  ┌──────────────┐  ┌──────────────┐                    │
│  │ App + libs   │  │ App + libs   │   只隔離 process、   │
│  │ (沒有 Kernel)│  │ (沒有 Kernel)│   檔案系統、網路    │
│  └──────────────┘  └──────────────┘   共用 Host Kernel  │
│         Docker Engine                                   │
│         Host OS Kernel（共用）                          │
│         硬體                                            │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│  venv — 連虛擬化都不算，只是一個資料夾                    │
│  venv/                                                  │
│    └── Lib/site-packages/    ← 套件裝這                 │
│  Python 啟動時換個 sys.path 而已                         │
└─────────────────────────────────────────────────────────┘
```

### 重要觀念釐清

**venv 根本不是虛擬化技術**，它只是：
- 複製一份 Python 執行檔
- 給一個空的 site-packages
- 改 `sys.path` 指向那個新 site-packages

完全沒有 sandbox、沒有 kernel 隔離、不能限制 CPU/記憶體。venv 裡的程式照樣能讀你 `C:\` 任何檔案、照樣能連網路。

| 名稱 | 隔離層級 | 大小 | 用途 |
|---|---|---|---|
| **VM** | 硬體 + Kernel | GB 等級 | 跑不同 OS、強隔離（雲端伺服器、資安） |
| **Container** | OS / Process | MB 等級 | 部署應用、保證環境一致 |
| **venv** | 只隔離 Python 套件 | KB~MB | 同一台機器跑不同 Python 專案 |

## Part 3: 沒有必要用虛擬環境裝嗎？

**短答：個人玩玩、只搞一個專案 → 確實不用。多專案 / 要分享 → 強烈建議。**

### 不用 venv 的痛點（這些都是真實會遇到的）

**1. 套件版本衝突**

```
專案 A 需要  numpy 1.24
專案 B 需要  numpy 2.0（API 不相容）
全域只能裝一個 → 只要切專案就要重灌
```

**2. 全域被塞爆**

`sentence-transformers` 會拖進來 `torch`（~2GB）、`transformers`、`scikit-learn`、`scipy`、`numpy`... 你以後想清掉只能一個一個 `pip uninstall`，且不知道哪些是別的東西在用的。

venv 的話：刪掉 `venv/` 資料夾，**乾乾淨淨**。

**3. 重現不了別人的環境**

別人說「我這專案用 numpy 1.24 + torch 2.1」，你全域是 numpy 2.0 + torch 2.5 → bug 一堆。

`pip freeze > requirements.txt` 在全域做會把**所有**套件都列出來（連別專案的也算進去），別人安裝會被一堆無關套件汙染。

**4. 系統 Python 被搞壞**

某些 Linux 發行版的系統工具依賴特定版本的 Python 套件，你 `pip install` 升級了 → 系統指令壞掉。Windows 比較沒這問題，但 Mac/Linux 常見。

### 你目前狀況的實用建議

如果你只是學 RAG、跑 sentence-transformers 玩玩 → **裝在全域沒事**，繼續用就好。

如果以後：
- 要開另一個專案 → 至少從第二個專案開始用 venv
- 要把 RAG 包成 Docker 部署 → Docker 內仍會 `pip install`，但 Docker 本身就是隔離環境（你不需要在 Docker 裡再開 venv）
- 要寫 `requirements.txt` 給別人 → **必須**用 venv，不然 freeze 會夾帶垃圾

### Docker vs venv：兩種選擇

實務上常見配置：

| 場景 | 推薦 |
|---|---|
| 本地單純學習 | 全域 or venv 都行 |
| 本地多專案開發 | venv |
| 部署到伺服器 / 跨機器 | Docker |
| 團隊協作 | Docker（保證 OS、Python、套件全部一致） |

你的 FutureSign 專案就是用 Docker（記憶裡的 `docker compose -f docker-compose.dev.yml`）— Docker 連 OS 都鎖死了，連 venv 都省了。

## 2. Embedding model ≠ Vector Database（完全不同東西）

很多人會混淆。它們是「分工合作」，不是同一個東西。

### 兩者的職責

| | Embedding model (bge-m3) | Vector Database (pgvector) |
|---|---|---|
| **做什麼** | 把文字**轉成**向量 | **儲存 + 搜尋**向量 |
| **輸入** | 文字 `"Docker AutoMigrate 失敗"` | 向量 `[0.12, -0.34, ..., 0.78]` (1024 維) |
| **輸出** | 向量 `[0.12, -0.34, ..., 0.78]` | 最相似的 K 個向量 |
| **儲存資料？** | ❌ 不儲存任何東西，純函式 | ✅ 把向量存到磁碟 |
| **能搜尋？** | ❌ 不會搜 | ✅ 用 cosine similarity 找最近的 |
| **API 形式** | `model.encode(text) → vector` | `SELECT ... ORDER BY embedding <-> query_vec LIMIT 5` |

### 比喻：相機 vs 相簿

```
相機（embedding model）：把現實場景拍成照片
                              ↓
相簿（vector DB）：把照片存起來，未來能翻找出相似的照片
```

相機只負責「拍」，不負責「存」；相簿只負責「存」，不負責「拍」。少了任一個都不行。

### 完整的資料流

```
你的筆記 (markdown)
     ↓
[chunking]                          ← 切成小塊
     ↓
"Docker AutoMigrate 失敗時..."       ← 純文字 chunk
     ↓
[bge-m3 encode]                     ← embedding model 做這步
     ↓
[0.12, -0.34, ..., 0.78]            ← 1024 維向量
     ↓
[INSERT INTO chunks ...]            ← pgvector 存起來

────────── 查詢時 ──────────

"AutoMigrate 怎麼處理？"             ← 你的問題
     ↓
[bge-m3 encode]                     ← 再用 embedding model 把問題也轉向量
     ↓
[0.15, -0.31, ..., 0.81]            ← 問題的向量
     ↓
[SELECT ... ORDER BY embedding <-> query_vec]   ← pgvector 算誰最像
     ↓
最相似的 5 個 chunks                 ← 搜出筆記片段
```

### 為什麼要分兩個？

因為它們的工作性質完全不同：
- **embedding model** 是「AI 神經網路」，吃 GPU/CPU 算力，狀態為零
- **vector DB** 是「資料庫」，吃磁碟空間和索引演算法，要負責 ACID、index、persistence

把它們分開讓你可以：
- 換 embedding model 不用換 DB（改用 OpenAI 也只是換 encode 那一步）
- 換 DB 不用換 model（從 pgvector 換 Qdrant，bge-m3 還是同一個）

| 真正的單元測試 | Smoke test |
|---|---|
| 寫 fixture、mock、邊界值 | `python -c "from xxx import Yyy; print(Yyy.ZZZ)"` |
| 慢，要寫 fixture | 快，30 秒內知道有沒有死 |
| 答「程式邏輯對嗎？」 | 答「程式至少跑得起來嗎？」 |

## 14:06 — 為什麼向量的長度是用1024來形容 ...  因為你出現1024個浮點數 可是你這些點所形成的是一條直線嗎

**Claude:**

超好的問題，這個概念卡住的人超多。釐清兩件事：

## 「1024」不是 1024 個點，是「一個點」需要 1024 個座標

### 用維度想，不要用「點數」想

| 空間 | 一個點需要幾個數字 | 例子 |
|---|---|---|
| 1 維（線）| 1 個 | `5`（在數線上的位置） |
| 2 維（平面）| 2 個 | `(3, 4)` |
| 3 維（立體）| 3 個 | `(3, 4, 7)` |
| **1024 維**| **1024 個** | `(0.012, -0.045, 0.078, ..., 0.023)` |

bge-m3 給你的 1024 個浮點數 = **一句話 = 1024 維空間裡的「一個點」**。不是 1024 個點，是 1 個點。

### 視覺化（用 2D 類比）

把每句話想成是一支「從原點出發的箭」：

```
2D 平面（2 個座標就夠）:

    y
    ↑
    │     ↗ 向量A = (3, 4)
    │   ↗
    │ ↗
    └─────────→ x
    ↘
      ↘
        ↘ 向量B = (3, -4)
```

- 「向量 A」是從原點 `(0,0)` 射到 `(3,4)` 的那支箭
- 它的「方向」記錄了 A 的語意
- 它的「長度」=`√(3² + 4²)` = 5

bge-m3 做的事：把 2D 升級到 1024D。原理一樣，只是箭住在 1024 維空間，**人類無法畫出來**，但數學完全有效。

## 你說的「長度」其實是兩個不同概念

中文「長度」有歧義，這是你卡住的點：

| 中文說「長度」 | 英文 | 意思 | 數值 |
|---|---|---|---|
| 「向量**有 1024 個**長度」| dimension | 幾個座標 | **1024** |
| 「向量**的長度**是多少」| magnitude / norm | 箭多長 | bge-m3 規一化成 **1** |

`v.shape == (1024,)` 印出來是「**這支箭活在 1024 維空間**」，不是「箭長 1024」。

實際算「箭的長度」：

```python
import numpy as np
v = embedder.encode_one("Hello")
print(v.shape)              # (1024,)        ← 這是維度
print(np.linalg.norm(v))    # 0.9999999...   ← 這才是真的「長度」
```

## 為什麼長度被規一化成 1（normalize）？

因為我們只在乎「**箭指哪個方向**」，不在乎「**箭多長**」。

```
       ↗ A
     ↗
   ↗
 ↗
原點

       ↗ B（跟 A 同方向，但短一點）
     ↗
   ↗
原點
```

A 跟 B 雖然長度不同，**方向一樣 → 語意一樣**。所以全部硬拉成「長度=1」，比較時只看方向夾角（cosine）。

```
夾角 0°   → cosine = 1   （完全同方向、同義）
夾角 90°  → cosine = 0   （無關）
夾角 180° → cosine = -1  （完全反向、相反意思，罕見）
```

## 回到你的疑問

> 「這些點所形成的是一條直線嗎」

**不是**。這 1024 個浮點數合起來定義 **一個點 / 一支箭**。每個浮點數是這支箭在某個維度上的座標。

如果你想看「直線」，那是兩個向量之間連的線——但 bge-m3 一次只給你一支箭，多執行幾次才會有多支箭。

> 為什麼向量被normalize_embeddings=True規一化 長度就會是1？  A 跟 B 雖然長度不同，方向一樣 → 語意一樣。所以全部硬拉成「

**Claude:**

兩個概念再分清楚：**維度 = 幾個座標**，**長度 = 那支箭多長**。它們完全獨立。

## 維度跟長度沒關係

```
向量 (3, 4)        ← 2 個座標 = 2 維。長度 = √(3² + 4²) = 5
向量 (0.6, 0.8)    ← 2 個座標 = 2 維。長度 = √(0.6² + 0.8²) = 1
```

兩個都是 **2 維**，但一個長度 5、一個長度 1。**規一化只把長度改掉，維度永遠不動**。

## Normalize 在做什麼：所有座標除以原本的長度

公式：`新向量 = 舊向量 / 舊長度`

### 親手算一次（2D 例子）

```
原向量 v = (3, 4)
舊長度 = √(3² + 4²) = √25 = 5

normalize:
  新 v = (3/5, 4/5) = (0.6, 0.8)
  
驗證新長度:
  √(0.6² + 0.8²) = √(0.36 + 0.64) = √1 = 1   ✅
```

座標還是 **2 個**（2 維沒變），只是每個座標等比縮小，總距離剛好 = 1。

### 1024 維 bge-m3 的版本

```
原向量 = (0.5, -0.3, 0.8, ..., 0.2)         ← 1024 個座標
舊長度 = √(0.5² + (-0.3)² + 0.8² + ... + 0.2²) = 假設算出來 = 17.3

normalize:
  新向量 = (0.5/17.3, -0.3/17.3, 0.8/17.3, ..., 0.2/17.3)
        = (0.029, -0.017, 0.046, ..., 0.012)   ← 還是 1024 個座標！

新長度 = √(0.029² + 0.017² + 0.046² + ... + 0.012²) = 1   ✅
```

**1024 個數字一個都沒少**，只是每個都被同一個倍率（÷17.3）縮小了。

## 為什麼「長度 1」≠「維度少」

| 操作 | 影響 | 不影響 |
|---|---|---|
| Normalize | 每個座標的數值大小（變小） | 座標的**個數**（永遠 1024） |
| 換成不同 model（如 MiniLM-384） | 座標的**個數**（變成 384） | — |

換句話說：
- **想改維度** → 換 model（bge-m3=1024、MiniLM=384、OpenAI ada-002=1536）
- **想改長度** → normalize（把所有座標等比縮放）

## 視覺化（2D）

```
原向量 (3, 4) 長度 5：

  4 ┤      ●─── (3, 4)
    │    ╱
  3 ┤  ╱
    │╱
  0 ●──────────→
    0  1  2  3

        ↓ normalize ↓

規一化後 (0.6, 0.8) 長度 1：

  4 ┤
    │
  3 ┤
    │      ●─── (3, 4)（原向量，灰色）
  1 ┤   ●──── (0.6, 0.8)
    │ ╱方向完全一樣
  0 ●──────────→
    0  1  2  3
```

**箭頭縮短了，但指的方向完全一樣。** 維度（兩個座標 x, y）也沒變。

## 一句話總結

> Normalize 只是把箭等比縮短到「長度 1」，**1024 個座標的個數一個都沒少**。維度由 model 決定（換 model 才會變），規一化動的是長度，不是維度。

## 教學時間 — Task 9 在做什麼？

### 一句話：把前面三個工具串成一條輸送帶

```
.md 檔案
  ↓ ① 讀檔 + 算 MD5（指紋）
  ↓ ② 跟 DB 裡的舊指紋比對 → 沒變就跳過
  ↓ ③ chunker.chunk_markdown()    ← Task 6
  ↓ ④ embedder.encode([每個 chunk]) ← Task 7（耗時主因）
  ↓ ⑤ db.insert_chunks_batch()    ← Task 8
DB 裡多了 N 筆 (chunk + 1024 維向量)
```

### 三個關鍵設計

**① 兩種模式**

| 模式 | 指令 | 行為 |
|---|---|---|
| Incremental（預設） | `python scripts\ingest.py` | 只處理「指紋變了」的檔案 |
| Full（強制重跑） | `--full` | 先 `TRUNCATE chunks` 清空，全部重新 embed |
| Dry-run | `--dry-run` | 走流程但不真的寫 DB，純報數 |

我們現在跑的是 `--full`。

**② MD5 指紋當「省力護身符」**

```python
def md5_of(path: Path) -> str:
    return hashlib.md5(path.read_bytes()).hexdigest()
```

每個檔案算一次 MD5（128 位 hash）。下次 ingest 時：

```python
if existing_hashes.get(rel) == new_hash:
    skipped += 1
    continue
```

—— 沒改的檔案完全不碰，連 chunker 都不用跑。
411 篇筆記裡可能只改了 3 篇 → 下次只處理那 3 篇 → 5 秒完事。

> 為什麼用 MD5 而不是 modified time？檔案被 git pull / 複製貼上時 mtime 會變但內容沒變，MD5 才是真正反映「內容有沒有動」的指標。

**③ 「先刪再插」處理檔案改寫**

```python
db.delete_file_chunks(rel)
embeddings = embedder.encode([c.content for c in chunks])
db.insert_chunks_batch(rows)
```

如果你改了一篇筆記，可能 chunk 數從 5 變成 7。如果單純 update 會錯位。所以策略是：**先把這個檔案的所有舊 chunks 刪光，再把新 chunks 整批插入**。

> 為什麼這樣安全：每個 chunk 在 DB 裡用 `(file_path, chunk_index)` 當 unique key，重新插入時 chunk_index 從 0 重新開始算，乾淨。

embedder 在跑 60 個 chunks 的 batch 時，CPU 全力推論 ~75 秒。這 75 秒裡 psycopg2 的 socket 完全沒講話 → Docker 的 NAT 層判定它「死掉了」就斷它。

### 為什麼 keepalive 沒救？

我先加了 `keepalives_idle=30`（30 秒沒講話就送 keepalive）。理論上應該有用，但 Docker Desktop on Windows 的網路層**不一定把 keepalive 封包 forward 到容器**，所以等於沒設。已知問題，社群討論多次。

### 真正救活 ingest 的是 retry

```python
try:
    insert_batch(rows)
except OperationalError:
    reconnect()
    insert_batch(rows)   # retry once
```

關鍵原理：
- 連線斷了，**這次操作失敗 ≠ DB 壞了**
- 重新 `psycopg2.connect()` 就拿到一條新鮮的 socket
- 同樣的 INSERT 重跑就成功

這是分散式/網路程式的常識：**transient failure 要會 retry，不是吐到使用者臉上**。

### 為什麼 retry 一次就夠（不需要 retry N 次）

| 連線斷掉的原因 | retry 結果 |
|---|---|
| TCP 被中介層 reset（這次的情況）| 重連 → 成功 ✅ |
| DB 真的當機重啟 | 重連也會失敗 → 應該 fail loud ✅ |

retry 無限次只會在 DB 真的死掉時把錯誤訊息延後好幾分鐘才看到，對 debug 沒幫助。

### Postgres 「recovery mode」是什麼？

```

## Q1: WAL (Write-Ahead Log) 是什麼

**「先記帳，再做事」的策略。**

Postgres 收到 `INSERT INTO chunks ...` 時，**不會直接寫進資料表檔案**。順序是：

```
1. 寫一行到 WAL：「我打算把這 N 個 chunks 寫進 file_path=X」
2. fsync() 強迫 OS 把這行寫進硬碟（不能只留在記憶體 buffer）
3. 回應 client「OK，搞定」
4. 之後找空檔再慢慢把資料真正寫進資料表檔案
```

為什麼這樣設計？

| 沒有 WAL | 有 WAL |
|---|---|
| 寫資料表 = 改硬碟上幾百個地方（B-tree 節點、index、heap...） | 寫資料表 = 只 append 一行到 WAL 檔尾（連續寫入超快） |
| 中途斷電 → 資料表半好半壞，要怎麼修？ | 斷電 → 開機重讀 WAL，「上次說要做的，沒做完的我幫你做完」 |

WAL 就是資料庫的**黑盒子飛航紀錄器**。

實體位置：`/var/lib/postgresql/data/pg_wal/` 裡的一堆 16MB 檔案。

---

## Q2: SIGTERM 是什麼

**「終止訊號」(Signal: Terminate)。** Unix/Linux 的標準關機請求。

Process 之間溝通的訊號代號（部分常見的）：

| 訊號 | 號碼 | 意思 | Process 可不可以拒絕？ |
|---|---|---|---|
| `SIGTERM` | 15 | 「請關閉」（禮貌請求） | 可以攔截、清理後再退出 |
| `SIGINT` | 2 | Ctrl+C | 可以攔截 |
| `SIGKILL` | 9 | 強制砍掉 | **不能拒絕**，OS 直接殺 |
| `SIGHUP` | 1 | 終端關閉 | 可攔截 |

`docker stop <container>` 預設就是先送 SIGTERM 給容器內的主 process，等 10 秒沒退出再升級到 SIGKILL。Postgres 收到 SIGTERM 後會：
1. 拒絕新連線
2. 等現有連線結束（或關掉它們）
3. checkpoint（把記憶體 buffer 寫回硬碟）
4. 安全關機

如果是 SIGKILL 或斷電 → 沒機會做 step 3 → 開機要靠 WAL 恢復。

## Q3: Transaction rollback 包括哪些？

**只包括資料庫內的操作。** 不包括外部副作用。

Transaction（交易）= 「一組要嘛全成功、要嘛全失敗」的操作。例子：

```sql
BEGIN;
  INSERT INTO chunks VALUES (...);
  UPDATE files SET ingested_at = NOW();
  DELETE FROM staging WHERE ...;
COMMIT;  -- ← 沒走到這行，前面三個都會被 rollback
```

**Rollback 會 undo 的：**
- INSERT/UPDATE/DELETE（資料表內容）
- DDL（CREATE TABLE、ALTER TABLE — Postgres 連 schema 改動都能 rollback）
- 索引修改
- pgvector 的向量寫入

**Rollback 不會 undo 的：**
- 已經透過 trigger 發送的 HTTP 請求
- 已經發出去的 email
- 寫到檔案系統的檔案
- 你 print 到 console 的文字
- 序號（sequence）的消耗（這已經被打掉的編號，rollback 不會還給你）

> 例子：你的 Python 程式跑完 `INSERT` 之後，再呼叫 `slack.send_message("done")`。如果 transaction rollback 了，slack 訊息**已經發出去了**，但資料庫沒寫入 → 不一致。

這就是「分散式 transaction 很難」的根源。

## Q5: 為什麼「只讀」= recovery mode？

**因為 recovery 期間，資料是「不確定狀態」，寫入會危險。**

Postgres 重啟時 recovery 流程：

```
時間 0:  容器啟動
時間 1:  Postgres 打開資料檔案 → 「上次有沒有乾淨關閉？」
時間 2:  讀 WAL 從 last checkpoint 開始
時間 3:  REDO：把已 commit 的 transaction 重做一次
         「啊，WAL 說 ID=12345 的 INSERT 已 commit 但資料檔案沒寫到」→ 補寫
時間 4:  UNDO：把沒 commit 的 transaction 清掉
         「ID=12346 的 INSERT 開始了但 WAL 沒看到 COMMIT」→ 丟棄
時間 5:  到達 consistent state（所有資料一致了）
時間 6:  開放寫入
```

**Step 2-5 期間：**
- 資料表還在被修改（系統正在「補完」舊操作）
- 如果你這時送一個新 INSERT 進來：
  - 它的 ID 可能跟某個正在被 UNDO 的撞號
  - 它寫進去的位置可能被 REDO 覆蓋掉
  - WAL 順序錯亂 → 下次再 crash 救不回來
- 安全做法：**全部寫入請求一律拒絕**，只允許 `SELECT`

所以 Postgres 在 step 2-5 時對 client 說：

```
FATAL: the database system is in recovery mode
```

= 「我還在整理東西，現在問什麼我都不答、給什麼我都不收，等我幾秒。」

這狀態 Postgres 內部叫 **hot standby mode**（也就是高可用主從架構中 standby 節點的常駐狀態），單機重啟時是一個短暫過渡。

### Q3: 「主 process」怎麼定義？

**容器啟動時 Docker 跑的那個 process（PID 1）。**

每個 Docker 容器只有一個「主」process — 它是 Dockerfile 裡 `CMD` 或 `ENTRYPOINT` 指定的那個。容器的生死跟著它走：

```
abby-rag-postgres 容器
├─ PID 1: postgres (主 process) ← 它死，容器就死
   ├─ PID 27: postgres: checkpointer
   ├─ PID 28: postgres: background writer
   ├─ PID 29: postgres: walwriter
   └─ PID 50+: postgres: <client1>  ← 每個連線一個
```

Postgres 本身就是 multi-process 架構，但對 Docker 來說只看 PID 1。所以 `docker stop` 送 SIGTERM 給的就是它。

> 對照：傳統 Linux server 上跑 Postgres 是 systemd 在管，不靠 PID 1 概念。

### Q4: Checkpoint 怎麼解釋？

**「把記憶體裡欠的帳，一次寫回硬碟。」**

複習一下流程：
- INSERT 進來 → 先寫 WAL（小、快、append-only）→ 資料先進 **記憶體 buffer**
- 真正的資料表檔案還沒更新

問題：記憶體會越欠越多。所以 Postgres 每隔一段時間（預設 5 分鐘）做一次 **checkpoint**：

```
checkpoint = 把記憶體 buffer 裡所有「髒頁面」（被改過還沒寫盤的）
            一口氣 flush 到硬碟上的資料表檔案
            然後在 WAL 裡標記「到這裡的東西都已經安全落地了」
```

好處：下次 crash 時，recovery 只要從**最後一次 checkpoint** 開始讀 WAL，不用回放整部歷史。

> 比喻：你打麻將打了一整天，每個牌局結算先寫在小白板（WAL+buffer）。等到吃飯休息（checkpoint）才把總帳謄到帳本（資料表檔案）。隔天就算白板被擦掉，帳本還在。

## 21:01 — 解釋corpus

**Claude:**

**Corpus = 「文本庫」**，是 NLP / 資訊檢索領域的標準術語。

### 字面

拉丁文「body」（身體、整體）。在 NLP 指**一整套用來研究/處理的文本集合**。

### 我們專案裡的 corpus

```
Abby-notes/  ← 這就是「我的個人筆記 corpus」
├── 373 個 .md 檔
├── 切成 4172 個 chunks
└── 都進了 pgvector
```

當我說「不在 corpus 裡」= 「不在 ingest 過的那 373 篇筆記裡」。例如 `MEMORY.md` 住在 `C:\Users\User\.claude\`，那是另一個地方，不屬於這個 corpus。

### 常見用法

| 句子 | 意思 |
|---|---|
| 「擴充 corpus」 | 加更多文件進去 |
| 「在 corpus 上 train model」 | 拿這批文字訓練 ML model |
| 「corpus size 1 億 tokens」 | 整個文本庫的詞數 |
| 「out-of-corpus」 | 沒在這批資料裡的東西 |

### 跟其他類似詞的差別

| 詞 | 用法 |
|---|---|
| **Corpus** | NLP / RAG 慣用，指「拿來處理的整批文本」 |
| **Dataset** | ML 通用，可以是文字、圖片、表格任何東西 |
| **Knowledge base** | 偏資料庫角度，強調「可被查詢的知識集合」 |
| **Document collection** | 偏資訊檢索（IR）術語 |

在 RAG 語境下這四個常常互換使用。我習慣用 corpus 因為它最精準指向「文字內容的整體」這個意思。
