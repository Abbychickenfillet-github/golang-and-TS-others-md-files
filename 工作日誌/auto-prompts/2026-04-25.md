## 12:28 — 有沒有cached query 這東西

**Claude:**

有，但「cached query」不是一個正式術語，實務上會拆成**三層不同的快取**，每層解決的問題不一樣：

## 三層 query 相關的快取

```
使用者 query「退款怎麼辦」
    │
    ├─【層1】Semantic Cache（語意快取）── 這最接近你說的「cached query」
    │   用 embedding 比對過去的 query，相似就直接回舊答案（不打 LLM）
    │
    ├─【層2】Prompt Caching（廠商原生）
    │   快取 prompt 的「前綴」（system + 文件），query 本身幾乎不快取
    │
    └─【層3】KV Cache（模型內部）
        attention 層的 key/value 向量快取，推論時用的底層機制
```

---

## 層 1：Semantic Cache —— 最常被稱為「cached query」的東西

### 核心概念

> 使用者問「退款怎麼辦」→ 之前有人問過「我要怎麼退款」→ **語意相似度 > 0.95 → 直接回傳舊答案，跳過 LLM**

### 流程

```
新 query 進來
    │
    ▼
把 query embed 成向量
    │
    ▼
去 cache 裡找語意最近的舊 query
    │
    ├─ 相似度 > 閾值 ──► 直接回傳舊答案（~10ms）
    │
    └─ 相似度 < 閾值 ──► 打 LLM、算完後把新 query + 答案存進 cache
```

### 代表實作
- **GPTCache**（開源）— 最經典的一套
- **Redis Semantic Cache**（Redis 官方模組，用 RediSearch + 向量）
- **LangChain** 的 `RedisSemanticCache` / `GPTCache` wrapper
- **Portkey / Helicone** 這類 LLM gateway 大多內建這功能

### 適合 / 不適合

| ✅ 適合 | ❌ 不適合 |
|---|---|
| 客服 FAQ（同一批問題被問到爛） | 有個人化狀態的 query（「我的訂單狀態」不能共用答案）|
| 文件問答（答案不常變）| 需要即時資料（股價、庫存）|
| 高頻重複的查詢 | 創意生成（每次都要新的）|

### 最大的坑

**語意相似 ≠ 答案相同**。例子：

```
Q1: 「退款政策是什麼？」    → 答案 A
Q2: 「退貨政策是什麼？」    → 答案應該是 B，但語意相似度 > 0.9 會誤命中 A
```

所以閾值調太低會亂答、調太高幾乎不命中。實務上會：
- 加**關鍵字過濾**（「退款」「退貨」不同關鍵字就不命中）
- 加 **user_id 隔離**（abby 的 query 不能命中 bob 的答案）

---

## 層 2：Prompt Caching —— 不是快取 query 本身

這就是上一輪講的那個。**快取的是 prompt 前綴**（system + 文件），不是 query：

```
[system + 10頁文件] [使用者 query]
 └──── 這段快取 ────┘   └── 這段每次都算 ──┘
```

為什麼 query 本身不快取？因為**每次使用者問的都不同**，快取沒意義。（除非你同時也套上面的 Semantic Cache。）

---

## 層 3：KV Cache —— 模型內部的快取，不是應用層

LLM 推論時，每個 token 經過 attention 會算一組 Key/Value 向量。**算過的 token 就不再重算**——這是模型**內建**的機制，你不用管。

- 對開發者**透明**（API 不暴露）
- Prompt Caching 底層其實就是把這層 KV Cache 做**跨 request 持久化**

---

## 實務上的疊加用法

一個成熟的 LLM 產品通常**三層一起用**：

```
使用者 query
    │
    ▼
【Semantic Cache】命中 ? ──► 直接回答（最便宜）
    │ miss
    ▼
【Prompt Cache】前綴命中 ? ──► 省 90% token 費用、快 85%
    │ miss
    ▼
【KV Cache】模型內部自動處理
    │
    ▼
LLM 生成答案 → 存回 Semantic Cache
```

## 一句話總結

- **你想問的「cached query」，八九成是指 Semantic Cache**——用 embedding 比對過去 query 去跳過 LLM
- 和 Prompt Caching **不同層**：Semantic Cache 在**應用層**（跳過整個 LLM call），Prompt Caching 在**LLM 呼叫內部**（省 token 錢）
- **RAG + Semantic Cache + Prompt Caching** 是生產級 LLM 產品的標準三件套

## 13:31 — 為何 5m cache token 比 base input tokens還要多呢

**Claude:**

好問題，這個價格差反直覺但有道理。

## 為什麼 cache write 比 base input 貴 25%？

先拆開「base input 1x」和「5m cache write 1.25x」到底在付什麼錢：

| 項目 | Base Input (1x) | 5m Cache Write (1.25x) | 差在哪 |
|---|---|---|---|
| **Prefill 算力**（跑 attention 算 K/V） | ✅ 要算 | ✅ 要算 | **一樣** |
| **算完之後的 K/V** | 丟掉 | **留在 VRAM 5 分鐘** | 多出這段 |
| **佔用的 GPU 記憶體** | 瞬間釋放 | **持續佔 5 分鐘** | 多出 5 分鐘租金 |

**關鍵洞察**：**寫入 cache 並沒有省任何算力**，prefill 一樣要跑一次。多付的那 25%，**買的不是 token 錢，是「VRAM 佔用 5 分鐘」的租金**。

## 為什麼 VRAM 要收錢？

回到 KV Cache 筆記講的那件事——**GPU VRAM 是 LLM infra 最貴的資源**：

- Claude 的 context 很長（200K tokens 可以佔幾十 GB VRAM）
- 你把 KV 留在那邊 5 分鐘 → 這塊 VRAM 這 5 分鐘不能服務別人
- Anthropic 的 GPU 容量 = 能同時服務多少使用者 × batch size
- 你獨佔了 VRAM → 整體吞吐量下降 → 這是**實打實的機會成本**

所以 1.25x 裡面的**多出 0.25x**，本質是**補償 Anthropic 這 5 分鐘的 VRAM 機會成本**。

## 讀取為什麼反而便宜到 0.1x？

讀取只是「從 cache 把已經算好的 K/V 搬進 attention 層」——**prefill 完全跳過**。Prefill 是 LLM 最吃算力的那段（尤其長 prompt），省掉它就幾乎沒算力成本，只剩記憶體存取，所以收 10% 就夠 cover。

## 損益平衡點：用幾次才划算？

假設同一份前綴要重用 N 次：

```
不用快取的總成本：N × 1x = N
用快取的總成本：  1.25x（第一次寫入）+ (N-1) × 0.1x（之後讀取）
                = 1.15 + 0.1N
```

令兩者相等：N = 1.15 + 0.1N → **N ≈ 1.28**

**結論**：**只要同份前綴 5 分鐘內被用超過 2 次，就划算**。用越多次賺越多：

| 使用次數 | 無快取成本 | 有快取成本 | 省多少 |
|---|---|---|---|
| 1 次 | 1x | 1.25x | **虧 25%**（唯一會虧的情況）|
| 2 次 | 2x | 1.35x | 省 32% |
| 10 次 | 10x | 2.15x | 省 78% |
| 100 次 | 100x | 11.15x | 省 89% |

## 所以 1h cache write 收 2x 也合理

Anthropic 還有一個 1 小時 TTL 版本，收 **2x**（100% 加價）——因為 VRAM 要佔用 12 倍久（5 分鐘 → 60 分鐘），機會成本更高。但平均下來也只有用超過幾次就划算。

## 設計動機

這個價格結構其實是**在逼你做對的事**：

1. **鼓勵「真的會重用才快取」**——只用一次會虧，強迫你想清楚這段 prompt 會不會被重用
2. **防止濫用**——不然大家什麼都標 `cache_control`，Anthropic 的 VRAM 會被垃圾 cache 塞爆
3. **補償 infra 成本**——VRAM 真的是錢

## 一句話總結

> **Cache write 的 25% 加價，是 VRAM 的 5 分鐘租金，不是 token 費。設計上就是「用 1 次會虧、用 2 次打平、用 3 次以上開始賺」**——強迫開發者只在真的重用場景才啟用 caching。

## 問題 2：KV 是什麼的縮寫？

- **K = Key（鍵）**
- **V = Value（值）**

連同第三個 **Q = Query（查詢）**，合稱 **QKV**，是 Transformer attention 層的三個向量。

### ⚠️ 術語撞車警告（重要！）

你應該已經發現：**「query」這個字在 LLM 世界有兩個完全不同的意思**：

| 「query」 | 出現在哪 | 意思 |
|---|---|---|
| **使用者 query** | RAG / 產品端 | 使用者打進去的**查詢語句**（「退款怎麼辦？」）|
| **Attention Query (Q)** | Transformer 內部 | attention 層算出來的一種**向量** |

**兩個根本不是同一件事，只是名字撞車**——這也是為什麼很多人看 Transformer 論文會一頭霧水。下面回答問題 1 時還會再碰到這個撞車。

---

## 問題 1：query embed 後會被存進向量資料庫嗎？

**不會**。這是超常見誤解，讓我拆清楚。

### RAG 的兩個階段用 embedding 的方式**不一樣**

```
【階段 A】Indexing（建庫，一次性 / 批次進行）
─────────────────────────────────────────
   文件 → chunk → embed → 【存進】向量資料庫
                              ↑ 永久保存

【階段 B】Retrieval（每次使用者問問題時）
─────────────────────────────────────────
   使用者 query → embed → 【臨時向量】 → 拿去跟資料庫裡的向量算相似度 → 取 top-k
                              ↑ 用完就丟、不存
```

### 視覺化

```
向量資料庫裡面（只有文件的向量）：
┌──────────────────────────────────────────┐
│ chunk_001: [0.12, 0.45, ...]  來自文件A │
│ chunk_002: [0.78, 0.33, ...]  來自文件A │
│ chunk_003: [0.56, 0.91, ...]  來自文件B │
│ chunk_004: [0.23, 0.67, ...]  來自文件B │
│ ... (成千上萬筆)                        │
└──────────────────────────────────────────┘
         ↑ 這些是「被索引」的東西

使用者 query「退款怎麼辦？」
    ↓ embed (臨時)
[0.33, 0.88, 0.12, ...]  ← 一個用後即丟的向量
    ↓ 拿去跟上面每一筆算 cosine similarity
找出相似度 top-3：chunk_002 (0.91), chunk_007 (0.87), ...
    ↓
把那 3 個 chunk 的原文塞進 prompt
    ↓
query 向量的任務結束 → 丟掉
```

### 兩種 embedding 的差別整理

| | Indexing 時的 embedding | Retrieval 時的 embedding |
|---|---|---|
| **對象** | 文件 chunks | 使用者 query |
| **時機** | 建庫（一次性） | 每次 query 進來（即時） |
| **會被儲存嗎？** | ✅ **存進向量資料庫** | ❌ **用完就丟** |
| **目的** | 讓未來能被搜到 | 當搜尋的「指針」去比對 |
| **生命週期** | 永久（直到資料更新才重 embed） | 毫秒級 |

### 所以「問題也會被索引嗎？」的正解

**預設不會**。向量資料庫只索引文件，**query 只是臨時拿來比對的向量**，比完就消失。

### 例外：Semantic Cache 會存 query 向量

還記得 `llm-caching-layers.md` §1 的 Semantic Cache 嗎？**那層快取才會真的存 query 向量**——但它是**另一個資料庫**（快取庫），不是原本的文件向量庫：

```
文件向量資料庫            Semantic Cache 資料庫（另外一個！）
─────────────            ────────────────────────
只存文件 chunks          存「過去的 query → 對應答案」
的向量                   的 query embedding + 答案
```

所以如果系統有開 Semantic Cache，**query 確實會被 embed 並存起來**，但這不是 RAG retrieval 的動作，是另一層快取機制。

# git reset HEAD~1 對已 push commit 有用嗎？

## 答案：本地有用、遠端沒用

`git reset` **只動你的本地** HEAD 指標，**碰不到遠端分支**。

### 實際會發生什麼

```
操作前：
  本地 HEAD ──► C（最新 commit）
  遠端 origin ──► C
  （兩邊同步）

執行 git reset HEAD~1 後：
  本地 HEAD ──► B（退一步）
  遠端 origin ──► C（沒動！）
  （本地和遠端分歧了）
```

這時候你會遇到：

| 動作 | 結果 |
|---|---|
| `git pull` | 遠端的 C 又被拉回來，等於白 reset |
| `git push` | **被拒絕**（error: failed to push some refs）—— 因為這是 non-fast-forward |
| `git status` | 顯示「本地落後遠端 1 個 commit」 |

---

## 要讓遠端也跟著退一步 → 必須 force push

### 兩種 force push

```bash
# 危險版：直接覆蓋遠端
git push --force

# 安全版：如果遠端在你 fetch 後被別人動過就拒絕
git push --force-with-lease
```

**一律用 `--force-with-lease`**——它會檢查「我最後看到的遠端是不是還是那個樣子」，如果別人剛好推了東西上去，你會被擋下來而不是蓋掉同事的 commit。

### Force push 的三大風險

1. **遠端歷史被改寫**——原本的 commit C 從遠端消失
2. **拉過 C 的人會爆炸**——他們的本地 git 跟遠端對不起來，要手動 rebase
3. **如果 CI/部署有綁 commit hash，會壞掉**

### 什麼情況**絕對不能** force push

| 分支 | 能 force push 嗎 |
|---|---|
| `main` / `master` | ❌ **絕對不行**（公司分支，會害到全隊）|
| `develop` / `stage` | ❌ **不行**（共用分支）|
| 已開 PR 且被 review 過的 feature 分支 | ⚠️ 最好先跟 reviewer 講 |
| **只有你自己用的 feature 分支、沒人拉過** | ✅ 可以，這是它的正確使用情境 |

---

## 更安全的替代方案：`git revert`

如果要退掉的 commit 已經 push 到**團隊分支**，正確做法是 `git revert`：

```bash
git revert HEAD            # 產生一個新 commit 去「抵消」上一個 commit
git push                   # 正常 push，不用 force
```

### reset vs revert 差在哪

```
原本的歷史：   A ── B ── C（要退掉 C）

git reset HEAD~1（需要 force push）：
    A ── B                    ← 歷史被改寫、C 消失

git revert HEAD（正常 push）：
    A ── B ── C ── C'         ← 新增一個 C'，C' 把 C 做的事反過來
                               C 仍然在歷史裡，只是效果被抵消
```

| | `git reset` + force push | `git revert` |
|---|---|---|
| 改寫歷史 | ✅ 改寫 | ❌ 不改，加新 commit |
| 需要 force push | ✅ 需要 | ❌ 不用 |
| 對團隊友善 | ❌ 會害到別人 | ✅ 完全安全 |
| 適用情境 | 你自己的分支、沒人拉過 | 已 push 到團隊分支 |

---

## 決策樹

```
你要退掉的 commit 已經 push 出去了？
    │
    ├─ 沒有 ──► git reset HEAD~1 即可（單純改本地）
    │
    └─ 有 ──► 這個分支有別人用嗎？
              │

```

echo "path/to/unwanted-folder/" >> .gitignore

git add .gitignore
git commit --amend --no-edit

git push --force-with-lease origin <你的分支名>

git rm --cached -r path/to/unwanted-folder/

# 2. 加進 .gitignore

git commit -m "chore: untrack files mistakenly added in previous commit"

## 19:28 — git reset HEAD~1 沒寫SOFT會發生什麼事

**Claude:**

不寫 `--soft` = **預設是 `--mixed`**，差別在**哪一層會被一起退**。

## git reset 三種模式（要搞懂 git 的三個區域才懂）

先認識 git 的三層結構：

```
┌──────────────────────────────────────────┐
│ HEAD / Commit history （提交歷史）        │  ← 已經 commit 的
├──────────────────────────────────────────┤
│ Index / Staging area （暫存區）          │  ← git add 之後、還沒 commit
├──────────────────────────────────────────┤
│ Working tree （工作目錄）                 │  ← 你電腦上實際的檔案
└──────────────────────────────────────────┘
```

### 三種 reset 模式對這三層做什麼

| 模式 | HEAD 退 | Index 清 | Working tree 清 | 一句話 |
|---|---|---|---|---|
| `--soft` | ✅ | ❌ | ❌ | 只退 commit，改動還 staged，可以直接 commit |
| **`--mixed`（預設）** | ✅ | ✅ | ❌ | 退 commit + unstage，檔案還在但要重新 add |
| `--hard` | ✅ | ✅ | ✅ | 三層都退，**改動徹底消失**（危險）|

---

## 視覺化：假設上一個 commit 改了 `foo.txt`，三種模式跑完的狀態

### `git reset --soft HEAD~1`

```
commit 歷史  ：退一步
Index        ：foo.txt 的改動【仍 staged】 ← 綠色
Working tree ：foo.txt 改動仍在

→ git status 顯示：
   Changes to be committed（已 staged）: modified foo.txt
   
→ 直接 git commit 就能做成新 commit（不用再 add）
```

**用途**：commit 訊息寫錯、想拆/合 commit——**內容不動，只重做 commit**。

---

### `git reset HEAD~1`（= `--mixed`，預設）

```
commit 歷史  ：退一步
Index        ：清空 ← 回到 B 的狀態
Working tree ：foo.txt 改動仍在

→ git status 顯示：
   Changes not staged for commit（紅色）: modified foo.txt
   
→ 要先 git add，才能再 git commit
```

**用途**：想**重新挑要 commit 什麼**——改動都還在，但你要重新決定要 add 哪些。這是**最常用**的 reset 模式，所以被設為預設。

---

### `git reset --hard HEAD~1`

```
commit 歷史  ：退一步
Index        ：清空
Working tree ：foo.txt 也被還原成 B 的樣子

→ git status 顯示：
   nothing to commit, working tree clean
   
→ 你做的改動徹底消失（除了 reflog 還救得回來 commit 本身，
   但如果改動從沒 commit 過就真的沒了）
```

**用途**：**徹底丟棄最近的改動**。⚠️ **不會先問你、直接毀掉 working tree**。用前請深呼吸。

---

```bash
git reset --soft HEAD~1    # 退 commit 但保留 staged
git reset                  # 把所有東西 unstage
```

其實這兩步合起來 = `git reset --mixed HEAD~1` = **`git reset HEAD~1`**（預設就是 mixed）。

所以方案 C 的前兩行可以簡化成一行：

```bash
git reset HEAD~1           # 預設 mixed：commit 退、unstaged、檔案還在
git add <你要的檔案>
git commit -m "..."
git push --force-with-lease
```

---

## 誤用 `--hard` 救援：`git reflog`

如果你不小心 `git reset --hard` 把 commit 搞丟了，還有一線生機：

```bash
git reflog                 # 看所有 HEAD 動過的記錄（包含被 reset 掉的）
# 找到你要救的 commit hash，例如 a7f3c9e
git reset --hard a7f3c9e   # 跳回去
```

**但要注意**：
- **reflog 只救 commit**——被 reset 掉的 commit 可以回來
- **reflog 救不回「從未 commit 的改動」**——如果你改了 code 但沒 commit 就 `--hard`，那些改動**永遠消失**
- reflog 預設保留 90 天，之後被 GC 就真的沒了

---

## 記憶口訣

| 情境 | 用哪個 |
|---|---|
| 只想改 commit 訊息、不動內容 | `--soft` |
| 想重新挑/重新組改動 | `--mixed`（= 不帶參數）|
| 確定要丟掉全部最近改動 | `--hard`（先 stash 或 backup）|

**最安全的習慣**：`--hard` 之前先 `git stash` 一下，把 working tree 先存起來當保險。

|---|---|---|
| **只想改 commit 訊息、內容完全不動** | `--soft` | mixed 要重新 `git add .` 把全部加回來，多此一舉 |

# NACL 是什麼？

**NACL = Network Access Control List（網路存取控制清單）**，是 AWS 在 **Subnet 層級**的防火牆。

對應到你早上學的概念——你的 VPC 裡面每個 Subnet **邊界**都會掛一個 NACL，管制「什麼樣的流量可以進出這個 Subnet」。

---

## NACL 在 AWS 網路層的位置

```
    Internet / 其他 VPC
         │
         ▼
┌────────────────────────────────────┐
│  VPC                                │
│  ┌──────────────────────────────┐  │
│  │  Subnet                       │  │
│  │   ↓↑                          │  │
│  │  【NACL】← Subnet 邊界防火牆   │  │  ← 層 1 防線
│  │   ↓↑                          │  │
│  │  ┌─────────────────────────┐ │  │
│  │  │  EC2                    │ │  │
│  │  │  【Security Group】     │ │  │  ← 層 2 防線
│  │  │   ↓                     │ │  │
│  │  │  實際機器                │ │  │
│  │  └─────────────────────────┘ │  │
│  └──────────────────────────────┘  │
└────────────────────────────────────┘
```

**流量要穿過兩層才能到 EC2**：先過 NACL（Subnet 邊界）→ 再過 Security Group（機器本身）。

---

## NACL vs Security Group —— 最常搞混的比較

| 維度 | **NACL** | **Security Group** |
|---|---|---|
| **作用層級** | **Subnet** | **EC2 / ENI**（每台機器）|
| **狀態** | **Stateless**（無狀態）| **Stateful**（有狀態）|
| **規則類型** | ✅ Allow + ✅ **Deny** | ✅ Allow only（只能允許，不能禁止）|
| **規則判斷順序** | **按編號由小到大**，第一條符合就生效 | **所有規則一起看**，只要有一條允許就放行 |
| **預設行為** | 預設全 Deny（自訂 NACL）/ 全 Allow（預設 NACL）| 預設全 Deny（要自己加 allow 規則）|
| **比喻** | Subnet 入口的**警衛** | 機器本身的**門鎖** |

---

## Stateless vs Stateful —— 最重要的差別

這個差別是 NACL 最容易踩雷的點。

### Stateful（Security Group）—— 有記憶

```
客戶端 ──allow入站80──► EC2
EC2    ──回應────────► 客戶端   ← 自動放行，不用設出站規則
                              （因為 SG 記得「這是剛才放進來的請求的回應」）
```

**只要設入站 allow，回應的出站自動放行**。

### Stateless（NACL）—— 沒記憶

```
客戶端 ──allow入站80──► Subnet  ← 要設入站 allow
Subnet ──回應────────► 客戶端  ← ❌ 被擋！必須另外設出站 allow
```

**入站和出站要分別設**。回應的封包 NACL 當作「全新流量」處理，沒設出站規則就會被擋。

這就是為什麼 NACL 比 SG 難設定——你得**兩邊都想清楚**。

---

## 什麼時候會用到 NACL？

大部分情況**只用 Security Group 就夠了**。NACL 出場時機：

1. **要明確 deny 某些 IP / 網段**（SG 做不到，只能 allow）
2. **整個 Subnet 的粗粒度防護**（例如 private subnet 直接禁止任何對外 traffic）
3. **合規要求**（HIPAA / PCI-DSS 需要雙層防護）
4. **封鎖已知攻擊 IP**（快速全 Subnet 生效）

日常開發 90% 用不到 NACL——**預設 NACL 是全 allow**，所以你沒動它也不會踩雷。

---

## 一句話記憶

> **Security Group 是機器的門鎖，NACL 是 Subnet 的警衛。SG 有記性（stateful）、NACL 沒記性（stateless）；SG 只會放行、NACL 還能明確禁止。**

## 關於「EC2 其實是 ENI」—— 他說得沒錯，我昨天畫錯了

### ENI = Elastic Network Interface（彈性網路介面）

**ENI 是虛擬網路卡**。你以為 EC2 有 IP、有 SG——**其實都綁在 ENI 上，不是 EC2 本身**。

### 修正版流程圖

我昨天畫的 NACL 流程圖裡，「EC2」那格**應該分成 ENI + 機器兩層**：

```
    Internet
       │
       ▼
┌─────────────────────────────────────┐
│  Subnet                              │
│  ┌───────────────────────────────┐  │
│  │  【NACL】← Subnet 邊界          │  │  (層 1：Subnet 邊界)
│  └──────────────┬────────────────┘  │
│                 ▼                    │
│  ┌───────────────────────────────┐  │
│  │  【ENI】 ← 真正持有 IP / MAC   │  │  (層 2：網卡)
│  │   【Security Group】 綁這裡！   │  │
│  └──────────────┬────────────────┘  │
│                 ▼                    │
│  ┌───────────────────────────────┐  │
│  │  EC2（OS + 你的程式）          │  │  (層 3：實際運算)
│  └───────────────────────────────┘  │
└──────────────────────────────────────┘
```

**SG 不是綁在 EC2 上、是綁在 ENI 上**——這是我昨天寫 NACL 筆記時簡化掉的細節。

### ENI 持有什麼？

| 屬性 | 屬於誰 |
|---|---|
| Private IP | ✅ **ENI** |
| Public IP / Elastic IP | ✅ **ENI** |
| MAC Address | ✅ **ENI** |
| Security Group | ✅ **ENI**（不是 EC2！）|
| 要放哪個 Subnet | ✅ **ENI**（決定這張網卡落在哪個 AZ）|
| CPU / RAM / 硬碟 | ✅ **EC2** |

**一句話**：**EC2 是算力、ENI 是網路身分**——兩者正交，可以獨立管理。

### 為什麼這個設計很重要？

1. **ENI 可以 detach / attach**——EC2 掛了，把 ENI 搬到備用機，**IP 不變、SG 不變**，前端完全感覺不到
2. **一台 EC2 可以多張 ENI**——例如一張對外（public）、一張對內（private），各綁不同 SG 各走不同 Subnet
3. **不只 EC2 會用 ENI**——Lambda、RDS、ECS Fargate 在 VPC 裡連線**都透過 ENI**。ENI 是**整個 VPC 的網路原子單位**

### 對昨天 NACL 筆記的影響

嚴格講，**「SG 在 EC2 層級」這句話不精確**，應該說「**SG 在 ENI 層級**」。但對日常使用者來說——一台 EC2 預設只有一張 primary ENI、SG 講在 EC2 上也不算錯太多。真正開始要講究的場景：

- 多網卡架構
- ENI 獨立移動（故障轉移）
- Lambda in VPC（沒有 EC2 但有 ENI）

這時候搞不清楚 EC2 vs ENI 就會卡住。

## 混合雲（Hybrid Cloud）— 一句話定義

**雲端（AWS / Azure / GCP）+ 地端（on-premise / 自己機房）同時運作、彼此互通**，把不同 workload 分散在兩邊各取所長。

---

## 優點（10 點，按重要性）

| # | 優點 | 為什麼 / 例子 |
|---|---|---|
| 1 | **漸進式上雲** | 老系統留地端、新系統放雲端，不用 big bang migration、風險可控 |
| 2 | **資料主權 / 合規** | 敏感資料留地端（金融、醫療、政府法規 e.g. PCI-DSS、HIPAA、金管會），一般運算放雲端 |
| 3 | **成本最佳化** | 穩定流量地端攤銷便宜、突發流量用雲端彈性擴展 |
| 4 | **Cloud Bursting（爆量擴展）** | 平常地端跑、峰值（黑五、雙 11、報稅日）多出來的流量導到雲端 |
| 5 | **災難復原（DR）** | 地端為主、雲端為備（或反過來），機房整個炸了還能切換 |
| 6 | **延遲分流** | 低延遲服務留地端（工廠 IoT、店面 POS），運算密集上雲端（AI 訓練、批次） |
| 7 | **避免廠商鎖定** | 不把所有雞蛋放 AWS 一個籃子，換廠商 / 斷雲時還有地端撐 |
| 8 | **既有投資保留** | 已買的伺服器、license 不浪費（沉沒成本不變廢鐵）|
| 9 | **IT 團隊技能過渡** | 既有運維團隊慢慢轉型 DevOps，不用一夕換腦 |
| 10 | **內網就近使用** | 公司內部使用者連地端只要幾 ms、對外服務走雲端 CDN |

---

## ⚠️ 也要知道的代價（混合雲不是萬能）

| 缺點 | 為什麼 |
|---|---|
| **架構複雜度暴增** | 兩套 ops、兩套 monitoring、兩套 IAM——人力成本變兩倍 |
| **網路打通難又貴** | VPN 慢、Direct Connect / ExpressRoute 一個月幾萬到幾十萬 |
| **安全模型分裂** | AWS IAM ≠ 地端 AD，要做身分整合（SAML / IAM Identity Center）|
| **資料同步延遲與一致性** | 兩邊資料庫怎麼同步？最終一致性還是同步寫入？容易踩坑 |
| **疑難排錯困難** | 問題出在哪？地端網路？AWS？中間 VPN？ — 排查路徑變長 |

## AWS 上的混合雲常用元件

| 元件 | 做什麼 |
|---|---|
| **AWS Direct Connect** | 拉專線到 AWS，繞過公網（高頻寬、低延遲、固定收費）|
| **Site-to-Site VPN** | 用公網建 IPSec 通道（便宜、上手快、頻寬不穩）|

| **AWS DataSync** | 地端 → 雲端的大量資料同步 |

**Claude:**

「一般資料庫沒有 realtime」這個說法不完全對——更精準的說法是：

> **一般資料庫有「變更感知」機制，但沒有「直接推送到前端」的功能。Supabase 的價值是把這條鏈路包好、做成一行 code 的 SDK。**

讓我拆清楚。

---

## 1. 傳統資料庫的 Realtime 能力分三個層級

### 層 1：完全沒有 → 只能 Polling（最笨）

```
前端每 1 秒問一次「有新資料嗎？」
    ↓
即使沒變化也打了 API、浪費頻寬
    ↓
不是 realtime，是「定時刷新」
```

### 層 2：有「變更感知」（但只給伺服器端用）

主流關聯式資料庫**其實都內建變更感知機制**，只是這些 API 是給**伺服器端**訂閱的、不是給前端：

| 資料庫 | 變更感知機制 |
|---|---|
| **PostgreSQL** | `LISTEN / NOTIFY`（內建 pub/sub）、**WAL**（Write-Ahead Log，所有變更紀錄）、Logical Replication |
| **MySQL** | **Binlog**（binary log，可串流出來）|
| **MongoDB** | **Change Streams**（最接近 realtime 的設計）|
| **Redis** | Pub/Sub、Keyspace Notifications |

**問題**：這些都不會直接推給前端——你還要自己架 **WebSocket server** + **訂閱資料庫變更** + **路由給對的使用者**。

### 層 3：原生 realtime to client（產品定位就是這個）

| 產品 | 怎麼做到 |
|---|---|
| **Firebase Firestore** | 整個產品 realtime first，client SDK 直接 subscribe |
| **Supabase** | PostgreSQL + Realtime Server 包好 |
| **PocketBase** | SQLite + 內建 WebSocket |
| **Hasura** | PostgreSQL + GraphQL Subscriptions |
| **AWS AppSync** | DynamoDB / Aurora + GraphQL Subscriptions |

---

## 2. Supabase Realtime 內幕——其實就是包裝 PostgreSQL 的 WAL

Supabase 沒發明新東西，**它把 PostgreSQL 早就有的功能串起來**：

```
你前端寫一行：
supabase.from('messages').on('INSERT', cb).subscribe()
                              ↓
                              ↓  Supabase 在背後做的事 ↓
                              ↓
PostgreSQL 寫入 INSERT
    ↓
WAL（Write-Ahead Log）紀錄這條變更
    ↓
Logical Replication slot 把 WAL 串流出來
    ↓
Supabase Realtime Server（Elixir / Phoenix 寫的）解析 WAL
    ↓
透過 WebSocket 推給訂閱了 'messages' 表的前端
    ↓
你前端的 callback 被觸發
```

**5 個層級，Supabase 全部代勞**。自架 PostgreSQL 要這樣做就要寫幾百行 infrastructure。

---

## 3. 自架 vs Supabase 的工程量對比

如果你想「自架 PostgreSQL + 自己做 realtime」要寫的東西：

| 要做的事 | 工程量 |
|---|---|
| 架 WebSocket server（Node.js / Go） | 幾百行 |
| 訂閱 WAL（用 `wal2json` 或 logical replication）| 進階 PG 知識 |
| Row-Level Security 整合（避免訂閱洩漏別人資料）| 安全坑很多 |
| Reconnection、heartbeat、斷線重訂閱 | 很煩 |
| Channel routing（誰能看哪個 table 哪一列）| 自己設計 |
| **Supabase 一行 code 全包** | `.on('INSERT', cb).subscribe()` |

這就是 BaaS（Backend as a Service）的賣點——**用工程時間換錢**。

---

## 4. 為什麼 Firebase / Supabase 都把這個當賣點？

因為**現代前端應用幾乎都需要 realtime**：

- 聊天訊息（不能用刷新看新訊息）
- 通知推送
- 多人協作（Notion、Figma 那種多游標）
- 即時儀表板（股票、訂單狀態）
- 多裝置同步

傳統做法（REST API + Polling）做這些**體驗很差**。Realtime DB 是 BaaS 的**差異化武器**，所以 Firebase 從第一天就把 realtime 當核心、Supabase 也跟進。

---

## 5. 一句話總結

> **PostgreSQL / MySQL 都有「變更感知」能力（WAL、Binlog），但只給伺服器端用。Supabase 的 Realtime 是把「PostgreSQL 變更感知 → WebSocket 推給前端」這條鏈路做成開箱即用的 SDK——不是創新功能，是降低工程門檻。**

# IAM = Identity and Access Management

**IAM 是 AWS 的權限管理系統**——回答一件事：**「誰（who）能對什麼資源（what）做哪些動作（how）」**。

## 1. 四個核心概念

```
        Policy（規則）
       /  附加在  \
      /            \
   User    Group    Role
   (人)   (一群人)  (可被假扮的身分)
```

| 概念 | 是什麼 | 例子 |
|---|---|---|
| **User** | 人類個人或服務帳號，**有長期憑證**（access key / 密碼）| `abby@company.com`、`ci-bot` |
| **Group** | 多個 User 的集合，方便集中管理權限 | `Developers`、`Admins` |
| **Role** | **可被「假扮」的身分**，無長期憑證、給臨時 token | `EC2-S3-Reader`、`Lambda-DB-Writer` |
| **Policy** | JSON 文件，定義「允許/拒絕什麼動作」 | 附在 User / Group / Role 上 |

⚠️ **Policy 自己不能單獨生效**——必須附加到 User / Group / Role 才算數。

---

## 2. Policy 長什麼樣（JSON 範例）

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::my-bucket/*"
    },
    {
      "Effect": "Deny",
      "Action": "s3:DeleteObject",
      "Resource": "*"
    }
  ]
}
```

四個欄位：

| 欄位 | 內容 |
|---|---|
| **Effect** | `Allow` 或 `Deny`（**Deny 永遠覆蓋 Allow**）|
| **Action** | 動詞，格式 `服務:動作`，如 `s3:GetObject`、`ec2:RunInstances` |
| **Resource** | ARN（Amazon Resource Name），如 `arn:aws:s3:::my-bucket/*` |
| **Condition** | 額外條件（可選），如「只有從特定 IP」、「必須開 MFA」|

---

## 3. User vs Role —— 最重要的對比

| 維度 | **User** | **Role** |
|---|---|---|
| 有長期憑證 | ✅ access key / 密碼 | ❌ 沒有 |
| 給誰用 | 人類 / CI bot | **AWS 服務 / 機器**（EC2、Lambda、ECS）|
| 怎麼取得權限 | 用憑證登入 | 透過 `sts:AssumeRole` 拿**臨時 token**（最多 12 小時）|
| 例子 | 員工帳號 | EC2 instance role 給 EC2 用來存 S3 |
| 安全風險 | 憑證洩漏 = 永久風險 | 沒長期 key 可洩漏 |

### ⭐ 為什麼推薦 EC2 用 Role 而不是塞 Access Key？

**舊作法（不安全）**：

```
EC2 上把 access key 存在 .env
  ↓
機器被入侵 → access key 永久洩漏
  ↓
換 key 要動所有機器（運維噩夢）
```

**Role 作法（推薦）**：

```
EC2 launch 時掛上 IAM Role
  ↓
AWS 自動發給 EC2 臨時 token（每小時自動換）
  ↓
沒長期 key 可洩漏，撤銷只要解除 role
```

---

## 4. IAM 在 AWS 安全模型的位置（呼應前面 AWS 筆記）

你之前學的網路層安全（NACL / SG）是**封包層**——管「TCP 封包能不能進來」。

**IAM 是 API 層的安全**——管「身分能不能呼叫 AWS API」：

```
攻擊者想存取你的 S3 bucket
    │
    ├─ 網路層：封包要先穿過 NACL / SG     ← 你之前學的
    │
    └─ API 層：呼叫 s3:GetObject 要 IAM 允許   ← IAM 管這裡
```

**兩層獨立、都要做**：
- 沒 IAM、只有網路防火牆 → 內網有人就能亂呼 API
- 沒網路防火牆、只有 IAM → 攻擊者能掃描到內部 endpoint

---

## 5. 安全 Best Practices（背起來）

| 原則 | 解釋 |
|---|---|
| **最小權限**（Least Privilege）| 只給必要權限、不要 `*:*` |
| **服務之間用 Role 不用 access key** | EC2、Lambda、ECS 都應該用 Role |
| **Group 管權限、User 只放在 Group 裡** | 權限變化動 Group 就好、不要直接附 Policy 給 User |
| **MFA 一定要開** | 至少根帳號和管理 user 必開 |
| **避免使用 root 帳號** | 建管理員 user、root 鎖在保險箱 |
| **定期 rotate access keys** | 不能用一輩子（90 天輪替是常見要求）|

---

## 6. 一句話總結

> **IAM 不管網路通不通，管「你拿什麼權限呼叫 AWS API」——是 AWS 安全模型的最頂層、和 NACL/SG 是**互補**而非替代關係。EC2 / Lambda / ECS 之間互相呼叫，用 Role 是最佳實踐。**

## 17:02 — 所以塞ACCESS KEY也是可以達成的做法只是不安全？

**Claude:**

**完全正確**——「塞 access key」是 **100% 可以達成、跑得動的做法**。功能完全 work、AWS API 照樣呼叫成功。

「能不能用」和「該不該用」是兩件不同的事：

| | 結果 |
|---|---|
| **能用嗎（功能上 work）** | ✅ 完全可以——程式照跑、API 照呼叫 |
| **該用嗎（安全敢不敢上 prod）** | ❌ 不建議——洩漏機率高 + 後果嚴重 |

## 「不安全」≠「一定會出事」

「不安全」是個**風險組合**，不是「會壞掉」：

```
洩漏機率（中高）  ×  洩漏後果（很嚴重）  =  期望損失（很大）
   ↑                       ↑
   .env 不小心 push、       永久有效後門、
   image 帶 key、            可能整個帳號被接管、
   員工離職還拿著           帳單炸幾千美金
```

很多開發者塞了 access key 一輩子沒出事——但**一旦出事就是大事**。

## 歷史脈絡：以前大家都這麼做

- **IAM Role 2012 年才推出**，之前唯一選項就是塞 access key
- 老專案、Stack Overflow 答案、舊教學文章**滿地都是「塞 access key」風格**
- AWS 自己的官方文件也只是寫**「Not recommended」**（不建議），不是「禁止」
- 一些情境**仍然合理**：地端伺服器呼叫 AWS、第三方 SaaS 要存你 S3、CI/CD 老系統還沒升級到 OIDC

## 一句話

> **塞 Access Key 不是「錯的做法」，是「不夠好的做法」**——能跑、簡單、上手快，但風險高過收益。**有 IAM Role 可用時就不要用 access key，這是「升級到更好的選項」，不是「修一個壞掉的東西」**。

> ## 1. 核心觀念：EC2 vs VPC
> 
> ### EC2 (Elastic Compute Cloud)
> - **類別**：Compute（運算服務）
> - **本質**：虛擬機器
> - **你可以**：開機、關機、SSH 進去、裝軟體、跑程式
> - **比喻**：雲端的一台電腦
> 
> ### VPC (Virtual Private Cloud)
> - **類別**：Networking（網路服務）
> - **本質**：**一套虛擬網路配置（不是機器）**
> - **包含**：IP 範圍、Subnet 劃分、路由規則、防火牆規則
> - **比喻**：整棟辦公大樓的網路規劃 + 門禁系統
> 
> ### 兩者的關係
> - EC2 **一定要**放在某個 VPC 的某個 Subnet 裡
> - VPC 可以是空的，但通常會有 EC2 / RDS 等服務住進去
> - 新帳號在每個 Region 會有一個「default VPC」自動建好
> 
> ### 對照表
> 
> | 項目 | EC2 | VPC |
> |---|---|---|
> | 類別 | 運算 Compute | 網路 Networking |
> | 是什麼 | 虛擬電腦 | 虛擬網路環境 |
> | 所在層級 | Subnet（AZ 層級） | Region 層級 |
> | 單獨存在？ | 不能，必須在 Subnet 裡 | 可以是空的 |
> | 收費方式 | 按小時 + EBS 儲存 | VPC 本身免費；NAT Gateway、Endpoint 等才收錢 |
> 
> ---
> 
> ## 2. 層級結構
> 
> ```
> Region（例如 ap-northeast-1 東京）
> ├── AZ-1a    ┐
> ├── AZ-1c    ├── 實體資料中心群（會停電、會斷網）
> └── AZ-1d    ┘
> 
> VPC（Region 層級，橫跨所有 AZ）
>   ├── Subnet A  ──→ 綁在 AZ-1a
>   │     └── EC2
>   ├── Subnet B  ──→ 綁在 AZ-1c
>   │     └── EC2
>   └── Subnet C  ──→ 綁在 AZ-1d
>         └── EC2
> ```
> 
> ### 每一層是什麼
> - **Region**：AWS 的地理區域，例如 ap-northeast-1（東京）、us-west-2（俄勒岡）
> - **AZ (Availability Zone)**：Region 底下獨立運作的資料中心群
> - **VPC**：Region 層級的虛擬網路（重點：橫跨 AZ）
> - **Subnet**：VPC 內的子網路（重點：綁定單一 AZ，不能改）
> - **EC2**：虛擬機，住在某個 Subnet 裡
> 
> ---
> 
> ## 3. 為什麼 Subnet 要綁一個 AZ
> 
> 這是 AWS 的硬性規定 — 建立 Subnet 時必須選一個 AZ，之後不能改也不能跨 AZ。
> 
> ### 物理原因
> AZ 是**不同實體機房**。一段 IP 範圍對應實際機器，機器住在某個實體地點，所以 Subnet 必須綁定一個 AZ。
> 
> ### 設計目的（更重要）
> AWS 故意設計這個限制，目的是**強迫使用者思考「失效隔離 (failure isolation)」**。
> 
> 如果 Subnet 可以跨 AZ，很多人會圖方便把所有 EC2 塞進同一個 Subnet，結果 AZ 一掛就全倒。AWS 把這個規則寫進設計，等於強迫你建架構時想：
> 
> > 如果 AZ-1a 整個掛了，我的服務還會活著嗎？
> 
> ### 所以 HA 要這樣做
> 1. 建立多個 Subnet，每個綁不同 AZ
> 2. 在每個 Subnet 放一台 EC2，跑一樣的服務
> 3. 前面放 Load Balancer 自動分流
> 
> ---
> 
> ## 4. Public Subnet vs Private Subnet
> 
> **重要：物理上 Public 和 Private Subnet 沒有任何差別 — 差別完全來自「路由表設定」。**
> 
> ### Public Subnet
> - **特徵**：路由表有一條 `0.0.0.0/0 → IGW` 的規則
> - **裡面的 EC2 可以**：
>   - 擁有 Public IP
>   - 直接被網際網路連到
>   - 主動連出去到網際網路
> - **典型用途**：Web 伺服器、跳板機、Load Balancer
> 
> ### Private Subnet
> - **特徵**：路由表**沒有**指向 IGW 的規則
> - **裡面的 EC2 只能**：
>   - 在 VPC 內部通訊
>   - 如果要連外網，需要透過 NAT Gateway（放在 Public Subnet）
> - **典型用途**：資料庫、內部 API、批次工作
> 
> ### 判斷方法
> 看路由表裡有沒有指向 IGW 的規則 — 就這一條規則決定 Subnet 的「公私」。
> 
> ---
> 
> ## 5. 關鍵網路元件
> 
> ### Internet Gateway (IGW)
> - **是什麼**：VPC 對外（網際網路）的「大門」
> - **特性**：每個 VPC 最多一個 IGW，且是 Region 層級（不會掛）
> - **怎麼用**：
>   1. 建立 IGW 並 attach 到 VPC
>   2. 在 Public Subnet 的路由表加 `0.0.0.0/0 → IGW`
> - **費用**：IGW 本身免費（但透過它的流量可能收費）
> 
> ### NAT Gateway
> - **是什麼**：讓 Private Subnet 的 EC2「能出去但不能被連進來」
> - **使用場景**：Private Subnet 的 DB 要下載安全更新、連外部 API
> - **特性**：要放在 Public Subnet，然後 Private Subnet 的路由表指向它
> - **費用**：**按小時 + 流量收費，蠻貴的**
> - **注意**：這是 AWS 意外帳單的常見來源，不用時記得刪掉
> 
> ### Route Table（路由表）
> - **是什麼**：每個 Subnet 都有一張，決定「要連到某個 IP 時走哪條路」
> - **內容範例**：
>   ```
>   10.0.0.0/16  → local      （VPC 內部通訊）
>   0.0.0.0/0    → igw-xxxxx  （走 IGW 上網際網路 —— Public 的關鍵）
>   ```
> - **路由邏輯**：由上到下比對，最精確的規則優先
> 
> ### Security Group
> - **層級**：掛在 EC2 上
> - **狀態**：**Stateful**（有狀態）— 允許進來的連線，自動允許回應流量
> - **規則類型**：只能「允許」，沒有「拒絕」
> - **預設**：對外全開、對內全關
> - **比喻**：每台 EC2 門口的保全
> 
> ### NACL (Network ACL)
> - **層級**：掛在 Subnet 上
> - **狀態**：**Stateless**（無狀態）— 進出都要各自定義規則
> - **規則類型**：允許 + 拒絕都可以
> - **預設**：對內對外全開（很寬鬆）
> - **比喻**：整個 Subnet 外面的警衛室
> 
> ### Security Group vs NACL 對照
> 
> | | Security Group | NACL |
> |---|---|---|
> | 層級 | EC2 | Subnet |
> | 狀態 | Stateful | Stateless |
> | 規則 | 只能允許 | 允許 + 拒絕 |
> | 預設值 | 對內全關 | 全開 |
> | 評估方式 | 所有規則一起評估 | 由上到下 |
> | 初學者頻率 | 常用 | 進階才用 |
> 
> ---
> 
> ## 6. Load Balancer（ALB）
> 
> ### 是什麼
> Application Load Balancer 處理 HTTP/HTTPS 流量，把使用者請求分散到多台 EC2。
> 
> ### 架構位置
> - 本身**跨 AZ**（不綁單一 AZ）
> - 建立時要選「橫跨哪些 Public Subnet」（至少兩個不同 AZ）
> - 使用者流量先打到 ALB，ALB 再分流到後面的 Web EC2
> 
> ### 為什麼要用 LB
> 1. **HA**：AZ-1a 的 Web 掛了，ALB 自動全導到 AZ-1c
> 2. **負載平衡**：多台 EC2 平均分擔流量
> 3. **單一入口**：使用者只記一個網址
> 4. **健康檢查**：定期「戳」EC2，戳不到就不分流量過去
> 
> ### 典型流量路徑
> ```
> 使用者 → DNS → ALB（Public Subnet）→ Web EC2（可能 AZ-1a 或 AZ-1c）→ DB EC2（Private Subnet）
> ```
> 
> ### ALB vs NLB vs CLB
> - **ALB** (Application) — HTTP/HTTPS 層 7，**最常用**
> - **NLB** (Network) — TCP/UDP 層 4，極高效能
> - **CLB** (Classic) — 舊版，新專案別用了
> 
> ---
> 
> ## 7. 高可用性（HA）架構
> 
> ### 最小可用 HA 架構
> 
> ```
>                  [使用者]
>                     ↓
>            [IGW]
>                     ↓
>            [Application Load Balancer]
>              （跨 AZ-1a + AZ-1c）
>               ↙              ↘
>     [Web EC2 AZ-1a]      [Web EC2 AZ-1c]
>            ↓                    ↓
>     [DB Primary]         [DB Standby]
>     AZ-1a                AZ-1c
>     （RDS Multi-AZ 會自動切換）
> ```
> 
> ### HA 檢查清單
> - [ ] ALB 至少設定兩個 AZ 的 Public Subnet
> - [ ] Web 層至少兩台 EC2 分別在不同 AZ
> - [ ] 資料庫使用 RDS Multi-AZ（或自己做 replication）
> - [ ] Route Table、Security Group 沒卡住流量

> ## 8. CIDR Block 基礎
> 
> ### 什麼是 CIDR
> 格式：`IP/前綴長度`
> 
> 例如 `10.0.0.0/16` = `10.0.0.0` 到 `10.0.255.255`（共 6 萬多個 IP）
> 
> ### 常見換算表
> | 前綴 | IP 數量 | 用途 |
> |---|---|---|
> | /32 | 1 | 單一 IP |
> | /24 | 256 | 單一 Subnet（常用）|
> | /16 | 65,536 | 單一 VPC（常用）|
> | /8 | 16,777,216 | 太大，不常用 |
> 
> 規則：前綴越小，範圍越大
> 
> ### VPC / Subnet 的 CIDR 約定
> - VPC CIDR 通常是 `/16`
> - Subnet CIDR 是 VPC 的子集，通常 `/24`
> - 每個 Subnet AWS 會保留 5 個 IP（第 1、2、3 個和最後 1 個），實際可用 = 256 − 5 = 251
> 
> ### 範例
> ```
> VPC      : 10.0.0.0/16   (10.0.0.0 ~ 10.0.255.255)
> Subnet 1 : 10.0.1.0/24   → AZ-1a, Public
> Subnet 2 : 10.0.2.0/24   → AZ-1a, Private
> Subnet 3 : 10.0.3.0/24   → AZ-1c, Public
> Subnet 4 : 10.0.4.0/24   → AZ-1c, Private
> ```
>

| **技術棧** | 統一（例如全部 Go） | 可混用（Go + Node + Python） |

## 22:19 — database sharding 意思?

**Claude:**

# Database Sharding 是什麼

**一句話：** 把一張「大到一台 DB 扛不動」的表，**按某個 key 切成好幾份**，分散到不同 DB 實例上。

## 直覺例子

假設 `orders` 表有 **10 億筆**，一台 MySQL 查詢很慢。

**Sharding 做法：** 依 `user_id % 4` 切成 4 份：

```
DB-0: user_id % 4 == 0 的訂單  (2.5 億)
DB-1: user_id % 4 == 1 的訂單  (2.5 億)
DB-2: user_id % 4 == 2 的訂單  (2.5 億)
DB-3: user_id % 4 == 3 的訂單  (2.5 億)
```

每次查詢先算 `user_id % 4` → 只打對應那台 DB。

## 跟其他概念的差別

| 技術 | 做什麼 | 解決什麼 |
|------|-------|---------|
| **Index** | 同一張表加索引 | 查詢慢 |
| **Partition** | 同一個 DB 內部切分區 | 單表太大，掃描慢 |
| **Replication** | 主從複製，多份一樣的資料 | 讀取流量大 / 容災 |
| **Sharding** | 切成多份，**每份都不一樣** | **寫入** 太多、單機裝不下 |

**關鍵：** Replication 是「複製」，Sharding 是「切開」。

## Sharding Key 怎麼選

常見策略：

| 策略 | 範例 | 優點 | 缺點 |
|------|------|------|------|
| **Range** | `user_id 1~100萬 → DB0` | 範圍查詢快 | 熱點（新用戶都在最後一台） |
| **Hash** | `user_id % N` | 分佈均勻 | 擴容難（要 rehash 搬資料） |
| **Consistent Hash** | 環狀 hash | 擴容只搬一部分 | 實作複雜 |
| **Geo** | 亞洲 / 美洲各一台 | 延遲低 | 跨區查詢麻煩 |

## 痛點（為什麼不輕易用）

1. **跨 shard 查詢地獄** — `SELECT * FROM orders WHERE amount > 1000` 要打所有 shard 再合併
2. **跨 shard JOIN 幾乎不可能** — 訂單在 DB-0、會員在 DB-2，JOIN 寫不出來
3. **跨 shard transaction 難** — 沒有 ACID，要用分散式交易（2PC / Saga）
4. **擴容痛** — 從 4 台加到 8 台，資料要搬
5. **Sharding key 選錯 = 災難** — 後期要換 key，幾乎要重灌

## 對照你的 FutureSign

**目前完全不需要 sharding。** 判斷依據：

| 指標 | Sharding 門檻 | FutureSign 現況 |
|------|-------------|----------------|
| 單表資料量 | > 10 億筆 | 遠低於 |
