## 04:33 — <-> C:\coding\futuresign\Abby-notes\RAG\pgvector-setup-guide.md 這個符號是什麼意思

**Claude:**

在 pgvector 裡，`<->` 是 **L2 距離（歐幾里得距離 / Euclidean distance）** 的運算子。

## 用法

```sql
SELECT description, embedding <-> '[1, 0, 0]' AS distance
FROM items
ORDER BY distance
LIMIT 3;
```

意思是：算出每個 `embedding` 跟 `'[1, 0, 0]'` 這個查詢向量之間的「直線距離」，距離越小代表越相似。

## pgvector 三個距離運算子（檔案 4.4 節）

| 運算子 | 距離類型 | 對應索引 ops | 何時用 |
|--------|---------|--------------|--------|
| `<->` | **L2 (Euclidean)** | `vector_l2_ops` | 最直覺，幾何空間直線距離 |
| `<=>` | **Cosine** | `vector_cosine_ops` | 文字 embedding（OpenAI）主流選擇 |
| `<#>` | **負內積 (Inner Product)** | `vector_ip_ops` | 已正規化的向量 |

## 直觀理解 L2

兩個向量 `a = [1, 0, 0]` 跟 `b = [0.9, 0, 0.1]`：

$$
\text{L2}(a, b) = \sqrt{(1-0.9)^2 + (0-0)^2 + (0-0.1)^2} = \sqrt{0.02} \approx 0.141
$$

就是國中數學的「兩點間距離公式」推廣到 N 維空間。所以筆記第 456 行才會出現 `香蕉 | 0.14...` —— 那就是這條公式算出來的。

## 為什麼長這樣

`<->` 是 PostgreSQL 自訂運算子（pgvector 這個 extension 註冊的），不是 SQL 標準語法。視覺上像「兩個點之間連一條線」，剛好對應 L2 距離的幾何意義。

## 1. `vector` 這個 type 哪裡來的？要先「啟用 extension」

PostgreSQL 內建的型別只有 `int`、`text`、`varchar`、`timestamp` 那些，**沒有 `vector`**。

`vector` 是 pgvector 這個 extension 註冊進去的「自訂型別 (user-defined type)」。所以順序一定是：

```sql
CREATE EXTENSION IF NOT EXISTS vector;   -- 1. 先把 pgvector 載入這個 DB
                                         --    → 註冊 vector type
                                         --    → 註冊 <->, <=>, <#> 運算子
                                         --    → 註冊 hnsw, ivfflat 索引

CREATE TABLE items (
  embedding vector(3)                    -- 2. 註冊完才能用
);
```

如果沒先 `CREATE EXTENSION` 就直接寫 `vector(3)`，會噴錯：

```
ERROR: type "vector" does not exist
```

**驗證 vector 真的是一個 type**：

```sql
\dT vector
-- 或
SELECT typname, typtype FROM pg_type WHERE typname = 'vector';
```

只要你前面那個 Docker 容器有跑、且執行過 `CREATE EXTENSION vector;`，**現在就能建 `vector(3)` 沒問題**。

## 2. ⚠️ 大誤解澄清：`vector(3)` 不是 3 欄！

我前面**沒**說過「3 等於 3 欄、1536 維 = 1536 欄」——這是要避免的誤解。實際上：

| 誤解 ❌ | 實際 ✅ |
|---------|---------|
| `vector(3)` = 3 個欄位 | `vector(3)` = **1 個欄位**，裡面裝一個「3 個浮點數的陣列」 |
| 1536 維 = 1536 欄 | 1536 維 = **1 個欄位**，裡面裝 1536 個 float |

**儲存的視覺化**（你看到的就是這樣）：

```
items 表
┌────┬─────────────┬───────────────────────────────┐
│ id │ description │ embedding                     │
├────┼─────────────┼───────────────────────────────┤
│ 1  │ 蘋果        │ [1.0, 0.0, 0.0]               │ ← 一格欄位，裡面是陣列
│ 2  │ 香蕉        │ [0.9, 0.0, 0.1]               │
│ 3  │ 狗          │ [0.0, 1.0, 0.0]               │
└────┴─────────────┴───────────────────────────────┘
                      ↑
                整個 [...] 都在「一個欄位」裡
```

對，**就是像陣列在存座標**——這個直覺完全正確。底層 pgvector 把 N 個 float 包成一個 binary blob 塞進這個欄位，並額外定義了 `<->`、`<=>`、`<#>` 三個運算子讓你能對「整個欄位」做向量運算。

### 為什麼設計成「一格塞整個陣列」，而不是「拆成 1536 欄」？

如果真的拆 1536 欄會出大事：

| 問題 | 說明 |
|------|------|
| **查詢爆炸** | `SELECT col1, col2, ..., col1536` 要列舉一千五百次 |
| **建不了索引** | HNSW / IVFFlat 索引要看「整個向量」才有意義，拆欄做不到 |
| **運算寫死** | `<->` 一行解決，拆欄要手寫 1536 個減法平方相加 |
| **schema 僵死** | 從 1536 維改 768 維 model 要重建 schema |

所以 pgvector 用「一個複合型別 + 專屬運算子」這條路。

## 3. 歐幾里德距離（L2 distance）補充

### 公式

$$
d(A, B) = \sqrt{\sum_{i=1}^{n} (a_i - b_i)^2}
$$

白話：**對應位子相減 → 平方 → 全部加起來 → 開根號**。

### 幾何意義：兩點的「直線距離」

#### 2D（國中畢氏定理就是它）

```
y
│
4 ┤        ● B (3, 4)
│       ╱│
│      ╱ │
│   d ╱  │ 4
│    ╱   │
│   ╱    │
0 ●──────┼──→ x
  A     3
(0,0)
```

$$
d = \sqrt{(3-0)^2 + (4-0)^2} = \sqrt{9+16} = \sqrt{25} = 5
$$

#### 3D（pgvector 筆記 4.3 節的「蘋果 vs 香蕉」）

```
A = [1,   0, 0]      ← 蘋果
B = [0.9, 0, 0.1]    ← 香蕉
```

$$
d(A,B) = \sqrt{(1-0.9)^2 + (0-0)^2 + (0-0.1)^2}
     = \sqrt{0.01 + 0 + 0.01}
     = \sqrt{0.02}
     \approx 0.1414
$$

→ 對應筆記裡 `香蕉 | 0.14...` 那行的由來。

#### 1536 維（OpenAI text-embedding-3-small）

公式不變，只是 i 從 1 跑到 1536：

$$
d(A, B) = \sqrt{\sum_{i=1}^{1536} (a_i - b_i)^2}
$$

人腦無法視覺化 1536 維空間，但**電腦不在乎**，CPU 一條 SIMD 指令能算 8 對 float 的減法平方，1536 維幾微秒就跑完。資料量大時用 HNSW 索引避免「暴力比對全部資料」。

### 為什麼叫「L2」？

因為它屬於 **Lp 範數家族**的 p=2 版本：

| Lp | 名稱 | 公式精髓 | 直覺 |
|----|------|---------|------|
| L1 | Manhattan 距離 | 差值取**絕對值**相加 | 棋盤格走法、計程車距離 |
| **L2** | **Euclidean 距離** | 差值**平方**相加開根號 | **直線距離**（最常用）|
| L∞ | Chebyshev 距離 | 取**最大**那維的差 | 國王走棋盤的步數 |

pgvector 三個運算子各自取捨：

| 運算子 | 距離 | 對誰敏感 | 主流用途 |
|--------|------|---------|---------|
| `<->` | L2 | 向量「長度」+「方向」都算 | 數值特徵、影像 embedding |
| `<=>` | Cosine | **只看方向**，忽略長度 | 文字 embedding（OpenAI 推薦）|
| `<#>` | 負內積 | 同方向越強越「近」（負越多）| 已正規化向量、最快 |

**為什麼文字 embedding 通常用 `<=>` 而不是 `<->`？**
因為文字 embedding 經過 normalize 後，「重要的是方向（語意），不是長度（強度）」。L2 會被向量長度干擾，cosine 不會。但學習階段先用 `<->`（最直覺）沒問題。

## 14:16 — _ops是什麼

**Claude:**

## `_ops` = **operator class**（運算子類別）的後綴慣例

是 PostgreSQL 的索引機制術語，告訴索引「比較這個欄位時，**要用哪一種距離 / 順序定義**」。

### 拆解 `vector_l2_ops`

```
vector  _  l2  _  ops
   ▲       ▲      ▲
 型別     方法   operator class
```

| 部分 | 意思 |
|------|------|
| `vector` | 適用的資料型別（pgvector 的 vector type）|
| `l2` | 用 L2 距離（歐幾里得，剛剛在 0 節學的）|
| `ops` | **op**erator clas**s** 的縮寫（PostgreSQL 慣例後綴）|

### 為什麼要有 operator class？

因為**同一個 vector type 有 3 種「相似」定義**：L2、Cosine、Inner Product。建索引時要先講好「我要按哪一種排序」，不然 HNSW 不知道怎麼排這棵圖。

| operator class | 對應運算子 | 距離 | 適用 |
|---|---|---|---|
| `vector_l2_ops` | `<->` | L2 (Euclidean) | 數值、影像 |
| `vector_cosine_ops` | `<=>` | Cosine | 文字 embedding 主流 |
| `vector_ip_ops` | `<#>` | 負內積 | 已正規化向量 |

### 鐵則：**索引 ops 要跟查詢運算子配對**

```sql
-- ✅ 配對正確 → 索引被使用，超快
CREATE INDEX ON items USING hnsw (embedding vector_l2_ops);
SELECT * FROM items ORDER BY embedding <-> '[1,0,0]' LIMIT 5;

-- ❌ 配對錯誤 → 索引廢掉，退回全表掃描
CREATE INDEX ON items USING hnsw (embedding vector_l2_ops);
SELECT * FROM items ORDER BY embedding <=> '[1,0,0]' LIMIT 5;
--                                    ▲
--                            Cosine，索引是 L2，不認識
```

→ **要用哪個距離查詢，就建哪個 ops 的索引**；想兩種都用就建兩個索引。

### `_ops` 不是 pgvector 發明的

是 PostgreSQL **通用索引擴充機制**，內建 type 也有：

| operator class | 用途 |
|---|---|
| `int4_ops` | 整數預設排序 |
| `text_pattern_ops` | `LIKE 'abc%'` 前綴搜尋（跟一般 `text_ops` 不同）|
| `gin_trgm_ops` | pg_trgm 模糊搜尋 |
| `jsonb_path_ops` | JSONB 路徑搜尋 |

任何資料型別只要想被索引，都要為「每種比較方式」註冊一個 operator class —— pgvector 只是延續這套慣例。
