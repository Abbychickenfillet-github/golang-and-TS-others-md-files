# 21. 2026-03-21 禮品券轉讓設計議題（待假日後決定）

## 核心矛盾

### 兩個獨立維度
1. **發放規則**：一人只能領一份 VS 一張票一份（`multiply_by_ticket`）
2. **轉讓規則**：贈品隨票券轉 VS 贈品單獨轉（`transfer_with_ticket`）

### 問題：`transfer_with_ticket` 放在哪一層？

#### 方案層級（現在的設計，有問題）
```
票券A 綁了：
  方案ㄅ transfer_with_ticket=true  → 跟著轉
  方案ㄆ transfer_with_ticket=false → 不跟著轉
  方案ㄇ transfer_with_ticket=true  → 跟著轉

消費者轉票券A → ㄅ轉了、ㄆ沒轉、ㄇ轉了
→ 超亂，消費者搞不懂為什麼有的轉有的沒轉
```

#### 票券層級（比較合理）
```
票券A：transfer_with_coupon=true（所有贈品跟著轉）
票券B：transfer_with_coupon=false（贈品不跟著轉）

消費者轉票券A → 所有綁定的方案ㄅㄆㄇ 全部一起轉
消費者轉票券B → 贈品都不動
→ 簡單，一刀切
```

## 建議方向

### 移到 tickets 表
```sql
-- 從 event_coupon_program 移除（或保留但不用）
-- ALTER TABLE event_coupon_program DROP COLUMN transfer_with_ticket;

-- 加到 tickets 表
ALTER TABLE tickets ADD COLUMN transfer_with_coupon TINYINT(1) NOT NULL DEFAULT 1
  COMMENT '轉讓票券時贈品是否跟著轉' AFTER is_transferable;
```

### 邏輯簡化
- 轉票時：查 `tickets.transfer_with_coupon`
- `true` → 所有 `source_order_item_id` 匹配的 claims 一起轉
- `false` → 只轉票，贈品不動
- 不再需要查每個方案的設定

## 單獨轉贈品
- 消費者端的「單獨轉贈品」按鈕先註解起來
- 後端 API 已做好（`TransferCouponClaimByConsumer`），隨時可開
- 但如果 `transfer_with_coupon=true`，單獨轉贈品會讓追蹤變複雜
- 建議：`transfer_with_coupon=true` 時不允許單獨轉贈品
