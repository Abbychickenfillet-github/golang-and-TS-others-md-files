## Print 欄位是什麼？

`Print` 告訴財政部這張發票是否印出紙本：

| Print | 意思 | 何時使用 |
|-------|------|----------|
| `"0"` | 不列印（雲端發票） | 有載具、有捐贈時 |
| `"1"` | 列印紙本 | 有統編 + 無載具時**必須** |

### Print 與其他欄位的搭配規則

| 情境 | Print |
|------|-------|
| 捐贈（Donation=1） | 必須 `"0"` |
| 有統編 + 無載具 | 必須 `"1"` |
| 有統編 + CarrierType 1 或 2 | 必須 `"0"` |
| 有統編 + CarrierType 3（手機條碼） | `"0"` 或 `"1"` 皆可 |
| 無統編 + 有載具 | 必須 `"0"` |

---

## CarrierType 載具類型對照表

| CarrierType | 名稱 | CarrierNum 格式 |
|---|---|---|
| `""`（空） | 無載具 | 空字串 |
| `"1"` | 綠界電子發票載具 | 空字串（由 email/phone 自動帶入） |
| `"2"` | 自然人憑證 | 16 字元：2 大寫英文 + 14 數字 |
| `"3"` | 手機條碼 | 8 字元：`/` + 7 個 `[0-9A-Z+\-.]` |

### 手機條碼格式（CarrierType = 3）

- 固定 8 個字元
- 第 1 個字元必須是 `/`
- 後 7 個字元只能是：`0-9`、`A-Z`（大寫）、`+`、`-`、`.`
- 正則：`^\/[0-9A-Z+\-.]{7}$`
- **ECPay 會向財政部驗證號碼是否真實存在**，假號碼會失敗

### Stage 測試怎麼填載具？

ECPay stage 環境**沒有提供假的手機條碼**，驗證仍打財政部。測試方式：

1. **用真的手機條碼**：去 https://www.einvoice.nat.gov.tw/ 用手機申請一個
2. **改用 CarrierType = `"1"`（綠界載具）**：CarrierNum 留空，不需驗證
3. **不帶載具**：CarrierType = `""`，Print = `"1"`

## 禮贈品核銷後消費者即時更新 — refetchInterval 頻率問題

**問題**：如果用 TanStack Query 的 `refetchInterval: 30000`（30 秒），核銷完成後消費者要等 30 秒才看到更新，太慢了。改成 3 秒 `refetchInterval: 3000` 會不會導致 DB 流量暴增？

**分析**：

- 3 秒輪詢 = 每個使用者每 3 秒發 1 個 SELECT 查詢
- 即使 1000 人同時在線 = ~333 queries/sec，對 MySQL 來說是小菜一碟
- 查的是 `event_coupon_claim` 單表 + member_id 索引，查詢極快
- **結論：3~5 秒的 refetchInterval 完全不會造成 DB 壓力問題**

**建議做法**：

1. **短期**：`refetchInterval: 5000` + `refetchOnWindowFocus: true`，體驗夠好
2. **中期**：搭配 `visibilitychange` 事件，頁面不可見時停止輪詢（TanStack Query 預設支援）
3. **長期**：如果要真正即時（<1s），用 WebSocket/SSE 推送核銷事件

## 時間戳記 UTC 顯示問題

**問題**：消費者頁面 `http://localhost:3000/event/{id}#coupon` 的時間顯示是 UTC+0，但使用者在 UTC+8 時區。

**原因**：後端用 `time.Time.Format("2006-01-02 15:04:05")` 輸出 UTC 時間，字串沒帶 `Z` 或 `+00:00` 後綴。JavaScript `new Date("2026-03-13 06:30:00")` 解析為本地時間而非 UTC，導致時間錯誤。

**修復方式**：前端 `formatDateTime()` 函數在解析前補上 `Z` 後綴，確保當作 UTC 解析再轉成本地時間。

**影響頁面**：
- `EventCouponSection.tsx` — 活動頁禮贈品區
- `CouponClaimButton.tsx` — 浮動按鈕 modal
- `MyTicketsPage.tsx` — 我的票券 coupons tab
