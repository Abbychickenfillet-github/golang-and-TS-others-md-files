## 發現的問題

### 問題 1：override_booth_price 不生效
主辦方在後台設了個別攤位價格 `override_booth_price = 450`，但前端列表顯示的還是時間區間定價。

**根因**：`GetBoothPrice()` 的優先順序寫反了：
```
舊（錯誤）：時間區間 > 類型預設 > override（永遠被蓋掉）
新（正確）：override > 時間區間 > 類型預設
```

### 問題 2：早鳥價套用到所有區域
在 `event_booth_type_pricing` 設了 `target_areas = ["台灣品牌"]` 的早鳥價 8980 元，但所有區域（法國 GCC、VIP、亞洲品牌等）都顯示 8980。

**根因**：
- `GetCurrentPricing()` 查詢時完全不看 `target_areas` 欄位
- `BuildPriceCache()` 按 `event_booth_type_id` 快取，同類型所有攤位共用一個價格
- Handler 用 `priceCache[boothTypeID]` 取價格時也不看攤位的 `area`

### 問題 3：改成每個攤位呼叫 GetBoothPrice 後效能爆差
第一次修的時候把 handler 迴圈改成每個攤位各呼叫一次 `GetBoothPrice`（查 DB），變成 N+1 查詢，載入超慢。

**修正**：改用 `BuildPriceCacheV2` 批次查（1-2 次 DB）+ `ResolvePriceForBooth` 純記憶體匹配（0 次 DB）。

## 學到的觀念

### Go 大寫 = public，但 public ≠ 給前端

Go 的大寫是 **package 層級** 的 public（其他 package 可以 import），不代表會暴露給 API。

```
handler package 可以用 service.BoothPriceCacheData（因為大寫）
但前端拿到的是 dto.BoothWithPrice（有 json tag 的才會序列化）
```

| 層級 | 給誰用 | 大寫的意義 |
|------|--------|-----------|
| `models.` | DB ↔ Service | 讓 service 能用 model struct |
| `service.` | Service ↔ Handler | 讓 handler 能用 service 的型別 |
| `dto.` | Handler ↔ 前端 | 有 `json` tag 才會變成 API response |

### BoothPriceCacheData 的資料來源

```go
type BoothPriceCacheData struct {
    PricingsByTypeID  // ← 來自 DB: event_booth_type_pricing 表
    DefaultPrices     // ← 來自 DB: event_booth_type.default_price
}
// booth.override_booth_price 不在 cache 裡，在 booth 本身的欄位上
```

`ResolvePriceForBooth(booth, cacheData)` 同時看兩邊：
- `booth.OverrideBoothPrice` → override
- `cacheData.PricingsByTypeID` → 時間定價（按 area 匹配）
- `cacheData.DefaultPrices` → 預設價格

### N+1 問題與批次查詢

```
❌ N+1：迴圈 100 個攤位，每個各查一次 DB = 100 次查詢
✅ 批次：先查 1-2 次 DB 拿所有定價，迴圈中純記憶體比對 = 2 次查詢
```

### interface 的 method 不能隨便刪

刪 interface 的方法 = 所有實作（包括 mock）都要一起改，否則編譯失敗。
確認沒有人呼叫後才能安全刪除。
