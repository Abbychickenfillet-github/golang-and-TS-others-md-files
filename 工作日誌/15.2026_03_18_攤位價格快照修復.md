# 2026-03-18 攤位價格快照修復

## 一、Stage 攤位價格沒記錄到

### Abby 說的話
> 「stage 這邊發現攤位價格沒有記錄到」
> 「他這邊就是要記錄 display_price 了」
> 「怎麼還一定要有 override_booth_price? 沒有設定也沒關係啊」
> 「現在是只有攤位價格不見了」

### 問題分析
前端選攤位時，後端 API 已經計算好 `display_price`（含三層定價邏輯：override > 時間區間定價 > default），前端確認頁也用 `display_price` 顯示金額。

但付款頁呼叫 `registration API` 建立訂單時，`BoothRegistrationItem` **只傳了 `booth_id`**，沒有帶 `display_price`。後端收到後用 `calculateBoothPrice()` 重新算，而這個函式只查 `override_booth_price` → `EventBoothTypePricing`，沒有 fallback 到 `default_price`，導致 `booth_price_snapshot` 記錄為 0。

## 四、Stage DB 票券欄位缺失

### Abby 說的話
> 「編輯票券儲存變更 modal 400 bad request 是為什麼」
> 回應：`Unknown column 'category_id' in 'field list'`
> 「我的 tickets 還少 hide_remaining_count 這個欄位」

### 問題分析
Stage DB 的 `tickets` 表缺少欄位和表，因為 `AUTO_MIGRATE_ENABLED=false` 沒跑 migration。

### Stage vs Prod 差異

| 項目 | Prod | Stage (修復前) |
|------|------|---------------|
| `tickets.category_id` | ✅ | ❌ |
| `tickets.hide_remaining_count` | ✅ | ❌ |
| `ticket_category` 表 | ✅ | ❌ |
| `ticket_prerequisite` 表 | ✅ | ❌ |

### 欄位說明
- **`category_id`**：票券分類 ID，關聯到 `ticket_category` 表（如「入場券」「體驗券」）
- **`hide_remaining_count`**：是否隱藏剩餘票數
- **`ticket_category`**：票券分類表（每個活動可有多個分類）
- **`ticket_prerequisite`**：票券前置條件表（多對多，必須先買 A 才能買 B）

### Claude 做了什麼
1. 手動對 `future_sign_stage` 執行 SQL 補齊 2 個欄位 + 2 張表
2. 驗證 stage 現在有 21 欄，與 prod 一致
3. SQL 檔案存到 `backend-go/sql/add_ticket_category_and_prerequisite.sql`
4. 建立 GitHub Issue [#382](https://github.com/yutuo-tech/futuresign_backend/issues/382)（已關閉）

## 六、修復 CI flaky test — TestLoginWithRateLimit_DifferentIPs

### 問題
`TestLoginWithRateLimit_DifferentIPs` 在 CI 偶發失敗（IP1 第 3 次請求預期 429 但拿到 200）。

### 根因
`cleanupTestRedisForIntegration` 使用 `FlushDB` 清掉整個 Redis DB，當多個 rate limit 測試**並行**執行時，一個測試的 cleanup 會清掉其他測試正在用的 rate limit key。

### 修法
- `setupTestRedisForIntegration`：連線後先 `FlushDB`（確保乾淨起始狀態）
- `cleanupTestRedisForIntegration`：只 `CloseRedis`，不 `FlushDB`（不影響並行測試）

### 修改檔案
- `backend-go/internal/handler/auth_handler_ratelimit_test.go`
