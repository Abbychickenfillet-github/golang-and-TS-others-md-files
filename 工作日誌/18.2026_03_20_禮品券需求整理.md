## 禮品券規則確認

### 已確認
- **購票自動發放**：消費者購票後系統自動發放夜市券
- **擴香石數量 = 票券數量**：買 N 張單日票 → 在有名額的情況下發 N 張擴香石券，不限每人張數。 他是怎麼去記錄說 轉出者它會有幾個禮品? 之前是因為購買數量是都限1個
也就是1張票1個禮贈品但現在規則改了   
- **票券預設可轉讓**：連帶的禮品券也一起轉移，但線下禮贈品票券的「可轉讓」要做成**可切換的開關**，讓主辦決定

### 需要開發的切換開關（主辦/系統可控）

| # | 設定項目 | 預設值 | 說明 |
|---|---------|--------|------|
| 1 | 禮品券是否可轉讓 | ✅ 是 | 線下禮贈品票券要可切換，主辦可關閉轉讓功能 |
| 2 | 票券轉讓時禮品券是否跟隨轉移 | ❌ 否 | 可切換，主辦可開啟讓禮品券跟著票券一起轉 |
| 3 | 取消後名額是否自動遞補 | ❓ 待確認 | 之前說不要自動遞補（前100名邏輯），現在需求不明確，建議做切換讓主辦決定。免費贈品影響不大 |

### 轉讓邏輯補充
- 轉讓時要回溯「當初買了幾張票」來決定可轉讓幾張禮品券
- 例：A 買了 3 張單日票 → 拿到 3 張擴香石 → 轉讓 1 張票時，對應的 1 張擴香石也跟著轉

## 新增核心需求：禮品券數量與部分轉讓

### 場景
夜市券：一個人買票後獲得 **5 張** $50 禮品券。
- 他可以自己全部用
- 也可以把 2 份給 A、1 份給 B、2 份給 C
- **必須非常有彈性**，可以自由分配轉讓

### 現有資料
```json
{
  "name": "4/12 $50夜市券",
  "coupon_value": 50,
  "max_quantity": 1000,
  "trigger_type": "ON_PURCHASE",
  "target_role": "CONSUMER",
  "claim_method": "ON_SITE",
  "require_ticket": 1,
  "is_free": 1
}
```

### 目前系統缺什麼

#### event_coupon_program 表
- [ ] `quantity_per_claim INT DEFAULT 1` — 每次購票發放幾張（夜市券場景 = 5）
- [ ] `is_transferable BOOL DEFAULT true` — 是否允許轉讓
- [ ] `transfer_with_ticket BOOL DEFAULT false` — 票券轉讓時是否跟隨

#### coupon_claim 表
- [ ] `quantity INT DEFAULT 1` — 持有數量（支援 > 1）
- [ ] 或者改為一次領取產生 N 筆 claim records（各 quantity=1）

#### 轉讓機制
- [ ] 部分轉讓：5 張轉 2 張 → 自己剩 3 張 + 對方得 2 張
- [ ] 用 transfer_log（entity_type = 'coupon', is_partial = true, remaining_amount = 3）
- [ ] 轉讓對象：用 email 搜尋會員

### 討論紀錄

#### source_order_item_id 追蹤所有入口
- 購票自動發放（payment_service.go）→ 有 source_order_item_id ✅
- 後台手動發放（ClaimCoupon）→ NULL（手動不來自特定票） ✅
- 消費者自領（ClaimCouponByConsumer）→ require_ticket 時查 order_item ✅
- 一筆 order_item 只會有一種票券（ticket_id），amount 是數量

#### 消費者自領檢查流程
1. 方案 require_ticket=true → 檢查會員是否買過指定票種
2. 同時查出匹配的 order_item → 寫入 source_order_item_id
3. 這樣未來做「轉票時禮品券跟隨」就有追蹤依據

#### 票券轉讓怎麼拆分
- order_item { id=A, amount=5, holder=我 }
- 轉 2 張給小明 → A.amount=3, 新建 B { amount=2, holder=小明 }
- 禮品券同理：claim.quantity -= N, 新建 newClaim { quantity=N }

### Phase 3: 消費者退款入口
- 前台「我的訂單」加申請退款按鈕
- 後端已有 CancelOrder + RefundOrder 流程
- 前端只需接上 UI

## 現有相關資料表

### coupon_program_ticket（多對多關聯）
```go
// CouponProgramTicket 消費券方案與票種的多對多關聯表
type CouponProgramTicket struct {
    ID              string // UUID
    CouponProgramID string // 消費券方案 ID
    TicketID        string // 票種 ID
}
```
**用途**：指定「哪些票種」觸發「哪個禮品券方案」。
例如：擴香石方案綁定 4/11 單日票 → 買 4/11 單日票才會發擴香石。

**這次會用到嗎？** ✅ 會。購票觸發發放時要查這張表，確認該票種有沒有綁定的方案。

### transfer_log（轉讓紀錄）
```go
// 現有 entity_type
TransferEntityTypeTicket = "ticket"
TransferEntityTypeBooth  = "booth"

// 需新增
TransferEntityTypeCoupon = "coupon"
```
**用途**：禮品券部分轉讓時記錄。
- `entity_type = "coupon"`, `entity_id = coupon_claim.id`
- `is_partial = true`, `quantity = 2`, `remaining_amount = 3`

目前是一筆 claim = 一張券（unique: member + program）。
需要改為支援 `quantity > 1`，或改為一次發多筆。
