# 19. 2026-03-20 禮品券與退款完整需求整理

## 用戶原話（逐條）

### 1. 票券轉讓與退款
> 票券原本轉讓出去就不能退款，但現在改成**轉回去可以退款**

- 現狀：holder != buyer 就擋退款（`ErrOrderTicketTransferred`）
- 新邏輯：如果票券被**轉回原購買者**（holder == buyer），就可以退款
- 也就是說：A 買票 → 轉給 B → B 轉回 A → A 可以退款

### 2. 轉讓次數限制
> 轉讓1次轉完目前是這個邏輯就不能再轉出去了。但先創建新的後端邏輯。

- 目前：轉一次就不能再轉
- 未來：可能開放多次轉讓，但先把後端邏輯建好

### 3. QR Code 獨立
> 要確定同1人的多張票截圖 QR code 都可以獨立

- 一個人買 3 張票 → 3 個獨立 QR Code
- 不能共用同一個 QR Code

### 4. 夜市券核銷 — 名字欄位
> 夜市券的票券核銷紀錄到底要新增一個打字的名字欄位

- 核銷時要輸入**領取者姓名**（手寫/打字）
- 目的：現場確認身份

### 5. 核銷防重複
> 可以重複選名字不重複第二次核銷要提示說這個人已經在其他張核銷過了

- 同一個名字第二次核銷 → 提示「此人已在其他張核銷過」
- 不擋住，只提示

### 6. 方案自訂欄位
> 主辦設定跟消費者端可以再設定方案的時候要帶什麼值但如果還有什麼地址需求的話存JSON

- 名字、電話、email → 寫死的固定欄位
- 其他自訂需求（地址等） → 存 JSON 擴充欄位

### 7. event_coupon_claim 改成每次一筆
> event_coupon_claim 最好改成每次都是新的一筆資料列，因為之後他如果現場要補簽名她就算沒有轉移到其他帳號也要一列一列的寫上名字

- **重大設計變更**：不用 `quantity` 欄位了
- 改成一張券一筆 claim row
- 買 5 張 → 建 5 筆 claim（各 quantity=1）
- 好處：每筆可獨立簽名、獨立核銷、獨立 QR Code
- 轉讓：直接改那一筆的 member_id

### 8. 核銷確認頁
> 核銷要確認說掃完要跑一個頁面是「是否確定核銷？」

- 掃 QR Code → 顯示確認頁（券資訊 + 領取者 + 確認按鈕）
- 不要掃完直接核銷

### 9. 主辦 vs 品牌商設定頁不同
> 主辦那邊設定方案頁面跟品牌商設定方案頁面不同

- 主辦方：完整設定（所有欄位）
- 品牌商：簡化版（只看自己相關的方案）

### 10. 免費票退款
> 免費票要可以退

- 目前免費票（total_amount=0）可能沒有退款流程
- 需要支援免費票退款（釋放庫存）

### 11. 主辦自己 approve 退款
> 要可以控制主辦：能不能自己 approve 自己的退款

- 新設定欄位：`allow_organizer_refund_approval BOOL`
- true → 主辦可以自己核准退款
- false → 需要管理員核准

### 12. 退款手續費
> 退刷手續費再我們現在改成這樣

- 退款時要扣手續費（ECPay 退刷費）
- 需要設計手續費計算邏輯

### 13. 同方案同會員可重複領取
> 同一個方案同一個會員是可以重複領取的

- 配合 #7（每次一筆 claim row）
- 移除 unique index 已做 ✅
- 但 `ClaimCouponByConsumer` 裡面的「已領取過」檢查要拿掉或改成「已達上限」

## 設計變更影響

### claim 從 quantity 改成一筆一張

```
舊方案（quantity=5）:
claim_1: { member=A, quantity=5 }

新方案（一筆一張）:
claim_1: { member=A, quantity=1 }
claim_2: { member=A, quantity=1 }
claim_3: { member=A, quantity=1 }
claim_4: { member=A, quantity=1 }
claim_5: { member=A, quantity=1 }
```

**影響**：
- `quantity_per_claim = 5` → 建 5 筆 claim row（不是 1 筆 qty=5）
- 轉讓：直接改那一筆的 `member_id`（不需要拆分）
- 核銷：每筆獨立核銷 + 獨立簽名
- QR Code：每筆一個獨立 QR Code
- `TransferCouponClaim`：簡化為改 member_id，不需要部分轉讓邏輯

### 轉讓簡化

```
舊方案（部分轉讓）:
A.qty=5 → A.qty=3 + 新建 B.qty=2

新方案（直接改 owner）:
claim_1: member=A → member=B（轉讓）
claim_2: member=A → member=B（轉讓）
claim_3: member=A（保留）
claim_4: member=A（保留）
claim_5: member=A（保留）
```

#### 退款自審規則（活動 > 公司覆蓋）

```
判斷邏輯：
if event.default_approve_refund != NULL {
    return event.default_approve_refund  // 活動有設定 → 用活動的
} else {
    return company.default_approve_refund // 活動沒設定 → 用公司的
}
```

| 公司設定 | 活動設定 | 結果 |
|---------|---------|------|
| 1 (可，預設) | NULL (未設定) | ✅ 可自審 |
| 1 (可) | 0 (不可) | ❌ 不可（活動覆蓋） |
| 0 (不可) | NULL (未設定) | ❌ 不可 |
| 0 (不可) | 1 (可) | ✅ 可（活動覆蓋） |

DB 欄位：
- `company.default_approve_refund TINYINT(1) NOT NULL DEFAULT 1`（預設可自審）
- `events.default_approve_refund TINYINT(1) DEFAULT NULL`（NULL=繼承公司）

## 確認：轉讓票券時禮品券自動跟隨

```
A 買票 order_item_1 (amount=3, ticket_id=4/11單日票)
  → program: quantity_per_claim=5, transfer_with_ticket=true
  → 發 3×5 = 15 筆 claims (各 source_order_item_id=order_item_1)

A 轉 1 張票給 B:
  order_item_1.amount = 2（A 剩 2 張票）
  新建 order_item_2 (amount=1, holder=B)

  transfer_with_ticket=true 觸發：
  → 5 筆 claims 的 member_id 改成 B
  → transfer_log 記錄 5 筆 coupon 轉讓

A 剩下 10 筆 claims, B 有 5 筆
```

**關鍵**：`source_order_item_id` 是追蹤的 key。轉票時查所有 `source_order_item_id = 被轉的 order_item`，按比例轉 claims。
