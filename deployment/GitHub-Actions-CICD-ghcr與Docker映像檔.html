重點整理

1. shell: /usr/bin/bash -e {0} 是「執行模板」不是命令

常出現在 CI/CD 的 YAML、Makefile、VS Code tasks.json，告訴系統「用哪個 shell、加什麼選項、命令放哪」。

  • shell: — 標籤（key），標明後面是 shell 路徑與參數，是設定檔語法。
  • /usr/bin/bash — Bash 的絕對路徑，避免依賴 PATH，行為一致。
  • -e — exit immediately（fail-fast）：任一命令失敗就立刻停止整個腳本。
  • {0} — 預留位置，由父程式動態替換成實際命令。

2. /usr/ 不是 user，是 Unix System Resources

最常見誤解：/usr/ 不是 user 的縮寫，而是 Unix System Resources。

目錄意義對應 Windows
/usr所有使用者共用的程式、函式庫Program Files + System32
/home/<user>你個人的家目錄C:\Users\Abby

3. ghcr.io 與 OCI

  • ghcr.io：GitHub Container Registry，作用等同 Docker Hub，但與 Repo/Actions 緊密整合。
  • OCI（Open Container Initiative）：Docker/Google/Red Hat/AWS 共同制定的容器標準，含 Image Spec 與 Runtime Spec。

4. 陷阱：映像檔名稱必須全小寫

invalid reference format: repository name (Abbychickenfillet-github/...) must be lowercase

這是 OCI Image Spec 的硬性規定：registry 路徑不允許大寫。問題來源是 GitHub 使用者名稱含大寫 A。

5. github.repository ≠ github.repository_owner

變數內容範例
github.repository_owner只有擁有者Abbychickenfillet-github
github.repository擁有者/儲存庫全名Abbychickenfillet-github/next-one-time-tracker

兩者都可能含大寫，用在 ghcr 路徑前都要轉小寫。

6. 正解：用 $GITHUB_ENV 動態轉小寫，別改帳號

# 【新增步驟】把 repository 名稱轉小寫，存進環境變數
- name: Set repository name to lowercase
  run: echo "REPO_LC=$(echo '${{ github.repository }}' | tr '[:upper:]' '[:lower:]')" >> $GITHUB_ENV

- name: Push to GitHub Container Registry
  run: |
    docker tag next-one-app ghcr.io/${{ env.REPO_LC }}/next-one-app:latest
    docker push ghcr.io/${{ env.REPO_LC }}/next-one-app:latest
  • name: 和 run: 屬於同一個 step，別在 run: 前多加 -（會拆成兩步、語法錯）。
  • >> $GITHUB_ENV：把 KEY=value 附加進特殊檔案，後續 step 用 ${{ env.KEY }} 讀。
  • 不需要去網頁 Settings 手動設變數；run 這行就是「動態產生」。
  • 不影響 Zeabur：REPO_LC 只活在臨時 runner，Job 結束就銷毀。
  • 千萬別改 GitHub 使用者名稱：remote、ghcr 路徑、連結、依賴全失效。

7. runs-on: ubuntu-latest：雲端 runner，不是你的電腦

跟你本機用 Windows 還是 Mac 無關——這是 GitHub 配的臨時虛擬機，Job 結束自動銷毀。偏好 Linux 因為：

  • 與生產環境一致（Docker base image、部署目標都是 Linux）。
  • 成本低：免費額度中 macOS runner 分鐘數消耗是 Linux 的 10 倍。
  • 速度快、工具相容。只有 build iOS/Windows 桌面程式才換 runner。

8. 映像檔容量爆炸（Image Accumulation）

  • 策略一 少推冗餘 tag：latest 被覆蓋後舊的變成 dangling / untagged 映像檔，容量還在；${{ github.sha }} 永久唯一每次多一版。
  • 策略二 遠端自動清理：GHCR Package settings → Lifecycle rules，刪 untagged 或只留最近 N 版。
  • 策略三 Multi-stage Build：Next.js output:'standalone' 把體積從 ~1GB 縮到 ~150MB。
FROM node:18-alpine AS deps
RUN apk add --no-cache libc6-compat
WORKDIR /app
COPY package.json package-lock.json ./
RUN npm ci

FROM node:18-alpine AS builder
WORKDIR /app
COPY --from=deps /app/node_modules ./node_modules
COPY . .
RUN npm run build          # next.config.js 需設 output: 'standalone'

FROM node:18-alpine AS runner
WORKDIR /app
ENV NODE_ENV production
COPY --from=builder /app/.next/standalone ./
COPY --from=builder /app/.next/static ./.next/static
COPY --from=builder /app/public ./public
EXPOSE 3000
CMD ["node", "server.js"]

本機清理：docker system prune -a --volumes。

快問快答（自我測驗）

  • -e 的作用？ → 任一命令失敗就立刻停止（fail-fast）。
  • /usr/ 是什麼縮寫？ → Unix System Resources，不是 user。
  • 為什麼映像檔名稱錯誤？ → OCI 規定必須全小寫，帳號含大寫 A。
  • 怎麼修最好？ → 用 tr 轉小寫寫入 $GITHUB_ENV，別改帳號名。
  • latest 被覆蓋後舊的去哪？ → 變 untagged/dangling，容量還在。
