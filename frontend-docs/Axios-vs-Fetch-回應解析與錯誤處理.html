重點整理

1. pg pool.query() 回傳的結果物件

用 pg（Node.js PostgreSQL library）執行查詢後，回傳的是一個結果物件 (Result)，常用屬性：

  • rows：查詢結果的資料列陣列（最常用）
  • rowCount：影響／回傳的資料列總數
  • command：執行的 SQL 指令（'SELECT'、'INSERT'…）
  • oid：物件識別碼
const result = await pool.query('SELECT id, name FROM users WHERE id < 3');
// result 長這樣：
// { rows:[{id:1,name:'張小華'},{id:2,name:'林小君'}], rowCount:2, command:'SELECT', oid:0 }

// 解構賦值：直接從物件取出需要的屬性
const { rows, rowCount, command } = result;

oid: 0 是什麼？ OID 是 PostgreSQL 早期用來識別資料庫物件的 4-byte 內部編號。像 SELECT 這種只讀查詢不會產生新物件，所以 oid 預設為 0。現代開發幾乎用不到它，看到 0 不用理會，專注在 rows 就好。

2. API 回傳結構：扁平 vs 巢狀

res.json({ data: users[0] })（扁平）vs res.json({ data: { user: users[0] } })（巢狀）：

寫法 優點 缺點
data: users[0]（扁平） 前端 response.data 直接拿，最簡潔 語義不明確、未來難擴充（無法再塞 permissions／role）
data: { user: ... }（巢狀） 語義清楚、命名一致（單數 user／複數 users）、易擴充 前端多一層 .user

取捨核心：一致性／可擴充 ↔ 簡潔。功能單純就扁平避免過度設計；要留擴充彈性就巢狀。

3. Fetch vs Axios 核心差異

fetch：瀏覽器原生、低階。回傳的 response 是串流 (stream)，資料還沒解析 → 必須手動 await response.json()；而且只有網路層錯誤（斷網）才會 reject，收到 404／500 仍算成功，要自己檢查 response.ok。

axios：高階函式庫，把 fetch 的麻煩都包好了： - 自動解析 JSON，直接用 response.data 拿到物件 - 4xx／5xx 會自動 throw，可以直接在 try...catch 統一處理 - 攔截器 (interceptors)：全域自動加 Token、遇 401 自動導向登入 - 內建取消請求（fetch 要搭 AbortController）

// fetch：手動解析
const res = await fetch(url);
if(!res.ok) throw new Error(res.status);   // 要自己判斷
const data = await res.json();

// axios：自動解析、自動丟錯
const res = await axios.get(url);
const data = res.data;

陷阱：data = response.data ?? response.json() 不能通用！ 因為 fetch 的 response 沒有 .data、axios 的 response 沒有 .json()。兩者都回 Promise，但 resolve 後的物件結構不同：fetch 資料在物件「內部」要呼叫方法取；axios 資料已解析好放在 .data。Promise.resolve() 只是包一層外層 Promise 狀態，不會改變內層物件結構。

4. 不寫 catch(error) 會怎樣？

  • 語法層面：try 一定要配 catch 或 finally，否則直接 SyntaxError 無法執行。
  • 後端完全沒 try-catch：DB 斷線 → UnhandledPromiseRejection → 整台 Node 伺服器 crash。
  • 後端有 catch 但忘了 res.json() 回應：前端請求一直卡在 Pending，直到瀏覽器逾時，Loading 圈圈轉不停。
  • 前端不 catch：axios 遇 500 自動 throw，程式中斷 → setUsers() 不會執行 → Loading 卡死、Console 一堆 Uncaught (in promise)。

正確後端防禦寫法：

try {
  const { rows } = await pool.query(sql);
  res.json({ data: rows });
} catch (error) {
  console.error(error);                              // 給工程師看的 log
  res.status(500).json({ status:'error', message:'伺服器內部錯誤' }); // 給前端的交代
}

一句話：寫 catch 不是為了阻止錯誤發生，而是讓錯誤發生時能優雅收拾（後端回 500、前端跳提示），不至於整個系統瞬間崩潰。
