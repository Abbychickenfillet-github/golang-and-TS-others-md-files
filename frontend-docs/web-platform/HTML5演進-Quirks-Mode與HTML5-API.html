HTML5 演進、Quirks Mode 與 HTML5 API（Web Platform vs Runtime）

重點整理

1. HTML 規範現況：版本號已被「Living Standard」取代

  • 官方最新規範由 WHATWG 維護，稱為 HTML Living Standard，網址 https://html.spec.whatwg.org/ 。
  • W3C 在 2014 年發布 HTML5 正式推薦標準，後續更新到 5.1、5.2，最後一版在 2017 年。
  • 現在 已沒有固定版本號，標準隨時滾動更新。

2. 新舊 <!DOCTYPE> 寫法差異

舊版（HTML 4.01 Strict）必須帶一長串 DTD 網址：

<!DOCTYPE HTML PUBLIC "-//W3C//DTD HTML 4.01//EN" "http://www.w3.org/TR/html4/strict.dtd">
  • 為什麼那麼長？因為舊 HTML 基於 SGML，瀏覽器要下載 DTD（Document Type Definition）才知道有哪些標籤、語法是否合法。

現代 HTML5 不再基於 SGML，宣告大幅簡化：

<!DOCTYPE html>
<html lang="zh-Hant">
<head>
  <meta charset="UTF-8">
  <title>現代網頁標題</title>
</head>
<body>
</body>
</html>
  • 核心差異：宣告簡化（只剩 <!DOCTYPE html>）＋ 語意化標籤（<header>/<footer>/<article> 取代滿滿的 <div id="header">）。

3. DOCTYPE 是「排版開關」，不是裝飾 ⭐

標準統一後，舊網站是照舊行為寫的；若瀏覽器突然全照新標準跑，舊網站會全部跑版。瀏覽器靠 DOCTYPE 判斷該用哪種模式：

模式 觸發條件 行為
標準模式 有寫 <!DOCTYPE html> 照 W3C／WHATWG 標準解析 CSS 與排版
Quirks Mode（怪異模式） 沒寫或寫錯 DOCTYPE 模擬 1990 年代舊瀏覽器 bug 行為以向後相容
  • 最經典災情：box model 算法不一樣。舊 IE 的 width 會把 padding、border 算進去，標準模式不算 → 版面寬度全部對不上、跑版。
  • 結論：DOCTYPE 是啟用正常排版的開關。

4. Runtime vs Web Platform：為什麼是兩回事

  • JavaScript Runtime（如 Deno、Node.js）：只負責「執行 JS 程式碼」，讓 JS 能在電腦／伺服器上跑。
  • Web Platform（瀏覽器環境）：除了有 JS 引擎，還額外提供大量 Web APIs（DOM、Fetch、Canvas…）。
  • 差別在「提供的 API 範圍」：把含 window／localStorage／DOM 的程式碼丟去 Node/Deno 執行，會直接報錯 window is not defined／localStorage is not defined。

[!note] Markdown → DOM 的流程印證三者關係 Markdown 不能直接被渲染成 DOM，要先轉成 HTML：Runtime 負責轉換邏輯，Web Platform 提供渲染／操作 DOM 的 API，最後才把純文字變成可互動畫面。

5. HTML5 經典 API（瀏覽器的超能力）

API 做什麼 改變了什麼
Canvas <canvas> JS 直接繪製 2D 圖形／動畫（WebGL 可 3D） 淘汰 Flash 做動畫與遊戲
Web Storage（local/session） 瀏覽器內存 key-value 資料 取代容量小（~4KB）又每次隨請求送伺服器的 Cookie；本機可存 ~5MB
Geolocation navigator.geolocation 取經緯度（需同意） 地圖／外送／叫車不必手動輸入地址
Audio/Video <audio>/<video> 原生影音播放與 JS 控制 不必再裝 RealPlayer／Flash

6. 進階 API：讓網頁變身 App（PWA）

  • History API：pushState/replaceState 改網址列但不重整 → React/Vue/Angular 的 SPA 路由基礎。
  • Drag and Drop API：把桌面檔案拖進網頁、用 JS 偵測讀取。
  • Web Workers：開背景執行緒處理耗時計算，避免單執行緒的 JS 卡死畫面。
  • Service Worker / Cache API：當瀏覽器與網路之間的代理，攔截請求並快取 → 離線瀏覽＋推播通知的核心。
