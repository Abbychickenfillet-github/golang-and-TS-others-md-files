# IndexedDB — 瀏覽器內建的本地資料庫

> 跟你做 RAG 用的 pgvector **完全不同類別**。
> 一個住瀏覽器、一個住伺服器;一個查 key、一個算相似度。

---

## 一句話

> **IndexedDB = 瀏覽器內建的「本地端 NoSQL 資料庫」**,用來在使用者瀏覽器裡存大量結構化資料。

---

## 重點特性

| 特性 | 說明 |
|---|---|
| **在哪** | 瀏覽器本機(每個網站有獨立一份,互相看不到) |
| **存什麼** | 結構化資料 —— 直接存 JavaScript 物件(NoSQL 風格) |
| **容量** | **幾百 MB ~ 幾 GB**(localStorage 只有 5–10 MB) |
| **持久性** | 永久(關瀏覽器、重啟電腦都還在,除非使用者清資料) |
| **查詢方式** | 用 **key** 查、或建 **index(索引)** 加速欄位查詢 |
| **非同步** | 全部 API 都是 async(Promise / event) |

---

## 典型用途

- **離線應用 / PWA**:沒網路也能用,資料先存本機,有網再同步
- **快取大量資料**:避免每次都打 API(電商商品清單、文章列表)
- **儲存使用者本機資料**:草稿、設定、瀏覽歷史
- 實例:Google Docs 離線編輯、Notion 桌面版

---

## 跟向量資料庫(pgvector)的差別

| | **IndexedDB** | **向量資料庫(pgvector)** |
|---|---|---|
| 位置 | **瀏覽器本機** | **後端伺服器** |
| 存什麼 | 一般物件(字串、數字、JSON、Blob) | **高維向量**(BGE-M3 1024 維) |
| 查詢 | **精確比對**(key、index、range) | **相似度搜尋**(餘弦、HNSW) |
| 「index」意思 | 為欄位建索引加速 key 查詢(B-tree) | 為向量建索引加速相似度搜尋(HNSW、IVF) |
| 比喻 | 把資料先**存進櫃子等下次拿** | 找**最像**的內容(語意檢索) |

> 「index」這個字在兩邊都是「加速查詢的資料結構」,但**用來加速的查詢類型完全不同**:一個是「找特定值」,一個是「找最像的向量」。

---

## 一個有趣的灰色地帶:瀏覽器本地 RAG

最近的實驗性專案會這樣做:

```
transformers.js (瀏覽器跑小 embedding 模型)
        ↓
   產生向量
        ↓
存進 IndexedDB(當儲存層)
        ↓
JS 一筆筆算餘弦相似度
```

→ 這時 IndexedDB 變成「**存向量的容器**」,但**它本身沒有向量搜尋能力**,要靠 JS 算 → 資料量大就會慢。

**本質還是不同**:IndexedDB 是「儲存」,pgvector 是「儲存 + 向量搜尋引擎」。

---

## 一句話總結

> **IndexedDB 是瀏覽器內建的本地 NoSQL 資料庫,用來在使用者端存大量結構化資料(離線、快取、草稿)。 跟向量資料庫沒關係 —— 它不做相似度搜尋,只做 key/index 查詢。**
