# IIFE 算原生 function 嗎？

> 路徑：frontend-docs / javascript / 設計模式 function / iife-是否算原生function
> 相關：[[原生函式與js引擎-v8]]、[[陣列遍歷-forEach與callback]]
> 一句話結論：**不算。** IIFE 不是「引擎內建的原生函式」，而是一種「立即執行」的**設計模式**——你自己定義一個函式，然後馬上呼叫它。它用的是原生語法，但本身不是原生函式。

---

## 📌 我這次想問的重點（先給結論）

1. **IIFE 算原生 function 嗎？** → ❌ 不算，它是設計模式。
2. **IIFE 只會執行一次嗎？** → ✅ **YES**（見最下面「只執行一次嗎」一節）。
3. **IIFE 可以被 `return` 嗎？** → ✅ 可以，而且 return 正是它拿來「給外部使用」的關鍵（見「IIFE 可以 return」一節）。
4. **「ES6 前沒有模組系統，但 IIFE 是模組化手法」是什麼意思？** → 模組化是「目標」，模組系統是「語法工具」，差別見「模組化 vs 模組系統」一節。

---

## 先釐清「原生 function」這個詞

「原生 / built-in function」通常指**瀏覽器或 JS 引擎內建、不用自己寫就能直接用**的函式：

```js
parseInt("42")        // 內建
Array.prototype.map   // 內建
Object.keys(obj)      // 內建
Math.random()         // 內建
```

IIFE 不在這個清單裡，因為它是**你自己寫的**函式。（內建函式跟引擎 V8 的關係見 [[原生函式與js引擎-v8]]）

---

## IIFE 是什麼

IIFE = **I**mmediately **I**nvoked **F**unction **E**xpression（立即執行函式表達式）。
定義完函式後，立刻用 `()` 呼叫它：

```js
(function () {
  console.log("定義完馬上執行");
})();

// 箭頭函式版本
(() => {
  console.log("一樣的效果");
})();
```

> 補充你的疑問：「它是不是執行到那裏才印出來？」→ **對。** JavaScript 是**同步、由上往下**執行，程式跑到 IIFE 那一行時，它就**當場立刻被呼叫**並印出來，不是等誰呼叫、也不是非同步。

### 兩對括號各自在做什麼

```js
(function () { ... })();
// ↑ 第一對括號：把 function 包成「表達式」(expression)，
//              引擎才不會把它當成「函式宣告」(declaration)
//                       ↑ 第二對括號：立刻呼叫這個函式
```

為什麼一定要第一對括號？這牽涉到「表達式 vs 陳述式」👇

---

## ❓ 第二對括號 `()`，編輯器不會以為是打錯字嗎？

**不會。** 編輯器和 JS 引擎判斷對錯靠的是**語法規則（grammar）**，不是「看起來怪怪的」。

關鍵規則只有一條：

> **任何「會求出一個函式值」的東西，後面接 `()`，就是合法的「呼叫」。**

```js
foo()                    // foo 是個函式值 → () 呼叫它 ✅
Math.max(1, 2)           // Math.max 是函式值 → () 呼叫它 ✅
(function () {})()        // (function(){}) 求出一個函式值 → () 呼叫它 ✅
```

差別只在於「前面那個函式值是怎麼來的」：

- `foo` 是**用名字**指到函式值；
- `(function () {})` 是**用第一對括號**當場生出一個（匿名的）函式值。

對 parser 來說，這兩種「前面是函式值、後面接 `()`」**長得一模一樣**，都符合「呼叫表達式（CallExpression）」的規則，所以**完全合法、不會報錯、編輯器也不會標紅**。

```js
(function () { ... })();
//             ↑ 第一對括號結束，整個 (function(){...}) 已經是「一個函式值」
//                ↑ 後面緊接 ()，就跟 foo() 一樣，是「呼叫前面那個值」
```

換句話說：**編輯器不是看到「兩個括號黏在一起」覺得奇怪，而是看到「一個值 + `()`」覺得很正常**——這跟 `foo()` 在它眼裡是同一件事。

唯一真的會被當成「錯」的，是**沒有第一對括號**的版本（見下方「表達式 vs 陳述式」）：

```js
function () {}();   // ❌ 以 function 開頭 → 被當成「宣告」(動作，不是值) → 後面 () 沒東西可呼叫 → SyntaxError
```

## 表達式（expression）vs 陳述式（statement）

這是理解「為什麼 IIFE 要包括號」的關鍵。

### 表達式 expression：會「求出一個值」

只要一段程式碼最後會**算出一個值**，它就是表達式。因為它代表一個值，所以**可以被賦值給變數、可以當參數傳進函式、可以放進更大的表達式裡**。

| 種類 | 例子 | 求出的值 |
|---|---|---|
| 字面表達式（literal） | `42`、`"hi"`、`true`、`[1,2]`、`{a:1}` | 那個字面值本身 |
| 算術表達式 | `1 + 2 * 3` | `7` |
| 邏輯表達式 | `a && b`、`x \|\| y` | `true` / `false` 或其中一個運算元 |
| 函式呼叫表達式 | `Math.max(1, 2)` | 呼叫後的**回傳值** `2` |
| 函式表達式 | `function () {}`、`() => {}` | 一個**函式值** |

```js
const n = 1 + 2;              // 算術表達式 → 求出 3 → 賦值給 n
foo(1 + 2);                   // 表達式當參數傳入
const fn = function () {};    // 函式表達式 → 求出函式值 → 賦值給 fn
```

### 陳述式 statement：執行一個「動作」，本身不是值

陳述式是「做一件事」的指令，它**不求值**，所以**不能拿去賦值或當參數**。

```js
if (x > 0) { ... }       // 陳述式
for (...) { ... }        // 陳述式
let y = 1;               // 宣告陳述式
function foo() { ... }   // 函式「宣告」陳述式
```

### 兩者一連起來，就懂 IIFE 的括號了

```js
function () {}();   // ❌ SyntaxError
```
因為一行**以 `function` 開頭**，引擎會把它當成「函式**宣告**陳述式」，而宣告是個動作、不是值，後面直接接 `()` 呼叫沒有意義 → 報錯。

用括號包起來，強迫它被當成**表達式**：
```js
(function () {})();   // ✅ 括號裡是函式「表達式」→ 求出一個函式值 → 才能立刻 () 呼叫
```

> 口訣：**括號把「函式宣告」變成「函式表達式」，有了值才能立刻呼叫。**

## IIFE 可以 `return`！（怎麼把結果給外部用）

IIFE 本質就是一個函式，**當然能 `return`**。而且因為「IIFE 整段 `(...)()` 是一個表達式、會求出一個值」，這個 `return` 出來的值就會變成那個表達式的值，**可以直接存進變數**。

### 最簡單的例子：算完把值交出來

```js
const hypotenuse = (function () {
  const a = 3, b = 4;
  return Math.sqrt(a * a + b * b);   // ← return 的值變成整段的值
})();

console.log(hypotenuse);   // 5  ← 外部拿到了
```

### 進階：return 一個物件 →「模組模式 Module Pattern」（最經典用法）

把**私有變數藏在裡面**，只 `return` 出想公開的方法，外部透過回傳的物件操作：

```js
const counter = (function () {
  let count = 0;                       // 🔒 私有：外部完全碰不到（鎖在閉包裡）

  return {                             // 📤 只暴露這些公開方法
    increment() { count++; return count; },
    decrement() { count--; return count; },
    get()       { return count; },
  };
})();

counter.increment();   // 1
counter.increment();   // 2
counter.get();         // 2
counter.count;         // undefined  ← 私有變數拿不到，達成封裝
```

重點：
- `count` 是私有的，外面改不到，只能透過 `increment / get` 這些公開介面 → 這就是**封裝**。
- 外部用的是 IIFE **回傳的那個物件**（存在 `counter` 裡），而不是 IIFE 函式本身。

## 為什麼要用 IIFE：製造獨立作用域

避免變數污染到全域：

```js
var x = "global";
(function () {
  var x = "只活在這個函式裡";   // 不會蓋掉外面的 x
})();
console.log(x);   // "global"
```

## 模組化（modularity）vs 模組系統（module system）

你抓到的矛盾點：「ES6 前**沒有模組系統**，但 IIFE 又是當年**模組化**手法」——關鍵是這兩個詞不一樣：

| | 模組化 modularity | 模組系統 module system |
|---|---|---|
| 是什麼 | 一種**目標 / 概念**：把程式切成獨立封裝、互不污染的單元，各自管好私有狀態，只暴露必要的公開介面 | 一套**語言/環境提供的正式語法與機制**，能正式宣告依賴、跨檔案載入 |
| 例子 | 用 IIFE 圍出作用域 + `return` 公開 API（就是上面的 Module Pattern） | ES Module 的 `import` / `export`；Node 的 `require` / `module.exports` |
| 誰提供 | 你自己土法煉鋼做出來的「效果」 | 語言 / 執行環境**內建的工具** |

所以那句話的真正意思：

> ES6 之前**沒有官方的模組系統（沒有 `import/export` 這種語法）**，但工程師還是想要**模組化的效果（封裝、不污染全域、只暴露公開介面）**，於是借用 IIFE 的「作用域隔離 + return 公開物件」**手工打造**出模組的樣子 —— 這套手法就叫 **Module Pattern（模組模式）**。
>
> ES6 之後有了正式的 `import/export` 模組系統，這種土法煉鋼就少用了，但讀舊程式碼還是常遇到。

一句話：**模組化是「想達成的目標」，模組系統是「達成它的官方工具」。當年沒有官方工具，就用 IIFE 自己做。**

## IIFE 只會執行一次嗎？→ ✅ YES

**會，只執行一次。** 原因：
- 那個函式**定義完當下就立刻被呼叫一次**，而且通常是**匿名、沒有被任何變數保存引用**，所以之後**沒有辦法再呼叫它第二次** → 外層函式體只跑一次。

⚠️ 容易混淆的點（釐清一下，免得誤會）：
- 如果 IIFE `return` 了物件 / 函式並存起來（像上面的 `counter`），那**回傳的那些方法**（`increment`、`get`）可以被呼叫**無數次**；
- 但 **IIFE 外層函式本身**仍然**只執行了那一次**（只做了一次初始化，`let count = 0` 只跑一次）。閉包讓那份私有狀態一直活著，所以後續每次 `counter.increment()` 操作的都是**同一個** `count`。

```js
const counter = (function () {
  let count = 0;            // ← 整個 IIFE 只跑一次，count 只初始化一次
  return { inc() { return ++count; } };
})();

counter.inc();  // 1
counter.inc();  // 2   ← 方法被呼叫多次，但外層 IIFE 沒有再執行
```
