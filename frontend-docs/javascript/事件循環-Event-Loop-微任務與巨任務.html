重點整理

JavaScript 是單執行緒（Single Thread）的語言，同一時間只能執行一個任務（只有一個 Call Stack 在運作）。但網頁要處理大量非同步操作（API 請求、計時器、使用者點擊），為了不讓畫面在等待時「凍結」，瀏覽器用 Event Loop 機制達成「非同步、非阻塞（Non-blocking）」。

四個核心組成：

  • 呼叫堆疊 Call Stack：追蹤目前正在執行的函式，遵循後進先出 LIFO。
  • Web APIs：瀏覽器提供的環境（非 JS 引擎本身），負責 setTimeout、fetch、DOM 事件等的倒數或等待。
  • 工作佇列 Callback / Task Queue：非同步任務完成後，其回呼函式被移入這裡排隊。
  • Event Loop：工作非常單純——持續監控 Call Stack，當 Call Stack 完全清空時，就把佇列中的第一個任務推進 Call Stack 執行。

微任務 vs 巨任務（決定執行先後）

類型 優先權 常見來源
微任務 Microtask 極高 Promise.then()、async/await、MutationObserver
巨任務 Macrotask 較低 setTimeout、setInterval、I/O、UI 渲染

面試守則：每當 Call Stack 清空，Event Loop 會一次清空目前所有微任務，才會去執行下一個巨任務；而在執行下一個巨任務前，瀏覽器會視情況決定是否進行畫面渲染。

隨堂考（是非／輸出順序）

console.log('1');
setTimeout(() => { console.log('2'); }, 0);
Promise.resolve().then(() => { console.log('3'); });
console.log('4');

正確輸出順序：1 → 4 → 3 → 2。
原因：1、4 是同步碼先執行；3 是微任務，優先於巨任務 2（setTimeout 即使設 0ms 也要排到巨任務佇列）。

追加 2026-07-02：Call Stack 到底怎麼運作（洋芋片罐子比喻）

把 Call Stack 想成一個有底的洋芋片罐子／疊盤子架，規則就是 LIFO（後放進去的先拿出來）。每呼叫一個函式就 Push 一片進罐子頂端，函式執行完（遇到 return 或結尾）就 Pop 掉。

function eatBreakfast() {
  makeCoffee();
  console.log("吃完早餐了！");
}
function makeCoffee() {
  console.log("咖啡泡好了！");
}
eatBreakfast(); // 程式從這裡開始

堆疊變化：[eatBreakfast] → 遇到 makeCoffee 變 [makeCoffee, eatBreakfast]（eatBreakfast 暫停等待）→ makeCoffee 印完被 Pop 回 [eatBreakfast] → eatBreakfast 印完 Pop → [空]。JS 只有一個罐子，所以叫單執行緒——最上面那片沒拿走，下面的只能死守等待。

追加：單執行緒 vs 同步 vs 非同步（別再搞混）

單執行緒講的是「廚房只有一個廚師」（資源限制）；同步/非同步講的是「廚師做事的模式」（流程設計）。

觀念 核心定義 以 Call Stack 為主的運作 生活化比喻
單執行緒 Single Thread 同一時間只能做一件事 整個 JS 引擎只有一個 Call Stack 廚房只有一個廚師、一雙手
同步 Synchronous 前一件沒做完，下一件絕不開始 任務進 Stack 若花 5 秒就阻塞（Blocking）5 秒，網頁凍結 廚師守在鍋邊 10 分鐘什麼都不做
非同步 Asynchronous 遇到耗時任務先交代出去，自己繼續做 丟給 Web APIs 倒數／等資料，Stack 立刻清空往下跑 廚師把水壺放上爐子就轉頭去切菜

常見迷思澄清：「非同步是不是一次能執行多個任務？」不是。 任何一微秒裡 JS 的 Call Stack 永遠只有一行程式碼在跑。非同步的真相是：JS 本人（廚師）永遠單執行緒，是瀏覽器環境（Web APIs，多執行緒）在旁邊幫忙等待與計時；等外包做完，JS 再找空檔處理結果。所以「同時燒水又切菜」是整個網頁系統看起來的樣子，不是 JS 引擎本人。

追加：宏/微任務的排隊是誰在管？

Web APIs 只負責「計時／等待」，不負責分類與排隊。 分類排隊是 JS 引擎內部的 Event Loop 在管：

  • setTimeout、DOM 事件 → 時間到後，Web API 把回呼丟進宏任務佇列 Macrotask Queue。
  • Promise.then() → 根本不經過 Web API（Promise 是 JS 原生自帶），狀態完成時引擎直接塞進微任務佇列 Microtask Queue。
  • Event Loop（餐廳經理）：Stack 清空後先把微任務佇列「全部」清光，才抓一個宏任務進 Stack；每執行完一個宏任務會再回頭檢查有沒有新微任務插隊。

追加：為什麼微任務要優先？（銀行櫃檯比喻）

微任務＝「因目前任務衍生、需要立刻接著做的收尾工作」（客人辦到一半說「順便刷個存摺」，順手辦完比叫他重新抽號碼牌有效率）。技術上兩大理由：

  1. 優化渲染效能：把所有微任務清空、資料徹底算好後，瀏覽器才一次性批量渲染（Batch Update），避免連續改 DOM 造成畫面閃爍。
  2. 確保資料即時性：Promise 拿到 API 資料要立刻更新狀態，若排到 setTimeout／scroll 後面會造成資料不同步（Race Condition）。

追加：宏/微任務是看「時間長短」分的嗎？

完全不是看時間，而是看「血統／來源」，且 W3C 規範定死。 引擎在執行前根本無法預知程式會跑多久。

類型 嚴格限定來源
微任務 Promise.then/catch/finally、async/await、MutationObserver、queueMicrotask()
宏任務 setTimeout/setInterval、I/O（讀檔／網路回傳）、UI 渲染／DOM 事件

反直覺例子：setTimeout(fn,0)（宏任務、瞬間結束）遇上一個內含 5 秒死迴圈的 Promise.then（微任務）→ 網頁會先卡死 5 秒印出微任務，才印出那個 0 秒的 setTimeout。證明微任務再慢也優先。

追加：為什麼 I/O 與 UI 渲染是宏任務？

  • I/O：是來自作業系統／硬體的「全新外部事件」（何時完成 JS 無法預測），不是引擎內部的收尾，沒資格插隊微任務 → 乖乖去宏任務排隊。
  • UI 渲染：是最笨重的工程（Style → Layout → Paint → Composite）。若做成微任務，改 1000 次就渲染 1000 次會當機；所以瀏覽器等微任務清空、資料算完後，才依固定頻率（60Hz≈每 16.6ms）批量渲染一次。

追加：完整的事件循環一輪順序（含 rAF）

requestAnimationFrame（rAF）是專為渲染而生的非同步：不屬宏也不屬微，有自己的隊伍，嚴格綁在渲染流水線最前端（下次重繪前執行），最適合做動畫位移計算，保證不掉影格。一個影格（16.6ms）內的嚴格順序：

  1. 宏任務：執行一個（點擊事件／定時器到期）。
  2. 微任務：執行並徹底清空（Promise.then）。
  3. rAF：若該更新畫面，先執行 requestAnimationFrame。
  4. UI 渲染：Style → Layout → Paint → Composite，畫面更新。
  5. Idle：有剩餘時間才跑 requestIdleCallback（零碎空閒做不緊急雜事）。

工作佇列 vs 呼叫堆疊（Gemini 手繪示意圖重點）

Abby 請 Gemini「畫給我看」，其產生的示意圖標題為 「工作佇列與呼叫堆疊：他們是不同的系統！」，核心對照（圖已轉成文字，未另存圖檔）：

  • Call Stack（呼叫堆疊）：主執行緒 Main Thread，立即執行 JS；LIFO；圖中疊著 main() → calcTotal() → draw()（執行中），靠 PUSH/POP 進出。
  • Web APIs / Node.js（外部環境）：處理耗時任務 setTimeout API、fetch API；完成後標記 Completed Task。
  • Job Queue / Task Queue（工作佇列）：存放已完成任務的回呼 callback1()、callback2()…，不立即執行、等待排隊。
  • Event Loop：監控 Stack 與 Queue，當 Stack 空了才從 Queue 取出 callback 放入 Stack。
  • 核心差異：Call Stack 是「現在」要執行的工作（Work in Progress）；Job Queue 是「未來」要排隊執行的工作（Waiting List）——兩者是不同的系統。
