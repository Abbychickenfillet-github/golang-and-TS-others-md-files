🧠 C. 自我作答打字 + 檢查

用自己的話回答，按「檢查」比對關鍵字，再看參考答案。

C1. 事件代理解決了哪兩個痛點？

① 效能／記憶體：1000 個 li 只綁 1 個監聽器（每個 addEventListener 都吃記憶體）。② 動態新增元素自動生效：監聽器在永遠存在的父層，新生的 li 不必再綁，被點時冒泡到父層就會觸發。

C2. 為什麼「動態新增」的 li 不用再綁事件，被點也能觸發？

因為監聽器綁在永遠存在的父層（ul / document）。新生的 li 本身沒綁任何東西，但被點時訊號會冒泡往上傳給父層，父層的監聽器就會被觸發並用 e.target 認出它。

C3. e.stopPropagation() 和 e.target 會衝突嗎？為什麼？

不衝突，分工不同：e.target 是「偵探」指出哪個元素被點；e.stopPropagation() 是「保全」阻止事件繼續往外擴散。

C4. 要讓 <a> 點了不要跳轉，用哪個方法？

e.preventDefault()。它阻止「瀏覽器內建預設行為」（a 跳轉、form 送出、右鍵選單）。注意：它和 stopPropagation 都不會阻止 target 本身的 JS（如 console.log）執行。

C5. 把事件綁在 document 上，舉兩個實戰用途。

① 全網頁動態圖片燈箱：判斷 e.target.tagName === 'IMG' 就放大（動態載入的圖也通用）。② 點空白處關閉彈窗：判斷 e.target.id !== 'my-image' 就關掉跳窗。

C6. 為什麼 DOM 需要「冒泡」這個階段？（它和捕獲是 W3C 定義的兩個傳播階段）

為了讓外層（祖先）元素能在事件浮上來時統一處理子元素的事件＝事件代理：① 省資源（只綁 1 個監聽器）② 動態新增的子元素不必再綁，被點時冒泡上來就會被父層接住處理。捕獲（由外往內）和冒泡（由內往外）是 W3C 定義的兩個傳播階段，中間夾著「目標」抵達點。

⭕❌ D. 是非題先判斷再點看答案

心裡先決定「對 / 錯」，再點「看答案」對照理由。

D1. 每寫一個 addEventListener，瀏覽器就會在記憶體裡建立一筆監聽器；綁太多會吃記憶體。
✅ 對。這正是「事件代理」要把監聽器集中綁在父層的原因——1000 個 li 只綁 1 個，而不是 1000 個，省記憶體也避免卡頓。
D2. e.stopPropagation() 會連 target 自己的 console.log 也一起擋掉。
❌ 錯。它只擋「往外傳播」，不會擋 target 自己的 JS。連 e.preventDefault()（擋瀏覽器內建行為）也一樣不會擋你寫的 console.log。
D3. e.target.tagName 拿到的是小寫字串 'li'。
❌ 錯。永遠是大寫 'LI'；寫 'Li' 會對不到。防呆用 .toLowerCase() 或 e.target.matches('li')。
D4. addEventListener 第三參數預設是 false，也就是在冒泡階段才執行。
✅ 對。要在捕獲階段執行才寫 true（或 {capture:true}）。
D5. 把外層監聽器改成捕獲（true），點內層時外層會比內層先執行。
✅ 對。捕獲由外往內，外層在「下樓」時就先跑了，所以順序翻轉成 外→內。
D6. 捕獲和冒泡是 W3C 標準定義的兩個傳播階段；冒泡讓外層能「代理」處理子元素的事件。
✅ 對。事件流＝捕獲（下樓）＋冒泡（上樓）兩個傳播方向，中間夾著「目標」抵達點。有了冒泡，外層才能在事件浮上來時統一處理（事件代理）。

🐞 E. 找錯題點看答案

下面這段想「點祖父就印出阿公」，但其實有好幾個錯。先自己抓，再看答案。

const grandpa = document.querySelectorbyId("grandpa");
grandpa.addEventListener(click, console.log("阿公"));
E1. 找出全部的錯與修正。
① querySelectorbyId 不存在 → 用 document.getElementById("grandpa")（不加 #）或 document.querySelector("#grandpa")（加 #）。
② click 沒加引號＝被當成變數 → 要寫字串 "click"。
③ 最關鍵：console.log("阿公") 是「立刻呼叫」，回傳值 undefined 才被當成 handler → 「阿公」在跑到那行就印一次、之後點擊毫無反應。要傳「函式本身」不是呼叫結果。
④ 想拿 e.target.id 必須用「會接收參數的函式」；e 是瀏覽器自動帶入 handler 的第一個參數。
✅ 正解： 
const grandpa = document.getElementById("grandpa");
grandpa.addEventListener("click", (e) => {
  console.log("阿公", e.target.id);
});
E2. addEventListener("click", fn()) 和 addEventListener("click", fn) 效果一樣。
❌ 錯。fn 是「傳函式本身」（點擊才執行、瀏覽器自動帶入 e）；fn() 是「立刻呼叫」、把回傳值當 handler。只有當 fn 會回傳另一個函式時 fn() 才有意義。
E3. getElementById 的參數要加 #，例如 getElementById("#grandpa")。
❌ 錯。getElementById 傳純 id（不加 #）：getElementById("grandpa")。加 # 是給 querySelector("#grandpa") 用的（它吃 CSS 選擇器）。
E4. 出現 Cannot read properties of undefined (reading 'id')，代表 .id 寫錯了。
❌ 錯。.id 沒寫錯；是「.id 前面那個東西是 undefined / null」。鏈式存取 a.b.id 報「讀 'id'」＝ a.b 是 undefined。常見 e.target 或 e.currentTarget 取到 null。
典型陷阱：事件結束後才讀事件（存進變數、放 setTimeout、await）→ currentTarget 會被重設為 null（但 target 還在）。所以「target 可以、currentTarget 爆」就是這個。修法：在 handler 內當下同步讀，或先 const id = e.currentTarget.id 存起來。

🎯 F. querySelector vs getElementById⭕❌ 故意寫錯

每題判斷對(⭕)或錯(❌)，再點看答案。重點看 # 的有無！

F1. document.getElementById("#parent") 抓得到 id="parent" 的元素。
❌ 錯。getElementById 傳純 id、不加 #。寫 "#parent" 會去找 id 字面叫「#parent」的元素→找不到、回傳 null。正解 getElementById("parent")。
F2. document.querySelector("#parent") 抓得到 id="parent" 的元素。
⭕ 對。querySelector 吃 CSS 選擇器，id 要加 #。
F3. document.querySelector("parent") 抓得到 id="parent" 的元素。
❌ 錯。沒加 # 會被當成標籤名 <parent>，找不到→null。要寫 querySelector("#parent")。
F4. document.getElementById("parent") 抓得到 id="parent" 的元素。
⭕ 對。這是正解寫法（純 id、不加 #）。
F5. document.querySelector(".item") 會回傳「所有」 class 是 item 的元素。
❌ 錯。querySelector 只回傳第一個符合的。要全部用 querySelectorAll(".item")（回傳 NodeList）。
F6. 找不到元素時，getElementById 和 querySelector 都回傳 null。
⭕ 對。接著對它 .addEventListener 或 .id 就會爆「Cannot read properties of null」。querySelectorAll 找不到則回傳空的 NodeList（不是 null）。

🐫 G. 命名慣例（camelCase）⭕❌

分清楚「屬性的名字」和「屬性回傳的值」。

G1. tagName 寫成 camelCase，是因為它是 JS 屬性（JS/DOM API 命名慣例）。
⭕ 對。JS/DOM 的屬性、方法都 camelCase：innerHTML、textContent、className、querySelector、getElementById、parentNode…
G2. el.tagName 回傳的值也是 camelCase，例如 'Li'。
❌ 錯。屬性名是 camelCase，但回傳值是大寫 'LI'（DOM 規範）。兩種大小寫別混。
G3. HTML 標籤要寫成 camelCase，例如 <Div> 才正確。
❌ 錯。HTML 標籤名大小寫不敏感，慣例寫小寫 <div>。camelCase 是 JS 屬性那邊的事。
G4. HTML 的 class="..."，在 JS 對應的屬性是 el.className。
⭕ 對。因為 class 是 JS 保留字＋camelCase 慣例，所以 JS 端叫 className。
G5. HTML 的 data-user-id="..."，在 JS 用 el.dataset.userId 取得。
⭕ 對。HTML 的 kebab-case data-user-id → JS 自動轉成 camelCase dataset.userId。
