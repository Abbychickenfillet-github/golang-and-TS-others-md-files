① 事件流是「單向一條龍」先建立直覺

點一個 <li>,訊號的旅程固定是:捕獲(由外→內) → 目標 → 冒泡(由內→外)。它不會擴散給兄弟元素,只會經過自己的祖先鏈。

② 現場：點一下,看順序怎麼跑親手玩

點最內層藍色「子 li」。每層可切 冒泡↔捕獲,或開 stopPropagation 看事件被攔截。

祖父 #grandpa div
父 #dad ul
（這裡會顯示觸發順序…）
把祖父切成「捕獲」再點 → 祖父會變第一個跑(捕獲由外往內,先經過它)。這證明順序是看「階段」決定的,不是看誰被點。

③ e.target vs e.currentTarget

看上面 console 那兩欄就懂了:

e.targete.currentTarget
意思真正被點的元素(事件起點)監聽器掛在誰身上(此刻經手那層)
點 kid 時永遠是 kidkid → dad → grandpa(逐層變)
會不會變整場不變每一層都不同
比喻:e.target = 信封上的寄件人(誰寄的,不會變);e.currentTarget = 此刻經手的收發室。一般 function 裡 e.currentTarget === this。

④ 現場：事件代理為什麼省資源又能接動態元素親手玩

監聽器只綁在 <ul> 一個。點任何 li 會變灰;按「新增」生出來的新 li,不用再綁也能直接點(因為冒泡會傳到 ul)。

綁定的監聽器數量：1 個（不管幾個 li）
  • 項目一
  • 項目二
  • 項目三
（點 li 看 e.target…）
const ul = document.querySelector('ul');
ul.addEventListener('click', (e) => {
  if (e.target.tagName === 'LI') {        // tagName 永遠大寫
    e.target.classList.toggle('on');     // 用 e.target 精準命中被點的 li
  }
});
沒有事件代理的災難:1000 個 li 要綁 1000 個監聽器(吃記憶體);而且後來才新增的 li 點了沒反應,要每次手動再綁。代理把這兩個問題一次解決。

⑤ stopPropagation vs preventDefault（常搞混的大魔王）

方法管的事經典場景
e.stopPropagation()阻止「傳播」(事件流到我為止,別再傳給其他節點)事件代理時防止冒泡亂觸發外層
e.preventDefault()阻止「瀏覽器內建預設行為」阻止 a 跳轉、form 送出重整、右鍵選單
兩者都不會阻止 target 本身的 JS(如 console.log)執行;它們也不衝突——e.target 是偵探(指出誰),stopPropagation 是保全(擋擴散)。
stopPropagation() 裝在哪個階段就中斷哪個階段(裝捕獲就到不了目標;裝冒泡就不再往上)。

⑥ React 怎麼寫

React 內部用 SyntheticEvent,預設也在冒泡階段。最推薦資料驅動:用 state 記住被點的 index,綁在 map 出來的 li,不碰原生 DOM。

const [activeIndex, setActiveIndex] = useState(null);

{items.map((item, index) => (
  <li key={index}
      onClick={() => setActiveIndex(index)}
      style={{ backgroundColor: activeIndex === index ? 'gray' : 'transparent' }}>
    {item}
  </li>
))}
若硬要用事件代理(綁 <ul>),仍要 e.target.tagName === 'LI'——記得大寫,寫 'Li' 會對不到;防呆用 .toLowerCase() 或 e.target.matches('li')。
