# 事件流與事件代理（Event Flow & Delegation）

> 互動考題版：[[事件流與事件代理-互動考題.html]]（點擊看答案／可輸入作答）

## 重點整理

### 事件流三階段（單向一條龍，不是廣播擴散）

點擊一個 `<li>`，訊號的旅程是<mark style="background: #FFF3A3A6;">捕獲 → 目標 → 冒泡</mark>，是「單向旅行」不是「擴散給所有兄弟元素」：

- <mark style="background: #ADCCFFA6;">捕獲（Capturing）</mark>：由外往內 `Window → Document → <ul> → <li>`
- 目標（Target）：抵達真正被點的 `<li>`
- <mark style="background: #ADCCFFA6;">冒泡（Bubbling）</mark>：由內往外 `<li> → <ul> → Document → Window`

點第三個 `<li>`，過程<mark style="background: #FF5582A6;">完全不會經過第一、第二個 `<li>`</mark>，兄弟元素全程不知情。

### e.target vs e.currentTarget

- `e.target`＝<mark style="background: #FFF3A3A6;">真正被點擊的元素</mark>（信封上的「寄件人」，例如第三個 `<li>`）
- `e.currentTarget`＝監聽器掛在誰身上（信封上的「收件地點」，例如 `<ul>`）
- 點同一個 `<li>` 時，`e.target` 從頭到尾<mark style="background: #FFF3A3A6;">不變</mark>（都是被點的 li）；`e.currentTarget` <mark style="background: #ADCCFFA6;">每層都不同</mark>（li→ul→document，誰在經手就是誰）。
- 直接綁在自己身上（點誰、綁誰是同一個）→ 兩者一樣；事件代理（綁父層）→ 兩者不同，這就是它存在的意義。
- 一般 `function` 裡 `e.currentTarget === this`。

### addEventListener 第三個參數：決定在哪一階段被觸發

```js
element.addEventListener('click', fn);        // 省略/false → 冒泡階段執行（預設、最常見）
element.addEventListener('click', fn, true);  // true → 捕獲階段執行
```

> 瀏覽器的物理順序永遠是「先捕獲（下樓）→ 目標 →再冒泡（上樓）」；我們能選的只是要在下樓還是上樓時攔截。平常什麼都不寫＝冒泡時觸發。

### 為什麼需要「冒泡」？（W3C 的兩個傳播階段）

事件流＝<mark style="background: #ADCCFFA6;">捕獲（由外往內，下樓）</mark>＋<mark style="background: #ADCCFFA6;">冒泡（由內往外，上樓）</mark>這兩個 <mark style="background: #FFF3A3A6;">W3C 標準定義的「傳播階段」</mark>，中間夾著「目標（target）」抵達點。

冒泡存在的價值：讓<mark style="background: #BBFABBA6;">外層（祖先）元素能在事件浮上來時，統一處理子元素的事件</mark>＝事件代理（省資源、動態新增元素自動生效）。

驗證：把外層監聽器改成捕獲（`true`），點內層時外層會「先」執行（捕獲由外往內），順序翻轉成 <mark style="background: #FFB8EBA6;">外→內</mark>——這證明<mark style="background: #FF5582A6;">順序是由「階段」決定，不是由「誰被點」決定</mark>。

### 事件監聽器與記憶體（重點）

> [!warning] 每寫一個 `addEventListener`，瀏覽器就會在「記憶體」裡建立一筆監聽器
> 綁太多（例如 1000 個 `<li>` 各綁一個）會默默吃掉記憶體、效能差的手機會卡頓。這正是<mark style="background: #BBFABBA6;">事件代理把監聽器集中綁在父層、只綁 1 個</mark>的根本原因。

### addEventListener 常見錯誤（handler 要「傳函式」、e 自動傳入）

```js
// ❌ 錯誤示範
const grandpa = document.querySelectorbyId("grandpa");   // 沒有這個方法
grandpa.addEventListener(click, console.log("阿公"));     // click 沒引號、console.log 被立刻呼叫
```

三個關鍵錯誤：

1. `document.querySelectorbyId` <mark style="background: #FF5582A6;">不存在</mark>。用 `getElementById("grandpa")`（傳 id、不加 `#`）或 `querySelector("#grandpa")`（傳 CSS 選擇器、加 `#`）。
2. 事件類型要是<mark style="background: #FFF3A3A6;">字串</mark> `"click"`，不加引號會被當成變數（ReferenceError）。
3. <mark style="background: #FF5582A6;">最關鍵</mark>：第二參數要傳「**函式本身**」，不是「呼叫函式的結果」。`console.log("阿公")` 會被立刻執行（印一次），把回傳值 `undefined` 當 handler → 點擊毫無反應、根本沒註冊到函式。

> [!tip] e 是瀏覽器「自動傳入」handler 的第一個參數
> 只要把 handler 寫成「會接收參數的函式」（`(e) => {...}` 或 `function(e){...}`），瀏覽器每次觸發就會自動把事件物件塞進 `e`，才能用 `e.target.id`。同理 `addEventListener("click", fn)` 傳參考（會帶入 e）；`addEventListener("click", fn())` 是<mark style="background: #FF5582A6;">立刻呼叫</mark>、把回傳值當 handler（除非 fn 會回傳另一個函式）。

```js
// ✅ 正解
const grandpa = document.getElementById("grandpa");
grandpa.addEventListener("click", (e) => {
  console.log("阿公", e.target.id);
});
```

### 實戰踩坑：阿公／阿爸／兒子三層 div（2026-06-16）

寫三層巢狀 div 練事件流時，踩到三個錯，整理成對照表：

| 錯誤寫法 | 報錯 | 正確 |
|---|---|---|
| `e.current.id` | `Cannot read properties of undefined (reading 'id')` | 沒有 `e.current` 這個屬性，是 `undefined`，讀 `.id` 就爆。要用 <mark style="background: #FFF3A3A6;">`e.currentTarget`</mark>（綁在誰身上）或 `e.target`（被點的） |
| `getElementById("sonVar")`（HTML 裡 id 其實是 `son`） | 下一行 `sonVar.addEventListener` → `Cannot read properties of null` | `getElementById` 找不到時<mark style="background: #FF5582A6;">回傳 `null`</mark>（不是報錯），對 `null` 呼叫方法才爆。id 要跟 HTML 完全一致 |
| CSS `#grandpa { text: red; }` | 不報錯，但<mark style="background: #FF5582A6;">顏色不生效</mark> | <mark style="background: #FFF3A3A6;">沒有 `text` 這個 CSS 屬性</mark>，改文字顏色用 `color: red;`。瀏覽器遇到無效屬性會「無聲忽略」，所以最難察覺 |

> [!tip] `undefined` vs `null` 的報錯差別
> - 讀「不存在的屬性」（如 `e.current`）→ 拿到 `undefined`，再 `.id` → `Cannot read properties of **undefined**`。
> - `getElementById` 沒找到 → 拿到 `null`，再 `.addEventListener` → `Cannot read properties of **null**`。
> 看報錯是 undefined 還是 null，能快速分辨是「打錯屬性名」還是「DOM 沒抓到」。

### 事件代理（Event Delegation）＝請家長代辦

把監聽器統一綁在共同父層（`<ul>` 或 `document`），利用冒泡，用 `e.target` 判斷是誰被點：

```js
const parentUl = document.querySelector('ul');
parentUl.addEventListener('click', function (e) {
  if (e.target.tagName === 'LI') {        // 確認點到的是 <li>
    e.target.style.backgroundColor = 'gray';
  }
});
```

解決兩大痛點：

1. <mark style="background: #BBFABBA6;">效能／記憶體</mark>：1000 個 `<li>` 只需綁 **1 個**監聽器，不是 1000 個。（每個 `addEventListener` 都會在記憶體建立一筆監聽關係，大量綁定會吃記憶體。）
2. <mark style="background: #BBFABBA6;">動態新增元素自動生效</mark>：監聽器綁在「永遠存在的父層」，之後才生出來的新 `<li>` 不必再綁，被點時冒泡到父層一樣會觸發。

### stopPropagation vs preventDefault（超級大魔王，常搞混）

| 方法 | 管的事 | 經典場景 |
|---|---|---|
| `e.stopPropagation()` | <mark style="background: #FF5582A6;">阻止「傳播」</mark>（事件流到我為止，不再傳給其他節點） | 事件代理防止冒泡亂觸發外層（如防跳窗亂關） |
| `e.preventDefault()` | <mark style="background: #FF5582A6;">阻止「瀏覽器內建預設行為」</mark> | 阻止 `<a>` 跳轉、`<form>` 送出重整、右鍵選單 |

- `stopPropagation()` 裝在哪個階段就中斷哪個：裝捕獲階段就到不了目標；裝冒泡階段（最常見）就不再往上傳。
- <mark style="background: #D2B3FFA6;">兩者都不會阻止 target 節點本身的 JS 程式碼（如 console.log）執行。</mark>
- `e.target` 與 `e.stopPropagation()` 不衝突：一個是「偵探」（指出是誰），一個是「保全」（阻止往外擴散），分工合作。

### tagName 大寫陷阱

`e.target.tagName` 回傳的標籤名<mark style="background: #FF5582A6;">永遠大寫</mark>（`'LI'`、`'DIV'`、`'IMG'`）。寫 `'Li'` 會對不到。防呆：

```js
if (e.target.tagName.toLowerCase() === 'li') { /* ... */ }
// 或更寬容、可丟 CSS 選擇器：
if (e.target.matches('li')) { /* ... */ }
```

### 為什麼 `tagName` 是 camelCase？（別把三種大小寫搞混）

`tagName` 是 **JS 屬性**，JS/DOM 的屬性與方法照慣例都用 camelCase（`innerHTML`、`textContent`、`className`、`querySelector`、`getElementById`、`parentNode`…）。這跟「值是大寫」無關，是兩件事：

| 東西 | 大小寫 | 例子 | 原因 |
|---|---|---|---|
| HTML 標籤本身 | 小寫（慣例） | `<div>` `<li>` | HTML 標籤名大小寫不敏感 |
| JS 屬性／方法名 | <mark style="background: #ADCCFFA6;">camelCase</mark> | `tagName` `innerHTML` | JS/DOM API 命名慣例 |
| `el.tagName` 回傳值 | <mark style="background: #FF5582A6;">大寫</mark> | `'LI'` `'DIV'` | DOM 規範：HTML 文件回傳大寫 |

延伸：HTML 小寫/kebab 進 JS 會變 camelCase——HTML `class` → JS `el.className`（`class` 是 JS 保留字）；HTML `data-user-id` → JS `el.dataset.userId`。

口訣：<mark style="background: #FFF3A3A6;">HTML 小寫、JS 屬性 camelCase、tagName 的「值」大寫</mark>，三層別混。

### 綁在 document 的實戰用途

- 全網頁動態圖片燈箱：`document` 上判斷 `e.target.tagName === 'IMG'` 就放大（圖片是動態載入也通用）。
- 點空白處關閉彈窗：`document` 上判斷 `e.target.id !== 'my-image'` 就關閉跳窗。

### React 怎麼寫？

React 內部用 SyntheticEvent，預設也在冒泡階段處理。

**作法一（最推薦，資料驅動）**：用 state 記住被點的 index，`onClick` 綁在 map 出來的 `<li>`，不碰原生 DOM：

```jsx
const [activeIndex, setActiveIndex] = useState(null);
// ...
{items.map((item, index) => (
  <li key={index}
      onClick={() => setActiveIndex(index)}
      style={{ backgroundColor: activeIndex === index ? 'gray' : 'transparent' }}>
    {item}
  </li>
))}
```

**作法二（事件代理，綁在 `<ul>`）**：仍要用 `e.target.tagName === 'LI'`（記得大寫）。

> 一句話死背核心：記憶體很貴，別給每個小孩請保鏢；在大門口（document／父元素）裝一個監視器，只要有小孩開門（冒泡），警衛（e.target）就能認出是哪個小孩並幫他開燈。
