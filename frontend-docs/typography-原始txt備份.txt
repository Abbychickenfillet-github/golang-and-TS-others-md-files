● 為什麼是 1.333

  1. 模組化比例（Modular Scale）的核心

  每一階等於前一階 × 固定比例。好處：

  - 視覺有數學一致性——階與階間看起來「成系統」而非隨手打的數字
  - 決策成本低——你不用在做卡片時再想「要 18 還是 20？」，抓 --text-lg 就對
  - 響應式好延伸——整個 scale 可以乘上根 size，所有比例同步縮放

  2. 常見比例光譜

  ┌───────┬────────────────┬───────────────────────────┬───────────────┐
  │ 比例  │      名稱      │           感覺            │     適用      │
  ├───────┼────────────────┼───────────────────────────┼───────────────┤
  │ 1.125 │ Major Second   │ 太接近，階層糊            │ ❌ 中文尤其糟 │
  ├───────┼────────────────┼───────────────────────────┼───────────────┤
  │ 1.2   │ Minor Third    │ 溫和、安靜                │ 文件、長閱讀  │
  ├───────┼────────────────┼───────────────────────────┼───────────────┤
  │ 1.25  │ Major Third    │ 安全常見（Tailwind 類似） │ 一般 SaaS     │
  ├───────┼────────────────┼───────────────────────────┼───────────────┤
  │ 1.333 │ Perfect Fourth │ 清晰、有戲但不誇張        │ 作品集、面試  │
  ├───────┼────────────────┼───────────────────────────┼───────────────┤
  │ 1.5   │ Perfect Fifth  │ 強烈戲劇                  │ 行銷 Landing  │
  ├───────┼────────────────┼───────────────────────────┼───────────────┤
  │ 1.618 │ Golden Ratio   │ 編輯感、雜誌味            │ Hero 專用     │
  └───────┴────────────────┴───────────────────────────┴───────────────┘

  3. 為什麼面試情境選 1.333（關鍵三點）

  a. 超過 impeccable 門檻——skill 規定「至少 1.25」才能建立清楚階層，1.333
  有安全邊際。

  b. 中文需要比英文更大的比例——這是中英排版最常被忽略的點：
  - 中文字是方塊、筆畫密（視覺密度高）
  - 同 point size 下內部複雜度比 Latin 字母高很多
  - 比例若太小，階層會被「字的內部細節」淹沒

  英文 1.2 看得清層級，中文就糊成一片。1.333 對中文是剛剛好的臨界值。

  c. 面試場景的眼動節奏——HR / 面試主管滑得快，H1→H2→body 要在 0.5
  秒內被視覺優先級「分層排序」。1.333 給你戲劇，但不會像 1.5
  那樣看起來像行銷網站（對工程職不加分）。

  4. 為什麼 xs/sm/base 刻意不守比例

  --text-xs:   0.75rem;   // 12px
  --text-sm:   0.9375rem; // 15px ← 打破 1.333
  --text-base: 1rem;      // 16px ← 打破 1.333

  這是故意的。小字級（UI
  chrome、標籤、註記、caption）用途是功能性而非閱讀性，需要 12 / 14 / 15 / 16
  這種 1-2px 微調的「UI 級距」，不是戲劇階層。嚴格套 1.333 會變成 9→12→16，9px
  在中文幾乎不可讀。

  規則：閱讀層（body 以上）嚴守比例；UI 層（body
  以下）用像素微調。這是成熟設計系統的通則（Radix、Vanta、Linear 都這樣）。

  ---
  簡而言之：1.333 = 跨越中文字密度門檻的最小值 ×
  面試場景的戲劇需求上限。再小階層糊，再大像行銷網站。

  可以當 npm 套件、會被打包進 Docker——但 CJK 有體積陷阱。

  三種常見做法

  ┌────────────────────────────┬────────────────────────┬──────────────────┐
  │            方式            │        字體來源        │  打包進 Docker?  │
  ├────────────────────────────┼────────────────────────┼──────────────────┤
  │ A. npm                     │                        │ ✅ 走 npm ci →   │
  │ 套件（@fontsource/*）      │ node_modules/          │ build → .next/st │
  │                            │                        │ atic/media/      │
  ├────────────────────────────┼────────────────────────┼──────────────────┤
  │                            │ 專案內 app/fonts/*.wof │ ✅ 走 COPY +     │
  │ B. next/font/local         │ f2（commit 到 repo）   │ build 重命名     │
  │                            │                        │ hash             │

  │ C. next/font/google（你現  │ Next.js build 時從     │ ✅ 但需要 build  │
  │ 在用的）                   │ Google 下載、自動      │ 階段有網路       │
  │                            │ self-host              │                  │
  └────────────────────────────┴────────────────────────┴──────────────────┘

  你目前專案的實際流向（用方式 C）

  Dockerfile build stage:
  ┌─────────────────────────────────────────┐
  │ npm ci                                  │
  │ npm run build ← 此時 Next.js 連 Google │
  │   ↓ 下載 Noto Sans TC .woff2           │
  │   ↓ 處理後放到 .next/static/media/     │
  └─────────────────────────────────────────┘
               ↓ COPY
  Dockerfile production stage:
  ┌─────────────────────────────────────────┐
  │ .next/static/media/noto-sans-tc-xxx.woff2 │  ← 字體最終在這裡
  └─────────────────────────────────────────┘

  結論：字體已經打包進你的 Docker image，不是 runtime 再去 Google
  抓。上線後使用者只跟你的 server 要字體，不會洩漏 IP 給 Google（GDPR 友善）。

  ⚠️ 你該注意的三個現實問題

  1. CJK 字體超級肥

  ┌────────────────────────┬──────────────┐
  │          字體          │ 單一字重大小 │
  ├────────────────────────┼──────────────┤
  │ Noto Sans TC（4 字重） │ ~40MB        │
  ├────────────────────────┼──────────────┤
  │ Geist（全字重）        │ ~200KB       │
  └────────────────────────┴──────────────┘

  中文字是英文字的 200 倍大。你現在 weight: ['300', '400', '500', '700']
  四個字重 ≈ 40MB 塞進 image。

  2. Next.js 對 CJK 不會自動 subset

  next/font/google 對 Latin 會按 unicode-range 切片，只載入你實際用到的字。但
  CJK 沒有「只載入這頁用到的 500 字」的自動機制——整包拖。

  3. Build 階段網路依賴（方式 C 的風險）

  如果你的 CI／Docker build 環境不能連外網（防火牆、私有
  registry、離線環境），next/font/google build 會失敗。這時要切 方式 
  A（@fontsource） 或 方式 B（本地檔）。

  🎯 建議給你的架構

  # Latin 走 next/font/google（build 時下載、極輕）
  npm install @fontsource-variable/geist

  # CJK 走 @fontsource（npm 套件、版本鎖定、無 build-time 網路需求）
  npm install @fontsource/noto-sans-tc
  npm install @fontsource/noto-serif-tc

  好處：
  - ✅ 版本鎖進 package-lock.json
  - ✅ 離線環境也能 build
  - ✅ Docker layer cache 友善（字體在 node_modules/ 層，改程式不會重下字體）
  - ✅ 部署後字體直接 bundle 在 image 裡

  缺點：Docker image 還是會因 CJK 肥。如果在意體積，後面可以再做「subset 成 2000
   常用字」的進階優化（用 pyftsubset 或 subset-font）。

❯   2. Next.js 對 CJK 不會自動 subset 是什麼意思                                

● 字體 subset（切片）是什麼

  Subset = 從完整字體檔裡只抽出「實際會用到的字」做成較小的檔。

  Latin 的自動切片怎麼運作

  Google Fonts 提供的 CSS 長這樣（簡化）：

  /* latin 切片 */
  @font-face {
    font-family: 'Geist';
    src: url(/geist-latin.woff2) format('woff2');
    unicode-range: U+0000-00FF, U+0131, U+0152-0153...;  /* 基本 ASCII + 西歐 */
  }
  /* latin-ext 切片 */
  @font-face {
    font-family: 'Geist';
    src: url(/geist-latin-ext.woff2) format('woff2');
    unicode-range: U+0100-017F, U+0180-024F...;  /* 東歐、土耳其 */
  }
  /* cyrillic 切片 */
  @font-face {
    font-family: 'Geist';
    src: url(/geist-cyrillic.woff2) format('woff2');
    unicode-range: U+0301, U+0400-045F...;  /* 俄文 */
  }

  瀏覽器的聰明：每個 @font-face 有 unicode-range 宣告它涵蓋的字元範圍。瀏覽器掃
  描頁面真的出現的字元，只下載包含那些字元的切片檔。

  - 你頁面全是英文 → 只下載 latin.woff2（~20KB）
  - 出現一個法文 é → 再下載 latin-ext.woff2
  - 沒出現俄文 → cyrillic.woff2 永遠不下載

  Next.js 把這些切片 self-host 到 .next/static/media/，機制完全保留。

  CJK 為什麼不能這樣自動做

  問題 1：沒有清楚的切片邊界

  Latin 可以按「西歐/東歐/越南文」這種語言/地區切。但 CJK：

  - 整個 U+4E00 – U+9FFF（CJK Unified Ideographs）是一整塊 20,000+ 漢字
  - 「基本 vs 進階」沒有語言學界線——你不知道使用者會打哪 2000 字
  - 一篇部落格文章可能用到 散落在整個區塊的 1500 字

  Google Fonts 其實有硬切成 ~100 個小檔按字頻分（最常用 500 字、次常用 500
  字…），但這是 by chunk 不是 by meaning，效果遠不如 Latin。

  問題 2：Next.js 的 subsets 選項根本沒有 CJK

  看 Next.js 官方支援的 subset 名稱：

  Noto_Sans_TC({
    subsets: ['latin'],       // ✅
    subsets: ['latin-ext'],   // ✅
    subsets: ['cyrillic'],    // ✅
    subsets: ['greek'],       // ✅
    subsets: ['vietnamese'],  // ✅
    subsets: ['chinese-traditional'], // ❌ 不存在
    subsets: ['chinese-simplified'],  // ❌ 不存在
    subsets: ['japanese'],            // ❌ 不存在
  })

  所以你無法像 Latin 那樣叫 Next.js「只幫我打包繁中字集」。

  這對你目前的程式有個現實後果

  你的 app/layout.js:15-20：

  const notoSansTC = Noto_Sans_TC({
    subsets: ['latin'],  // ← 你只跟 Next.js 說要 latin
    weight: ['300', '400', '500', '700'],
  })

  實際結果（要驗證但很可能如此）：
  - Next.js 只下載並 self-host Noto Sans TC 的 Latin 切片
  - 你頁面上的所有中文字，Noto Sans TC 字體檔裡沒有對應 glyph
  - 中文 fallback 到你 font-family stack 的下一個：-apple-system, 
  BlinkMacSystemFont, 'Segoe UI'...
  - 結論：你以為在用 Noto Sans TC 顯示中文，實際上 Windows 使用者看到的是 Segoe
  UI 找不到中文字再 fallback 到系統中文字體（微軟正黑、PMingLiU），macOS
  使用者看到 PingFang——每個平台長得不一樣

  怎麼辦？三個策略

  ┌─────────────────┬─────────────────────────┬─────────────────────────────┐
  │      策略       │          作法           │            取捨             │
  ├─────────────────┼─────────────────────────┼─────────────────────────────┤
  │ A. 整包拖（simp │ 移除 subsets 改用       │ ~10MB/字重，首載慢，但字形  │
  │ lest）          │ preload: false，讓字體  │ 統一                        │
  │                 │ lazy-load 全字集        │                             │
  ├─────────────────┼─────────────────────────┼─────────────────────────────┤
  │ B. 換用         │ 安裝 @fontsource/noto-s │ 版本鎖、離線                │
  │ @fontsource +   │ ans-tc（或 Pro 版 @font │ build、可手動選 subset 檔案 │
