① Critical Rendering Path 是什麼

關鍵渲染路徑指的是瀏覽器把 HTML、CSS、JavaScript 轉換成螢幕上「實際像素」所必須經過的一連串步驟。優化 CRP，就是讓「首次有意義的畫面」更快出現（影響 FCP / LCP 等指標）。

一句話記憶：把字串（程式碼）變成像素（畫面）的最短關鍵流程。

1. DOM Tree：解析 HTML，建立節點樹。

2. CSSOM Tree：解析 CSS，建立樣式樹（CSS 是渲染阻塞資源）。

3. Render Tree：把 DOM ＋ CSSOM 合併，只留下會顯示的節點（display:none 不會進 render tree，但 visibility:hidden 會）。

4. Layout（又叫 Reflow / 重排）：計算每個節點的幾何資訊——確切的位置、寬高、相對關係。

5. Paint（又叫 Repaint / 重繪）：把節點轉成實際像素，填上顏色、文字、邊框、陰影、背景。

6. Composite（合成）：把多個圖層（layers）依正確順序疊合成最終畫面，常由 GPU 處理。

③ 核心問題：重排（Reflow）和重繪（Repaint）哪一個先？

結論：重排（Reflow / Layout）先，重繪（Repaint / Paint）後。
在流程順序上是 Layout → Paint。瀏覽器一定要先算出「東西在哪、多大」，才有辦法「在那個位置塗上顏色」。

為什麼是這個順序？用蓋房子比喻

🏗️ 重排 = 量地、放隔間、定家具位置（決定幾何）。
🎨 重繪 = 油漆牆面、貼壁紙（決定外觀）。

你不可能在「還沒決定牆在哪」之前就先油漆，所以一定先量好（重排），再上色（重繪）。

更關鍵的一條規則（面試加分）

🔁 重排「一定」會引發重繪；但重繪「不一定」會引發重排。
因為位置/大小一旦改變（重排），那塊區域的像素必須重畫（重繪）。但若只是改顏色（重繪），幾何沒變，就不需要重排。所以重排比重繪更昂貴，效能優化要優先避免重排。

④ 什麼操作會觸發？

會觸發「重排」的常見動作

// ❌ 改幾何 → 重排（接著一定重繪）
el.style.width = "300px";
el.style.display = "block";
el.style.margin = "20px";
document.body.appendChild(newNode);   // 增刪節點
window.resize / 字體大小改變 / 內容變多

// ⚠️ 讀取「即時幾何值」會強制同步重排（layout thrashing）
el.offsetWidth; el.offsetHeight; el.getBoundingClientRect();
el.scrollTop; getComputedStyle(el);

只觸發「重繪」、不重排的動作

// ✅ 只改外觀 → 只重繪
el.style.color = "#fff";
el.style.backgroundColor = "tomato";
el.style.visibility = "hidden";       // 注意：仍佔位，所以不重排
el.style.boxShadow = "0 2px 8px #000";
陷阱題：visibility:hidden vs display:none
visibility:hidden：元素還佔空間 → 只重繪。
display:none：元素從版面消失 → 觸發重排（連幾何都變了）。

⑤ 優化技巧（減少重排重繪）

  1. 批次修改樣式：用 class 一次切換，或先 cssText 一次寫完，不要一行一行改。
  2. 讀寫分離：避免「讀幾何 → 寫樣式 → 又讀幾何」交錯，會造成 layout thrashing（強制多次同步重排）。先全部讀、再全部寫。
  3. 離線操作：先 display:none 或用 DocumentFragment 改完再一次插回 DOM。
  4. 用 transform / opacity 做動畫：走合成層，跳過重排重繪。搭配 will-change 提前提示瀏覽器。
  5. 動畫用 requestAnimationFrame：在下一次重繪前執行，和瀏覽器的刷新節奏對齊，避免掉幀。見 [[requestAnimationFrame]]。
// ❌ Layout thrashing：讀-寫-讀-寫交錯，每次都強制重排
for (const el of boxes){
  el.style.width = el.offsetWidth + 10 + "px"; // 讀後馬上寫
}

// ✅ 讀寫分離：先讀完，再寫完
const widths = boxes.map(el => el.offsetWidth); // 全部讀
boxes.forEach((el,i)=> el.style.width = widths[i] + 10 + "px"); // 全部寫

Q5（申論）請說明完整的 Critical Rendering Path 流程，並解釋為什麼「重排一定先於重繪」。再舉一個前端優化重排重繪的實際做法。

① CRP 流程：解析 HTML 建立 DOM → 解析 CSS 建立 CSSOM → 兩者合併成 Render Tree（只含可見節點）→ Layout（重排） 計算每個節點的位置與大小 → Paint（重繪） 把節點轉成像素、上色 → Composite 把圖層合成最終畫面。

② 為何重排先於重繪：重繪需要知道「要在哪裡、多大範圍」塗顏色，而這些幾何資訊正是重排階段算出來的。沒有幾何就無法上色，所以順序必為 Layout → Paint。延伸：重排改變幾何，該區域必須重畫，因此重排一定引發重繪；而重繪只改外觀、幾何不變，不會反過來引發重排，故重排成本更高。

③ 優化實例：用 transform: translate() 取代修改 top/left 來做位移動畫，讓動畫走 GPU 合成層，跳過重排與重繪；或採「讀寫分離」避免 layout thrashing；或用 class 批次切換樣式。

Q6（申論）什麼是 layout thrashing（強制同步重排）？為什麼會發生？怎麼避免？

瀏覽器原本會把多次 DOM 修改合併，在適當時機一次重排（批次最佳化）。但當你在修改樣式後馬上讀取即時幾何值（如 offsetWidth、getBoundingClientRect()），瀏覽器為了給你正確數字，被迫立刻同步重排。若在迴圈中「寫一次、讀一次」反覆交錯，就會觸發多次強制重排，效能急遽下降，這就是 layout thrashing。

避免方法：讀寫分離——先把所有需要的幾何值一次讀完存起來，再統一進行所有寫入；或使用 requestAnimationFrame 安排寫入，避免在同一幀內讀寫交錯。
