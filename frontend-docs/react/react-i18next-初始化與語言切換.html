react-i18next 初始化整合與語言切換

重點整理

1. i18n.use(initReactI18next) 為什麼要這樣傳遞

把 i18n 實例整合進 React 生態系的關鍵步驟，主要有四個原因：

  • 提供 Context 全域存取：把 i18n 實例綁到 React 的 Context，讓所有 Hook（useTranslation）／HOC（withTranslation）都能直接取用，不必每個元件手動 import。
  • 自動觸發 UI 更新（Re-render）：切換語言（i18n.changeLanguage）時通知元件重新渲染。沒有這層綁定，元件感知不到語系改變，文字不會自動更新。
  • 處理預設值與安全性：協助 React 特有設定，例如 interpolation: { escapeValue: false }——因為 React 本身已防 XSS，不需 i18next 再轉義一次。
  • 生命週期管理：確保 i18next 初始化與 React 載入流程正確銜接，避免翻譯資源還沒載入就報錯。
import i18n from "i18next";
import { initReactI18next } from "react-i18next";

i18n
  .use(initReactI18next) // 核心步驟：把 i18n 傳遞給 react-i18next
  .init({
    resources,
    lng: "en",
    interpolation: {
      escapeValue: false, // React 已安全處理，不需重複轉義
    },
  });

2. lng: 'en' 不是把語言鎖死，而是「啟動預設語系」

  • lng 是 Initial Language（初始語系）：使用者第一次開網頁時顯示哪種語言。
  • 寫死 'en' → 不會偵測瀏覽器語系。
  • 進階做法：搭配 i18next-browser-languagedetector，讓初始語系變動態（自動抓使用者瀏覽器設定）。
  • 若完全不設 lng，可能因找不到語系資源而直接顯示原始的 Key。

3. i18n.changeLanguage 在哪裡呼叫

是 i18n 實例上的 方法（Method），通常綁在「語系切換按鈕」的點擊事件。

方法 A — React 元件內（推薦，用 Hook 取得）：

import { useTranslation } from 'react-i18next';

function LanguageSwitcher() {
  const { i18n } = useTranslation();
  return (
    <button onClick={() => i18n.changeLanguage('fr')}>
      切換至法文
    </button>
  );
}

方法 B — 一般 JS/TS 檔案內（匯入全域實例）：

import i18n from './i18n'; // 匯入設定好的 i18n 實例
const toggleLanguage = (lng) => {
  i18n.changeLanguage(lng);
};

[!warning] 注意 執行 changeLanguage('fr') 後，i18next 會去 resources 找對應語系檔；若 resources 沒定義該語系，畫面就無法正確顯示翻譯。

自我測驗

是非題：初始化寫 lng: 'en' 代表整個專案語言被鎖死成英文。
✗ 錯。 `lng` 只是啟動時的預設語系，之後仍可用 `changeLanguage` 切換；要自動偵測瀏覽器語系可搭配 `i18next-browser-languagedetector`。
填空：React 環境設定 escapeValue: false 的原因是 ____ 已內建防護機制。
React 本身已防範 XSS，不需 i18next 再做一次轉義。
申論題：不呼叫 i18n.use(initReactI18next) 會有什麼後果？
i18n 實例不會綁進 React Context：`useTranslation` 等 Hook 取不到實例，且切換語言時元件收不到通知、不會 re-render，介面文字無法自動更新。
