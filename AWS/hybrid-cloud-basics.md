# 混合雲（Hybrid Cloud）基礎筆記

> 為什麼不是「全上雲」或「全地端」二選一？
> 因為大多數公司的真實場景，都需要兩邊互通取長補短。

---

## 1. 一句話定義

**混合雲 = 雲端（AWS / Azure / GCP）+ 地端（on-premise，自己機房）同時運作、彼此互通**，把不同 workload 分散在兩邊各取所長。

```
                  ┌─── 雲端（AWS）
                  │      - 突發流量、彈性服務
                  │      - 全球節點
                  │
   你的系統 ─────┤  互通（VPN / Direct Connect）
                  │
                  └─── 地端（on-prem）
                         - 敏感資料、合規要求
                         - 內部用戶就近存取
                         - 既有硬體投資
```

---

## 2. 優點（10 點，按重要性）

| # | 優點 | 為什麼 / 例子 |
|---|---|---|
| 1 | **漸進式上雲** | 老系統留地端、新系統放雲端，不用 big bang migration、風險可控 |
| 2 | **資料主權 / 合規** | 敏感資料留地端（金融、醫療、政府法規 e.g. PCI-DSS、HIPAA、金管會），一般運算放雲端 |
| 3 | **成本最佳化** | 穩定流量地端攤銷便宜、突發流量用雲端彈性擴展 |
| 4 | **Cloud Bursting（爆量擴展）** | 平常地端跑、峰值（黑五、雙 11、報稅日）多出來的流量導到雲端 |
| 5 | **災難復原（DR）** | 地端為主、雲端為備（或反過來），機房整個炸了還能切換 |
| 6 | **延遲分流** | 低延遲服務留地端（工廠 IoT、店面 POS），運算密集上雲端（AI 訓練、批次）|
| 7 | **避免廠商鎖定** | 不把所有雞蛋放 AWS 一個籃子，換廠商 / 斷雲時還有地端撐 |
| 8 | **既有投資保留** | 已買的伺服器、license 不浪費（沉沒成本不變廢鐵）|
| 9 | **IT 團隊技能過渡** | 既有運維團隊慢慢轉型 DevOps，不用一夕換腦 |
| 10 | **內網就近使用** | 公司內部使用者連地端只要幾 ms、對外服務走雲端 CDN |

---

## 3. ⚠️ 代價（混合雲不是萬能）

| 缺點 | 為什麼 |
|---|---|
| **架構複雜度暴增** | 兩套 ops、兩套 monitoring、兩套 IAM——人力成本變兩倍 |
| **網路打通難又貴** | VPN 慢、Direct Connect / ExpressRoute 一個月幾萬到幾十萬 |
| **安全模型分裂** | AWS IAM ≠ 地端 AD，要做身分整合（SAML / IAM Identity Center）|
| **資料同步延遲與一致性** | 兩邊資料庫怎麼同步？最終一致性還是同步寫入？容易踩坑 |
| **疑難排錯困難** | 問題出在哪？地端網路？AWS？中間 VPN？——排查路徑變長 |

## 4. 典型 workload 分配模式

### 模式 A：穩定地端 + 突發雲端（Cloud Bursting）

```
日常：流量 100% 走地端（伺服器都吃飽）
峰值：超出地端容量的部分自動 spin up 雲端 EC2 接手
峰值結束：雲端 EC2 自動 terminate
```

**典型場景**：電商、報稅系統、賽事直播

---

### 模式 B：核心地端 + 周邊雲端（Edge）

```
核心：交易系統、客戶資料庫 → 留地端
周邊：CDN、媒體處理、推薦系統、AI 推論 → 上雲端
```

**典型場景**：金融、醫療、政府

---

### 模式 C：地端做開發、雲端做生產（罕見）

```
DEV：地端便宜、安全、隨便玩
PROD：雲端彈性、全球節點、高可用
```

---

### 模式 D：雲端為主、地端為備（DR）

```
平時：100% 跑在雲端
災難：AWS 整個 Region 出事 → 切到地端最小集
```

## 5. AWS 上的混合雲常用元件

### 網路打通

| 元件 | 做什麼 | 何時用 |
|---|---|---|
| **AWS Direct Connect** | 拉專線到 AWS，繞過公網（高頻寬、低延遲、固定收費）| 大流量、穩定要求高、長期 |
| **Site-to-Site VPN** | 用公網建 IPSec 通道（便宜、上手快、頻寬不穩）| 小流量、PoC、臨時 |
| **AWS Transit Gateway** | 多 VPC + 多地端 + 多 VPN 一個 hub 管 | 多分公司、多 VPC 複雜拓撲 |

### 把雲服務搬進地端

| 元件 | 做什麼 |
|---|---|
| **AWS Outposts** | AWS 把實體伺服器送進你的機房，跑真的 AWS 服務（EC2、EBS、RDS 在地端） |
| **AWS Snow 系列**（Snowball / Snowmobile）| 大量資料離線搬家（裝箱寄到 AWS）|
| **AWS Wavelength** | 把 AWS 服務塞進電信商的 5G 基地台（超低延遲）|

### 資料同步與儲存

| 元件 | 做什麼 |
|---|---|
| **AWS Storage Gateway** | 地端應用透明存取 S3 / EBS，當作本地檔案系統 |
| **AWS DataSync** | 地端 → 雲端的大量資料同步 |
| **AWS Database Migration Service (DMS)** | 把地端資料庫遷移到 RDS / Aurora |

### 統一管理

| 元件 | 做什麼 |
|---|---|
| **AWS Systems Manager** | 統一管理地端 + 雲端的 EC2 / 機器（patch、inventory、Run Command）|
| **AWS Config** | 跨地端 + 雲端的合規檢查 |
| **IAM Identity Center** | 把地端 AD / Okta 整合進 AWS（SSO）|

## 6. 決策樹：要不要採用混合雲？

```
你已經有地端機房 + 在跑業務嗎？
    │
    ├─ 沒有 ──► 直接全上雲（Cloud-Native），沒必要混合
    │
    └─ 有 ──► 法規 / 敏感資料要留地端嗎？
              │
              ├─ 是 ──► 必選混合雲（資料留地端、運算上雲）
              │
              └─ 否 ──► 既有投資多大？團隊轉型風險？
                        │
                        ├─ 投資大 / 團隊不熟 ──► 混合雲過渡
                        │
                        └─ 投資小 / 團隊強 ──► 直接全上雲（一次到位）
```

---

## 7. 一句話總結

> **混合雲不是「為了用而用」，是「為了讓對的 workload 跑在對的位置」**——敏感/穩定/低延遲留地端、彈性/突發/運算密集上雲端，雙方互通取長補短。
