JavaScript 事件循環與閉包（面試核心）

重點整理

一、事件循環 Event Loop

JavaScript 是單執行緒（Single Thread）語言，同一時間只有一個 Call Stack 在運作，一次只能做一件事。為了不讓頁面在等待（API、計時器、點擊）時凍結，瀏覽器用 Event Loop 達成非阻塞（Non-blocking）的非同步行為。

四個核心組成：

  • 呼叫堆疊 Call Stack：追蹤目前執行中的函式，遵循「後進先出（LIFO）」。
  • Web APIs：瀏覽器提供的環境（非 JS 引擎本身），處理 setTimeout、fetch、DOM 事件等的倒數/等待。
  • 工作佇列 Callback / Task Queue：非同步任務完成後，其 callback 移到這裡排隊。
  • 事件循環 Event Loop：持續監控 Call Stack，當 Stack 清空時把佇列中第一個任務推入執行。

微任務 vs 巨任務（決定執行先後）：

類型 優先權 常見來源
微任務 Microtask 高 Promise.then()、async/await、MutationObserver
巨任務 Macrotask 低 setTimeout、setInterval、I/O、UI 渲染

面試守則：每當 Call Stack 清空，Event Loop 會一次清空目前「所有」的微任務佇列，才會去執行下一個巨任務；執行下一個巨任務前，瀏覽器才視情況進行畫面渲染。

隨堂考（輸出順序）：

console.log('1');
setTimeout(() => { console.log('2'); }, 0);
Promise.resolve().then(() => { console.log('3'); });
console.log('4');

正確順序：1 → 4 → 3 → 2。1、4 是同步先跑；3 是微任務，優先於巨任務 2。

二、閉包 Closure

一句話定義：閉包是一個內層函式，能「記憶」並存取外層函式的範疇（Scope），即使外層函式已經執行完畢、離開 Call Stack。閉包不是要特別開啟的功能，而是函式建立時自然誕生的底層機制。

兩大基石：

  • 詞法範疇 Lexical Scope：變數存取權限在「程式碼被寫下來那一刻（宣告位置）」就決定，不是執行時決定。
  • 垃圾回收 Garbage Collection：函式執行完局部變數本應釋放，但若仍被內層函式引用，瀏覽器就不敢釋放，會保留在記憶體（Heap）中。

應用：用閉包建立私有變數（資料封裝）。

function createCounter() {
  let count = 0;               // 鎖在範疇內，外部摸不到
  return {
    increment() { count++; console.log(count); },
    decrement() { count--; console.log(count); }
  };
}
const counter = createCounter();
counter.increment(); // 1
counter.increment(); // 2
// console.log(count); // ReferenceError: count is not defined

經典面試題（var vs let）：

for (var i = 1; i <= 3; i++) {
  setTimeout(() => { console.log(i); }, 1000);
}
  • 直覺：依序 1, 2, 3
  • 實際：同時輸出 4, 4, 4
  • 原因：var 是函式作用域，迴圈裡其實成了同一個（近乎全域的）i；callback 執行時迴圈早跑完，i 已是 4，三個閉包指向同一個 i。
  • 修正：把 var 改成 let（區塊作用域，每次迭代各建立獨立範疇），輸出 1, 2, 3。

三、閉包 + 傳址陷阱（魔王題）

function createWallet(initialAmount) {
  let money = initialAmount;
  let history = [];
  return {
    checkBalance() { return money; },
    getHistory()   { return history; },   // 回傳「同一個」陣列參照
    spend(amount)  { if (amount <= money) { money -= amount; history.push(`Spent ${amount}`); } }
  };
}
const myWallet = createWallet(100);
const record = myWallet.getHistory();
myWallet.spend(30);
record.push('Found 100 on the street');
myWallet.spend(20);
console.log(myWallet.checkBalance()); // 50
console.log(myWallet.getHistory());   // ['Spent 30', 'Found 100 on the street', 'Spent 20']

關鍵：陣列/物件是傳址（Pass by Reference），record 與閉包內 history 指向同一塊記憶體，外部 record.push 直接污染了內部私有狀態（但 money 不受影響，仍是 70 → 50）。

防漏修正：回傳時切斷參照，回傳複本：

getHistory() { return [...history]; } // 展開運算子回傳新陣列

四、觀念釐清

  • return money 不會變成全域變數。money 既不在全域、也沒「回到」函式內部，而是被封裝在專屬該實例的閉包環境裡；return 只是把值「複印」一份丟給外面，本體仍留在閉包口袋中。
  • createWallet 才是函式物件（Function Object，可被呼叫）；myWallet 是普通物件（Plain Object），myWallet() 會 TypeError。物件裡綁定的 spend、checkBalance 稱為方法（Method）。
  • C++ 也有閉包（C++11 起的 Lambda），但無 GC：須明確「捕捉」[value]（傳值複製，安全）或 [&ref]（引用，類似 JS）。回傳引用捕捉了 Stack 區域變數會造成懸空引用（Dangling Reference）→ 未定義行為。JS 靠 GC 自動把變數留在 Heap，不會有這問題。
