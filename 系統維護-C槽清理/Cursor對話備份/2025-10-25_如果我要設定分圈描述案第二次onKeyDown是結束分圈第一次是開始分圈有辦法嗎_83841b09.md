## API 回應格式設計

從 API 端點的代碼可以看到，這個 API 採用了一種**標準化的回應格式**：

```javascript
// API 回應格式
return NextResponse.json({
  status: 'success',  // 或 'error'
  data: parsedData,   // 實際的資料內容
  message: '分圈計時器資料載入成功', // 狀態訊息
})
```

## 為什麼使用這種格式？

### 1. **統一的 API 回應結構**
```javascript
// 成功回應
{
  status: 'success',
  data: { /* 實際資料 */ },
  message: '操作成功'
}

// 錯誤回應  
{
  status: 'error',
  message: '錯誤訊息'
}
```

### 2. **前端處理的便利性**
```javascript
// 在前端可以統一處理
if (data.status === 'success' && data.data) {
  // 處理成功的情況
  const lapData = data.data
  setTitle(lapData.title || '')
  setDesc(lapData.desc || '')
  // ...
} else {
  // 處理錯誤的情況
  console.error(data.message)
}
```

### 3. **錯誤處理的一致性**
```javascript
// 無論是網路錯誤還是業務邏輯錯誤，都能統一處理
if (!response.ok) {
  throw new Error(`HTTP error! status: ${response.status}`)
}

const result = await response.json()
if (result.status !== 'success') {
  throw new Error(result.message || '操作失敗')
}
```

## 這種設計的優點：

1. **可預測性**：前端開發者知道回應一定會有 `status` 和 `message`
2. **錯誤處理**：可以區分 HTTP 錯誤和業務邏輯錯誤
3. **擴展性**：未來可以輕鬆添加新的欄位（如 `code`、`timestamp` 等）
4. **除錯友善**：錯誤訊息清晰，便於問題排查

## 對比其他常見格式：

```javascript
// 直接回傳資料（不推薦）
return NextResponse.json(lapData)

// 使用 data 包裝（推薦）
return NextResponse.json({
  status: 'success',
  data: lapData,
  message: '成功'
})
```

所以這裡使用 `data` 格式是一個很好的 API 設計實踐，讓前後端的資料交換更加規範和可靠！

## 1. `[LapTimer]` 特殊標記的用法

這是一個**命名空間標記**的設計模式：

```javascript
// 在資料庫中儲存時加上標記
const lapTimerTitle = `[LapTimer] ${title || '分圈計時器'}`

// 查詢時使用標記來區分
title: {
  startsWith: '[LapTimer]', // 只查詢分圈計時器相關的記錄
}

// 回傳給前端時移除標記
title: lapTimerData.title.replace('[LapTimer] ', '')
```

### 為什麼需要這個標記？

1. **區分不同類型的 TimeLog**：
   - `[LapTimer] 跑步訓練` - 分圈計時器
   - `工作會議` - 一般時間記錄
   - `[Project] 專案開發` - 專案時間記錄

2. **避免資料混淆**：同一個 `TimeLog` 表可能有多種用途

3. **查詢效率**：可以快速篩選特定類型的記錄

## 2. 為什麼查詢「未結束的」活動？

這是**狀態管理**的設計：

```javascript
endTime: null, // 只查詢未結束的活動
```

### 原因分析：

1. **即時狀態恢復**：
   - 用戶刷新頁面時，需要恢復「進行中」的計時器
   - 如果查詢已結束的，就無法繼續計時

2. **分圈計時器的特性**：
   - 分圈計時器通常是「長時間進行」的活動
   - 用戶可能中途關閉瀏覽器，重新開啟時需要繼續

3. **資料庫設計邏輯**：
   ```javascript
   // 未結束 = 進行中
   isRunning: lapTimerData.endTime === null
   
   // 已結束 = 歷史記錄
   if (endTime !== null) {
     // 這是已完成的活動，不需要恢復
   }
   ```

## 3. Cookie 與 401 狀態碼的因果關係

**因果關係：Cookie 不存在 → 401 未授權**

```javascript
// 步驟 1: 檢查 Cookie 是否存在
const cookie = (await cookies()).get('ACCESS_TOKEN')?.value
if (!cookie) {
  // 步驟 2: Cookie 不存在 → 直接回傳 401
  return NextResponse.json(
    { status: 'error', message: '未授權訪問' },
    { status: 401 }  // HTTP 狀態碼
  )
}

// 步驟 3: Cookie 存在但無效 → 也回傳 401
const session = await decrypt(cookie)
if (!session?.payload?.userId) {
  return NextResponse.json(
    { status: 'error', message: '未授權訪問' },
    { status: 401 }
  )
}
```

### HTTP 狀態碼的意義：

- **401 Unauthorized**：表示「未授權」，需要身份驗證
- **403 Forbidden**：表示「已授權但無權限」
- **500 Internal Server Error**：表示「伺服器內部錯誤」

### 流程說明：

1. **Cookie 不存在** → 立即回傳 401
2. **Cookie 存在但無效** → 解密失敗，回傳 401  
3. **Cookie 有效** → 繼續執行業務邏輯

所以是 **Cookie 不存在導致 401**，而不是相反。

這個設計是合理的：

- 查詢未結束的活動是為了恢復進行中的計時器狀態
- 401 狀態碼正確反映了身份驗證失敗的情況

## 1. `'[LapTimer]'` 這裡一定要有引號嗎？

**是的，一定要有引號！**

```javascript
// ✅ 正確 - 有引號，這是字串
title: {
  startsWith: '[LapTimer]', // 字串文字
}

// ❌ 錯誤 - 沒有引號，程式會報錯
title: {
  startsWith: [LapTimer], // 這會被當作變數，但沒有定義
}
```

**原因：**
- 引號告訴 JavaScript 這是一個**字串文字**（string literal）
- 沒有引號的話，JavaScript 會認為 `[LapTimer]` 是一個**變數名**或**陣列**
- 如果變數沒有定義，就會出現 `ReferenceError`

## 2. 所以我也可以用 `[Project]專案開發` 這樣？
