## Cursor
這段 `CASE` 是在把舊欄位的值轉成新字串狀態：

```
UPDATE member
SET email_verified = CASE
  WHEN email_verified IN ('1', 1, TRUE) THEN 'verified'
  WHEN email_verified IN ('verified', 'pending', 'failed', 'not_yet') THEN email_verified
  ELSE 'not_yet'
END;
```

- 第一個 `WHEN`：如果舊資料存的是 `1` / `TRUE`（也就是「已驗證」的 boolean 寫法），就改成字串 `'verified'`。
- 第二個 `WHEN`：如果資料本來就已經是新格式的四種字串之一，就照抄，不再改變。這行是避免你重複跑 migration 時把新值又改掉。
- `ELSE`：剩下任何不是上述情況的值（例如 `0`、`FALSE`、`NULL` 等），全部視為「尚未驗證」，統一寫成 `'not_yet'`。

所以只有兩類結果：舊的 true → `'verified'`；其他 → `'not_yet'`。第二個 `WHEN` 只是保護第二次/重複執行 migration 時不會把已經是 `'pending'` 等新狀態的資料再改掉，看起來像「兩種狀態」其實是在容錯。

### 為什麼 `order` 表既有 `id` 又有 `uuid`？

打開 `backend/app/models/order.py` 可以看到：

```python
class OrderBase(SQLModel):
    uuid: str = Field(default_factory=lambda: str(uuid.uuid4()), ...)

class Order(OrderBase, table=True):
    id: str = Field(default_factory=lambda: str(uuid.uuid4()), primary_key=True, ...)
```

所以資料表同時有：

- `id`: 真正的主鍵（primary key），給其他表的 foreign key 用，或讓資料庫索引最佳化。
- `uuid`: 業務用的 “訂單編號”，透過 `uuid.uuid4()` 生成，方便對外顯示、與第三方系統交握時使用。

這種設計蠻常見：`id` 保持內部一致性，`uuid` 當作公開的識別碼，避免曝露自動增加的流水號。若希望只保留一個欄位，需要重新設計 schema，把 `uuid` 改為 primary key，並調整所有外鍵/程式碼引用；目前的程式是刻意保留兩者。

所以現在的行為是合理的：`Order` 表照樣有 UUID 欄位，是因為模型繼承 `OrderBase`；如果不需要，可以改模型並跑新的 migration。

## Cursor
`docker compose up -d --build` 這個指令會：

- `--build`：在啟動容器前先重新 build 所有有變動的 image（包含 frontend、backend 等 compose 裡定義的服務）。  
- `-d`：detached mode，建完後在背景跑，終端不會卡在 log。

所以當你執行 `docker compose up -d --build`，其實前端的 Docker image 也會重新 build（只要 compose 裡有定義 frontend 服務，且 Dockerfile 沒跳過）。指令本身沒有「只重後端不重前端」的區別，除非你把 frontend 的 service 註解掉或沒寫 build 內容；不然 backend、frontend 都會被建置並啟動。

如果擔心前端版本和後端不同步，建議：

1. 每次修完程式先跑 `npm run build`（或 `npm run lint`）在本機確認。
2. Docker 要整包測試時就跑 `docker compose up -d --build`，等所有服務啟起來再看 log。  
3. 如果只想重建某個服務，可以用 `docker compose build frontend` 或 `docker compose up -d --build frontend`。

總結：`docker compose up -d --build` 會把 docker-compose.yml 裡的所有服務（前後端）都重新 build 並啟動，只是加了 `-d` 讓它在背景跑而已。
