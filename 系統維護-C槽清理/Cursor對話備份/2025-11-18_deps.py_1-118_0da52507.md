### 容器內實際路徑如何確認？
1. 先讓容器跑起來（或在 `docker compose up` 時按住讓它不要退出），然後執行：
   ```powershell
   docker compose exec backend bash
   ```
   如果服務沒起來，可以改跑：
   ```powershell
   docker compose run --rm backend bash
   ```
2. 進去後 `ls /` 看看有什麼目錄。一般情況下，Dockerfile 裡可能把程式放在 `/app`，而你的程式實際是在 `/app/app/pre_start.py`。確定之後再修改 `scripts/prestart.sh`。

### `prestart.sh` 正確寫法
假設容器裡 `pre_start.py` 的完整路徑是 `/app/app/pre_start.py`，腳本要調整成：

### Windows PowerShell/ CMD 怎麼改？
你本地透過 Docker Compose build 時用的是 PowerShell 或 CMD。但 `prestart.sh` 是在容器中執行的，所以只要修改腳本，再 `docker compose up --build` 即可，不需要另外用 PowerShell 跑 `uv run python ...`。

### 關於 docker daemon 與 port already allocated

### 建議流程
1. 進容器確認實際路徑，修改 `scripts/prestart.sh`.
2. `docker compose down`.
3. `docker compose up --build`.
4. 如果還看到 port 3003, 3000 等被占用，先停掉本地其他服務或改 port。

## Cursor
`set -e -x` 是在 shell 腳本裡開啟兩個常用的執行選項：

- `set -e`（或 `set -o errexit`）：一旦腳本中任意命令回傳非 0（失敗），整個腳本立即退出；避免錯誤被忽略。
- `set -x`（或 `set -o xtrace`）：執行每個命令前都會輸出該命令（含展開的參數）到標準錯誤，方便除錯。

通常在自動化腳本中兩者一起用，既能防止錯誤繼續執行，也能清楚看到每一步跑了哪些命令。

`docker compose run --rm backend bash` 需要取得終端輸入，所以不能在 `docker compose up` 目前的同一個視窗裡使用。你可以：

1. 開第二個終端視窗，在專案根目錄 (`C:\coding\template`) 執行：

   這會啟動一個新的 backend 容器並進入 `bash`，即使原來的 `docker compose up` 正在跑也沒關係。

2. 如果只想在現有容器裡開互動 shell，可以用：
   ```powershell
   docker compose exec backend bash
   ```
   但這需要 backend 服務本身已在 `docker compose up` 中啟動成功。如果因為錯誤而沒起來，就改用前面的 `run --rm`。

3. 在第一個視窗讓 `docker compose up --build` 持續輸出即可。需要輸入指令時，在另一個視窗執行即可，不會互相干擾。

這樣就能同時觀察 `up` 的輸出，又能開殼進容器確認檔案位置。
