這兩張表現在語意有點重疊：`order_electricity` 看起來是專門放「額外電力加購」的需求（瓦數、電壓等），但又沒有價格欄位；`order_item` 則是所有購買項目（含價格、數量、狀態）。如果把「加購電力」當成攤位必需品的一種商品，就應該落在訂單項目裡，這樣統計金額、付款狀態、退款都能共用相同流程。

建議做法：

1. **把價格放回 `order_item`**  
   - 每一個加購電力需求，建立一個 `order_item`，`product_id` 指向「電力加購」類型的商品（或設備 SKU）。  
   - `amount` = 插座或瓦數單位數，`price` = 單位價格。若有不同電壓/安培，可透過 `product_id` 或 `type` 區分。

2. **`order_electricity` 改成純屬性表**  
   - 只存放「電力需求細節」：wattage、voltage、quantity 等，並加上一個 `order_item_id` (FK) 指回剛剛的訂單項目。  
   - 這樣它就變成 `order_item` 的延伸資訊（類似 attributes），語意會比較準：`order_item` 負責金額與狀態，`order_electricity` 負責技術細節。

3. **流程**  
   - 使用者選電力 → 建立 `order_item`（金額、付款）+ 建立對應的 `order_electricity`（瓦數、電壓）。  
   - 取消或修改時，更新/刪除 `order_item` 同時 cascade `order_electricity`。

如果你覺得 `order_electricity` 這個名稱容易混淆，也可以改叫 `order_electricity_details` 或 `electricity_requirement`. 但重點是價格、付款狀態等財務資訊應全部維持在 `order_item` 表，這樣報表和金流都不會被拆散。

### 「Public」的意思？

在這份自動產生的 models 裡，命名慣例通常是：

- `SomethingBase`: 資料表的基本欄位
- `SomethingCreate` / `SomethingUpdate`: POST/PUT 用的 payload
- `SomethingPublic`: 後端 API 回傳給前端的公開資料結構 (DTO)

所以 `ProductPublic` 就是「後端回傳的商品資料長什麼樣」；`ProductCreate` 是「前端送資料時應該帶哪些欄位」。這些純粹是 TypeScript 型別，幫我們在寫程式時對照後端 schema，不會真的送到 API。

---

### 之後如果要新增型別，要怎麼做？

1. **後端先確定 schema**（最好更新 OpenAPI）。
2. 在 `frontend/src/client/models.ts` 補上對應的型別定義（或重新產生）。
3. 如果也有個別檔案（例如 `client/models/product.ts`）想拆出來寫，可以照目前做法寫 interface/type，再視需要在 `client/models/index.ts` 重新 export。
4. 保存後重新跑 `npm run lint` 或 `npm run build` 確認 TypeScript 沒抱怨。

現在檔案已更新，如果還有其他欄位要補或要重新產生整個 models.ts，再告訴我一起處理。
