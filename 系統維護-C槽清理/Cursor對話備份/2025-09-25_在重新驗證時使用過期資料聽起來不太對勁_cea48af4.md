## SWR 的 Stale-While-Revalidate 策略解釋

**Stale-While-Revalidate** 並不是"使用過期資料"，而是一個**效能優化策略**：

### 實際運作流程：

1. **首次請求**：獲取資料並快取
2. **資料變舊**：快取中的資料變成 "stale"（過期）
3. **用戶再次訪問**：
   - **立即顯示**：先顯示快取中的舊資料（stale data）
   - **背景更新**：同時在背景發送新請求獲取最新資料
   - **更新顯示**：新資料到達後，自動更新 UI

### 為什麼這樣設計？

```javascript
// 傳統方式（會讓用戶等待）
const data = await fetch('/api/users')  // 用戶要等 2 秒
return <UserList data={data} />

// SWR 方式（立即顯示，背景更新）
const { data } = useSWR('/api/users')  // 立即顯示舊資料
return <UserList data={data} />        // 新資料到達後自動更新
```

### 實際範例：


## 1. 為什麼 `typeof window !== 'undefined'` 會導致 Hydration 錯誤？

### Hydration 的本質
Hydration 是 Next.js 將**伺服器端渲染的 HTML** 與**客戶端 React 組件**進行"同步"的過程。

```javascript
// 伺服器端渲染 (SSR)
const serverHTML = renderToString(<MyComponent />)
// 結果：<div>載入中...</div>

// 客戶端 Hydration
const clientHTML = render(<MyComponent />)
// 期望：<div>載入中...</div> (必須與伺服器端完全一致)
// 實際：<div>上午12:38:16</div> (不匹配！)
```

### 問題的核心
```javascript
// 伺服器端 (Node.js 環境)
typeof window !== 'undefined'  // false (window 不存在)
// 渲染結果：載入中...

// 客戶端 (瀏覽器環境)  
typeof window !== 'undefined'  // true (window 存在)
// 渲染結果：currentTime.toLocaleTimeString()
```

2) Prisma 關聯欄位說明

`steps` 和 `user` 是關聯欄位，不會在資料庫表內實際儲存。

```prisma
model TimeLog {
  id        Int       @id @default(autoincrement())
  title     String
  startTime DateTime  @map("start_time")
  endTime   DateTime? @map("end_time")
  userId    Int?      @map("user_id")  // ← 實際的外鍵欄位
  memo      String?   // ← 實際的資料欄位
  
  steps     Step[]    // ← 關聯欄位，不儲存在 TimeLog 表
  user      User?     @relation(fields: [userId], references: [user_id]) // ← 關聯欄位
}
```

- `steps`：一對多關聯，實際外鍵在 `Step` 表的 `timeLogId`
- `user`：多對一關聯，實際外鍵是 `TimeLog` 表的 `userId`
- `memo`：實際資料欄位，會儲存在資料庫

## Cursor
是的，你說得對！在 React Bootstrap 中，`Col` 是一個 React 組件，所以 `md={6}` 是作為 props 傳遞給組件的。

## React Bootstrap vs 原生 Bootstrap 的差異：

### **React Bootstrap 方式：**
```jsx
<Col md={6} className="mb-3">
  <div>內容</div>
</Col>
```

### **原生 Bootstrap 方式：**
```html
<div class="col-md-6 mb-3">
  <div>內容</div>
</div>
```

## 主要差異：

1. **語法結構：**
   - React Bootstrap：`<Col md={6}>` - 使用 JSX 語法和 props
   - 原生 Bootstrap：`class="col-md-6"` - 使用 CSS 類名

2. **響應式斷點：**
   - React Bootstrap：`xs={12} sm={6} md={4} lg={3} xl={2}`
   - 原生 Bootstrap：`col-12 col-sm-6 col-md-4 col-lg-3 col-xl-2`

3. **偏移和排序：**
   - React Bootstrap：`offset={{ md: 2 }}` 或 `order={{ md: 1 }}`
   - 原生 Bootstrap：`offset-md-2` 或 `order-md-1`

## 在你的代碼中：
```jsx
<Col md={6} className="mb-3">
  <div className={styles.problemItem}>
    <h6>📅 Google Calendar</h6>
    <p className="text-muted small">
      只能記錄事件，無法精確到秒的時間戳
    </p>
  </div>
</Col>
```

這裡 `md={6}` 表示在中等螢幕（≥768px）及以上時，這個欄位佔 6/12 = 50% 的寬度，`className="mb-3"` 則是額外的 CSS 類名用於底部邊距。

React Bootstrap 的好處是提供了更語義化和類型安全的組件 API，同時保持了 Bootstrap 的響應式網格系統。
