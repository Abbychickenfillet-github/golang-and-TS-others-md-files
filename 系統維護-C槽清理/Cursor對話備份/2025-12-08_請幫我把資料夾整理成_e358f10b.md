## 观察结果

### 1. 普通JWT
- 同一组账号密码在同一秒内生成的JWT相同（因为 `iat` 时间戳相同）
- 如果跨秒生成，JWT会不同（因为 `iat` 不同）

### 2. 带 jti 的JWT
- 每次生成的JWT都不同（因为每次都有新的 UUID 作为 `jti`）
- 即使在同一秒内生成，JWT也不同

## 关于黑名单机制的问题

**问题：登出时将 jti 存入黑名单表，下次还能正常登录吗？**

**答案：可以正常登录。**

### 工作原理：

1. 登录时：每次登录都会生成新的 JWT，包含新的 `jti`（UUID）
2. 登出时：将当前 JWT 的 `jti` 存入黑名单表
3. 下次登录：生成新的 JWT，包含新的 `jti`（不在黑名单中）
4. 验证时：检查 `jti` 是否在黑名单中
   - 旧的 Token（jti 在黑名单中）→ 验证失败
   - 新的 Token（jti 不在黑名单中）→ 验证成功

## 问题解答

### 为什么不能直接用 `node sign.mjs`？

**原因：**
- `sign.mjs` 需要从 `dev.env` 文件读取 `SECRET_KEY`
- 直接运行 `node sign.mjs` **不会自动加载** `dev.env` 文件
- 所以 `process.env.SECRET_KEY` 是 `undefined`，导致错误

**解决方法：**
```bash
# 方法 1：使用 --env-file 参数
node --env-file=dev.env sign.mjs

# 方法 2：使用 npm script（推荐）
npm run dev
```

## 为什么 `All env {}` 会显示 `PSModulePath` 和 `PyCharm`？

### 原因

`process.env` 会显示所有环境变量，包括：

1. 系统环境变量（始终存在）
   - `PSModulePath` - PowerShell 模块路径
   - `PyCharm` - PyCharm IDE 路径
   - `USERNAME` - 用户名
   - `Path` - 系统路径
   - 等等...

2. 项目环境变量（需要 `--env-file=dev.env` 才会加载）
   - `SECRET_KEY` - 从 `dev.env` 加载
   - `SIGNED_SECRET_KEY` - 从 `dev.env` 加载
   - `心情` - 从 `dev.env` 加载

### 对比测试

已创建 `test-env.mjs` 来演示：

**不使用 `--env-file`：**
```bash
node test-env.mjs
```
- ✅ 可以看到 `PSModulePath`、`PyCharm`（系统环境变量）
- ❌ `SECRET_KEY` 是 `undefined`（未加载 dev.env）

**使用 `--env-file`：**
```bash
node --env-file=dev.env test-env.mjs
```
- ✅ 可以看到 `PSModulePath`、`PyCharm`（系统环境变量，仍然存在）
- ✅ `SECRET_KEY` 是 `'445552'`（已加载 dev.env）

### 关键理解

`process.env` 是“累积”的：
- 系统环境变量 → 始终存在（不需要 `dev.env`）
- `--env-file=dev.env` → 将 `dev.env` 中的变量添加到 `process.env`（不会清除系统变量）

因此，即使没有 `--env-file`，也能看到 `PSModulePath` 和 `PyCharm`，因为它们是系统环境变量。
