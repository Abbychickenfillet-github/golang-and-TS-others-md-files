**短答**：看情境而定。
- 在「型別位置」中，`|` 是「聯合型別（union）」= 或。
- 在「運算式中」單個 `|` 是「位元 OR」，邏輯 OR 要用 `||`。

### 範例
- 型別（or/聯合）：
```ts
type InputType = 'title' | 'desc';
let x: number | string;
```

- 位元 OR（單個 `|`，作用在數字位元上）：
```ts
const flags = 0b0010 | 0b1000; // 0b1010
```

- 邏輯 OR（兩個 `|`）：
```ts
const result = a || b; // a 為 truthy 則回 a，否則回 b
```

你在 `VoiceInput` 裡的 `inputType: 'title' | 'desc'` 就是「聯合型別」的用法。

## Cursor
對，`memo || ''` 表示：若 `memo` 為真值則使用 `memo`，否則使用空字串 `''`。

## 邏輯 OR (`||`) 運算符的行為

```javascript
memo || ''
```

意思是：
- 如果 `memo` 是 truthy → 使用 `memo`
- 如果 `memo` 是 falsy（`null`, `undefined`, `''`, `0`, `false`, `NaN`）→ 使用 `''`

## 為什麼需要這樣寫？

確保 `sanitizeInput` 收到的參數是字串，避免 `null`/`undefined` 導致錯誤：

```javascript
// ❌ 如果 memo 是 null 或 undefined，可能出錯
sanitizeInput(memo)  // 如果 memo 是 null，sanitizeInput 會收到 null

// ✅ 確保永遠傳入字串
sanitizeInput(memo || '')  // 如果 memo 是 null，會傳入 ''
```

## 實際範例

```javascript
// 情況 1：memo 有值
const memo = "我的備註"
sanitizeInput(memo || '')  // 傳入 "我的備註"

// 情況 2：memo 是 null
const memo = null
sanitizeInput(memo || '')  // 傳入 ''（空字串）

// 情況 3：memo 是 undefined
const memo = undefined
sanitizeInput(memo || '')  // 傳入 ''（空字串）

// 情況 4：memo 是空字串
const memo = ''
sanitizeInput(memo || '')  // 傳入 ''（空字串）
```

## 在您的代碼中

```javascript
setMemo: (memo) => {
  const safeMemo = sanitizeInput(memo || '')  // 確保 memo 永遠是字串
  set({ memo: safeMemo })
},
```

這樣可避免 `memo` 為 `null` 或 `undefined` 時出錯。

### 更現代的做法（可選）

如果使用 TypeScript 或想更明確，也可以用空值合併運算符 `??`：

```javascript
sanitizeInput(memo ?? '')  // 只有 null 或 undefined 時才用 ''
```

兩者差別：
- `||`：所有 falsy 值都會替換（`null`, `undefined`, `''`, `0`, `false`）
- `??`：只有 `null` 或 `undefined` 才會替換

對於字串清理，通常 `||` 已足夠。

```javascript

  endTime: new Date('2025-01-15T12:00:00'),    // Date 物件
  createdAt: new Date()                         // Date 物件
```

## 為什麼不能用 `toLocaleString()`？

問題：
```javascript
const date = new Date('2025-11-01T11:28:00')
date.toLocaleString('zh-TW')
// 結果："2025/11/1 上午11:28:00"
// 或："2025/11/1, 上午 11:28:00"

// 問題：無法解析回 Date 物件
new Date("2025/11/1 上午11:28:00")  // ❌ Invalid Date
```

原因：
- 格式不標準，依瀏覽器與語言環境而異
- 無法可靠地用 `new Date()` 還原
- 跨瀏覽器/語言環境不一致

## 為什麼不能直接用 `new Date(dateString)`？

問題：
```javascript
// ❌ 錯誤：會被當作 UTC 時間
new Date('2025-11-01T11:28:00')
// 在 ES5 規範中，這種格式（沒有時區信息）會被當作 UTC
// 結果：解析為 UTC 11:28，但您要的是本地時間 11:28

// 例如：台灣時間 11:28 (UTC+8)
new Date('2025-11-01T11:28:00')  // 被當作 UTC 11:28
// 顯示時：台灣時間 = UTC 11:28 + 8小時 = 19:28 ❌（完全錯誤）
```

原因：
- `YYYY-MM-DDTHH:mm:ss` 格式（不含時區）在 ES5 被視為 UTC
- 會導致時差錯誤

## 當前解決方案的原理

### 儲存時：使用標準格式但不含時區
```javascript
formatDateForStorage(new Date('2025-11-01T11:28:00'))
// 返回："2025-11-01T11:28:00"（本地時間字串）
```

### 讀取時：使用構造函數形式（明確本地時間）
```javascript
parseLocalDate('2025-11-01T11:28:00')
// 解析為：new Date(2025, 10, 1, 11, 28, 0)
// ✅ 使用構造函數形式 = 本地時間，不是 UTC
```

## 兩種 `new Date()` 的差別

```javascript
// ❌ 字串形式：可能被當作 UTC
new Date('2025-11-01T11:28:00')
// ES5 規範：沒有時區 → 視為 UTC

// ✅ 構造函數形式：明確指定本地時間
new Date(2025, 10, 1, 11, 28, 0)
// 參數形式：year, month (0-based), day, hours, minutes, seconds
// 明確是本地時間
```

## 總結

- 不用 `toLocaleString()`：格式不標準，無法可靠解析
- 不用 `new Date(dateString)`：不含時區時可能被當作 UTC

當前做法：儲存標準格式字串，讀取時用構造函數明確本地時間，避免時差問題。
