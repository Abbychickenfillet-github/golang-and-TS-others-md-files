## 加 `-d` vs 不加 `-d` 的差異

### 不加 `-d`（前景執行）
```bash
docker compose -f docker-compose.yml -f docker-compose.personal.yml up
```

優點：
- 即時看到所有容器的日誌輸出
- 方便調試和查看啟動過程
- 按 `Ctrl+C` 可停止所有容器

缺點：
- 終端機被佔用，無法執行其他命令
- 關閉終端機視窗會停止容器

### 加 `-d`（背景執行）
```bash
docker compose -f docker-compose.yml -f docker-compose.personal.yml up -d
```

優點：
- 容器在背景執行，終端機可繼續使用
- 關閉終端機視窗不會停止容器
- 適合長期運行

缺點：
- 看不到即時日誌（需用 `docker compose logs -f` 查看）

## 建議使用方式

### 開發/調試時（不加 `-d`）
```bash
# 查看啟動過程和日誌
docker compose -f docker-compose.yml -f docker-compose.personal.yml up
```

### 正常使用時（加 `-d`）
```bash
# 背景執行，釋放終端機
docker compose -f docker-compose.yml -f docker-compose.personal.yml up -d

# 需要查看日誌時
docker compose logs -f backend
docker compose logs -f frontend
```

## 總結

- 不加 `-d`：可以，適合開發/調試時查看日誌
- 加 `-d`：適合正常使用，容器在背景運行

根據需求選擇即可。

## CORS 的工作原理

CORS 使用精確匹配，Origin 由三部分組成：
1. 協議（scheme）：`http` 或 `https`
2. 主機名（host）：`localhost`
3. 埠號（port）：`80`、`5003`、`5004` 等

因此：
- `http://localhost` = `http://localhost:80`（預設 HTTP 埠號）
- `http://localhost:5004` 是不同的 origin
- `http://localhost:3001` 是不同的 origin

## 為什麼需要明確列出？

瀏覽器會檢查請求的 Origin 是否在允許列表中。例如：
- 從 `http://localhost:5004` 發送請求 → Origin 是 `http://localhost:5004`
- 如果列表中只有 `http://localhost` → 不會匹配 → CORS 錯誤

## 結論

需要明確列出每個埠號：
- `http://localhost:5004`（Frontend 個人開發）

## Cursor
可以用 Docker Compose 的 `--env-file` 參數指定另一個環境檔，或在 Compose 檔內為服務設定 `env_file`。兩種做法：

### 1) 指令層級指定（最簡單）
在專案根目錄：
```bash
docker compose --env-file .env.personal -f docker-compose.personal.yml up -d
```
- `.env.personal` 會覆蓋預設的 `.env`。
- `docker-compose.personal.yml` 內若引用 `${VAR}`，會從 `.env.personal` 取值。

### 2) 在 `docker-compose.personal.yml` 內指定
對每個服務加上：
```yaml
services:
  backend:
    env_file:
      - .env.personal
      - .env        # 可選：作為 fallback
  frontend:
    env_file:
      - .env.personal
  official_website:
    env_file:
      - .env.personal
  prestart:
    env_file:
      - .env.personal
```
之後直接：
```bash
docker compose -f docker-compose.personal.yml up -d
```

建議做法：如果只是要切換整體環境變數，用方法 1；如果要服務各自指定，或保留 fallback，用方法 2。

Base64 是一種二進位到文本 (binary-to-text) 的編碼，它不是二進位本身，而是將二進位數據 (binary data) 轉換成一套包含 64 個可列印 ASCII 字符的文本表示法，目的是讓二進位資料能在文本為主的環境（如 email, JSON, HTML）中安全傳輸與儲存。簡而言之，它把 3 個位元組 (24 bits) 的二進位資料轉換成 4 個 Base64 字符 (4 x 6 bits)。

## Cursor
Base64 不是專門給圖片的，它是通用的「二進位→可列印文字」編碼。常見用途：
- 圖片／音訊／PDF 等二進位內容嵌入 HTML、CSS、JSON
- 透過只允許文字的通道（例如早期 email、某些 API）傳遞二進位資料
- 簽章、Token（如 JWT 的段落）

特點與取捨：
- 體積會變大約 33%（3 bytes 變 4 chars）
- 瀏覽器預覽圖片時可用 `FileReader.readAsDataURL`（Base64 data URL），但若只是本地預覽，`URL.createObjectURL(file)` 更省記憶體與 CPU，預覽完記得 `URL.revokeObjectURL()`。
- 傳輸場景若不受文字限制，直接傳二進位／multipart/form-data 效率更好。
