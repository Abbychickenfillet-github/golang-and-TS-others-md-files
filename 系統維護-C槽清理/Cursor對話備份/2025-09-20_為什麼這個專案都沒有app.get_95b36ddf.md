## 這是一個 **Next.js 專案**，不是 Express.js 專案

### 主要差異：

1. **Next.js 使用 App Router 架構**：
   - API 路由使用 `export async function GET()` 而不是 `app.get()`
   - 檔案結構在 `app/(api)/api/` 目錄下
   - 每個路由檔案定義對應的 HTTP 方法函數

2. **Express.js 使用傳統路由**：
   - 使用 `app.get()`, `app.post()` 等方法
   - 通常在單一檔案中定義所有路由

### 你的專案實際使用的路由方式：

從搜尋結果可以看到，你的專案使用的是 Next.js 的 App Router 模式：

```javascript
// 例如：app/(api)/api/products/route.js
export async function GET(request) {
  // 處理 GET 請求
}

export async function POST(req) {
  // 處理 POST 請求
}
```

### 為什麼會這樣：

1. **Next.js 15** 使用 App Router 作為預設路由系統
2. **檔案系統路由**：路由由檔案結構決定，而不是程式碼定義
3. **API Routes**：在 `app/(api)/api/` 目錄下的檔案自動成為 API 端點

### 如果你想要 Express.js 風格的 `app.get()`：

你需要建立一個獨立的 Express 伺服器，或者使用 Next.js 的傳統 Pages Router（在 `pages/api/` 目錄下）。

你的專案目前使用的是現代 Next.js App Router 架構，這是更推薦的方式！

## 1. `import { NextResponse } from 'next/server'` 的重要性

### **為什麼需要 NextResponse？**

## 2. LEFT JOIN 的含義

### **LEFT JOIN 解釋：**

## 3. GROUP BY 的必要性

### **為什麼需要 GROUP BY？**

**GROUP BY 不會自動按照順序分組！** 這是常見的誤解。

#### **A. 沒有 GROUP BY 的問題：**
```sql
-- 錯誤查詢 (沒有 GROUP BY)
SELECT 
  tl.id,
  tl.title,
  COUNT(s.id) as step_count
FROM "TimeLog" tl
LEFT JOIN "Step" s ON tl.id = s.time_log_id
-- 結果會是錯誤的！
```

**結果會是：**
```
id | title   | step_count
1  | "學習程式" | 2
1  | "學習程式" | 2  -- 重複記錄！
2  | "運動"    | 0
3  | "休息"    | 0
```

#### **B. 有 GROUP BY 的正確結果：**
```sql
-- 正確查詢 (有 GROUP BY)
SELECT 
  tl.id,
  tl.title,
  COUNT(s.id) as step_count
FROM "TimeLog" tl
LEFT JOIN "Step" s ON tl.id = s.time_log_id
GROUP BY tl.id, tl.title, tl.start_time, tl.end_time
```

**結果：**
```
id | title   | step_count
1  | "學習程式" | 2        -- 只有一條記錄
2  | "運動"    | 0
3  | "休息"    | 0
```

### **為什麼要 GROUP BY 所有欄位？**

```sql
GROUP BY tl.id, tl.title, tl.start_time, tl.end_time
```

**原因：**
1. **SQL 規則**：使用聚合函數(COUNT)時，所有非聚合欄位都必須在 GROUP BY 中
2. **資料完整性**：確保每條 TimeLog 記錄只出現一次
3. **避免重複**：防止 JOIN 產生的重複記錄

## �� 完整範例對比

### **錯誤的查詢：**
```sql
-- ❌ 沒有 GROUP BY
SELECT tl.id, tl.title, COUNT(s.id) as step_count
FROM "TimeLog" tl
LEFT JOIN "Step" s ON tl.id = s.time_log_id
-- 結果：重複記錄，COUNT 不正確
```

### **正確的查詢：**
```sql
-- ✅ 有 GROUP BY
SELECT 
  tl.id,
  tl.title,
  tl.start_time,
  tl.end_time,
  COUNT(s.id) as step_count
FROM "TimeLog" tl
LEFT JOIN "Step" s ON tl.id = s.time_log_id
GROUP BY tl.id, tl.title, tl.start_time, tl.end_time
ORDER BY tl.start_time DESC
LIMIT 10
-- 結果：每條 TimeLog 記錄只出現一次，COUNT 正確
```

3. **GROUP BY**：使用聚合函數時必須使用，避免重複記錄

2. **雙向通訊**：伺服器可以主動推送
3. **即時性**：適合即時應用

## 3. TE 是什麼的縮寫？

**TE = Transfer-Encoding**

### **Transfer-Encoding 標頭：**
```http
Transfer-Encoding: chunked
Transfer-Encoding: gzip, chunked
TE: gzip, deflate
```

#### **作用：**
- **分塊傳輸**：將大檔案分成小塊傳輸
- **壓縮編碼**：指定支援的壓縮方式
- **動態內容**：不需要預先知道內容大小

#### **實際例子：**
```http
# 伺服器回應
HTTP/1.1 200 OK
Transfer-Encoding: chunked
Content-Type: text/html

5\r\n
Hello\r\n
6\r\n
 World\r\n
0\r\n
\r\n
```

**解釋：**
- `5\r\n` = 5 個字元
- `Hello\r\n` = 實際內容
- `6\r\n` = 6 個字元
- ` World\r\n` = 實際內容
- `0\r\n\r\n` = 結束標記

## 4. 完整的連接類型比較

### **Connection 標頭的所有可能值：**

```http
# 持久連接
Connection: keep-alive
Keep-Alive: timeout=5, max=100

# 關閉連接
Connection: close

# 升級連接
Connection: upgrade
Upgrade: websocket

# 傳輸編碼
Connection: TE
TE: gzip, deflate

# 多個值
Connection: keep-alive, upgrade
```

### **實際應用場景：**

#### **A. 一般網頁請求：**
```http
Connection: keep-alive
Keep-Alive: timeout=5
```

#### **B. WebSocket 升級：**
```http
Connection: upgrade
Upgrade: websocket
Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==
```

#### **C. HTTP/2 升級：**
```http
Connection: upgrade
Upgrade: h2c
HTTP2-Settings: <base64url encoding of HTTP/2 SETTINGS payload>
```

#### **D. 壓縮傳輸：**
```http
Connection: TE
TE: gzip, deflate
```

## 5. 資源使用比較

### **資源使用對比：**

| 連接類型 | 建立開銷 | 維持開銷 | 適用場景 |
|----------|----------|----------|----------|
| **HTTP/1.0** | 高 | 無 | 簡單請求 |
| **HTTP/1.1 Keep-Alive** | 中 | 低 | 一般網頁 |
| **WebSocket** | 中 | 極低 | 即時通訊 |
| **HTTP/2** | 中 | 極低 | 現代網頁 |

### **實際例子：**

```javascript
// 聊天應用
// 使用 WebSocket 比 HTTP 輪詢更節省資源

// ❌ HTTP 輪詢（浪費資源）
setInterval(() => {
  fetch('/api/messages')  // 每秒發送完整 HTTP 請求
}, 1000)

// ✅ WebSocket（節省資源）
const ws = new WebSocket('ws://localhost:3001')
ws.onmessage = (event) => {
  // 只有小的資料幀，沒有 HTTP 標頭
}
```

## 總結

1. **5秒後重新開始**：✅ 是的，新請求會建立新連接
2. **WebSocket 不浪費資源**：✅ 實際上比 HTTP 輪詢更節省
3. **TE = Transfer-Encoding**：✅ 用於分塊傳輸和壓縮
4. **選擇合適的連接類型**：根據應用場景選擇最適合的方式

**關鍵是要理解每種連接類型的適用場景，而不是單純比較資源使用！**
