## 📋 命令分解

```bash
"prisma-safe": "taskkill /f /im node.exe 2>nul || true && npx prisma generate"
```

### 1. **`taskkill`** - Windows 任務終止命令
- 用來強制結束正在運行的進程

### 2. **`/f`** - Force（強制）
- **強制終止**進程，不等待進程正常結束
- 相當於 Linux 的 `kill -9`
- 如果沒有 `/f`，taskkill 會嘗試優雅地結束進程

### 3. **`/im`** - Image Name（映像名稱）
- 指定要終止的**進程名稱**
- `/im node.exe` = 終止所有名為 `node.exe` 的進程
- 其他選項：`/pid`（進程ID）、`/fi`（篩選器）

### 4. **`node.exe`** - 目標進程
- 所有 Node.js 進程的執行檔名稱

### 5. **`2>nul`** - 錯誤重定向
- **`2`** = 標準錯誤輸出（stderr）
- **`>`** = 重定向運算子
- **`nul`** = Windows 的「黑洞」設備（類似 Linux 的 `/dev/null`）
- **作用**：將錯誤訊息丟棄，不顯示在終端

### 6. **`||`** - OR 運算子（邏輯或）
- 如果**左邊命令失敗**，執行右邊命令
- 如果**左邊命令成功**，不執行右邊命令

### 7. **`true`** - 總是成功的命令
- 在 Windows 中總是返回成功狀態
- 用來確保整個命令鏈不會因為 taskkill 失敗而中斷

### 8. **`&&`** - AND 運算子（邏輯與）
- 如果**左邊命令成功**，執行右邊命令
- 如果**左邊命令失敗**，不執行右邊命令

## 🔄 執行流程

```bash
taskkill /f /im node.exe 2>nul || true && npx prisma generate
```

### 步驟分解：

1. **執行** `taskkill /f /im node.exe 2>nul`
   - 嘗試強制終止所有 node.exe 進程
   - 錯誤訊息被丟棄（`2>nul`）

2. **檢查結果**：
   - **成功**：有 node.exe 進程被終止 → 跳到步驟 4
   - **失敗**：沒有 node.exe 進程 → 執行步驟 3

3. **執行** `true`（因為 `||`）
   - 確保命令鏈繼續執行

4. **執行** `npx prisma generate`（因為 `&&`）
   - 生成 Prisma Client

## 🎯 實際效果

### 情況 1：有 Node.js 進程在運行
```bash
taskkill /f /im node.exe 2>nul  # ✅ 成功終止
|| true                        # ❌ 不執行（因為左邊成功）
&& npx prisma generate         # ✅ 執行
```

### 情況 2：沒有 Node.js 進程
```bash
taskkill /f /im node.exe 2>nul  # ❌ 失敗（沒有進程可終止）
|| true                        # ✅ 執行（因為左邊失敗）
&& npx prisma generate         # ✅ 執行
```

## 💡 為什麼這樣設計？

1. **安全終止**：確保沒有 Node.js 進程鎖定 Prisma 檔案
2. **容錯處理**：即使沒有進程可終止，命令仍會繼續
3. **靜默執行**：不顯示不必要的錯誤訊息
4. **自動化**：一個命令完成所有必要步驟

## 📋 標準輸出流

在 Windows 命令列中，有三個標準輸出流：

### 1. **標準輸出（stdout）** - 流編號 `1`
- 正常的輸出訊息
- 預設顯示在終端

### 2. **標準錯誤（stderr）** - 流編號 `2`
- 錯誤訊息和警告
- 預設也顯示在終端

### 3. **標準輸入（stdin）** - 流編號 `0`
- 使用者輸入

## 🔄 `2>nul` 的運作

```bash
taskkill /f /im node.exe 2>nul
```

### 分解說明：

- **`2`** = 標準錯誤流（stderr）
- **`>`** = 重定向運算子
- **`nul`** = Windows 的「黑洞」設備

### 實際效果：

### 情況 2：沒有 Node.js 進程

## 🔧 其他重定向範例

### 1. **只丟棄錯誤，保留正常輸出**
```bash
taskkill /f /im node.exe 2>nul
# 成功訊息被丟棄，錯誤訊息也被丟棄
```

### 2. **將錯誤重定向到檔案**
```bash
taskkill /f /im node.exe 2>error.log
# 錯誤訊息寫入 error.log 檔案
```

### 3. **將錯誤重定向到標準輸出**
```bash
taskkill /f /im node.exe 2>&1
# 錯誤訊息和正常輸出都顯示在終端
```

### 4. **完全靜默（丟棄所有輸出）**
```bash
taskkill /f /im node.exe >nul 2>nul
# 或
taskkill /f /im node.exe >nul 2>&1
```

## 🔍 一對一關聯的兩種方式

### 1. **真正的 One-to-One（需要 `@unique`）**

```prisma
model User {
  id        Int         @id @default(autoincrement())
  email     String      @unique
  profile   UserProfile? // 一個用戶最多有一個個人資料
}

model UserProfile {
  id       Int    @id @default(autoincrement())
  userId   Int    @unique @map("user_id") // @unique 確保一對一
  bio      String?
  avatar   String?
  user     User   @relation(fields: [userId], references: [id])
}
```

**特點**：
- 每個 User 最多只能有一個 UserProfile
- 每個 UserProfile 只能屬於一個 User
- **需要 `@unique` 約束**

### 2. **One-to-Zero-or-One（不需要 `@unique`）**

```prisma
model User {
  id        Int         @id @default(autoincrement())
  email     String      @unique
  profile   UserProfile? // 一個用戶可能有或沒有個人資料
}

model UserProfile {
  id       Int    @id @default(autoincrement())
  userId   Int    @map("user_id") // 沒有 @unique
  bio      String?
  avatar   String?
  user     User   @relation(fields: [userId], references [id])
}
```

**特點**：
- 每個 User 可能有 0 或 1 個 UserProfile
- 每個 UserProfile 只能屬於一個 User
- **不需要 `@unique` 約束**
