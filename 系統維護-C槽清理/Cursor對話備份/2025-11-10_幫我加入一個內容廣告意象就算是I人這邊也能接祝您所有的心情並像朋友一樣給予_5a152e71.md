- `hour12`：可选，控制是否使用 12 小时制（AM/PM）
  - `false`：24 小时制（如 "14:30:25"）
  - `true`：12 小时制（如 "下午 2:30:25" 或 "2:30:25 PM"）

- `year`：年份显示格式
  - `'numeric'`：完整年份（如 "2024"）
  - `'2-digit'`：两位年份（如 "24"）

- 其他常用选项：
  - `month`: `'numeric'`（1-12）、`'2-digit'`（01-12）、`'long'`（一月）、`'short'`（1月）
  - `day`: `'numeric'`（1-31）、`'2-digit'`（01-31）
  - `hour`: `'numeric'`（0-23）、`'2-digit'`（00-23）
  - `minute`: `'numeric'`（0-59）、`'2-digit'`（00-59）

## 回答

### 1. `hour12: false` 是固定的 key 吗？

不是固定的 key，而是可选的配置项。

- `hour12` 是 `toLocaleString()` 的标准选项之一
- 可选值：
  - `false`：24 小时制（如 "14:30:25"）
  - `true`：12 小时制（如 "下午 2:30:25"）
  - 不设置：使用 locale 的默认值

### 2. `year: 'numeric'` 是什么意思？

这是年份的显示格式选项：

- `'numeric'`：完整年份（4 位数），如 `2024`
- `'2-digit'`：两位年份，如 `24`

### 当前配置的效果

```javascript
dateObj.toLocaleString('zh-TW', {
  hour12: false,        // 24 小时制，不使用 AM/PM
  year: 'numeric',     // 完整年份：2024
  month: '2-digit',     // 两位月份：01, 02, ..., 12
  day: '2-digit',       // 两位日期：01, 02, ..., 31
  hour: '2-digit',      // 两位小时：00, 01, ..., 23
  minute: '2-digit',    // 两位分钟：00, 01, ..., 59
  second: '2-digit',    // 两位秒数：00, 01, ..., 59
})
```

输出示例：`2024/01/15 14:30:25`

### 为什么使用这些选项？

1. `hour12: false`：24 小时制更清晰，避免 AM/PM 混淆
2. `year: 'numeric'`：显示完整年份，避免跨年混淆
3. `'2-digit'`：统一格式，便于排序和对齐

已创建说明文档：`docs/toLocaleString-options-explanation.md`，包含更多示例和说明。

这些选项都是可选的，可根据需求调整。

## Cursor
在網頁上做「即時」語言/地區格式切換，通常有兩種做法：

---

### 1. 前端程式控制（最常見、使用者體驗最好）

1. **偵測目前語系**：一開始可以用 `navigator.language` 或 `navigator.languages[0]` 作為預設值。
2. **提供切換介面**：例如下拉選單，把常用語系（`zh-TW`, `en-US`, `ja-JP` …）列出來。
3. **根據選擇重新格式化**：在使用 `toLocaleString()` 時，把選擇的字串放進 `locales` 參數。例如：

   ```javascript
   const [locale, setLocale] = useState(navigator.language);

   const formatted = new Date().toLocaleString(locale, {
     hour12: false,
     year: 'numeric',
     month: '2-digit',
     day: '2-digit',
     hour: '2-digit',
     minute: '2-digit',
     second: '2-digit',
   });
   ```

   使用者切換 `locale` state ⇒ 重新 render ⇒ 時間格式即時變化。

4. **搭配 i18n 套件**：文字內容可以搭配 `react-intl`, `i18next`, `next-intl` 等工具，同步切換語言。

---

### 2. 倚靠使用者裝置/瀏覽器設定（不用寫程式，但不夠即時）

- 如果呼叫 `toLocaleString()` 時不帶 `locales` 參數，JS 會使用使用者瀏覽器的預設語系。
- 使用者要改變語系，只能去**調整作業系統或瀏覽器的語言設定**，再重新整理頁面。
- 這方式無法在同一頁面內「即時」切換，也難以控制不同使用者的體驗。

---

### 建議做法

- 如果需要同一頁面內馬上切換，就必須在程式裡讀取使用者選擇並把它傳給國際化 API。
- 如果只是要「尊重使用者的裝置設定」，可以不傳 `locales`，讓瀏覽器自動決定，但切換就不會即時。

結論：**想要即時切換，就得透過自己的 UI + state，把選擇的語系傳給 `toLocaleString()`（或 `Intl.DateTimeFormat`）等 API。單靠瀏覽器原生設定無法做到即時切換。**

### 架構說明

- `log.id`  
  在 `/app/dashboard/page.js` 的 `TimeLogList` 中，`log` 來自 state `timeLogs`。  
  `timeLogs` 是透過 `fetch('/api/timelogs')` 取得後端傳回的結果，成功後呼叫 `setTimeLogs(result.data.timeLogs)` 更新的，所以 `id` 就是後端送出的那筆時間記錄的主鍵。

- `sharedLogIds`  
  不是 Prisma 的語法，它是前端本地 state：`const [sharedLogIds, setSharedLogIds] = useState(new Set())`。  
  我們在登入後會呼叫 `fetchSharedLogIds()`，打 `/api/featured-shares?userId=...`，把回傳的 `timeLogId` 全部放進這個 `Set`。`TimeLogList` 只要看到 `sharedLogIds` 裡有這個 `log.id`，就顯示「已分享」/ 實心 ICON。

### 你看到「已分享」消失的可能原因

1. **前端沒重新載入 `sharedLogIds`**  
   分享成功後，我們在 `handleShareTimeLog` 只把 `numericLogId` 加到 `sharedLogIds` 的 state；如果你重新整理頁面，`fetchSharedLogIds` 一定會跑一次，把資料庫裡的資料抓回來。  
   但如果你是在分享後、還沒 refresh 之前又做了什麼操作（例如錯誤的 PATCH/DELETE），就可能使 `Set` 被覆蓋或清空。

2. **後端 `featuredShare` 表真的沒有這筆**  
   Dashboard 要顯示實心標記，前端會看 `sharedLogIds.has(log.id)`。倘若你手動刪除了 `featuredShare` 紀錄，那 `fetchSharedLogIds` 回傳的就不包含該 ID，自然也不顯示已分享。

3. **ID 型別不一致**

## Cursor
`Set` 和 `Map` 雖然都屬於 ES6 的集合型別，但用途不同：

- `Set`
  - 只存放「值」，沒有對應的 key。
  - 內部會自動去除重複元素（以 `SameValueZero` 判斷）。
  - 適合用來判斷「這個值是否存在？」、或保持唯一化的列表。
  - 在專案裡：`sharedLogIds` 用來記錄有哪些 `timeLogId` 已分享，方便用 `sharedLogIds.has(timeLogId)` 直接問「這個活動有分享嗎？」

- `Map`
  - 存的是 key → value 的對應關係，可以把任何型別當 key。
  - 不會自動去重（key 相同會覆蓋）。
  - 適合保存「ID → 物件／資料」的映射。
  - 在專案裡：`sharedShareMap` 用來記錄 `timeLogId` 對應的 `featuredShareId`，取消分享時需要這個 shareId 才能打 DELETE，所以 `Map` 很合適。

簡單說，`Set` 就像「唯一化的值清單」，`Map` 則像「字典／哈希表」，能幫你找到一個 key 對應的 value。

### 2. 單向數據流 vs 子傳父

子組件通過回調通知父組件，這不違反單向數據流。

單向數據流的意思是：
- 數據從父組件流向子組件（通過 props）
- 子組件可以通過回調函數通知父組件狀態變化

例如：
```jsx
// ✅ 這是正確的單向數據流
function Parent() {
  const [count, setCount] = useState(0)
  
  // 父組件定義處理函數，通過 props 傳給子組件
  const handleIncrement = () => {
    setCount(count + 1)  // 父組件自己更新狀態
  }
  
  return <Child onIncrement={handleIncrement} count={count} />
}

function Child({ onIncrement, count }) {
  // 子組件只負責「通知」父組件，不直接修改父組件狀態
  return <button onClick={onIncrement}>Count: {count}</button>
}
```

這不是「子組件傳值給父組件」，而是「子組件通知父組件，父組件自己更新狀態」。


- `TimeLogList`、`PaymentHistorySection`、`UserInfoSection` 都是 `Dashboard` 的子組件（父子關係存在）
- 它們彼此之間是兄弟關係（不是父子關係）

### 數據流

```jsx
export default function Dashboard() {
  const [sharedLogIds, setSharedLogIds] = useState(new Set())
  
  // 父組件定義處理函數
  const handleShare = (log) => {
    // 父組件更新狀態
    setSharedLogIds(new Set([...sharedLogIds, log.id]))
  }
  
  return (
    <>
      {/* 數據向下流動（props） */}
      <TimeLogList 
        sharedLogIds={sharedLogIds}
        onShare={handleShare}  // 回調函數向下傳遞
      />
      
      {/* 兄弟組件之間不直接通信 */}
      <PaymentHistorySection />
      <UserInfoSection />
    </>
  )
}
```

### 總結

- 多個組件在同一頁上，它們都是該頁的子組件（父子關係存在）
- 它們彼此之間是兄弟關係（不是父子關係）
- 數據流：父 → 子（props），子 → 父（回調函數）
- 兄弟組件之間不直接通信，需要通過共同的父組件

所以你的 `/dashboard` 頁面中，`TimeLogList`、`PaymentHistorySection` 等都是 `Dashboard` 的子組件，它們之間是兄弟關係。
