## Cursor
`globals` 是 ESLint 的套件，用來定義全域變數。

## 主要功能

### 1. **定義全域變數**
告訴 ESLint 哪些變數是全域的，避免被標記為未定義：

```javascript
// 沒有 globals 時，ESLint 會報錯
console.log("Hello"); // ❌ 'console' is not defined
window.location.href = "/"; // ❌ 'window' is not defined
document.getElementById("app"); // ❌ 'document' is not defined
```

### 2. **常見的全域變數**
- 瀏覽器環境：`window`, `document`, `console`, `alert`, `setTimeout`
- Node.js 環境：`process`, `Buffer`, `global`
- 測試環境：`describe`, `it`, `expect`, `beforeEach`

### 3. **在 ESLint 配置中的使用**
```javascript
// eslint.config.js
import globals from 'globals';

export default [
  {
    languageOptions: {
      globals: {
        ...globals.browser, // 瀏覽器全域變數
        ...globals.node,    // Node.js 全域變數
        ...globals.jest     // Jest 測試全域變數
      }
    }
  }
];
```

### 4. **為什麼需要它**
- 避免誤報：`console.log`、`window` 等不應被標記為未定義
- 環境區分：區分瀏覽器、Node.js、測試環境
- 提升準確性：減少誤判

### 5. **實際例子**
```javascript
// 沒有 globals 配置時
function fetchData() {
  fetch('/api/data') // ❌ 'fetch' is not defined
    .then(response => response.json())
    .then(data => console.log(data)); // ❌ 'console' is not defined
}

// 有 globals 配置時
function fetchData() {
  fetch('/api/data') // ✅ 正常
    .then(response => response.json())
    .then(data => console.log(data)); // ✅ 正常
}
```

## 總結
`globals` 讓 ESLint 正確識別環境的全域變數，避免誤報，並提升檢查準確性。

## Cursor
在 `const [, setLastStepTime] = useState<Date | null>(null)` 中，前面的逗號是解構賦值的語法，用來跳過第一個元素。

## 解構賦值語法

```javascript
const [first, second, third] = [1, 2, 3]
// first = 1, second = 2, third = 3

// 如果只想取第二個值，可以這樣寫：
const [, second] = [1, 2, 3]
// second = 2，第一個值被跳過

// 如果只想取第三個值：
const [, , third] = [1, 2, 3]
// third = 3，前兩個值被跳過
```

## 在 React 中的應用

```javascript
// useState 返回一個數組：[state, setState]
const [value, setValue] = useState(initialValue)

// 如果我們只需要 setter，不需要 state 值：
const [, setValue] = useState(initialValue)
```

## 為什麼這樣做？

在 `TimeLogClient.tsx` 中：
```typescript
const [, setLastStepTime] = useState<Date | null>(null)
```

- 只需要 `setLastStepTime` 來更新狀態
- 不需要讀取 `_lastStepTime` 的值
- 用逗號跳過第一個元素，避免未使用變數警告

## 其他寫法

```typescript
// 方法1：使用下劃線表示未使用
const [_lastStepTime, setLastStepTime] = useState<Date | null>(null)

// 方法2：使用逗號跳過（推薦）
const [, setLastStepTime] = useState<Date | null>(null)

// 方法3：使用 ESLint 註釋
// eslint-disable-next-line @typescript-eslint/no-unused-vars
const [_lastStepTime, setLastStepTime] = useState<Date | null>(null)
```

方法2 最簡潔，且不會觸發 ESLint 警告。

- 用途：API 路由、認證、資料庫連線
- 運行環境：Node.js 服務器
- 特點：不會暴露到客戶端

## App Router 中的使用場景

### 客戶端組件 (`'use client'`)
```javascript

import { apiURL } from '@/config/client.config'

}
```

### 服務端組件 (API 路由)
```javascript
// app/(api)/api/users/route.js
import { serverConfig } from '@/config/server.config'

export async function GET() {
  // 使用服務端配置
  const jwtSecret = serverConfig.jwt.secret
}
```

## 為什麼不只用一個文件？

### 安全性
- 客戶端配置：可公開（API URL、前端設定）
- 服務端配置：需保密（JWT 密鑰、資料庫密碼）

### 環境差異
- 客戶端：瀏覽器環境限制
- 服務端：Node.js 環境，可存取檔案系統、環境變數

## 什麼時候需要 `setTimeout`？

### 1. **防抖 (Debounce)**
```javascript
// 防止用戶快速點擊造成重複請求
const [isLoading, setIsLoading] = useState(false)

const goLinePay = async () => {
  if (isLoading) return // 防止重複點擊
  
  setIsLoading(true)
  try {
    const res = await fetch(`${apiURL}/payment/line-pay/request?amount=${quantity * price}`)
    // ... 處理回應
  } finally {
    // 延遲重置狀態，防止快速重複點擊
    setTimeout(() => setIsLoading(false), 1000)
  }
}
```

### 2. **重試機制**
```javascript
const retryRequest = async (url, maxRetries = 3) => {
  for (let i = 0; i < maxRetries; i++) {
    try {
      const res = await fetch(url)
      if (res.ok) return res
    } catch (error) {
      if (i === maxRetries - 1) throw error
      // 等待後重試
      await new Promise(resolve => setTimeout(resolve, 1000 * (i + 1)))
    }
  }
}
```

### 3. **模擬延遲（測試用）**
```javascript
// 僅用於測試，生產環境不建議
const goLinePay = async () => {
  // 模擬網路延遲
  await new Promise(resolve => setTimeout(resolve, 500))
  
  const res = await fetch(`${apiURL}/payment/line-pay/request?amount=${quantity * price}`)
  // ...
}
```

## Line Pay 的情況

### 不需要 `setTimeout` 的原因：
1. `fetch` 本身有超時機制
2. 瀏覽器會處理網路超時
3. 可能造成不必要的延遲

### 建議的做法：
```javascript
const goLinePay = async () => {
  try {
    // 直接發送請求，不需要 setTimeout
    const res = await fetch(
      `${apiURL}/payment/line-pay/request?amount=${quantity * price}`,
      {
        method: 'GET',
        credentials: 'include',
        headers: {
          'Content-Type': 'application/json',
          Accept: 'application/json',
        },
        // 可以設定超時（可選）
        signal: AbortSignal.timeout(10000) // 10秒超時
      }
    )
    
    if (!res.ok) {
      throw new Error(`HTTP ${res.status}: ${res.statusText}`)
    }
    
    const resData = await res.json()
    // 處理回應...
    
  } catch (error) {
    console.error('Line Pay 請求失敗:', error)
    // 處理錯誤...
  }
}
```

## 總結

- 一般 API 請求：不需要 `setTimeout`
- 防抖/節流：需要 `setTimeout`
- 重試機制：需要 `setTimeout`
- 測試模擬：需要 `setTimeout`

Line Pay 請求建議直接使用 `fetch`，並加上適當的錯誤處理。

## Cursor
Line Pay Sandbox 的 10–15 分鐘自動登出是服務端 session 過期，不是前端 `setTimeout` 或 `refreshToken`。

## Line Pay Sandbox 登出機制

### 1. **服務端 Session 過期**
```javascript
// Line Pay 服務端會設定 session 過期時間
// 這是在 Line Pay 的伺服器端處理，不是前端
```

### 2. **不是 `setTimeout`**
- 前端 `setTimeout` 無法控制 Line Pay 的登入狀態
- 即使頁面一直開著，Line Pay 的 session 仍會過期

### 3. **不是 `refreshToken`**
- Line Pay 沒有提供 refresh token 機制
- 過期後需要重新登入

## 實際運作方式

### Line Pay 的 Session 管理：
```javascript
// Line Pay 服務端設定
sessionTimeout: 15 * 60 * 1000 // 15分鐘

// 當用戶在 Line Pay 頁面停留超過 15 分鐘
// Line Pay 會自動重定向到登出頁面
```

### 前端處理方式：
```javascript
// 在 Line Pay 回調頁面檢查 session 狀態
const checkLinePaySession = () => {
  // 檢查 URL 參數是否有 session 過期標記
  const urlParams = new URLSearchParams(window.location.search)
  if (urlParams.get('session_expired')) {
    alert('Line Pay 會話已過期，請重新登入')
    // 重新導向到 Line Pay 登入頁面
  }
}
```

## 建議的處理方式

### 1. **監聽 Line Pay 回調**
```javascript
// 在 Line Pay 回調頁面
useEffect(() => {
  const urlParams = new URLSearchParams(window.location.search)
  const transactionId = urlParams.get('transactionId')
  const sessionExpired = urlParams.get('session_expired')
  
  if (sessionExpired) {
    // 處理 session 過期
    handleSessionExpired()
  } else if (transactionId) {
    // 處理正常回調
    handlePaymentCallback(transactionId)
  }
}, [])
```

### 2. **提供重新登入選項**
```javascript
const handleSessionExpired = () => {
  if (confirm('Line Pay 會話已過期，是否重新登入？')) {
    // 重新發起 Line Pay 請求
    goLinePay()
  }
}
```

## 總結

- 自動登出：Line Pay 服務端 session 過期（10–15 分鐘）
- 不是 `setTimeout`：前端無法控制
- 不是 `refreshToken`：Line Pay 不提供此機制
- 處理方式：監聽回調參數，提供重新登入選項

這是 Line Pay 的安全機制，無法透過前端代碼改變。
