## 關於 Zustand 的使用時機

**通常建議在以下情況使用 Zustand：**

1. **狀態數量超過 3-5 個**：當組件內有太多 `useState` 時
2. **跨組件共享狀態**：多個組件需要共享同一狀態
3. **狀態邏輯複雜**：需要複雜的狀態更新邏輯
4. **避免 Context 嵌套**：Context Provider 過多造成嵌套地獄

## Cursor
## Flux 單向資料流

### 1. **Action（動作）**
- 定義要做什麼
- 例如：`{ type: 'ADD_ITEM', payload: { name: '新項目' } }`

### 2. **Dispatcher（調度器）**
- 接收 Action 並分發到 Store
- 確保更新順序一致

### 3. **Store（儲存庫）**
- 存放狀態與更新邏輯
- 接收 Action 後更新資料

### 4. **View（視圖）**
- 顯示 Store 的資料
- 使用者操作時發出 Action

## 實際例子（以你的 TimeLog 為例）

```javascript
// 1. Action - 使用者點擊「開始」按鈕
const startAction = {
  type: 'START_ACTIVITY',
  payload: { title: '學習 React' }
}

// 2. Dispatcher - 分發這個動作
Dispatcher.dispatch(startAction)

// 3. Store - 接收動作並更新狀態
const TimeLogStore = {
  state: {
    title: '',
    startTime: null,
    isRunning: false
  },
  
  handleAction(action) {
    switch(action.type) {
      case 'START_ACTIVITY':
        this.state.title = action.payload.title
        this.state.startTime = new Date()
        this.state.isRunning = true
        this.notifyViews() // 通知所有 View 更新
        break
    }
  }
}

// 4. View - 顯示更新後的狀態
function TimeLogView() {
  const { title, startTime, isRunning } = TimeLogStore.state
  
  return (
    <div>
      <h1>{title}</h1>
      <p>開始時間: {startTime}</p>
      <p>狀態: {isRunning ? '進行中' : '未開始'}</p>
    </div>
  )
}
```

## 為什麼是「單向」？
- 資料只朝一個方向流動：Action → Dispatcher → Store → View
- View 不直接改 Store，而是發 Action
- 避免雙向綁定造成的複雜度

## Zustand 如何簡化 Flux？
```javascript
// 傳統 Flux：需要 Action、Dispatcher、Store 三個部分
// Zustand：全部整合在一個 store 裡

const useTimeLogStore = create((set) => ({
  // Store 狀態
  title: '',
  startTime: null,
  isRunning: false,
  
  // Actions（直接定義在 store 裡）
  startActivity: (title) => set({ 
    title, 
    startTime: new Date(), 
    isRunning: true 
  }),
  
  endActivity: () => set({ 
    isRunning: false, 
    endTime: new Date() 
  })
}))

// 在組件中使用
function TimeLogView() {
  const { title, startTime, isRunning, startActivity } = useTimeLogStore()
  
  return (
    <div>
      <h1>{title}</h1>
      <button onClick={() => startActivity('學習 React')}>
        開始
      </button>
    </div>
  )
}
```

## 優點
1. 可預測：資料流向清楚
2. 易除錯：每個步驟可追蹤
3. 易測試：邏輯集中在 Store
4. 可擴展：新增功能不影響既有流程

這就是為什麼 Zustand 說它基於「simplified flux principles」— 保留單向資料流，但大幅簡化實作。

### 1. **`create()`** - Zustand 的核心

```javascript
import { create } from 'zustand'

// 基本用法
const useStore = create((set, get) => ({
```

### 2. **`persist()`** - 持久化中間件
```javascript
import { persist } from 'zustand/middleware'

// 包裝 create() 來添加持久化功能
const useStore = create(
  persist(
    (set, get) => ({
      // 狀態和 actions
    }),
    {
      name: 'storage-key', // localStorage 的 key
      partialize: (state) => ({ /* 選擇要持久化的狀態 */ })
    }
  )
)
```

### 主要函數說明

#### 1. **`create()`** - Zustand 核心函數

- 用途：將狀態自動保存到 localStorage
- 參數：store 函數 + 配置選項
- 功能：頁面重新載入後自動恢復狀態

#### 4. **`get()`** - 狀態獲取函數
- 用途：讀取當前 store 的完整狀態

- 回傳：當前狀態物件

- 用途：選擇要持久化的狀態
- 參數：`(state) => ({ ... })` 回調函數
- 功能：過濾不需要保存的狀態（如 `currentTime`、`isClient`）

- 用途：指定在 localStorage 中儲存的 key

#### **`partialize`** - 狀態選擇器

- 邏輯：只保存重要狀態，排除臨時狀態
- 好處：減少儲存空間，避免保存無用資料

1. 初始化：`create()` 建立 store

### 1. **Zombie Child Problem (殭屍子組件問題)**

```javascript
// 問題示例
function ParentComponent() {
  const [data, setData] = useState(null)
  
  useEffect(() => {
    // 非同步操作
    fetchData().then(result => {
      setData(result) // 如果父組件已卸載，這裡會報錯
    })
  }, [])
  
  return <ChildComponent data={data} />
}

function ChildComponent({ data }) {
  // 如果父組件卸載後，這裡的 data 更新會造成錯誤
  return <div>{data}</div>
}
```

**解決方案：**
```javascript
useEffect(() => {
  let isMounted = true
  
  fetchData().then(result => {
    if (isMounted) { // 檢查組件是否還存在
      setData(result)
    }
  })
  
  return () => {
    isMounted = false // 清理標記
  }
}, [])
```

### 1. **為什麼是 `lazy`？**

`lazy` 是 React 的**延遲載入函數**，名稱來自「懶惰載入」(Lazy Loading)概念：


**為什麼叫 lazy？**
- **懶惰**：只有需要時才載入，不是一開始就載入
- **延遲**：等到真正使用時才執行 import
- **節省資源**：避免載入不需要的程式碼

### 2. **lazy 的工作原理**

