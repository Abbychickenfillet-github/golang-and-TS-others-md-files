### 3. Dockerfile 在哪？

`Dockerfile` 是一個**文字檔案**，它包含了 Docker 建置鏡像所需的**所有指令**。

1. **位置**：
    
    - 通常，`Dockerfile` 檔案會被放置在**專案的根目錄（Root Directory）**下。
        
    - 它**沒有副檔名**，檔案名稱就是 `Dockerfile`（首字母大寫）。
        
2. **作用**：
    
    - 你透過這個檔案來告訴 Docker 引擎：
        
        - `FROM`：使用哪個基礎鏡像（例如 `node:18-alpine`）。
            
        - `COPY`：把專案程式碼複製到鏡像裡。
            
        - `RUN`：執行指令（例如 `npm install`）。
            
        - `CMD`：容器啟動時要執行的指令（例如 `npm start`）。
            
3. **如何使用**：
    
    - 當你在包含 `Dockerfile` 的目錄下執行 `docker build -t your-image-name .` 指令時，Docker 就會讀取這個檔案，並開始一步步建置你的鏡像。
        

所以，如果你在專案根目錄找不到它，那麼你的應用程式可能是依賴 Zeabur 的自動偵測（Autodetect）功能，讓 Zeabur 自動生成並執行一個適合你 Node.js 專案的 `Dockerfile` 腳本。

### 🎯 **Rerun 的運作方式**

**Rerun** 是**重新執行**同一個 workflow run，**不是**建立新的節點：

1. **相同的 Run ID**：保持原本的 `${{ github.run_id }}`
2. **相同的 Commit SHA**：使用相同的 `${{ github.sha }}`
3. **相同的觸發事件**：基於相同的 push/PR

### 📊 **Rerun vs 新 Push 的差異**

| 特性 | **Rerun** | **新 Push** |
|------|-----------|-------------|
| **Run ID** | 相同 | 不同 |
| **Commit SHA** | 相同 | 不同 |
| **觸發原因** | Manual rerun | Push event |
| **Docker 標籤** | 相同 SHA | 新 SHA |
| **Registry 映像** | 覆蓋舊的 | 新增新的 |

### 🐳 **Docker Registry 的影響**

**Rerun 時**：
```yaml
# 會推送到相同的標籤
docker push username/next-one-app:latest
docker push username/next-one-app:${{ github.sha }}  # 相同的 SHA
```

**結果**：
- ✅ `latest` 標籤會被**覆蓋**
- ✅ SHA 標籤會被**覆蓋**
- ❌ **不會**產生新的映像版本

### 🔍 **如何查看差異**

1. **Actions 頁面**：
   - 會顯示 "Re-run all jobs" 或 "Re-run failed jobs"
   - Run ID 保持相同

2. **Docker Registry**：
   - 映像標籤相同
   - 但映像內容可能不同（如果程式碼有修改）

### 💡 **什麼時候會產生新節點？**

**會產生新節點的情況**：
- ✅ **新的 Push**：新的 commit
- ✅ **新的 PR**：新的 pull request
- ✅ **手動觸發**：workflow_dispatch

**不會產生新節點的情況**：
- ❌ **Rerun**：重新執行現有的 workflow
- ❌ **重新執行失敗的 job**：只重新執行失敗的部分

### 🎯 **總結**

**Rerun 不會造成新的節點**，它只是：
- 🔄 重新執行相同的 workflow
- 🐳 覆蓋 Docker Registry 中的映像
- 📧 重新發送郵件通知
- 📝 更新 Actions 執行記錄

所以你可以放心使用 Rerun 來測試，不會產生多餘的節點！🚀
