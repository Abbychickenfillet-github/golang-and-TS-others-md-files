Why it couldn’t read .env
- You likely started the server from the repo root. Previously dotenv was only auto-loading when the working dir was `backend/`.
- On Windows PowerShell, `&&` isn’t supported, so combined commands like cd && npm run dev failed. That left the process in the wrong folder.

What I changed
- `backend/server.js` now explicitly loads `backend/.env`:
```12:22:backend/server.js
import dotenv from 'dotenv';
import path from 'path';
import { fileURLToPath } from 'url';

// Ensure .env is loaded from the backend folder even if started from repo root
const __filename = fileURLToPath(import.meta.url);
const __dirname = path.dirname(__filename);
dotenv.config({ path: path.join(__dirname, '.env') });
```

Where to put .env
- Keep secrets only in `backend/.env`. Do NOT put them in the frontend.
- Your file is correctly placed: `c:\coding\zustand-practice\backend\.env`

How to run correctly on PowerShell
- Start the backend:
  - Option A (from backend dir):
    1) cd backend
    2) npm run dev
  - Option B (from repo root, still works with my change):
    1) node backend/server.js
    2) or: npm --prefix backend run dev
- Start the frontend:
  - cd zustand
  - npm run dev

Quick checks
- After starting backend, visit http://localhost:4000/api/health → should return {"ok": true}.
- Then use the subscription page at http://localhost:5173/linepay.

Notes
- Once `.env` is changed, restart the backend process to apply.
- Secrets should never be exposed to the frontend; only backend uses `LINE_PAY_*`.
