## �� **這種寫法的特點**

1. **`fileExt` 是函數參數**，不是用 `var`、`const`、`let` 宣告的變數
2. **可以在函數內部重新賦值**（因為它是參數）
3. **如果調用時沒有傳入值，`fileExt` 會是 `undefined`**

## 🔄 **更現代的寫法**

你可以用 ES6 的默認參數語法來簡化：

```javascript
export const loadEnv = (fileExt) => {
  // 如果沒有指定 fileExt，根據 NODE_ENV 自動選擇
  if (!fileExt) {
    const env = process.env.NODE_ENV || 'development'
    fileExt = env.trim() === 'production' ? '.production' : '.development'
  }
  // ...
}
```

或者更簡潔的寫法：

```javascript
export const loadEnv = (fileExt = null) => {
  // 如果沒有指定 fileExt，根據 NODE_ENV 自動選擇
  if (!fileExt) {
    const env = process.env.NODE_ENV || 'development'
    fileExt = env.trim() === 'production' ? '.production' : '.development'
  }
  // ...
}
```

## 💡 **術語解釋**

- **函數參數**：函數定義時接收的變數
- **默認參數**：ES6 語法，可以給參數設定默認值
- **參數重新賦值**：在函數內部可以修改參數的值

## ⚠️ **注意事項**

1. **參數重新賦值**：雖然可以，但不是最佳實踐
2. **更好的做法**：使用 `const` 宣告新變數
3. **可讀性**：明確宣告變數會讓代碼更容易理解

所以 `fileExt` 是**函數參數**，不是用 `var`、`const`、`let` 宣告的變數！

- **最寬鬆**：Cookie 可以在**任何跨域請求**中使用
- **可以跨域**：包括 AJAX 請求、iframe 嵌入等
- **功能最完整**，但需要配合 `secure: true`

## 🔍 **三元運算子 vs 邏輯運算子**

### **1. 三元運算子 `? :`**
```javascript
{groups && groups.length > 0 ? groups.map((group) => (
  // 當條件為 true 時執行
)) : (
  // 當條件為 false 時執行
  <div>目前沒有參加的揪團</div>
)}
```

**完整的三元運算子**：有 `true` 和 `false` 兩個分支

### **2. 邏輯運算子 `&&`**
```javascript
{groups && groups.length > 0 && groups.map((group) => (
  // 只有當條件為 true 時才執行
  // 當條件為 false 時，什麼都不渲染
))}
```

**只有 `true` 分支**：當條件為 `false` 時，返回 `false`（React 不會渲染）

## �� **兩種寫法的比較**

### **三元運算子（完整）**
```javascript
{condition ? (
  <div>有數據時顯示</div>
) : (
  <div>沒有數據時顯示</div>
)}
```

### **邏輯運算子（簡化）**
```javascript
{condition && (
  <div>有數據時顯示</div>
)}
// 沒有數據時：不渲染任何東西
```

## 🔍 **`filter()` 函數解釋**

```javascript
(couponDataList || []).filter((coupon) => {
  const searchContent = searchTerm.toLowerCase()
  return (
    coupon.coupon_content.toLowerCase().includes(searchContent) ||
    coupon.coupon_code.toLowerCase().includes(searchContent) ||
    String(coupon.coupon_discount).includes(searchContent)
  )
})
```

### **`filter()` 的作用**：
- **遍歷數組**：檢查每個 `coupon` 項目
- **返回布林值**：`true` 保留，`false` 過濾掉
- **創建新數組**：只包含符合條件的項目

### **條件檢查**：
```javascript
return (
  coupon.coupon_content.toLowerCase().includes(searchContent) ||  // 優惠券內容包含搜尋詞
  coupon.coupon_code.toLowerCase().includes(searchContent) ||      // 優惠券代碼包含搜尋詞
  String(coupon.coupon_discount).includes(searchContent)          // 折扣金額包含搜尋詞
)
```

**`||` 的邏輯**：
- 只要**任何一個條件**為 `true`，就保留這個優惠券
- 如果**所有條件**都是 `false`，就過濾掉這個優惠券

## �� **TCP vs UDP 差異**

### **TCP (Transmission Control Protocol)**
- **連線導向**：需要先建立連線才能傳輸資料
- **可靠傳輸**：保證資料完整性和順序
- **錯誤檢查**：自動重傳丟失的封包
- **流量控制**：避免接收端過載
- **適用場景**：網頁瀏覽、檔案傳輸、電子郵件

### **UDP (User Datagram Protocol)**
- **無連線**：直接發送資料，不需要建立連線
- **不可靠傳輸**：不保證資料到達或順序
- **速度快**：沒有連線建立的開銷
- **無流量控制**：可能造成接收端過載
- **適用場景**：即時通訊、線上遊戲、串流媒體

## 🔄 **TCP 連線狀態解釋**

### **LISTENING (監聽狀態)**
```
TCP    0.0.0.0:3000           0.0.0.0:0              LISTENING       29096
```

**意思**：
- **伺服器端**：等待客戶端連線
- **0.0.0.0:3000**：監聽所有網路介面的 3000 端口
- **0.0.0.0:0**：外部位址為空（等待連線）
- **PID 29096**：執行這個服務的程序 ID

### **ESTABLISHED (已建立連線)**
```
TCP    127.0.0.1:5141         127.0.0.1:49763        ESTABLISHED     5148
```

**意思**：
- **雙向連線**：客戶端和伺服器已經建立連線
- **127.0.0.1:5141**：本機的 5141 端口
- **127.0.0.1:49763**：連線到本機的 49763 端口
- **PID 5148**：處理這個連線的程序

## 🔍 **"連線" 的定義**

### **TCP 的連線 (Connection)**
- **連線狀態**：需要維護連線狀態（SYN, ESTABLISHED, FIN 等）
- **連線建立**：需要三次握手建立連線
- **連線維護**：需要保持連線狀態，監控連線健康
- **連線關閉**：需要四次揮手關閉連線

### **UDP 的"無連線" (Connectionless)**
- **無狀態**：不維護連線狀態
- **直接發送**：直接發送資料包，不需要握手
- **無確認**：不等待接收確認
- **無關閉**：不需要關閉連線的過程

## 🎮 **線上遊戲的連線需求**

### **線上遊戲通常使用 UDP，但為什麼？**

**1. 即時性需求**
```
遊戲狀態更新：每秒 60 次
如果使用 TCP：等待確認 → 延遲增加 → 遊戲卡頓
如果使用 UDP：直接發送 → 低延遲 → 流暢遊戲
```

**2. 遊戲容錯機制**
```
UDP 丟包：遊戲客戶端可以預測/插值
TCP 重傳：會造成遊戲時間不一致
```

**3. 實際例子**
```
玩家移動：UDP 發送位置更新
如果丟包：其他玩家看到短暫停頓
如果 TCP：等待重傳會造成明顯延遲
```

### **但線上遊戲真的不需要連線嗎？**

**實際上需要！**
- **登入系統**：使用 TCP（需要可靠傳輸）
- **遊戲狀態同步**：使用 UDP（需要即時性）
- **聊天系統**：使用 TCP（需要可靠傳輸）

## 🔍 **`@/*` 和 `./` 的關係**

### **`@/*` 是什麼？**
```json
{
  "compilerOptions": {

    "paths": {
      "@/*": ["./*"]  // ← 這裡的設定
    }
  }
}
```

**`@/*` 是一個別名 (alias)**，用來簡化 import 路徑。

### **`./` 的含義**

**`./` 確實代表「同一層」**，但在這裡的上下文是：

```json
"baseUrl": ".",        // ← 設定根目錄為當前目錄
"paths": {
  "@/*": ["./*"]       // ← 相對於 baseUrl 的路徑
}
```

## 📁 **實際路徑解析**

### **你的專案結構**
```
frontend/
├── jsconfig.json
├── pages/
│   ├── index.js
│   └── member/
│       └── login.js
├── components/
│   └── header.js
└── hooks/
    └── use-auth.js
```

### **路徑解析過程**

**1. `baseUrl: "."`**
- 設定根目錄為 `frontend/` 目錄

**2. `"@/*": ["./*"]`**
- `@/*` 映射到 `./*`
- `./*` 相對於 `baseUrl`，也就是 `frontend/*`

### **實際使用例子**

```javascript
// 不使用別名 (相對路徑)
import Header from '../../../components/header'
import { useAuth } from '../../../hooks/use-auth'

// 使用 @ 別名 (絕對路徑)
import Header from '@/components/header'
import { useAuth } from '@/hooks/use-auth'
```

## 🎯 **為什麼這樣設計？**

### **1. 簡化路徑**
```javascript
// 原本：複雜的相對路徑
import Component from '../../../../components/Component'

// 現在：簡潔的絕對路徑
import Component from '@/components/Component'
```

### **2. 避免路徑錯誤**
```javascript
// 相對路徑：容易出錯
import Auth from '../../../hooks/use-auth'  // 如果移動檔案，路徑就錯了

// 絕對路徑：永遠正確
import Auth from '@/hooks/use-auth'  // 不管檔案在哪裡都正確
```

### **3. 重構友好**
- 移動檔案時不需要修改 import 路徑
- 路徑結構更清晰

## 💡 **其他常見的別名設定**

```json
{
  "compilerOptions": {
    "baseUrl": ".",
    "paths": {
      "@/*": ["./*"],           // 根目錄別名
      "@components/*": ["./components/*"],  // 組件別名
      "@hooks/*": ["./hooks/*"],            // hooks 別名
      "@utils/*": ["./utils/*"]             // 工具別名
    }
  }
}
```

## �� **總結**

1. **`@/*`**：自定義的別名，代表根目錄
2. **`./`**：相對於 `baseUrl` 的當前目錄
3. **`baseUrl: "."`**：設定根目錄為 `frontend/`
4. **實際效果**：`@/*` 指向 `frontend/*`

**所以 `@/*` 確實代表根目錄，而 `./` 是相對於 `baseUrl` 的路徑表示法！** 🎯
