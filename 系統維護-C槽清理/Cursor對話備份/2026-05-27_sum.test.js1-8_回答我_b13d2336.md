有，而且這正是單元測試最常見、也最基本的形式：**給定輸入 → 期待輸出**。

### 為什麼寫死預期值（expected）仍然有意義？
- **防回歸（Regression）**：今天 `sum(1,2)` 回傳 3，但未來有人改到 `sum.js`（或改成字串相加、型別轉換出錯等），測試會立刻抓到。
- **定義行為規格（Spec）**：測試其實是在說「`sum(a,b)` 應該做什麼」。這比註解更可靠，因為它會被執行。
- **驗證整合點**：就算函式很簡單，`require('./sum')`、匯出方式、呼叫方式都可能出錯；測試會直接爆。

### 但你說得也對：只測 1+2 太弱
如果只有這一個案例，確實容易覺得「像在背答案」。改進方式是把測試寫成**覆蓋更多情境**，例如：
- **多組案例**：`(0,0)→0`、`(-1,2)→1`、`(1.5,2.5)→4` 等
- **邊界/型別行為**：非數字輸入要不要丟錯？要不要轉型？（這要你先決定 spec）
- **性質測試（可選）**：例如交換律 `sum(a,b) === sum(b,a)`（這種不靠「單一答案」）

### 你第 7 行「佔位符」的理解也對

已加上註解。快速對照如下：

### `describe('sum', () => { ... })`
把多個測試**分組**。跑完後報告會顯示 `sum` 這個區塊，方便整理。

### `test('名稱', () => { ... })`
定義**一個**測試：
- 第一個參數：測試名稱（失敗時會顯示）
- 第二個參數：要執行的程式（通常用 `expect(...).toBe(...)` 檢查結果）

### `test.each([...])('adds %p + %p = %p', (a, b, expected) => { ... })`
同一套邏輯，用**多組輸入**各跑一遍。  
`[1, 2, 3]` 會變成一次 test，`[0, 0, 0]` 再變成另一次，以此類推。

### `%p` 是什麼？
在測試名稱字串裡的**佔位符**（print value）。  
Jest 會依序把 `%p` 替換成該列的 `a`、`b`、`expected`：

| 資料列 | 顯示的測試名稱 |
|--------|----------------|
| `[1, 2, 3]` | `adds 1 + 2 = 3` |
| `[0, 0, 0]` | `adds 0 + 0 = 0` |

某組失敗時，名稱會直接告訴你是哪一組，不用自己猜。

### `describe` 是在報告裡分區塊嗎？
**對。** `describe('sum', () => { ... })` 會把裡面的測試包成一組，Jest 報告會像這樣顯示：

```
sum
  ✓ adds 1 + 2 = 3
  ✓ adds 0 + 0 = 0
  ✓ JS behavior: string concatenation if given strings
```

`describe` 主要是**分組、整理**，讓報告好讀。  
沒有 `describe` 也可以跑 `test`，只是報告會比較散。

---

### `test` 是定義測試案例的名稱嗎？
**一半對。** `test` 其實做兩件事：

```javascript
test('這是測試名稱', () => {
  // 這裡才是「實際要跑的測試內容」
  expect(sum(1, 2)).toBe(3);
});
```

1. **第一個參數**：測試名稱（報告上顯示的那行）
2. **第二個參數**：真正執行的函式（裡面的 `expect` 才是檢查）

所以更精準地說：  
`test` 是**定義一個完整的測試案例**——包含「名稱」+「要執行的檢查邏輯」。

可以記成：
- `describe` = 資料夾（分類）
- `test` = 資料夾裡的一個檔案（一個具體測試）

## Cursor
Jest 的 **Snapshot（快照）** 不是螢幕截圖，而是把**某次程式的「輸出結果」存成文字檔**，之後每次跑測試再拿來比對。

---

### 快照的是「什麼」？

通常是這些東西被序列化成文字後存下來：

| 常見用途 | 快照的內容 |
|---------|-----------|
| React 元件 | 渲染出來的 HTML / JSX 字串 |
| 物件 / 陣列 | JSON 風格的文字 |
| 函式回傳值 | 複雜結構的完整輸出 |
| 錯誤訊息 | 完整的 error 字串 |

---

### 跟 `toBe(3)` 差在哪？

你現在寫的：

```javascript
expect(sum(1, 2)).toBe(3);  // 你自己寫預期值
```

Snapshot 是：

```javascript
expect(某個複雜輸出).toMatchSnapshot();  // Jest 第一次跑時「幫你存」預期值
```

第一次跑測試 → Jest 產生 `.snap` 檔，把輸出存進去。  
之後每次跑 → 拿**現在的輸出**跟**.snap 裡存的**比，不一樣就失敗。

---

### 簡單例子

假設有個函式回傳物件：

```javascript
function getUser() {
  return { name: '小明', age: 20, role: 'admin' };
}

test('user snapshot', () => {
  expect(getUser()).toMatchSnapshot();
});
```

第一次跑完會產生 `__snapshots__/xxx.snap`：

```
exports[`user snapshot 1`] = `
Object {
  "age": 20,
  "name": "小明",
  "role": "admin",
}
`;
```

之後若有人把 `role` 改成 `'user'`，測試就會失敗，因為跟快照不一致。

---

### 什麼時候用？

- **適合**：輸出很長、結構複雜（例如 React 整棵 component tree）
- **不太適合**：像 `sum(1,2)` 這種簡單數字，用 `toBe(3)` 更清楚

---

### 一句話記

**Snapshot = 程式輸出的「文字版存檔」**，用來防「不小心改壞 UI 或資料結構」卻沒發現。

你現在的 `sum` 練習用 `toBe` 就夠了；等學 React 或回傳複雜物件時，snapshot 會比較派得上用場。
