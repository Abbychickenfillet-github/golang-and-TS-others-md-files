# Epoch 與 Batch Size

> 相關概念：[[超參數-hyperparameter]]（batch_size、epoch、learning rate 都屬於超參數）

Epoch（世代 / 輪次）是機器學習中完整跑完一次整個訓練資料集的過程。
當模型把資料集裡所有的樣本都「看過一遍」並完成權重更新後，就代表結束了一個 Epoch。

## 🧩 核心概念與計算

* 定義： 模型學習完整資料集的次數。
* 為什麼不能只跑 1 個 Epoch？ 模型看一次資料通常學不乖。需要讓它反覆看很多遍（多個 Epoch），它才能抓到資料的規律。
* 與 Batch Size 的關係：
   * 假設總資料量：$1,000$ 筆。
   * 設定 `batch_size`：$100$ 筆。
   * 1 個 Iteration（迭代） = 處理 $100$ 筆資料。
   * 1 個 Epoch（世代） = 處理 $1,000$ 筆資料 = 需要跑 $10$ 個 Iteration。

## ⚖️ Epoch 設太大或太小的影響

* Epoch 太少（欠擬合 Underfitting）：
   * 模型看得不夠多。
   * 沒學到資料的特徵，訓練與測試的準確度都很低。
* Epoch 太多（過擬合 Overfitting）：
   * 模型看太多次，把不重要的雜訊和死背答案都記下來。
   * 訓練集的準確度極高，但遇到沒看過的測試資料時，準確度大跌。

## 🔀 推論 (Inference) vs 訓練 (Training) 的 batch_size 差異

`batch_size` 在「訓練」和「推論」兩個情境意思不同，這點在做 RAG 時最容易搞混。

| | 訓練 (Training) | 推論 / Embedding (Inference) |
|---|---|---|
| 在做什麼 | 更新模型權重，讓它學會 | 模型已學好，只是拿來算結果 |
| 一個「樣本」是什麼 | 一筆訓練範例（圖片、文字＋標籤） | 一段要編碼的 chunk |
| batch_size 的意義 | 一次學幾筆資料再更新一次權重 | 一次同時編碼幾個 chunk |
| 會不會影響結果準確度 | 會（影響收斂與泛化） | 不會，只影響速度與記憶體 |
| 有沒有 epoch / 過擬合問題 | 有 | 沒有（不更新權重） |

### 在你的 RAG 專案裡

你用 bge-m3 算 embedding 時，它是**已經訓練好的 pretrained 模型**（參考 [[pretrained-models-explained]]），你只是「拿來用」=> 屬於**推論**。

所以這裡的 `batch_size`：

* 指的是「一次把幾個 chunk 一起送進模型算向量」。
* **一個 chunk = 一個樣本。**
* 調大 → 編碼更快，但吃更多 GPU/記憶體；爆記憶體 (OOM) 就調小。
* **不會**改變算出來的向量品質，純粹是速度 / 記憶體的取捨。
* **沒有** epoch、過擬合、欠擬合這些問題——那些是「訓練」才會碰到的。

一句話：上面 epoch 那段在講「訓練」，你 RAG 日常碰到的 batch_size 是「推論」，兩者只是剛好同名。

## 💥 OOM (Out Of Memory，記憶體不足)

OOM 指程式想用的記憶體超過硬體實際能給的，系統就把它強制中止。在 RAG / embedding 情境，幾乎都是指 **GPU 顯示記憶體 (VRAM) 爆掉**。

當你一次把太多 chunk 送進 bge-m3 算向量（`batch_size` 設太大），這一整批資料要同時放進 GPU 記憶體運算；需要的記憶體 > 顯卡 VRAM（例如 8GB）就會 OOM，程式直接掛掉。典型訊息：

```
torch.cuda.OutOfMemoryError: CUDA out of memory.
Tried to allocate 2.00 GiB (GPU 0; 8.00 GiB total capacity)
```

### 怎麼解決

* **把 `batch_size` 調小**（最常見、最直接），例如 64 → 32 → 16，一次處理少一點。
* 換 VRAM 更大的 GPU，或改用 **CPU** 跑（慢但不爆 VRAM，改吃系統 RAM）。
* 縮短每個 chunk 的長度（序列越長越佔記憶體）。

### 兩種記憶體要分清楚

* **VRAM（顯卡記憶體）**：GPU 運算時用，跑模型最常爆這個。
* **RAM（系統記憶體）**：一般程式用；用 CPU 跑模型、或一次把超大檔案整個讀進來，也可能爆這個。

一句話：OOM 不是程式邏輯錯，而是「東西太多、記憶體裝不下」。在 RAG 裡遇到，第一個動作就是**把 batch_size 調小**再跑一次。

## 💡 實務調整建議

1. 如何觀察： 訓練時要同時監控「訓練損失值（Train Loss）」與「驗證損失值（Validation Loss）」。
2. 何時停止： 當驗證損失值不再下降、甚至開始往上升時，就代表模型開始過擬合，此時應該停止訓練。
3. 自動化工具： 實務上常使用 Early Stopping（早期停止法機制），讓程式自動在模型表現最好、最剛好的 Epoch 自動停下來，省時又防過擬合。
